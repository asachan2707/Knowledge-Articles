I Built a Python Tool That Automatically Discovers Trading Strategies For You

Automated strategy discovery means using code to systematically search, test, and validate trading rules across historical data — removing human bias from hypothesis generation. Here is the full architecture, the key Python libraries, and the pitfalls that will destroy your backtest results if you ignore them.

LibrariesVectorBT, backtesting.py, yfinance, pandas-ta, QuantStats
LevelIntermediate Python — assumes basic pandas familiarity
FocusStrategy discovery, backtesting, statistical validation

Most algorithmic trading tutorials teach you how to backtest one strategy you already thought of. Automated discovery inverts this: define a space of possible strategies, test all of them, and let the data tell you which ones have an edge — without you forming a prior hypothesis first.

Core idea: VectorBT's vectorised approach lets you test thousands of parameter combinations in seconds by running them as NumPy array operations instead of Python loops. This is the key that makes automated discovery practical.

Why manual strategy selection fails

When humans pick trading strategies to test, they carry confirmation bias — they gravitate toward patterns that intuitively make sense. Automated discovery has no such bias. It tests every combination in a defined parameter space and lets statistical filters separate genuine edge from noise. The result is a ranked shortlist of strategies that survived objective screening, not strategies that felt right.

The architecture

Five layers of the discovery pipeline

1. Data layer

import yfinance as yf
df = yf.download("SPY", start="2015-01-01", auto_adjust=True)
# Returns OHLCV DataFrame ready for backtesting

Use yfinance for free OHLCV data. For production, replace with alpaca-trade-api or a broker's data feed. Always download point-in-time data to avoid survivorship bias — testing only on stocks that survived to today inflates results.

2. Signal generation layer

Define a parameter grid. For a dual-moving-average strategy, the grid is every combination of fast and slow period within your chosen ranges: 

from itertools import product
fast_range = range(5, 50, 5)   # 5, 10, 15 ... 45
slow_range = range(20, 200, 10) # 20, 30, 40 ... 190
param_grid = list(product(fast_range, slow_range))
# 9 × 18 = 162 combinations

3. Backtesting layer — VectorBT for scale

backtesting.py is clean and readable — good for prototyping individual strategies. VectorBT is the choice for automated discovery: it packs all parameter combinations into NumPy arrays and runs them simultaneously using Numba kernels. 10,000 combinations runs in seconds. 

import vectorbt as vbt

# Run all combinations at once — no Python loops
fast_ma = vbt.MA.run(close, window=fast_range, short_name='fast')
slow_ma = vbt.MA.run(close, window=slow_range, short_name='slow')

entries = fast_ma.ma_crossed_above(slow_ma)
exits   = fast_ma.ma_crossed_below(slow_ma)

portfolio = vbt.Portfolio.from_signals(close, entries, exits, freq='1D')
stats = portfolio.stats()

4. Filtering layer

Apply statistical thresholds to separate surviving strategies from noise: 

survivors = stats[
    (stats['Sharpe Ratio'] > 1.0)    &
    (stats['Max Drawdown'] > -0.15)  &
    (stats['Win Rate'] > 0.45)       &
    (stats['Profit Factor'] > 1.3)
]

5. Out-of-sample validation

Split your data: train on 70%, validate on 30%. Re-run survivors on the held-out period. Any strategy that degrades significantly on out-of-sample data is overfit — discard it. Walk-forward validation (rolling windows) is more rigorous than a single split.

ML-enhanced discovery

Going beyond technical indicator sweeps

Instead of sweeping indicator parameters, use ML to discover which market features have predictive power:

  • Feature engineering: RSI, MACD, Bollinger Band position, volume ratios, rolling returns as model inputs
  • Labeling: the Triple-Barrier Method (Marcos Lopez de Prado) creates more honest trade labels than simple "up/down tomorrow" classification
  • Models: Random Forest or XGBoost for feature importance — identifies which indicators have actual predictive power before backtesting them

MlFinLab (Hudson & Thames) provides production-grade implementations of data structures, labelling, and ML models purpose-built for quantitative finance.

The critical pitfalls

What will destroy your backtest results

Overfitting / curve-fitting

A strategy optimised on training data that fails out-of-sample is the #1 failure mode. Always reserve 30%+ of data for validation. Walk-forward optimisation is better than a simple split.

Look-ahead bias

Using tomorrow's data to generate today's signal — accidentally common in vectorised backtesting. Verify signal timing explicitly. A signal calculated from today's close should not influence today's trade entry.

The multiple comparisons problem

Testing 10,000 strategies, roughly 500 will look good by pure chance at 5% significance. Use Bonferroni correction or the Deflated Sharpe Ratio (Bailey & de Prado) to adjust for multiple testing. Never report the raw Sharpe of a strategy found via grid search without deflating it.

Transaction costs and slippage

Naive backtests assume frictionless execution. Even 0.1% per trade destroys many short-term strategies. Model commissions and a realistic slippage assumption in every backtest.

Key Takeaway

Automated strategy discovery is a data pipeline, not a single script. Build it in layers: fetch data, generate a parameter grid, run everything with VectorBT in one vectorised call, filter survivors by Sharpe and drawdown, then validate the survivors out-of-sample. Correct for multiple comparisons. Model transaction costs. A strategy that survives all of that is worth taking seriously. One that only survives training data is not.
