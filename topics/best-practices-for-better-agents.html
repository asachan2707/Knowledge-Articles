Best Practices for Building Better AI Agents

What separates a demo agent from a production agent is a set of decisions most tutorials skip: how you feed it knowledge, how you scope its actions, how you tune retrieval, and how you make its answers verifiable. This guide covers all of it — grounded in real implementation patterns, not platform-specific steps.

Focus Agent design, RAG, prompt engineering
Published Jun 10, 2026
Level Intermediate to Advanced

An AI agent that works in a demo often fails in production for the same set of reasons: it doesn't know what it doesn't know, it retrieves the wrong context, its prompts are too vague, and there is no way to tell if its answers are accurate. Fixing all four of those problems is what this guide is about.

Core idea: Agent quality is mostly a function of data quality, retrieval precision, and prompt clarity — not model size. A well-designed agent on a mid-tier model consistently outperforms a poorly-designed agent on the best available model.

Foundation

What is RAG and why does every serious agent use it?

Retrieval-Augmented Generation (RAG) is the dominant pattern for grounding LLM responses in accurate, current, organisation-specific knowledge. Instead of relying solely on what the model learned during training, RAG adds a retrieval step: before generating a response, the agent searches a knowledge store for the most relevant information, injects it into the prompt, and generates a response grounded in that retrieved context.

The three-step loop at runtime:

  1. Retrieve — the user's query is converted to a vector and used to search a knowledge index for semantically similar content.
  2. Augment — retrieved chunks are injected into the prompt alongside the user's question.
  3. Generate — the LLM produces a response grounded in the augmented context, not just its training data.

RAG solves three critical limitations of base LLM responses: knowledge cutoff (training data becomes stale), hallucination risk (the model invents plausible-sounding facts), and lack of organisation-specific context (the model has never seen your internal documents, cases, or procedures).

Common RAG use cases

  • Answering questions using knowledge articles, documentation, and research papers
  • Generating summaries from call transcripts, emails, and meeting notes
  • Resolving support queries using historical case resolutions
  • Producing briefings from CRM notes, account history, and activity feeds
  • Triaging incoming requests by matching against known issue patterns
Architecture

How RAG works: offline preparation and online retrieval

RAG has two distinct phases. Getting both right is the difference between accurate agents and unreliable ones.

Offline preparation (indexing)

Before the agent can retrieve anything, your content must be indexed. This involves:

  1. Ingest — connect your content sources (documents, databases, object records, file stores) to the indexing pipeline.
  2. Chunk — break long documents into smaller, semantically coherent pieces. Each chunk should represent a self-contained factoid or set of related facts.
  3. Vectorize — convert each chunk to a numeric vector using an embedding model. This makes semantic similarity search possible.
  4. Store — write chunks and their vectors to a vector store or search index, alongside any metadata (source, date, category, language) needed for filtering.

Online retrieval (runtime)

When a user sends a query, the pipeline:

  1. Vectorizes the query using the same embedding model used during indexing.
  2. Performs a similarity search against the index to find the most relevant chunks.
  3. Applies any pre-retrieval filters (language, date, category, record scope).
  4. Injects the top-ranked chunks into the prompt template.
  5. Submits the augmented prompt to the LLM for generation.

Key insight: The query is the most critical element of RAG retrieval. The quality of retrieved chunks depends entirely on how well the query reflects user intent. A vague, poorly formed query retrieves vague, poorly matched chunks — and no amount of prompt engineering recovers from that upstream failure.

Content Quality

Curating content for accurate retrieval

RAG is only as good as the knowledge you put into it. Grounding an agent on inaccurate, outdated, or poorly structured content produces inaccurate, confidently-wrong responses. Content quality is the highest-leverage investment in any RAG system.

Structured vs. unstructured content

TypeExamplesRAG requirement
Structured Database records, CRM objects, spreadsheets, picklist fields Only long free-text fields (full sentences with semantic content) are useful for RAG. Categorical fields (IDs, status codes, short labels) should be used as metadata filters, not indexed content.
Unstructured PDFs, Word docs, emails, call transcripts, HTML pages, audio/video Chunked, vectorized, and indexed. Audio/video should be transcribed before indexing.

Content best practices

  • Be thorough, not brief. LLMs work best with complete, detailed context. A one-line article answer is worse than a paragraph with examples, synonyms, and edge cases. More detail improves retrieval recall and generation accuracy.
  • Use real-world examples. Describe scenarios conversationally — the way a user would actually phrase their question. This strengthens semantic alignment between the query and the stored content.
  • Structure articles coherently. Ensure sentences within a paragraph are logically related. Use headings (H1–H6 in HTML content) — chunking pipelines often use heading boundaries as natural split points.
  • Keep articles focused. One article, one topic. Mixing multiple topics in a single document dilutes retrieval precision — a query about topic A will retrieve a chunk that also contains unrelated content about topic B.
  • Include titles, headings, and metadata. These provide context that helps both retrieval and generation. Storing titles as prepend fields (see Chunking section below) is even better.
  • Add synonyms and abbreviations. If users might ask using informal language, jargon, or acronyms, include those in the content so semantic search can bridge the vocabulary gap.
  • Annotate media with descriptive text. Charts, diagrams, and images are not vectorized. Add alt-text or prose descriptions so the information they contain is searchable.
  • Perform regular content audits. Stale content produces stale answers. Establish governance: versioning, review cycles, expiry dates on time-sensitive content.
Chunking Strategy

How to optimise your chunking strategy

Chunking is the process of splitting source content into pieces that are small enough to be semantically precise but large enough to carry meaningful context. It is one of the most consequential decisions in any RAG system, and it is often treated as a configuration detail rather than a design decision.

Four roles a field can play in a RAG index

RoleWhat it doesExample
Index field Content is chunked, vectorized, and used for semantic search Case description, article body, call transcript
Prepend field Value is prepended to every chunk, enriching it with context without being indexed separately Article title, product name, document category
Filter field Added to the index schema but not vectorized; used to pre-filter retrieval results Language, status, date, product family
Return field Retrieved and passed to the prompt but not used in search; provides additional context for generation Source URL, record ID, author name

Chunk size trade-offs

  • Smaller chunks (100–200 tokens) — higher retrieval precision, less irrelevant content, but may lack enough context for the LLM to generate a complete answer.
  • Larger chunks (400–512 tokens) — more context per chunk, better for complex reasoning, but reduces precision and increases prompt size and cost.
  • Practical default — 256–512 tokens (~200–400 words) works well for most knowledge base and document use cases. Adjust based on information density and test results.

Field prepending — the underused optimisation

Prepending a title or category to every chunk dramatically improves retrieval. Without it, a chunk reading "Replace the filter every 6 months" is semantically opaque — the model has no idea what product, procedure, or context it belongs to. With prepending, the same chunk reads "Device X maintenance schedule: Replace the filter every 6 months" — now retrievable from any question about Device X maintenance.

What NOT to index

  • Categorical fields, picklists, status codes — use these as filter fields, not index fields. Short chunks (1–2 words) have no semantic context and produce erratic vector rankings.
  • Duplicate or highly similar fields — if you have Summary, Title, and Description all covering the same content, index the most detailed one and prepend the shorter version. Multiple similar chunks from the same document can dominate retrieval results, crowding out other relevant documents.
  • Boilerplate and legal disclaimers — footer text, copyright notices, and standard terms add noise. Exclude them from indexing using pre-processing rules.
Search Strategy

Vector search, keyword search, and hybrid search

How you search your index is as important as what you put in it. Three retrieval modes exist, and each fits different content and query patterns.

ModeHow it worksStrengthsWeaknesses
Vector search Converts query to a vector; retrieves chunks by semantic similarity Handles synonyms, paraphrasing, and intent — finds conceptually related content even if the exact words differ Can miss specific product names, model numbers, or jargon that appear verbatim in the query
Keyword search Matches exact or near-exact terms using inverted indexes (BM25 or similar) Precise for known terms, SKUs, codes, and proper nouns Fails on intent — "fix broken printer" won't match "printer troubleshooting guide" without the overlap
Hybrid search Runs both in parallel; merges and reranks results using a combined score Best of both — handles intent and exact terms; most robust for general-purpose agents Higher latency; roughly double the index compute cost

When to use hybrid search

Use hybrid search when your content contains specific terminology that users will query verbatim — product names, model numbers, error codes, medical terms, legal clauses. Vector search alone will miss exact-match lookups in these cases.

When your content is entirely conversational natural language with no specific jargon, vector search alone may be sufficient and cheaper.

Common mistake: Don't use hybrid search as a substitute for categorical filtering. If users filter by department, language, or product family, implement those as pre-retrieval filters on metadata fields — not as keyword queries against indexed content. Keyword search on categorical data produces single-word chunks with no semantic context and unpredictable rankings.

Reranking and relevance signals

After initial retrieval, a reranker can re-score results using additional signals beyond pure similarity — recency (newer content ranks higher for time-sensitive queries), popularity (more frequently accessed content ranks higher), and cross-encoder models (which evaluate query-chunk pairs more accurately than bi-encoder similarity alone). Reranking adds latency but meaningfully improves result quality for complex queries.

Agent Architecture

Designing agents for RAG: structure and scope

How you structure an agent's actions and knowledge sources has a larger effect on accuracy than any prompt tweak. Two architectural patterns cover most use cases.

Pattern 1: One action, multiple retrievers

A single agent action uses multiple retrievers — one per knowledge source — and an ensemble reranker merges and re-scores results before augmenting the prompt. The LLM sees the top-ranked chunks from across all sources and generates a unified response.

Use when: All knowledge sources are potentially relevant to any query. Users ask questions that may draw on multiple data sources simultaneously.

Benefits: Simpler agent instruction set; single action call; ensemble reranking prevents any one source from dominating; reduces over-calling multiple actions.

Pattern 2: Separate actions per knowledge domain

Each knowledge domain gets its own dedicated agent action, with clear scope instructions telling the agent's reasoning engine which action to call for which type of question.

Use when: Knowledge domains are cleanly separable and a single source can fully answer most questions. Example: defect queries always go to the defects database; pricing queries always go to the price book.

Benefits: Clearer retrieval scope; lower cost per call; easier to debug when a specific domain produces wrong answers.

Anti-pattern to avoid: Do not use multiple separate actions as a workaround for Approach 1 by instructing the agent to call all of them in sequence. The agent's reasoning engine may consider the task complete before all actions are called, and LLM instructions for how to interpret context belong in the prompt template — not in the agent's topic instructions.

Combining both patterns

Both patterns can coexist in the same agent. A common design: one dedicated action for a high-specificity domain (e.g., "Answer questions about known defects") plus one combined action for all other knowledge sources. This gives you precision where it matters and breadth where it doesn't.

Writing effective action descriptions

The agent's reasoning engine uses action descriptions to decide which action to call. Write descriptions that clearly define what the action can and cannot answer:

  • Be specific about scope: "Use this action when the user asks about product defects or quality issues" — not "Use this action to answer questions."
  • Include negative scope: "Do not use this action for pricing, availability, or shipping questions."
  • Match the language of the action description to the language of user queries — use the same terminology your users use, not internal system names.
Retrieval Precision

Using filters to improve retrieval precision

Pre-retrieval filters are one of the most effective and underused tools for improving agent accuracy. They constrain the search space before semantic similarity is computed — dramatically improving result relevance when the eligible content is a subset of the full index.

Static pre-filters

A fixed condition applied to every query. Examples: only retrieve published content (not drafts), only retrieve content in the user's language, only retrieve content tagged to the current product line. These are defined at design time and require no runtime input.

Dynamic pre-filters

Filter values provided at runtime from the query context. Examples: the user's account ID narrows retrieval to documents related to their account; the product category selected in the UI narrows retrieval to that product's knowledge base; the detected language of the query filters results to that language.

Dynamic filters are powerful because they effectively create personalised retrieval without building separate indexes per user or context. The filter condition is defined at design time (e.g., product_family = $placeholder); the value is resolved at runtime from the conversation context.

Pre-filter vs. post-filter: A pre-filter narrows the candidate set before retrieval — if you ask for 10 results, you get up to 10 results that satisfy the filter. A post-filter retrieves 10 results then applies the filter — potentially leaving you with 0. Always prefer pre-filters for mandatory constraints. Post-filters are useful only for access-control checks that cannot be expressed in the index schema.

Prompt Engineering

Writing effective RAG prompt templates

The prompt template is the final layer between good retrieval and a good response. A minimal prompt that simply says "answer this question using this information" leaves too much to the model's discretion and produces inconsistent, hallucination-prone responses.

What a good RAG prompt template contains

  • Explicit instruction to use the provided context. Tell the model to base its answer on the retrieved information, not on general training knowledge. Without this, the model will freely mix retrieved context with remembered facts.
  • A fallback instruction. What should the agent say when the retrieved context doesn't contain the answer? Define this explicitly: "If the provided context does not contain sufficient information to answer the question, say so clearly — do not guess."
  • Output format instructions. Define the expected structure: length, tone, format (bullet list, paragraph, markdown table), and whether to include source references.
  • Reasoning instructions for complex queries. For queries requiring synthesis across multiple chunks, instruct the model to "think through the context carefully before answering" or "consider the question from multiple angles before generating a response."
  • Language and tone constraints. Specify the output language (especially for multilingual agents), formality level, and any domain-specific terminology preferences.

A minimal vs. a production-grade prompt

Minimal (risky)

Answer this question: {question}

Using this information: {retrieved_context}

Production-grade (explicit)

INSTRUCTIONS
1. Read the QUESTION carefully.
2. Review the CONTEXT provided below.
3. Base your answer strictly on the CONTEXT — do not use outside knowledge.
4. If the CONTEXT does not contain enough information to answer, say:
   "I couldn't find a definitive answer in the available knowledge."
5. Be concise, professional, and direct.
6. If relevant, cite which part of the context supports your answer.

CONTEXT:
{retrieved_context}

QUESTION:
{question}

Common prompt engineering mistakes

  • No fallback instruction — the model invents an answer when context is insufficient. Always define what happens when context is missing.
  • Vague context instructions — "use the information below" is weaker than "base your answer strictly on the CONTEXT — do not use outside knowledge." Precision reduces hallucination.
  • Mixing agent-level and prompt-level instructions — instructions for how to interpret or synthesise retrieved content belong in the prompt template, not in the agent's topic instructions. Agent instructions define when to call the action; prompt instructions define how to use what was retrieved.
  • Over-long prompts — every instruction token competes with context tokens for the model's attention. Keep instructions tight; use the context window for retrieved knowledge.
Query Quality

Improving query quality before retrieval

The query sent to the retrieval system is not always the raw user message. For complex conversational agents, transforming the user's input into a better-formed retrieval query significantly improves result quality.

Query rewriting

A small LLM call can rewrite the user's raw message into a retrieval-optimised query before the search is performed. This is especially valuable when:

  • The user's message is long and conversational ("Hi, I've been having this issue where my account seems to be showing the wrong balance and I'm not sure if it's a display bug or an actual error…") — the retrieval query should be "account balance display error troubleshooting."
  • The query references context from earlier in the conversation ("the same issue as before") — the rewriter resolves coreferences before searching.
  • The user uses informal language or abbreviations — the rewriter normalises vocabulary to match the knowledge base.

Iterative retrieval

For complex queries, a single retrieval pass is sometimes insufficient. An iterative pattern runs an initial retrieval, summarises the results, then uses the summary plus the original query to rewrite and run a second retrieval. The final augmented prompt uses the second retrieval's results. This adds latency but substantially improves answer quality for queries requiring synthesis across multiple pieces of knowledge.

When the user's context shapes the query

In many production agents, the retrieval query is not just the user's message — it is the user's message plus context derived from the conversation or the active record. For a service agent handling a support case, the query might combine the user's question with the case description and product category. For a sales agent, the query might combine the user's question with the account name and recent activity summary. This scoped querying dramatically improves retrieval relevance without requiring separate indexes per account or product.

Model Selection

Choosing the right LLM for your agent

For most RAG applications, the greatest complexity occurs in retrieval and augmentation — not in generation. The LLM's primary job is to synthesise clearly presented context, not to reason from scratch. This has practical implications for model choice.

Use caseModel tier neededRationale
Simple Q&A from a well-structured knowledge base Smaller / faster model If the relevant chunk is retrieved accurately, a smaller model can extract and present it cleanly. High reasoning capability is not required.
Multi-source synthesis, drawing conclusions from several chunks Mid-tier model The model needs to reconcile potentially conflicting information and identify the most relevant evidence across chunks.
Complex reasoning, ambiguous queries, or adversarial inputs Frontier model Strong reasoning capability, longer effective context window, and better instruction-following reduce hallucination risk for high-stakes use cases.

Context window matters more than benchmark scores

For RAG, the effective context window is often the most important LLM parameter. Each retrieved chunk consumes tokens. A model with a 16k token context window can handle roughly 10–15 document chunks before running out of room. A model with a 128k–200k context window can handle large document sets, long call transcripts, and multi-turn conversation history simultaneously.

Rule of thumb: 1 token ≈ ¾ of a word. 512-token chunks are roughly 380 words each. If you retrieve 10 chunks at 512 tokens, that's 5,120 tokens before the system prompt, conversation history, or user message are counted.

Embedding model selection

The embedding model converts text to vectors and directly determines retrieval quality. Two practical considerations:

  • Consistency — use the same embedding model for both indexing and query vectorization. Switching models means re-indexing all content.
  • Multilingual support — multilingual embedding models (e.g., multilingual E5) can retrieve across languages — a French query can surface German articles. Verify your target languages are well-represented in the model's training data before deploying.
Trust & Verifiability

Building trust through citations and verifiable responses

Trust is the primary barrier to AI agent adoption. Users who cannot verify an agent's answer treat it with suspicion — and they should. Citations are the mechanism that converts a plausible-sounding response into a verifiable one.

Why citations matter

  • Verifiability — users can check the source directly and confirm the answer is accurate.
  • Accountability — citations make it clear the answer derives from a specific piece of documented knowledge, not from the model's memory.
  • Discovery — a citation gives users a path to deeper reading, enabling self-service beyond the agent's answer.
  • Debugging — when an answer is wrong, a citation immediately tells you which source document is the problem — a governance and content quality issue, not just an AI issue.

Two citation approaches

ApproachHow it worksBest forTrade-off
Platform-managed citations The infrastructure automatically appends source references from retriever metadata Standard knowledge base Q&A; teams that want consistent formatting without prompt customisation Less flexibility in citation format
Prompt-based citations The prompt template instructs the LLM to include source references in its response Custom formats, inline citations, domain-specific citation styles Consumes context window; requires testing to prevent hallucinated source IDs

Practical citation design

  • Ensure citation URLs point to content the end user can actually access. A citation to an internal document that requires elevated permissions — that the user doesn't have — is worse than no citation, because it creates friction and distrust.
  • For prompt-based citations, instruct the model to extract the source ID verbatim from the retrieved context rather than generating it. LLMs will confidently fabricate plausible-looking source IDs if asked to recall them.
  • When images or charts are part of the source material, include their descriptive text in the index so that visual information can be cited and traced back to the original document.
Debugging

Diagnosing and fixing poor agent responses

When an agent gives a wrong or unhelpful answer, the root cause is almost always one of four things. Debugging systematically is faster than tuning blindly.

SymptomLikely root causeFix
Answer is wrong but confident Retrieved wrong chunks; or model used training knowledge instead of context Inspect retrieved chunks for the failing query. Improve content quality or chunking. Strengthen prompt instruction to use only the provided context.
Answer is "I don't know" for a question that should be answerable Retrieval failed to surface relevant content Check if the content exists in the index. Check if filters are too restrictive. Check if chunk size is too small to carry sufficient context. Consider switching to hybrid search.
Answer is partially correct but incomplete Relevant information is spread across multiple chunks; not all retrieved Increase the number of chunks retrieved. Consider larger chunk sizes. Add an ensemble retriever if multiple sources contain relevant pieces.
Answer is correct but cites the wrong source Hallucinated source ID in prompt-based citations Switch to platform-managed citations, or instruct the model to extract source IDs verbatim — never recall or generate them.
Good answers in testing, poor answers in production Content has become stale; or production queries differ from test queries Audit knowledge base for outdated content. Build a representative production query benchmark and evaluate against it regularly.

Build a benchmark query set

The single highest-value investment in long-term agent reliability is a maintained benchmark set of representative queries with known correct answers. Run your agent against this set on every significant change — prompt update, model upgrade, content change, chunk size adjustment. Without a benchmark, you are flying blind: you cannot tell if a change improved or regressed agent quality.

Advanced Patterns

Orchestration, pipelines, and non-generative RAG

Retrieval does not have to end in a generated text response. As agents become more sophisticated, retrieval becomes an input to broader orchestration pipelines.

Using retrieval inside automation flows

A retriever can be called mid-workflow as a classification or matching step — not to generate an answer, but to look up similar past cases, check for existing knowledge, or tag an incoming item against a taxonomy. In these scenarios, the retrieved chunk is used programmatically (e.g., routed to a specific team, used to pre-populate a form, or evaluated for a threshold match) rather than surfaced to an end user.

Chained prompt pipelines

Complex agent tasks can be decomposed into multi-step prompt chains: one step extracts intent, a second step retrieves relevant content, a third step synthesises a draft, and a fourth step evaluates and refines the draft before output. Each step uses a small, focused prompt. This is more reliable than a single monolithic prompt attempting to do all four things at once.

Multilingual agents

Modern multilingual embedding models preserve semantic similarity across languages — a query in one language can retrieve content written in another. For global deployments:

  • Detect the query language and pass it as a dynamic pre-filter if your index is monolingual and organised by language.
  • For cross-lingual retrieval, use a multilingual embedding model and verify its performance on your target language pairs — some languages are significantly better represented than others.
  • Specify the output language explicitly in the prompt template. Without this, the model may respond in the language of the retrieved chunks rather than the user's language.
Quick Reference

Best practices checklist

  • Content: Thorough, focused articles. One topic per document. Include synonyms, examples, and headings.
  • Indexing: Only index long free-text fields. Use prepend fields for titles and categories. Exclude boilerplate and categorical short fields.
  • Chunking: 256–512 tokens for most use cases. Adjust based on information density. Prepend context labels to every chunk.
  • Search: Use hybrid search when queries contain specific terminology or product names. Use vector-only for purely conversational content.
  • Filters: Apply static pre-filters for mandatory constraints (published status, language). Use dynamic pre-filters to scope retrieval to the current record or context.
  • Agent structure: Scope action descriptions tightly. Prefer one combined action with ensemble retrieval over many actions called in sequence.
  • Prompts: Always include a fallback instruction. Explicitly instruct the model to use only the provided context. Define output format and language.
  • Citations: Include citations for every production agent. Point only to content the user can access. Never ask the LLM to recall or generate source IDs.
  • Benchmarks: Maintain a representative query benchmark. Run it on every change to prompt, model, or content.
  • Governance: Establish content review cycles. Stale knowledge produces stale answers regardless of how good the retrieval is.

Key Takeaway

Agent quality is primarily determined before the LLM sees a single token: by the quality of your knowledge content, the precision of your retrieval strategy, and the clarity of your prompt instructions. A well-designed retrieval pipeline with clear prompts on a mid-tier model will consistently outperform a poorly-designed one on the most capable model available. Build the foundation right, measure it with a benchmark, and iterate from data — not intuition.
