I'm a Software Architect, and I'm Terrified of Your "Vibe-Coding"

Vibe-coding — prompting an AI to generate code without deeply understanding what it produces — is a real and growing risk in production systems. This is not an argument against AI-assisted development. It is an argument for knowing what "responsible AI-assisted development" actually means, and what the failure modes look like from an architect's perspective.

PerspectiveSoftware architect with 15+ years, AI tools adopter
AudienceDevelopers using AI tools for real production code

The author uses AI tools every day. Claude Code is open in a terminal pane right now. The concern is not AI-generated code — it is code deployed without the developer understanding what it does, why it does it, or what happens when it fails. That used to be the definition of "bad code from a contractor." In 2026 it is becoming the default output of well-meaning, fast-moving developers.

Core idea: Vibe-coding creates an understanding debt, not just technical debt. Technical debt can be refactored. Understanding debt means nobody knows what the system does — which makes every outage, every security audit, and every new hire harder than it should be.

The incident that prompted this

A team inherited a Node.js service that had been "vibe-coded" into production over six weeks. 3,400 lines. The developer who wrote it (or rather, prompted it) had left the company. The service handled payment webhooks. Nobody on the new team could explain: why was there a 200ms setTimeout in the webhook handler? Why were some events processed twice? Why did the error handler re-throw in some branches but swallow in others? Every question led to "let me ask Claude" — which was fine for the original developer but useless for debugging a production incident at 2am.

The real risks

What vibe-coding breaks that isn't obvious

Security vulnerabilities you can't audit

AI generates secure-looking code that has subtle injection, auth bypass, or race conditions. If you don't understand the code, you can't spot them in review.

Incident response without a mental model

At 3am, you need to understand your system faster than you can re-prompt an AI. Vibe-coded systems produce on-call engineers who are helpless without AI access.

Compounding complexity

Each AI-generated addition makes the next one harder to verify. Eventually the system has emergent behaviour nobody designed and nobody understands.

Knowledge silos become knowledge voids

When the developer who vibe-coded a service leaves, the institutional knowledge of "ask Claude" leaves with them. New team members have nothing to build on.

Testing that doesn't test the right things

AI-generated tests often test implementation details rather than behaviour. You get 90% coverage and a system that breaks in production in obvious ways.

Over-engineered solutions

AI models solve problems at the level of complexity they've seen. A 10-line problem often becomes a 200-line "pattern-complete" solution nobody asked for.

The code problem illustrated

What vibe-coded vs understood code looks like

// Vibe-coded webhook handler — "it works, I don't know why"
app.post('/webhooks/payment', async (req, res) => {
  const sig = req.headers['stripe-signature'];
  let event;
  try {
    event = stripe.webhooks.constructEvent(req.body, sig, process.env.STRIPE_WEBHOOK_SECRET);
  } catch (err) {
    return res.status(400).send(`Webhook Error: ${err.message}`);
  }

  // Why is this here? The original dev doesn't know. Claude suggested it.
  await new Promise(r => setTimeout(r, 200));

  if (event.type === 'payment_intent.succeeded') {
    const paymentIntent = event.data.object;
    try {
      await db.orders.update({ where: { stripeId: paymentIntent.id }, data: { status: 'paid' } });
      await emailService.send(paymentIntent.metadata.userId, 'payment_success');
    } catch (error) {
      // Swallows error — order may not be updated but Stripe gets 200
      console.error('Error processing webhook:', error);
    }
  }

  res.json({ received: true });
});
// Understood webhook handler — every line has a reason
app.post('/webhooks/payment',
  express.raw({ type: 'application/json' }), // Must be raw buffer for signature verification
  async (req, res) => {
    // Verify the event came from Stripe, not an attacker
    const sig = req.headers['stripe-signature'];
    let event;
    try {
      event = stripe.webhooks.constructEvent(req.body, sig, process.env.STRIPE_WEBHOOK_SECRET);
    } catch (err) {
      logger.warn('Invalid Stripe webhook signature', { err: err.message });
      return res.status(400).json({ error: 'Invalid signature' });
    }

    // Respond immediately — Stripe expects 200 within 5s or it retries
    // Processing happens asynchronously to avoid timeout
    res.json({ received: true });

    // Process after responding
    try {
      await processWebhookEvent(event);
    } catch (err) {
      // Log but don't crash — Stripe will retry if we return non-200, but we already returned 200
      // For critical failures, use a dead-letter queue and alert
      logger.error('Webhook processing failed', { eventId: event.id, err: err.message });
      await deadLetterQueue.add('failed-webhook', { event });
    }
  }
);

async function processWebhookEvent(event) {
  // Idempotency: Stripe may send the same event multiple times (at-least-once delivery)
  const alreadyProcessed = await db.webhookLog.findUnique({ where: { stripeEventId: event.id } });
  if (alreadyProcessed) return;

  if (event.type === 'payment_intent.succeeded') {
    await handlePaymentSuccess(event.data.object);
  }

  // Record that we processed this event
  await db.webhookLog.create({ data: { stripeEventId: event.id, processedAt: new Date() } });
}

async function handlePaymentSuccess(paymentIntent) {
  // Both updates must succeed together — wrap in transaction
  await db.$transaction([
    db.orders.update({ where: { stripeId: paymentIntent.id }, data: { status: 'paid' } }),
    db.webhookLog.create({ data: { stripeId: paymentIntent.id, event: 'payment_success' } }),
  ]);

  // Email is best-effort — failure here should not roll back the order update
  await emailService.send(paymentIntent.metadata.userId, 'payment_success').catch(err =>
    logger.warn('Email send failed after payment', { err: err.message })
  );
}

The difference is not length — it is intent. Every decision in the second version has a reason a new engineer can read. The first version can only be understood by re-running the AI session that produced it.

What responsible AI-assisted development looks like

The architect's rules of thumb

  • You own every line. If you can't explain why a line is there, it doesn't ship. Use AI to draft — use your understanding to approve.
  • Read, don't just run. Reading AI-generated code before running it takes 5 minutes. Debugging unknown code in production takes hours.
  • Name the decisions. When AI suggests an approach, name it: "this is a saga pattern," "this is optimistic locking." Naming forces understanding.
  • Test the behaviour, not the code. Write tests for what the system should do for users — not for implementation details the AI chose.
  • Document the why. A one-line comment explaining the WHY (not the what) is worth three times its weight in review.
  • Resist over-engineering. When AI generates a factory/builder/strategy pattern for a 10-line problem, delete it and ask for something simpler.
  • Pair AI with architecture review. Have a senior review AI-generated systems architecture before it gets load-tested by production traffic.
The positive case

What AI-assisted development does well

  • Boilerplate elimination: Express route setup, CRUD scaffolding, test file creation — AI is excellent at this because understanding is not required.
  • Pattern suggestion: "Should I use a queue or a cron job here?" — AI explains trade-offs well when you provide context.
  • Code review assistance: AI spots missing edge cases and missing error handling faster than most code reviewers.
  • Documentation drafts: Generating a first-pass JSDoc or README is fast and then easily edited.
  • Refactoring toward known patterns: "Refactor this to use the strategy pattern" — excellent use of AI when you know what you want.

The common thread: AI is valuable when you know enough to evaluate its output. Vibe-coding removes that evaluation step and ships the evaluation debt to production.

Key Takeaway

Vibe-coding is not about AI being bad. It is about understanding debt — deploying code nobody can reason about at 3am, in a security audit, or when onboarding a new hire. AI makes you faster at writing code. It does not make you faster at understanding systems. That asymmetry is exactly where architectural risk accumulates. The fix is not to use less AI. The fix is to use it for drafting and own every approval.
