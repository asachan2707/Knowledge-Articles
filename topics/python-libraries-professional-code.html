Story · The 4-level data.get() chain

Every developer has written: data.get("user", {}).get("profile", {}).get("address", {}).get("city", "Unknown"). It works. It is also a future bug magnet — one extra nesting level breaks it.

The article uses this exact example to motivate glom, which turns nested access into a declarative path. Same idea applies across the library list: trade brittle chains for declarative tools.

Detailed breakdown

Key insights and why they matter

1) glom — nested data access without the chain

Pulls deeply nested values declaratively with defaults. Removes brittle .get() chains across messy API responses.

2) attrs / dataclasses — classes without boilerplate

attrs and dataclasses encode equality, repr, and validation rules in a few lines. Used everywhere senior code lives.

3) tenacity — retry done right

Retry decorators with backoff, jitter, and stop conditions. Every external call that can fail temporarily belongs in a tenacity wrapper.

4) structlog — context-rich structured logs

Logs you can actually query. Attach request_id, user_id, job_id — and your incident postmortems get 10x faster.

5) rich — terminal output that explains itself

Tables, syntax highlighting, progress bars in CLI tools. Tiny dependency, huge readability gain.

6) diskcache / cachetools — caching without Redis

When you don't need a server. Local caches with TTLs, perfect for small services and CLI tools.

Practical learning

How to apply this in real work

  • Pick one current data.get() chain in your code and rewrite it with glom.
  • Add tenacity to every external call (HTTP, DB connect, message broker publish).
  • Replace ad-hoc print debugging with structlog and one context-bound logger.
  • For one CLI tool you maintain, swap basic prints for rich tables / progress bars.
