Load Balancing

Distributes incoming traffic across a pool of servers so no single node is overwhelmed. The mechanism that makes horizontal scaling actually work — without it, adding more servers changes nothing.

Concept01 of 13
CategoryDistributed Systems
Used ByEvery web-scale system
CodeNode.js / JavaScript

Before load balancers, scaling meant buying a bigger server — vertical scaling. You hit a hard wall when one machine couldn't keep up. The solution was to spread load across many smaller machines, but that required something to decide which machine each request goes to. That something is a load balancer.

Mental model: A load balancer is a traffic cop standing in front of your server pool. Every incoming request arrives at the cop, who decides which server handles it, then forwards the request. The client never knows which server responded — it only ever talks to the cop.

How It Works

Request flow and internal mechanics

When a request arrives at the load balancer, three things happen: the LB selects a backend server using its configured algorithm, it forwards the request (either terminating the TCP connection itself or passing it through), and it tracks health of the pool to route around failures.

Layer 4 vs Layer 7 load balancing

This is the most important distinction architects make. Layer 4 (transport) balancers operate on IP and TCP — they forward packets without reading the application payload. Layer 7 (application) balancers read the HTTP request fully, enabling content-based routing.

Algorithms

Load balancing algorithms — when to use each

Round Robin

Requests are distributed sequentially: server 1, server 2, server 3, server 1... Simple and effective when all servers have identical capacity and requests take similar time. Breaks down when servers have different specs or request durations vary widely.

Weighted Round Robin

Same as round robin, but each server gets a weight. A server with weight 3 receives 3× the traffic of a weight-1 server. Use when your pool is heterogeneous — e.g., some nodes are c5.4xlarge and some are c5.2xlarge.

Least Connections

Routes each new request to the server with the fewest active connections. Better than round robin when request duration varies significantly (some requests take 10ms, others take 2s). The LB must track in-flight connection counts.

IP Hash (Sticky Sessions)

Hashes the client IP and always routes that client to the same server. Provides session affinity — useful when server-side session state exists. Problem: uneven distribution if many users share one IP (NAT), and if a server goes down, those sessions are lost.

Least Response Time

Routes to the server with the lowest combination of active connections and response time. Most accurate but requires the LB to continuously measure response times. Used in premium LBs like AWS ALB with target group metrics.

// Implementing Round Robin and Least Connections in Node.js

class RoundRobinBalancer {
  constructor(servers) {
    this.servers = servers.filter(s => s.healthy);
    this.index = 0;
  }

  next() {
    const healthy = this.servers.filter(s => s.healthy);
    if (healthy.length === 0) throw new Error('No healthy servers');
    const server = healthy[this.index % healthy.length];
    this.index = (this.index + 1) % healthy.length;
    return server;
  }
}

class LeastConnectionsBalancer {
  constructor(servers) {
    this.servers = servers;
    // Track active connection count per server
    this.connections = new Map(servers.map(s => [s.id, 0]));
  }

  next() {
    const healthy = this.servers.filter(s => s.healthy);
    if (healthy.length === 0) throw new Error('No healthy servers');

    // Pick server with fewest active connections
    return healthy.reduce((best, s) =>
      (this.connections.get(s.id) < this.connections.get(best.id)) ? s : best
    );
  }

  // Call when a request starts and ends
  increment(serverId) { this.connections.set(serverId, (this.connections.get(serverId) || 0) + 1); }
  decrement(serverId) { this.connections.set(serverId, Math.max(0, (this.connections.get(serverId) || 1) - 1)); }
}

// Health check loop — runs independently every 5 seconds
async function healthCheckLoop(servers, intervalMs = 5000) {
  async function checkServer(server) {
    try {
      const resp = await fetch(`http://${server.host}:${server.port}/health`, { signal: AbortSignal.timeout(2000) });
      server.healthy = resp.ok;
    } catch {
      server.healthy = false;
    }
  }

  setInterval(() => Promise.all(servers.map(checkServer)), intervalMs);
}

// Proxy implementation using the balancer
import http from 'http';
import httpProxy from 'http-proxy';

const servers = [
  { id: 's1', host: '10.0.0.1', port: 8080, healthy: true },
  { id: 's2', host: '10.0.0.2', port: 8080, healthy: true },
  { id: 's3', host: '10.0.0.3', port: 8080, healthy: true },
];

const balancer = new LeastConnectionsBalancer(servers);
const proxy = httpProxy.createProxyServer({});

healthCheckLoop(servers);

http.createServer((req, res) => {
  const server = balancer.next();
  balancer.increment(server.id);

  proxy.web(req, res, { target: `http://${server.host}:${server.port}` }, (err) => {
    balancer.decrement(server.id);
    res.writeHead(502); res.end('Bad Gateway');
  });

  res.on('finish', () => balancer.decrement(server.id));
}).listen(80);
Trade-offs

What you gain and what you give up

Pros

  • Horizontal scalability — add servers to add capacity
  • High availability — one server down, traffic reroutes automatically
  • SSL termination in one place
  • DDoS absorption surface
  • Zero-downtime deployments (drain, deploy, re-add)

Cons

  • Single point of failure if LB itself has no redundancy
  • Session affinity breaks stateful servers
  • Adds one network hop + latency
  • Sticky sessions cause uneven load
  • Layer 7 inspection costs CPU under high TLS load
The sticky session trap: IP hash routes a client to the same server, but if that server is deployed or crashes, the session is gone. The real fix is to not store session state on the server — use Redis or a shared token (JWT). Then any server can handle any request.
Real World

How production systems implement this

AWS: ALB (Layer 7) for HTTP microservices with path-based and host-based routing. NLB (Layer 4) for high-throughput, ultra-low-latency workloads like gaming or IoT. Global Accelerator for cross-region routing using anycast IPs.

Cloudflare: Uses anycast routing — all Cloudflare IPs are announced from every PoP globally. The BGP network routes your request to the nearest PoP automatically, before any application-layer balancing happens. This is why Cloudflare absorbs 100Tbps+ DDoS attacks: the attack traffic is spread across 250+ PoPs.

Netflix: Uses Zuul (their open-source API gateway/LB) in front of all microservices. Zuul integrates with Eureka (service registry) and Ribbon (client-side load balancing) so each service instance knows about every other service and load-balances requests itself, without a central LB for internal traffic.

Client-side vs server-side load balancing: Traditional load balancing is server-side — one central LB. Kubernetes and Netflix use client-side — each service has a list of target instances and balances requests itself. Client-side removes the central bottleneck but requires service discovery to keep instance lists fresh.
Common Mistakes

What goes wrong in production

  • Single LB with no redundancy: The LB is the new single point of failure. Always run LBs in HA pairs with a VIP (Virtual IP) that floats between them.
  • Health check endpoint that doesn't represent real health: A health check that returns 200 even when the database connection pool is exhausted is worse than useless — the LB sends traffic to a broken server.
  • Too-aggressive health check timeouts: A 500ms health check timeout causes servers under load to be marked unhealthy, reducing pool size exactly when you need capacity most.
  • Ignoring connection draining: Removing a server immediately drops all in-flight requests. Use connection draining (deregister with a drain timeout) so existing requests complete before the instance is removed.
  • Not accounting for slow backends: Round robin can pile up thousands of connections on a slow server. Least connections handles this better — but only if you measure active connections, not just request count.

Key Takeaway

A load balancer is the prerequisite for horizontal scaling — without it, additional servers are invisible to clients. Choose Layer 7 for HTTP services where content-based routing matters; Layer 4 for raw TCP where latency is critical. Use least-connections over round-robin for services with variable request duration. And always make your servers stateless — session affinity is a workaround for a design problem, not a feature.
