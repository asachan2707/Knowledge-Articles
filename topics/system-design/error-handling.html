Error Handling

Consistent, structured error responses that help API consumers understand what went wrong and what to do next. The difference between errors that reduce support tickets and errors that generate them — and the RFC 7807 Problem Details standard that most teams don't know exists.

Concept29 of 33
CategoryAPI Design
StandardRFC 7807 Problem Details for HTTP APIs
Key rule4xx = client error (they can fix it), 5xx = server error (you fix it)

A well-designed error response is a form of documentation. When an API returns {"error": "invalid"}, the developer opens a support ticket. When it returns a structured error with the field that failed, the validation rule that was violated, and a link to the docs, they fix their code in 2 minutes without contacting anyone. Error quality directly impacts developer experience and support load.

The two categories of errors: 4xx errors are the client's fault — wrong data, missing fields, invalid permissions, resource not found. 5xx errors are your fault — database down, code bug, dependency failure. 4xx errors should be descriptive enough for the client to self-serve a fix. 5xx errors should leak no internal details (don't expose stack traces or SQL errors) but should log everything internally.

Implementation

RFC 7807 Problem Details + consistent error middleware

import express from 'express';

const app = express();

// ─── RFC 7807 PROBLEM DETAILS FORMAT ──────────────────────────────────────
// Standardised error format: application/problem+json
// Fields: type (URI), title, status, detail, instance
// Custom extensions allowed (traceId, errors array, etc.)

// Example response:
// HTTP 422 Unprocessable Entity
// Content-Type: application/problem+json
// {
//   "type":     "https://api.example.com/errors/insufficient-balance",
//   "title":    "Insufficient Balance",
//   "status":   422,
//   "detail":   "Account balance of $10.00 is insufficient for transfer of $50.00",
//   "instance": "/transfers/txn_abc123",
//   "traceId":  "4bf92f3577b34da6a3ce929d0e0e4736"
// }

// Custom error classes for each error category
class AppError extends Error {
  constructor(message, status, code, details = {}) {
    super(message);
    this.status  = status;
    this.code    = code;     // Machine-readable code for client switch/case
    this.details = details;
  }
}

class ValidationError extends AppError {
  constructor(errors) {
    super('Validation failed', 400, 'VALIDATION_FAILED', { errors });
  }
}

class NotFoundError extends AppError {
  constructor(resource, id) {
    super(`${resource} not found`, 404, 'NOT_FOUND', { resource, id });
  }
}

class ConflictError extends AppError {
  constructor(message, field) {
    super(message, 409, 'CONFLICT', { field });
  }
}

class BusinessRuleError extends AppError {
  constructor(message, details = {}) {
    super(message, 422, 'BUSINESS_RULE_VIOLATION', details);
  }
}

// ─── GLOBAL ERROR HANDLER ─────────────────────────────────────────────────
app.use((err, req, res, next) => {
  const traceId = req.headers['x-trace-id'] ?? 'unknown';

  // Known application errors — safe to expose details to client
  if (err instanceof AppError) {
    const body = {
      type:     `https://api.example.com/errors/${err.code.toLowerCase().replace(/_/g, '-')}`,
      title:    err.message,
      status:   err.status,
      instance: req.path,
      traceId,
      ...err.details,
    };

    // Log 4xx at warn level, 5xx would be at error level
    console.warn({ msg: 'Client error', ...body, path: req.path, method: req.method });
    return res.status(err.status).json(body);
  }

  // Unexpected errors — log full details, return generic message
  console.error({
    msg:    'Unhandled error',
    error:  err.message,
    stack:  err.stack,
    traceId,
    path:   req.path,
    method: req.method,
  });

  // Never leak stack traces or internal error messages to clients
  res.status(500).json({
    type:     'https://api.example.com/errors/internal-error',
    title:    'An unexpected error occurred',
    status:   500,
    instance: req.path,
    traceId,  // Client can include this in a support ticket for correlation
  });
});

// ─── USAGE IN ROUTE HANDLERS ───────────────────────────────────────────────
app.post('/transfers', async (req, res, next) => {
  try {
    const { fromAccountId, toAccountId, amountCents } = req.validatedBody;

    const account = await accountService.findById(fromAccountId);
    if (!account) throw new NotFoundError('Account', fromAccountId);

    if (account.balanceCents < amountCents) {
      throw new BusinessRuleError('Insufficient balance', {
        available: account.balanceCents,
        required:  amountCents,
        currency:  'USD',
      });
    }

    const transfer = await transferService.execute({ fromAccountId, toAccountId, amountCents });
    res.status(201).json(transfer);
  } catch (err) {
    next(err); // Passes to error handler middleware
  }
});
Real World

Stripe and Twilio error design

Stripe: Every Stripe error has a type (card_error, api_error, idempotency_error), a code (card_declined, expired_card, insufficient_funds), and a human message. The code is machine-readable so client code can switch on it. The message is human-readable for display. This design means: no guessing at the cause from a status code, no parsing error message strings to determine the code path.

Twilio: Every error has a numeric code (e.g. 21211 for invalid To number), a message, and a URL to the documentation page for that specific error code. Developers can go directly from the error response to the exact explanation and fix — without searching the docs.

The traceId is your lifeline in production. Every error response should include a trace ID that links to the full request trace, logs, and span. When a user reports "I got an error," you give them the traceId from the response and you can immediately find the exact error, the full stack trace, the SQL query, and the upstream call that failed — without asking "what were you doing?" The traceId closes the loop between client-visible errors and server-side observability.

Key Takeaway

Consistent, structured errors are a first-class API feature. Use RFC 7807 Problem Details format (type, title, status, detail, instance) with a machine-readable code field. Use the right HTTP status code: 400 for malformed requests, 401 for unauthenticated, 403 for unauthorized, 422 for business rule violations, 429 for rate limiting, 5xx for server failures. Always include a traceId in error responses so support can correlate client reports to server logs. Never leak stack traces or SQL errors to clients in production — log them server-side and return a generic message with the trace ID.
