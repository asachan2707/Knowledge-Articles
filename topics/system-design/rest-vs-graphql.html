REST vs GraphQL

REST's resource-centric model vs GraphQL's query language for APIs — the overfetching problem, the N+1 query problem, schema evolution, and when each is genuinely the better choice. Plus gRPC for internal service communication where performance matters most.

CategoryAPI Design

REST dominated web APIs for 15 years. GraphQL emerged from Facebook's need to serve a mobile app with limited bandwidth — every byte mattered, and REST's fixed response shapes sent too much data on some calls and required too many calls on others. Today both are valid choices, and the right one depends on who your clients are and what they need.

The real decision: REST is the right default for most teams. GraphQL is worth the complexity when you have many different clients with different data needs (web, mobile, partner integrations), or when overfetching/underfetching are genuine pain points you've measured — not assumed.

Comparison

REST vs GraphQL — the practical trade-offs

REST: overfetching and multiple round trips Load a user profile page — needs: name, avatar, recent posts, post like counts GET /users/123 → { id, name, avatar, email, address, phone, ... } ← overfetching GET /users/123/posts → [ {id, title, body, createdAt, ...}, ... ] GET /posts/1/likes/count → { count: 42 } GET /posts/2/likes/count → { count: 17 } ... N+1 requests if you have N posts GraphQL: one request, exactly the data needed POST /graphql { user(id: "123") { name avatar posts(limit: 5) { title likeCount ← resolved in one DB query with JOIN } } } Response: exactly those 5 fields. No overfetching. One round trip. GraphQL N+1 problem — if NOT using DataLoader Query: users { posts { author { name } } } Resolves: 1 query for users, N queries for posts, N*M queries for authors → 100 users × 10 posts = 1001 queries Fix: DataLoader batches and deduplicates resolver calls

Implementation

GraphQL with DataLoader to solve the N+1 problem

import { ApolloServer, gql } from '@apollo/server';
import DataLoader from 'dataloader';
import { Pool } from 'pg';

const db = new Pool({ connectionString: process.env.DATABASE_URL });

// ─── SCHEMA ────────────────────────────────────────────────────────────────
const typeDefs = gql`
  type User {
    id:     ID!
    name:   String!
    email:  String!
    posts:  [Post!]!
  }

  type Post {
    id:        ID!
    title:     String!
    body:      String!
    likeCount: Int!
    author:    User!
  }

  type Query {
    user(id: ID!): User
    users: [User!]!
  }
`;

// ─── DATALOADERS — batch and deduplicate resolver calls ───────────────────
// Without DataLoader: resolving 10 users × 5 posts each = 10 posts queries
// With DataLoader: 1 batched query for all post IDs

function createLoaders() {
  return {
    userById: new DataLoader(async (userIds) => {
      const { rows } = await db.query(
        'SELECT * FROM users WHERE id = ANY($1)',
        [userIds]
      );
      // DataLoader requires results in same order as input IDs
      const byId = Object.fromEntries(rows.map(r => [r.id, r]));
      return userIds.map(id => byId[id] ?? null);
    }),

    postsByUserId: new DataLoader(async (userIds) => {
      const { rows } = await db.query(
        'SELECT * FROM posts WHERE user_id = ANY($1) ORDER BY created_at DESC',
        [userIds]
      );
      // Group posts by user_id
      const byUserId = {};
      rows.forEach(post => {
        if (!byUserId[post.user_id]) byUserId[post.user_id] = [];
        byUserId[post.user_id].push(post);
      });
      return userIds.map(id => byUserId[id] ?? []);
    }),
  };
}

// ─── RESOLVERS ────────────────────────────────────────────────────────────
const resolvers = {
  Query: {
    user:  (_parent, { id }, { loaders }) => loaders.userById.load(id),
    users: (_parent, _args, { db }) => db.query('SELECT * FROM users').then(r => r.rows),
  },

  User: {
    // This resolver will be called for every User in the result set.
    // Without DataLoader: N queries. With DataLoader: 1 batched query.
    posts: (user, _args, { loaders }) => loaders.postsByUserId.load(user.id),
  },

  Post: {
    author: (post, _args, { loaders }) => loaders.userById.load(post.user_id),
    likeCount: async (post) => {
      const { rows } = await db.query('SELECT COUNT(*) FROM likes WHERE post_id = $1', [post.id]);
      return parseInt(rows[0].count);
    },
  },
};

// ─── SERVER — loaders created per request (not shared) ───────────────────
const server = new ApolloServer({ typeDefs, resolvers });

// Each request gets its own DataLoader instances (prevents cache leaking between users)
const context = ({ req }) => ({ db, loaders: createLoaders(), user: req.user });
Real World

GitHub GraphQL API and when NOT to use GraphQL

GitHub GraphQL API: GitHub migrated from REST to GraphQL for their v4 API. Their key motivation: the REST API required many round trips to build complex views (issue + comments + reactions + labels + assignees). With GraphQL, integrators fetch all of it in one query, specifying only the fields they need. GitHub still maintains the REST API (v3) for simpler use cases.

When GraphQL is NOT worth it: Public APIs serving simple CRUD (GET /products/:id) don't need GraphQL's complexity. HTTP caching on REST is extremely valuable for read-heavy APIs and works for free — GraphQL's POST-only model requires custom cache keys. Simple internal services or webhooks don't benefit from a query language. Start with REST; introduce GraphQL when you have evidence of the problem it solves.

gRPC for internal services: When two microservices communicate thousands of times per second, gRPC's binary protocol (protobuf) and HTTP/2 multiplexing provide 5–10× better throughput than REST+JSON. Code generation from .proto files gives type-safe clients in any language. Used heavily by Google, Netflix, and Uber for internal service-to-service calls.

REST + sparse fieldsets is often good enough. Before adopting GraphQL, try: GET /users/123?fields=id,name,avatar. Many REST frameworks support this. It solves overfetching without the N+1 complexity, the DataLoader requirement, the POST-only caching constraint, and the learning curve of an entirely different API paradigm.

Key Takeaway

REST is the right default — it is simpler, HTTP-cacheable, and well-understood. GraphQL is worth the complexity when many different clients have different data needs and overfetching/underfetching are genuine measured problems. If you choose GraphQL, DataLoader is mandatory — without it, every relationship in your schema becomes an N+1 query. gRPC is the choice for internal microservice-to-microservice communication where throughput matters and both sides can run code generation. Don't let hype drive the choice — let the actual client data requirements drive it.
