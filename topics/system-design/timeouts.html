Timeouts

Connection, read, and deadline timeouts are the fuses in your distributed system. Without them, one slow or hung downstream service blocks your threads indefinitely, exhausts your connection pool, and brings down your entire service. Timeouts are what prevent a single slow dependency from becoming a total outage.

CategoryResilience

Key RuleEvery network call must have a timeout

A system without timeouts is a system with a latent single point of failure. Any service you call can become slow or unresponsive вЂ” database under load, upstream API degraded, network partition. Without a timeout, every thread waiting on that call is blocked forever. As traffic continues, you fill your thread pool, your connection pool, and your queue. Eventually your service stops accepting requests вЂ” not because anything is wrong with your code, but because you were waiting on something else.

The cascade: Service A calls B calls C. C becomes slow. Without timeouts on Aв†’B and Bв†’C, A's threads start piling up waiting for B (which is waiting for C). A's thread pool exhausts. A starts rejecting requests. The latency of C killed A вЂ” even though C is just slow, not down.

Timeout patterns in Node.js

import http from 'http';

// в”Ђв”Ђв”Ђ fetch() with AbortController deadline в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ
async function fetchWithTimeout(url, options = {}, timeoutMs = 5000) {
  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(), timeoutMs);

  try {
    const response = await fetch(url, { ...options, signal: controller.signal });
    if (!response.ok) throw Object.assign(new Error(`HTTP ${response.status}`), { status: response.status });
    return await response.json();
  } catch (err) {
    if (err.name === 'AbortError') {
      throw Object.assign(new Error(`Request to ${url} timed out after ${timeoutMs}ms`), { code: 'ETIMEDOUT' });
    }
    throw err;
  } finally {
    clearTimeout(timer);
  }
}

import { Pool } from 'pg';
const pool = new Pool({
  connectionString:         process.env.DATABASE_URL,
  connectionTimeoutMillis:  3_000,   // Max time to get a connection from pool
  idleTimeoutMillis:        30_000,  // How long idle connections stay in pool
  query_timeout:            10_000,  // pg query-level timeout (statement_timeout)
  statement_timeout:        10_000,  // Equivalent PostgreSQL server-side timeout
});

// Run any query with a per-query deadline
async function queryWithTimeout(sql, params, timeoutMs = 5000) {
  const client = await pool.connect();
  try {
    await client.query(`SET statement_timeout = ${timeoutMs}`);
    return await client.query(sql, params);
  } finally {
    client.release();
  }
}

// в”Ђв”Ђв”Ђ Cascading deadline: propagate parent deadline to child calls в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ
// When request A (deadline 10s) calls service B, pass the REMAINING time
// so B's timeout is bounded by A's remaining budget, not a fresh N seconds.

async function handleRequest(req, res) {
  const requestDeadlineMs = 10_000;
  const startedAt = Date.now();

  function remainingMs() {
    return requestDeadlineMs - (Date.now() - startedAt);
  }

  const user = await fetchWithTimeout(
    `http://user-service/users/${req.params.id}`,
    {},
    Math.min(3000, remainingMs())  // At most 3s, but less if budget is low
  );

  const orders = await fetchWithTimeout(
    `http://order-service/orders?userId=${user.id}`,
    {},
    Math.min(5000, remainingMs())  // Remaining budget for this call
  );

  res.json({ user, orders });
}

Real World

How Netflix and Google set timeouts

Netflix: Uses the "p99.9 Г— 2" heuristic for setting timeouts. Measure the 99.9th percentile latency of your downstream service. Set your timeout to 2Г— that value. This means fewer than 0.1% of legitimate slow-but-valid requests get cut off, while runaway calls are reliably terminated. Netflix's Hystrix library enforces timeouts on every command wrapping a downstream call.

Google SRE: Propagates deadlines through every call in a request chain via context objects. When the root deadline expires, every in-progress call in the chain immediately aborts. This prevents the "orphaned work" problem where a child call completes successfully long after its parent has already returned an error to the user вЂ” wasting CPU and I/O on work whose result will never be used.

The "no timeout" failure mode: Amazon's famous 2013 DynamoDB incident was partially driven by retry storms from clients without proper timeouts вЂ” clients kept requests open for minutes, eventually filling every available socket in the fleet. Timeouts are the first line of defence against this class of failure.

Propagate deadlines, not fresh timeouts. If the user has a 10-second budget and you've already spent 7 seconds, don't give your downstream call a fresh 5-second timeout вЂ” give it 3 seconds (the remaining budget). Otherwise your total call chain latency silently exceeds what the user is waiting for. This is called "deadline propagation" and is the standard in gRPC, Google's RPC framework, and well-designed microservices.

Pros

  • Prevents thread pool exhaustion from hung calls
  • Limits blast radius of a slow downstream service
  • Makes latency predictable вЂ” users get an error in N seconds, not infinity
  • Enables circuit breakers to open (need timeout counts as a failure)

Cons

  • Timeout too short в†’ false positives on legitimately slow operations
  • Timeout too long в†’ too slow to protect under real degradation
  • Without deadline propagation, child work continues after parent times out (wasted resources)
  • Requires tuning per-downstream-service, not one-size-fits-all

Key Takeaway

Every network call must have a timeout вЂ” this is non-negotiable. Set a connection timeout (how long to wait to establish TCP), a read timeout (how long to wait for first byte), and a deadline (total operation budget). Use the p99.9 Г— 2 heuristic as a starting point for read timeouts. Propagate deadlines through your call chain so child calls never outlive their parent budget. Pair timeouts with circuit breakers so that once a downstream starts timing out consistently, you stop trying and return fallbacks immediately rather than waiting out every timeout.
