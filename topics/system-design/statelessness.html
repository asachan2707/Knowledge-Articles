Statelessness

A service is stateless when any instance can handle any request — no request depends on what a previous request stored in local memory. The prerequisite for horizontal scaling: you can't add a second server if the first one holds session state that the second one doesn't have.

Concept32 of 33
CategoryArchitecture
Anti-patternSticky sessions, in-memory session stores
SolutionExternalise state to Redis, database, JWTs

When you start a web server, it handles one request at a time. When traffic grows, you add a second server behind a load balancer. But if your first server stored the user's session in its local memory, the second server can't find it — the user gets logged out. This is the stateful anti-pattern: state that lives in a single service instance, making that instance un-replaceable and un-scalable. Statelessness means externalising that state so any instance can pick up any request.

The 12-Factor App principle: "Store session state in a backing service such as Memcached or Redis." Twelve-factor apps share nothing between instances. Any state that needs to persist across requests must live in an external store, not in the process.

The Pattern

Stateful vs stateless — what changes

Stateful: session stored in-memory on server 1 User logs in ──▶ Server 1 stores { userId: 123, cart: [...] } in memory User request ──▶ Load Balancer ──▶ Server 2 — has no session — user sees logged-out state Scale Server 1 down for maintenance → all active sessions lost Deploy new version → rolling restart loses all in-flight sessions Stateless: session stored externally in Redis User logs in ──▶ Server 1 stores { userId: 123, cart: [...] } in Redis with sessionId Returns sessionId cookie to client User request ──▶ Load Balancer ──▶ Server 2 Server 2 reads sessionId cookie → fetches session from Redis Full session available — request succeeds Scale to 10 servers → all see same session state via Redis Deploy new version → rolling restart is safe — sessions survive JWTs: externalize auth state into the token itself Session store: look up userId 123 → is it valid? → DB/Redis query per request JWT: decode token → userId 123, roles=[admin], exp=1700003600 → No lookup needed. Implementation

Externalising session state with Redis

import express from 'express';
import session from 'express-session';
import RedisStore from 'connect-redis';
import { createClient } from 'redis';

const redis = createClient({ url: process.env.REDIS_URL });
await redis.connect();

const app = express();

// ─── EXTERNALIZED SESSION STORE ────────────────────────────────────────────
// Sessions live in Redis — any server instance reads them
app.use(session({
  store: new RedisStore({ client: redis }),
  secret:            process.env.SESSION_SECRET,
  resave:            false,
  saveUninitialized: false,
  cookie: {
    secure:   process.env.NODE_ENV === 'production', // HTTPS only in prod
    httpOnly: true,     // Not accessible via JavaScript (XSS protection)
    sameSite: 'strict', // CSRF protection
    maxAge:   24 * 60 * 60 * 1000, // 24h
  },
}));

// ─── STATELESS AUTH WITH JWT ───────────────────────────────────────────────
// No server-side state at all — token carries user identity
// Downside: can't revoke individual tokens without a blocklist (server-side state)

function issueTokens(userId) {
  const accessToken = jwt.sign(
    { sub: userId, roles: ['user'] },
    process.env.JWT_PRIVATE_KEY,
    { algorithm: 'RS256', expiresIn: '15m' }
  );

  const refreshToken = jwt.sign(
    { sub: userId, type: 'refresh' },
    process.env.JWT_PRIVATE_KEY,
    { algorithm: 'RS256', expiresIn: '7d' }
  );

  return { accessToken, refreshToken };
}

// ─── STATELESS SERVICE: no process-level state ────────────────────────────
// Wrong: caching in module-level variable (shared between requests but not instances)
let userCache = {}; // ← WRONG: each server instance has its own cache

// Right: cache in Redis (shared across all instances)
async function getCachedUser(userId) {
  const cached = await redis.get(`user:${userId}`);
  if (cached) return JSON.parse(cached);

  const user = await db.query('SELECT * FROM users WHERE id = $1', [userId]);
  if (user.rows.length) {
    await redis.setEx(`user:${userId}`, 300, JSON.stringify(user.rows[0]));
  }
  return user.rows[0];
}

// ─── IDENTIFYING STATEFUL PATTERNS IN YOUR CODEBASE ──────────────────────
// These patterns create server-instance-level state:
const ANTI_PATTERNS = [
  'let sessionData = {}',       // Module-level session storage
  'global.userSessions',        // Global state
  'app.locals.pendingJobs',     // Express app.locals for mutable state
  'setInterval for in-memory cache refresh', // Each instance caches independently
];

// ─── GRACEFUL HANDLING OF STATEFUL WORKLOADS ──────────────────────────────
// Some things ARE stateful by nature (WebSocket connections, streaming, upload progress).
// Solution: use a consistent hash or sticky routing for those specific endpoints only.
// Don't make your entire service stateful because one endpoint needs it.

// Example: WebSocket connections are stateful — sticky route only the WS endpoint
const wsRouter = express.Router();
wsRouter.ws('/ws', (ws, req) => {
  // This endpoint is routed via consistent hash on userId to the same server
  // All other endpoints remain stateless
});
Real World

The 12-Factor App and Kubernetes pod scheduling

Kubernetes expects statelessness: Kubernetes can restart, move, or scale your pods at any time. If your pod stores session state in memory, a pod restart (triggered by a deploy, OOM kill, or node failure) silently logs out all active users. Kubernetes' design assumes stateless pods — any pod in a deployment can be replaced by any other. Statelessness is not optional for Kubernetes-native services.

Sticky sessions as a crutch: Load balancers can route the same client to the same server ("sticky sessions" via IP hash or cookie). This lets a stateful service scale temporarily. It is not a solution — it prevents full horizontal scaling, makes rolling deploys harder, and doesn't survive server crashes. Sticky sessions are a migration aid, not an architecture.

The cloud-native implication: Every major cloud hosting model — Lambda, Cloud Run, App Engine, Kubernetes — assumes stateless instances that can be started, stopped, and replicated on demand. If your service is stateful, it can't leverage elastic scaling, can't run on these platforms correctly, and has a hidden single point of failure in every server instance holding live session data.

In-memory caches are a form of state. A module-level let cache = {} is state. Each server instance builds its own cache from scratch. When you add a second instance, it cold-starts with an empty cache, causing a load spike on your database. Externalise caches to Redis. A brief window of duplicate DB queries is acceptable; an unbounded cache-miss spike on every new instance is not.

Stateless pros

  • Horizontal scaling: add instances freely
  • Rolling deploys are safe — sessions survive restarts
  • Load balancer can route any request to any instance
  • Crash recovery: new instance starts clean, users unaffected

Stateless cons

  • External state store (Redis) becomes a dependency — must be HA
  • Redis call per request for session adds ~0.5ms latency
  • JWT revocation requires a server-side blocklist (re-introducing state)
  • Some workloads (streaming, WebSockets) are inherently stateful

Key Takeaway

Statelessness is the prerequisite for horizontal scaling. Any state stored in a service instance's memory — sessions, caches, counters, in-flight state — breaks the assumption that any instance can handle any request. Externalise session state to Redis (connect-redis or equivalent). Use JWTs for stateless authentication. Move in-memory caches to Redis or a shared cache. The only things a stateless service should store in memory are immutable config and read-only data loaded at startup. Sticky sessions are a migration aid, not a solution — they prevent true horizontal scaling and don't survive crashes.
