Rate Limiting

Controls how many requests a client can make in a given time window. Protects services from overload, prevents abuse, enforces fair usage across tenants, and is a prerequisite for any public-facing API.

Concept10 of 13
CategoryAPI Design / Security
Used ByEvery public API: Stripe, GitHub, Twitter, OpenAI
ImplementationRedis sliding window, token bucket, leaky bucket

Without rate limiting, a single misbehaving client — whether a scraper, a buggy integration, or a deliberate attacker — can exhaust your server capacity and degrade service for every other user. Rate limiting is the enforcement mechanism that ensures fair resource allocation.

Rate limiting at every layer: Effective rate limiting happens at multiple layers. At the network edge (Cloudflare, AWS WAF) for DDoS. At the API gateway for per-client quotas. At the service level for per-feature or per-tenant limits. These layers are complementary, not alternatives.

Algorithms

Four algorithms — trade-offs between simplicity and accuracy

Fixed Window Counter — simplest, with a boundary problem

Count requests in a fixed time window (e.g. 12:00:00–12:01:00). Reset counter at the window boundary. Problem: a client can make 100 requests at 12:00:59 and 100 more at 12:01:01 — 200 requests in 2 seconds while technically under the per-minute limit.

Sliding Window Log — most accurate, highest memory

Store a timestamp for every request in the last window. On each new request, remove timestamps older than the window, count remaining, reject if over limit. Perfectly accurate but stores O(requests) timestamps per client — expensive at scale.

Sliding Window Counter — the production standard

Hybrid: combine the previous fixed window count with the current window count, weighted by how far into the current window we are. Approximates the sliding log with O(1) memory per client. This is what most production systems use.

Token Bucket — allows bursting

Each client has a bucket with a maximum capacity. Tokens refill at a fixed rate (e.g. 10/second). Each request consumes one token. If bucket is empty, reject. Allows short bursts (drain the bucket) but enforces average rate. Used by AWS API Gateway and most CDNs.

Implementation

Sliding window counter in Node.js with Redis

import { createClient } from 'redis';
const redis = createClient({ url: process.env.REDIS_URL });
await redis.connect();

// Sliding window counter using Redis sorted sets
// Accurate, efficient, and works across multiple API gateway instances
async function isRateLimited(identifier, { limit = 100, windowMs = 60000 } = {}) {
  const key = `ratelimit:${identifier}`;
  const now = Date.now();
  const windowStart = now - windowMs;

  // Atomic pipeline: remove expired, add current, count, set TTL
  const [[, , count]] = await redis.multi()
    .zRemRangeByScore(key, 0, windowStart)              // Remove old entries
    .zAdd(key, [{ score: now, value: `${now}` }])       // Add current request
    .zCard(key)                                          // Count in window
    .expire(key, Math.ceil(windowMs / 1000))             // Auto-expire key
    .exec();

  return {
    limited:   count > limit,
    count:     count,
    remaining: Math.max(0, limit - count),
    resetAt:   new Date(now + windowMs).toISOString(),
  };
}

// Token bucket with Redis — allows bursting
async function tokenBucket(identifier, { capacity = 10, refillRate = 2, refillIntervalMs = 1000 } = {}) {
  const key = `tokenbucket:${identifier}`;
  const now = Date.now();

  // GET current state
  const [tokens, lastRefillStr] = await redis.hmGet(key, 'tokens', 'lastRefill');

  let currentTokens = parseFloat(tokens ?? capacity);
  const lastRefill   = parseInt(lastRefillStr ?? now);

  // Calculate tokens to add since last refill
  const elapsed  = now - lastRefill;
  const newTokens = (elapsed / refillIntervalMs) * refillRate;
  currentTokens = Math.min(capacity, currentTokens + newTokens);

  if (currentTokens < 1) {
    return { allowed: false, tokens: 0, retryAfterMs: Math.ceil((1 - currentTokens) / refillRate * refillIntervalMs) };
  }

  // Consume one token
  currentTokens -= 1;
  await redis.hSet(key, { tokens: currentTokens, lastRefill: now });
  await redis.expire(key, Math.ceil(capacity / refillRate) + 1);

  return { allowed: true, tokens: currentTokens };
}

// Express middleware — enforces per-user and per-IP limits
function rateLimitMiddleware({ perUserLimit = 100, perIpLimit = 200, windowMs = 60000 } = {}) {
  return async (req, res, next) => {
    const userId     = req.user?.sub;
    const ip         = req.ip;
    const identifier = userId ?? ip; // Prefer user ID; fall back to IP for unauthenticated

    const [userLimit, ipLimit] = await Promise.all([
      isRateLimited(`user:${identifier}`, { limit: perUserLimit, windowMs }),
      isRateLimited(`ip:${ip}`,           { limit: perIpLimit, windowMs }),
    ]);

    // Always expose headers so clients can back off gracefully
    res.setHeader('X-RateLimit-Limit',     perUserLimit);
    res.setHeader('X-RateLimit-Remaining', userLimit.remaining);
    res.setHeader('X-RateLimit-Reset',     userLimit.resetAt);

    if (userLimit.limited || ipLimit.limited) {
      return res.status(429).json({
        error:      'Too Many Requests',
        message:    'Rate limit exceeded. Back off and retry.',
        retryAfter: Math.ceil(windowMs / 1000),
        resetAt:    userLimit.resetAt,
      });
    }

    next();
  };
}

// Mount on router
app.use('/api/', rateLimitMiddleware({ perUserLimit: 100, perIpLimit: 300, windowMs: 60000 }));

// Stricter limits for expensive operations
app.use('/api/ai/', rateLimitMiddleware({ perUserLimit: 10, perIpLimit: 20, windowMs: 60000 }));
Real World

How Stripe, GitHub, and OpenAI implement rate limits

Stripe: Per-account limits based on account tier. Free accounts: 25 req/s. Production: higher. Stripe uses token bucket with burst allowance. They also rate limit per API key, not just per account — so one leaked key can be limited without affecting other keys on the same account.

GitHub: 5,000 requests/hour for authenticated users, 60/hour for unauthenticated (IP-based). GraphQL API: 5,000 points/hour where each operation costs different points based on complexity. This is rate limiting by cost, not just count — prevents expensive queries from abusing the quota.

OpenAI: Rate limits on tokens-per-minute, not just requests-per-minute. A single request consuming 10,000 tokens counts as much as 10 requests of 1,000 tokens each. This is resource-aware rate limiting — measuring actual consumption, not request count.

Rate limiting keys beyond just the user ID: Most production systems rate limit on multiple dimensions: user ID (per-account limit), API key (per-key limit), IP address (per-IP limit for unauthenticated), and endpoint (expensive endpoints get tighter limits). Layer these — a burst of requests from one IP is still caught even if the user hasn't hit their quota yet.

Pros

  • Protects services from overload and abuse
  • Fair usage enforcement across tenants
  • Forces good API client behavior (backoff, retry)
  • DDoS mitigation at application layer
  • Quota enforcement for billing tiers

Cons

  • Legitimate burst traffic may be rejected (misconfigured limits)
  • Distributed systems need centralized state (Redis) for accuracy
  • False positives behind NAT (many users, one IP)
  • Rate limit state can be a single point of failure if Redis goes down

Key Takeaway

Rate limiting is not optional for production APIs — it's the mechanism that keeps one client from starving all others. Use the sliding window counter algorithm (Redis sorted sets) for per-client quotas: accurate, O(1) memory, works across distributed gateway instances. Always expose rate limit headers (X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset) so clients can implement proper backoff rather than hammering your 429 responses. Layer limits by user, API key, and IP address for defense in depth.
