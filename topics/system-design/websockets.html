WebSockets

Full-duplex persistent connections that let servers push data to clients without polling. The protocol powering chat apps, live dashboards, multiplayer games, and collaborative editing tools — and the architectural challenge of scaling them horizontally.

Concept17 of 33
CategoryReal-Time Communication
Librariesws, Socket.IO, uWebSockets.js
Used BySlack, Figma, Discord, trading platforms

HTTP is request-response: the client speaks, the server answers, the connection closes. This works for fetching data but fails for pushing it. Polling workarounds (hit the server every second) waste bandwidth and introduce artificial latency. WebSockets replace the request-response model with a persistent, bidirectional channel — once established, either side can send data at any time, with no overhead per message.

WebSocket vs SSE vs polling: WebSocket is bidirectional (client and server both send). SSE (Server-Sent Events) is unidirectional server→client push over HTTP — simpler, works through proxies, but can't send from client. Polling is the fallback for environments where persistent connections aren't feasible. Use WebSocket for chat, collaborative tools, games. Use SSE for live feeds, progress bars, notifications.

Implementation

WebSocket server with horizontal scaling via Redis pub/sub

import { WebSocketServer, WebSocket } from 'ws';
import { createClient } from 'redis';
import http from 'http';

// ─── SINGLE-SERVER WEBSOCKET (no horizontal scaling) ─────────────────────
const server = http.createServer();
const wss    = new WebSocketServer({ server });

// Map of roomId → Set of WebSocket connections on THIS server instance
const rooms = new Map();

wss.on('connection', (ws, req) => {
  // Parse room from URL: ws://host/chat?room=general
  const url     = new URL(req.url, 'http://localhost');
  const roomId  = url.searchParams.get('room') ?? 'general';
  const userId  = url.searchParams.get('userId');

  // Add to room
  if (!rooms.has(roomId)) rooms.set(roomId, new Set());
  rooms.get(roomId).add(ws);

  ws.on('message', (data) => {
    const message = JSON.parse(data.toString());
    broadcastToRoom(roomId, { ...message, userId }, ws);
  });

  ws.on('close', () => {
    rooms.get(roomId)?.delete(ws);
  });

  // Heartbeat: detect broken connections (NAT timeout, mobile network switch)
  ws.isAlive = true;
  ws.on('pong', () => { ws.isAlive = true; });
});

function broadcastToRoom(roomId, message, sender) {
  const roomClients = rooms.get(roomId);
  if (!roomClients) return;
  const payload = JSON.stringify(message);
  for (const client of roomClients) {
    if (client !== sender && client.readyState === WebSocket.OPEN) {
      client.send(payload);
    }
  }
}

// Ping every 30s to detect dead connections
const heartbeat = setInterval(() => {
  wss.clients.forEach(ws => {
    if (!ws.isAlive) return ws.terminate();
    ws.isAlive = false;
    ws.ping();
  });
}, 30_000);

wss.on('close', () => clearInterval(heartbeat));


// ─── HORIZONTAL SCALING: Redis pub/sub fan-out ───────────────────────────
// Problem: if Client A is on server 1 and Client B is on server 2,
// server 1's broadcastToRoom only reaches clients on server 1.
// Solution: publish to Redis — every server subscribes and relays to local clients.

const publisher  = createClient({ url: process.env.REDIS_URL });
const subscriber = createClient({ url: process.env.REDIS_URL }); // separate client for subscribe
await publisher.connect();
await subscriber.connect();

// Subscribe to ALL room channels
await subscriber.pSubscribe('room:*', (message, channel) => {
  const roomId = channel.replace('room:', '');
  broadcastToRoom(roomId, JSON.parse(message), null); // null = relay to ALL local clients
});

// Modified message handler: publish to Redis instead of broadcasting locally
wss.on('connection', (ws, req) => {
  const url    = new URL(req.url, 'http://localhost');
  const roomId = url.searchParams.get('room') ?? 'general';
  const userId = url.searchParams.get('userId');

  if (!rooms.has(roomId)) rooms.set(roomId, new Set());
  rooms.get(roomId).add(ws);

  ws.on('message', async (data) => {
    const message = JSON.parse(data.toString());
    const payload = JSON.stringify({ ...message, userId });

    // Publish to Redis — all server instances (including this one) will receive it
    await publisher.publish(`room:${roomId}`, payload);
  });

  ws.on('close', () => rooms.get(roomId)?.delete(ws));
});

// ─── CLIENT-SIDE: reconnect with exponential backoff ─────────────────────
function createReconnectingWebSocket(url) {
  let ws, attempt = 0;

  function connect() {
    ws = new WebSocket(url);

    ws.onopen    = () => { attempt = 0; console.log('Connected'); };
    ws.onmessage = (event) => handleMessage(JSON.parse(event.data));
    ws.onclose   = () => {
      const delay = Math.min(30_000, 1000 * Math.pow(2, attempt++)) * (0.5 + Math.random() * 0.5);
      console.log(`Reconnecting in ${Math.round(delay)}ms (attempt ${attempt})`);
      setTimeout(connect, delay);
    };
    ws.onerror = (err) => console.error('WebSocket error:', err);
  }

  connect();
  return { send: (data) => ws?.readyState === WebSocket.OPEN && ws.send(JSON.stringify(data)) };
}

server.listen(3000);
Real World

Slack, Discord, and Figma — how they scale WebSockets

Slack: Maintains one WebSocket connection per logged-in client. Real-time events (messages, presence, typing indicators) are pushed over this connection. Slack's backend publishes events to a message bus (Kafka-based), which routes to the WebSocket gateway layer holding the relevant connections. Gateways fan out to local connections.

Discord: Handles millions of concurrent WebSocket connections. Uses a dedicated "Gateway" service that maintains connections, and "State" services behind it. Discord moved from Node.js to Elixir (Erlang VM) for their WebSocket gateway, exploiting the BEAM VM's lightweight process model — each WebSocket maps to an Erlang process with negligible overhead vs OS threads.

Figma: Collaborative editing requires sub-100ms propagation of cursor moves and edits. Uses WebSockets for the real-time channel. Operational Transforms (OT) or CRDT algorithms on top ensure concurrent edits converge to the same document state regardless of ordering.

WebSocket servers are stateful — this complicates horizontal scaling. Each client's connection is pinned to one server instance. When a message arrives for a client on a different instance, you need an out-of-band channel (Redis pub/sub, a message broker, or a coordination service) to relay it. This is fundamentally different from stateless HTTP servers where any instance can handle any request.

Key Takeaway

WebSockets provide full-duplex communication over a single persistent TCP connection, eliminating polling overhead and enabling true server push. The architectural challenge is horizontal scaling: connections are stateful and pinned to a server instance, so cross-instance messaging requires a pub/sub layer (Redis is the standard). Always implement heartbeats to detect broken connections, and exponential backoff with jitter on the client for reconnection. Choose WebSocket for bidirectional real-time workloads; use SSE for simpler server-to-client push where full duplex isn't needed.
