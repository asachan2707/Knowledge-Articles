Throttling

Slowing down excess traffic rather than rejecting it outright. The leaky bucket algorithm, queue-based throttling, and the key distinction from rate limiting — throttling shapes traffic flow while preserving request delivery; rate limiting stops it.

Concept30 of 33
CategoryResilience
Pair WithRate Limiting (Concept 10), Circuit Breaker (Concept 07)
AlgorithmLeaky bucket, queue with concurrency limit

Rate limiting stops requests once a quota is exceeded — the excess gets a 429 and the client must retry. Throttling takes a different approach: absorb excess requests into a queue and process them at a controlled rate. No requests are lost; they are delayed. This is the distinction between protecting a downstream service by shedding load (rate limiting) vs protecting it by smoothing load (throttling).

Rate limiting vs throttling: Rate limiting: "you've made 100 requests this minute — I'm returning 429, come back in 60 seconds." Throttling: "you've sent requests faster than I can process them — I'm queuing the excess and processing them at 100/minute." Rate limiting is a hard stop. Throttling is a flow regulator.

Algorithm

Leaky bucket — smoothing bursty traffic

Leaky bucket analogy Requests come in ──▶ [ bucket (queue) ] ──▶ leak out at FIXED rate ──▶ processed Bucket capacity = 100 (max queue size) Leak rate = 10 req/sec (processing rate) t=0: 50 requests arrive instantly ──▶ queued [ 50 in bucket ] t=0s: process 10 ──▶ delivered [ 40 in bucket ] t=1s: 60 more requests arrive [ 100 in bucket — at capacity ] t=1s: process 10 ──▶ delivered [ 90 in bucket ] t=1s: 10 more arrive ──▶ dropped (bucket full) vs Token Bucket: burst of 100 can be processed instantly if bucket is full Leaky bucket: always processes at fixed rate — smooths ALL bursts Queue-based throttling with concurrency limit Concurrency limit: 10 simultaneous requests to downstream Queue: unlimited (or capped) waiting requests t=0: 100 requests → 10 start immediately, 90 wait in queue t=0.5s: 5 complete → 5 more start from queue t=1.0s: 5 more complete → 5 more start ... All 100 eventually processed. No rejections, just queuing delay.
Implementation

Throttle with concurrency queue in Node.js

// ─── CONCURRENCY-LIMITED THROTTLE ─────────────────────────────────────────
// Process at most N tasks concurrently. Excess tasks wait in queue.
class Throttle {
  constructor({ concurrency = 10, queueLimit = 1000 } = {}) {
    this.concurrency = concurrency;
    this.queueLimit  = queueLimit;
    this.running     = 0;
    this.queue       = [];
  }

  async run(fn) {
    if (this.queue.length >= this.queueLimit) {
      throw new Error('Throttle queue full — shedding load');
    }

    return new Promise((resolve, reject) => {
      this.queue.push({ fn, resolve, reject });
      this._tick();
    });
  }

  _tick() {
    while (this.running < this.concurrency && this.queue.length > 0) {
      const { fn, resolve, reject } = this.queue.shift();
      this.running++;
      Promise.resolve().then(fn).then(resolve, reject).finally(() => {
        this.running--;
        this._tick(); // Process next queued item when slot frees up
      });
    }
  }
}

// Usage: throttle outgoing calls to a downstream service
const dbThrottle = new Throttle({ concurrency: 5, queueLimit: 500 });

async function queryWithThrottle(sql, params) {
  return dbThrottle.run(() => db.query(sql, params));
}

// ─── LEAKY BUCKET RATE SHAPER (outgoing requests) ─────────────────────────
// Shape a burst of outgoing requests to a fixed rate (N per second)
class LeakyBucket {
  constructor({ rate = 10, capacity = 100 }) {
    this.rate      = rate;      // requests per second to process
    this.capacity  = capacity;  // max queue size
    this.queue     = [];
    this.running   = false;
  }

  async add(fn) {
    if (this.queue.length >= this.capacity) {
      throw new Error('Leaky bucket overflow — request dropped');
    }
    return new Promise((resolve, reject) => {
      this.queue.push({ fn, resolve, reject });
      if (!this.running) this._start();
    });
  }

  _start() {
    this.running = true;
    const interval = 1000 / this.rate; // ms between each processed request

    const tick = () => {
      if (this.queue.length === 0) { this.running = false; return; }
      const { fn, resolve, reject } = this.queue.shift();
      Promise.resolve().then(fn).then(resolve, reject);
      setTimeout(tick, interval);
    };
    tick();
  }
}

// Send 100 emails at most 5 per second, regardless of how fast they arrive
const emailBucket = new LeakyBucket({ rate: 5, capacity: 500 });
emails.forEach(email => emailBucket.add(() => sendEmail(email)));

// ─── EXPRESS MIDDLEWARE: concurrency throttle per route ───────────────────
function throttleMiddleware({ concurrency = 20, queueLimit = 200 } = {}) {
  const throttle = new Throttle({ concurrency, queueLimit });

  return async (req, res, next) => {
    try {
      await throttle.run(() => new Promise((resolve) => {
        req._throttleResolve = resolve;
        next();
      }));
    } catch {
      res.status(503).json({
        error:      'Service temporarily overloaded',
        retryAfter: 5,
      });
    }
  };
}

// Throttle expensive endpoint to at most 10 concurrent executions
app.post('/reports/generate', throttleMiddleware({ concurrency: 10, queueLimit: 50 }), async (req, res) => {
  const report = await generateReport(req.body);
  req._throttleResolve?.();
  res.json(report);
});
Real World

AWS SDK adaptive throttling and Nginx rate shaping

AWS SDK adaptive retry mode: The AWS SDK's "adaptive" retry mode includes a client-side token bucket that throttles the rate of requests when receiving throttling responses. Rather than immediately retrying at full speed, it reduces the token refill rate until the service stops throttling. This is self-regulating throttling — it backs off proportionally to the amount of throttling received.

Nginx limit_req: limit_req zone=one burst=5 nodelay implements leaky bucket at the reverse proxy layer. burst=5 allows a queue of 5 excess requests. nodelay processes burst requests immediately (token bucket behaviour). Without nodelay, excess requests are delayed by the leaky bucket rate — a form of throttling, not rejection.

When to throttle vs rate limit: Throttle outgoing calls to third-party APIs with quota limits — you want to use the quota evenly, not burn it in 10 seconds. Throttle expensive internal operations (report generation, ML inference) to prevent overloading the backend while still accepting all requests. Rate limit untrusted external clients to prevent abuse.

Throttling is the right tool when request loss is unacceptable. If every request represents a financial transaction, a customer order, or an irreversible action — you can't afford to return 429 and tell the client to retry. Absorb the excess in a queue and process at a safe rate. Size your queue conservatively: a queue of 10,000 items sounds safe but if each takes 1 second, the last item waits 2.7 hours. Combine throttling with sensible queue limits and a maximum wait time.

Throttling pros

  • No requests lost — queued, not rejected
  • Protects downstream from burst load
  • Smooths traffic — more predictable load on downstream
  • Better user experience than hard rejections

Throttling cons

  • Adds latency — queued requests wait
  • Queue can grow unboundedly if not capped
  • Not suitable when SLA requires fast rejection (e.g. real-time bidding)
  • Memory overhead for large queues

Key Takeaway

Throttling and rate limiting solve related but different problems. Rate limiting stops requests above a quota (hard reject). Throttling delays excess requests in a queue (slow, not stop). Use throttling for outgoing calls to third-party APIs with quota limits, for expensive operations where every request must eventually be processed, and for protecting internal services from burst load. Use a concurrency-limited queue as the implementation. Cap your queue size — unbounded queues become unbounded memory usage. Combine throttling with a maximum wait time so requests don't queue indefinitely.
