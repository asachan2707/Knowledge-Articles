Observability

The three pillars — metrics, logs, and distributed traces — and how they work together to answer "what is broken, where, and why." The difference between a system that pages you at 3am with a mystery and one you can debug in minutes.

Concept18 of 33
CategoryOperations
ToolsOpenTelemetry, Prometheus, Grafana, Jaeger, Datadog
FrameworkRED metrics: Rate, Errors, Duration

Monitoring asks: "is this specific thing still working?" Observability asks: "why is the system behaving unexpectedly?" A monitored system tells you when CPU spikes. An observable system lets you figure out that the CPU spike was caused by a database query introduced in the 14:32 deploy, triggered by users in the EU timezone, because a missing index is forcing a full table scan on the orders table.

The three pillars: Metrics (numbers over time: request rate, error rate, latency percentiles), Logs (structured events with context: who did what, when, with what result), and Traces (the path a single request took through every service it touched). Metrics tell you something is wrong. Logs tell you what happened. Traces tell you where in the chain it went wrong.

Metrics, logs, and traces — how they complement each other

[3.8s] ← bottleneck └── publish order.created [45ms] Step 3 — Logs tell you WHY: { "traceId":"abc123", "query":"SELECT * FROM orders WHERE status='pending'", "duration_ms":3800, "rows_examined":8420000, "rows_returned":12, "missing_index":true } → Full table scan on 8.4M rows because orders.status has no index. Implementation

Structured logging, RED metrics, and OpenTelemetry tracing

import { NodeSDK }                from '@opentelemetry/sdk-node';
import { Resource }               from '@opentelemetry/resources';
import { trace, metrics, context, propagation } from '@opentelemetry/api';
import { PrometheusExporter }     from '@opentelemetry/exporter-prometheus';
import express from 'express';

// ─── OPENTELEMETRY SETUP ──────────────────────────────────────────────────
const sdk = new NodeSDK({
  resource: new Resource({ 'service.name': 'checkout-service', 'service.version': '2.1.0' }),
  // Exporter: Jaeger for traces, Prometheus for metrics
});
sdk.start();

const tracer = trace.getTracer('checkout-service');
const meter  = metrics.getMeter('checkout-service');

// ─── RED METRICS (Rate, Errors, Duration) ────────────────────────────────
// These three metrics cover the health of any service.

const requestCounter = meter.createCounter('http_requests_total', {
  description: 'Total HTTP requests',
});
const errorCounter = meter.createCounter('http_errors_total', {
  description: 'Total HTTP errors (4xx + 5xx)',
});
const requestDuration = meter.createHistogram('http_request_duration_ms', {
  description: 'Request duration in milliseconds',
  boundaries:  [5, 10, 25, 50, 100, 250, 500, 1000, 2500, 5000],
});

// ─── STRUCTURED LOGGING ───────────────────────────────────────────────────
// Always JSON. Always include trace context so logs correlate with traces.
function log(level, message, fields = {}) {
  const span = trace.getActiveSpan();
  const ctx  = span?.spanContext();
  console.log(JSON.stringify({
    timestamp:  new Date().toISOString(),
    level,
    message,
    traceId:    ctx?.traceId,   // Links this log line to a specific trace
    spanId:     ctx?.spanId,
    service:    'checkout-service',
    ...fields,
  }));
}

// ─── OBSERVABILITY MIDDLEWARE ─────────────────────────────────────────────
const app = express();

app.use((req, res, next) => {
  const start = Date.now();

  // Propagate W3C trace context from upstream (API gateway, load balancer)
  const parentContext = propagation.extract(context.active(), req.headers);

  const span = tracer.startSpan(`HTTP ${req.method} ${req.route?.path ?? req.path}`, {
    attributes: {
      'http.method':     req.method,
      'http.url':        req.url,
      'http.user_agent': req.headers['user-agent'],
      'user.id':         req.user?.id,
    },
  }, parentContext);

  context.with(trace.setSpan(parentContext, span), () => {
    res.on('finish', () => {
      const durationMs = Date.now() - start;
      const labels = { method: req.method, route: req.route?.path ?? req.path, status: String(res.statusCode) };

      requestCounter.add(1, labels);
      requestDuration.record(durationMs, labels);
      if (res.statusCode >= 400) errorCounter.add(1, labels);

      span.setAttribute('http.status_code', res.statusCode);
      span.end();

      log('info', 'HTTP request', {
        method:     req.method,
        path:       req.path,
        statusCode: res.statusCode,
        durationMs,
        userId:     req.user?.id,
      });
    });

    next();
  });
});

// ─── TRACED DATABASE CALL ─────────────────────────────────────────────────
async function getOrder(orderId) {
  const span = tracer.startSpan('db.getOrder', {
    attributes: { 'db.system': 'postgresql', 'db.operation': 'SELECT', 'db.statement': 'SELECT * FROM orders WHERE id = ?' },
  });

  try {
    const result = await db.query('SELECT * FROM orders WHERE id = $1', [orderId]);
    span.setAttribute('db.rows_returned', result.rows.length);
    log('debug', 'Database query', { query: 'getOrder', orderId, rows: result.rows.length });
    return result.rows[0];
  } catch (err) {
    span.recordException(err);
    span.setStatus({ code: 2, message: err.message }); // SpanStatusCode.ERROR
    log('error', 'Database error', { query: 'getOrder', orderId, error: err.message });
    throw err;
  } finally {
    span.end();
  }
}

Real World

OpenTelemetry, Prometheus, and SLOs

OpenTelemetry: The vendor-neutral standard for instrumentation. You instrument your code once using the OTel SDK, and the data is exported to whatever backend you prefer — Jaeger, Zipkin, Datadog, Honeycomb, or your own. OTel handles trace context propagation automatically across HTTP calls (via W3C TraceContext headers), so traces span multiple services without extra code in each service.

RED metrics: The simplest useful metric set for any service: Rate (requests per second), Errors (errors per second or error rate %), Duration (latency percentiles — p50, p95, p99). These three tell you whether a service is healthy right now. Anything else is detail you drill into after RED tells you something is wrong.

SLO-based alerting: Alert on SLO breaches, not raw metric values. Instead of "alert if p99 > 2s," use: "alert if error budget is burning faster than sustainable." This means fewer false positives (brief spikes don't page you) and alerts that are directly tied to user experience impact. Google's SRE book introduced this approach and it is now standard in mature organizations.

Logs without structure are expensive noise. Unstructured logs (plain text) require regex to query, can't be aggregated, and make correlating a log line with a trace impossible. Always log as JSON with a consistent field schema: timestamp, level, service, traceId, userId, and the event-specific fields. One structured log line is worth 100 unstructured ones in production debugging.

Pros

  • Reduces mean time to resolution (MTTR) from hours to minutes
  • SLO alerting dramatically reduces alert fatigue (fewer false positives)
  • Trace context links logs, metrics, and spans into one coherent story
  • OpenTelemetry: instrument once, change backend without re-instrumentation

Cons

  • High cardinality metrics (per-user, per-request labels) can OOM Prometheus
  • Distributed tracing adds ~1ms latency overhead per span
  • Structured logs at high volume are expensive to store and query
  • Getting teams to adopt consistent schema takes sustained effort

Key Takeaway

Observability is what lets you answer "why is this broken" rather than just "is this broken." Instrument every service with the RED metrics (Rate, Errors, Duration), structured JSON logs that include trace context, and OpenTelemetry distributed traces that link spans across services. Alert on SLO burn rate — not raw thresholds — to get signal without noise. When an alert fires, traces show you which service in the chain failed, logs show you the exact error and context, and metrics tell you the magnitude. Together, they cut incident MTTR from hours to minutes.
