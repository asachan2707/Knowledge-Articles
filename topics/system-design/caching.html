Caching

Stores frequently accessed data in fast memory so it doesn't need to be recomputed or re-fetched from a slow source. One of the highest-leverage performance improvements in distributed systems — a well-placed cache can reduce database load by 80–95%.

CategoryPerformance

Every database query takes time. Even fast queries — sub-millisecond on a local machine — become 5–20ms over the network. At 10,000 requests/second, that latency compounds into a bottleneck. Caching breaks the loop: compute the answer once, store it in memory, serve it from there for every subsequent request until the data changes.

The fundamental trade-off: Caching trades memory for speed, and consistency for performance. Every cache introduces the possibility of serving stale data. The design question is never "should we cache?" — it's "for how long can we tolerate stale data for this specific piece of information?"

Cache Strategies

The four write strategies — and when each breaks

Cache-Aside (Lazy Loading) — the most common

Application checks cache first. On miss: fetch from DB, populate cache, return. On hit: return from cache. The application owns the cache population logic. Cache only contains data that was actually requested — no pre-loading waste. Problem: the first request always misses (cold start). Can be mitigated with cache warming.

Cache-Aside (Read) App ──▶ Cache ──▶ HIT ──▶ return data │ └──▶ MISS ──▶ DB ──▶ write to cache ──▶ return data Cache-Aside (Write) App ──▶ DB (write) ──▶ invalidate cache key -- next read will repopulate from DB --

Write-Through — consistency first

Every write goes to the cache and the DB synchronously before returning to the caller. Cache is always consistent with DB. Cost: write latency doubles (two writes). Good for read-heavy workloads where consistency is important. Bad for write-heavy workloads.

Write-Behind (Write-Back) — throughput first

Write to cache immediately, persist to DB asynchronously in the background. Write latency drops to near zero. Risk: if the cache node crashes before the async write completes, data is lost. Used by databases internally (write-ahead log buffers) but risky at application level without durable queues.

Read-Through — transparent caching

Cache sits between app and DB. On miss, the cache itself fetches from DB (not the app). The app always reads only from the cache. Simplifies application code but the cache must know how to talk to your DB — not always possible with generic solutions.

import { createClient } from 'redis';

const redis = createClient({ url: process.env.REDIS_URL });
await redis.connect();

// Cache-aside pattern — the standard approach
async function getUserById(userId) {
  const cacheKey = `user:${userId}`;
  const TTL_SECONDS = 300; // 5 minutes

  // 1. Check cache
  const cached = await redis.get(cacheKey);
  if (cached) {
    return JSON.parse(cached); // Cache HIT
  }

  // 2. Cache MISS — fetch from DB
  const user = await db.query('SELECT * FROM users WHERE id = $1', [userId]);
  if (!user) return null;

  // 3. Populate cache with TTL
  await redis.setEx(cacheKey, TTL_SECONDS, JSON.stringify(user));

  return user;
}

// Write-through: update DB and cache atomically
async function updateUser(userId, data) {
  // Write to DB first (source of truth)
  const updated = await db.query(
    'UPDATE users SET name=$2, email=$3 WHERE id=$1 RETURNING *',
    [userId, data.name, data.email]
  );

  // Immediately update cache (write-through)
  const cacheKey = `user:${userId}`;
  await redis.setEx(cacheKey, 300, JSON.stringify(updated.rows[0]));

  return updated.rows[0];
}

// Invalidation on delete
async function deleteUser(userId) {
  await db.query('DELETE FROM users WHERE id = $1', [userId]);
  await redis.del(`user:${userId}`);
}

// Cache stampede prevention — only one worker fetches, others wait
const inFlight = new Map();

async function getUserSafe(userId) {
  const cacheKey = `user:${userId}`;
  const cached = await redis.get(cacheKey);
  if (cached) return JSON.parse(cached);

  // Prevent multiple simultaneous DB fetches for the same key
  if (!inFlight.has(cacheKey)) {
    inFlight.set(cacheKey,
      db.query('SELECT * FROM users WHERE id = $1', [userId])
        .then(async user => {
          if (user) await redis.setEx(cacheKey, 300, JSON.stringify(user));
          inFlight.delete(cacheKey);
          return user;
        })
    );
  }
  return inFlight.get(cacheKey);
}
Eviction Policies

Cache eviction — what gets removed when memory fills

PolicyRemovesBest for
LRU (Least Recently Used)Item not accessed for longestGeneral purpose — most used in production
LFU (Least Frequently Used)Item accessed fewest timesSkewed workloads where some keys are always hot
TTL (Time-To-Live)Expired itemsTime-sensitive data (prices, sessions)
FIFOOldest written itemSimple queues, not general caches
RandomRandom itemUniform access patterns, simple implementation
Redis default is LRU with maxmemory-policy allkeys-lru. In production, always set maxmemory explicitly and choose the right eviction policy. An unconfigured Redis will run out of memory and start refusing writes — usually discovered at 2am during a traffic spike.
Failure Modes

Cache stampede, thundering herd, and cold start

Cache stampede (thundering herd): A popular key expires. At the exact moment it expires, 10,000 requests arrive, all miss the cache simultaneously, and all fire a DB query. The database gets 10,000 simultaneous queries instead of 1. Solution: probabilistic early expiration (refresh a key slightly before it expires), mutex locks, or background refresh.

Cache penetration: Attackers query for keys that don't exist in either the cache or the DB. Every request misses cache and hits the DB. Solution: cache negative results (cache null with a short TTL), or use a Bloom filter to reject requests for non-existent keys before hitting DB.

Cache avalanche: Many keys are set with the same TTL, causing a mass expiration event. The cache empties in waves, flooding the DB. Solution: add jitter to TTLs — instead of exactly 300s, use 300 + random(0, 60).

// TTL jitter — prevents cache avalanche
function setWithJitter(redis, key, value, baseTTL) {
  const jitter = Math.floor(Math.random() * (baseTTL * 0.2)); // ±20% jitter
  return redis.setEx(key, baseTTL + jitter, JSON.stringify(value));
}

// Bloom filter for cache penetration prevention (using Redis SETBIT)
async function mightExist(userId) {
  // In production use a proper Bloom filter library like 'bloom-filters'
  // This is a conceptual illustration
  const bits = hashToBits(userId); // multiple hash functions → bit positions
  const results = await Promise.all(bits.map(bit => redis.getBit('users:bloom', bit)));
  return results.every(b => b === 1); // All bits set = probably exists
}

async function getUserSafe(userId) {
  // Fast rejection: if Bloom filter says definitely not in DB, don't even try
  if (!(await mightExist(userId))) {
    return null; // Definitely doesn't exist — no DB query
  }
  return getUserById(userId); // Might exist — proceed with cache-aside
}

Pros

  • Massive latency reduction (DB: 5ms → Cache: 0.2ms)
  • DB load reduction (80–95% hit rate typical)
  • Throughput multiplier — same DB handles 10× more app servers
  • Resilience — app survives temporary DB outages if cache is warm

Cons

  • Stale data — always a consistency gap
  • Invalidation complexity — especially with relational data
  • Cache stampede risk on popular key expiry
  • Another system to operate, monitor, and scale
  • Memory is expensive at scale
Real World

How production teams use caching

Twitter: Stores timelines in Redis. Every user's home timeline is a pre-computed list of tweet IDs stored in a sorted set. When you open Twitter, your timeline is read from Redis (sub-millisecond), not assembled from DB joins. New tweets from people you follow are fanned out and written to all followers' Redis timelines. This is called "fanout-on-write" or the "push model".

Facebook: Uses Memcached as its L1 cache (within a data center region) and TAO (their custom graph cache) for social graph data. The infamous "look-aside" cache paper from Facebook describes how they handle cache invalidation at billions-of-users scale using a lease system to prevent thundering herd.

Stripe: Idempotency keys are stored in Redis with a TTL. Every payment request carries an idempotency key; Stripe caches the result so duplicate requests return the cached result instead of charging twice.

Key Takeaway

Caching is not one thing — it is a family of strategies, each with different consistency, complexity, and durability trade-offs. Cache-aside with TTL-based expiration and jitter handles 90% of cases. The hard problems are invalidation (when does stale become wrong?), stampede prevention (what happens when a hot key expires?), and penetration (what happens when attackers query missing keys?). Design your eviction policy and failure modes before your cache goes to production.
