Sharding

Horizontal partitioning of a large dataset across multiple database nodes using a shard key. The technique that lets databases scale beyond the physical limits of a single machine.

Concept09 of 13
CategoryData Architecture
Used ByInstagram, Discord, MongoDB, Cassandra, MySQL at scale
Also calledHorizontal partitioning

A single database server has physical limits: a finite amount of RAM, CPU, and disk I/O. When your dataset grows beyond what one machine can handle — whether in storage, query volume, or write throughput — you have two options: vertical scaling (bigger machine, finite ceiling) or horizontal scaling (more machines). Sharding is horizontal scaling for databases.

Sharding vs replication: Replication copies the same data to multiple nodes for fault tolerance and read scaling. Sharding splits different data to different nodes for write scaling and capacity. They are complementary: production systems typically shard first, then replicate each shard for redundancy.

How It Works

Shard keys and routing

Shard 3 (users 2,5,8,11...) Read: get orders for user_id=1001 │ ▼ Shard Router: hash(1001) % 3 = shard 1 │ └──▶ Shard 1 ← query only this shard (no scatter-gather needed) Cross-shard query: get all orders (no shard key) │ └──▶ ALL shards ← scatter: query each shard in parallel aggregate results ← gather: merge sorted results expensive — avoid if possible

Choosing the shard key — the most important decision in sharding

The shard key determines which rows go to which shard. A bad shard key creates hotspots (one shard handles 80% of traffic while others are idle) or makes common queries expensive (every query must hit all shards). A good shard key distributes writes evenly and aligns with your most common access pattern.

Implementation

Shard router in Node.js

import crypto from 'crypto';
import { Pool } from 'pg';

// Shard pool — one connection pool per physical shard
const shardPools = [
  new Pool({ host: process.env.SHARD_0_HOST, database: 'orders', user: 'app', password: process.env.DB_PASSWORD }),
  new Pool({ host: process.env.SHARD_1_HOST, database: 'orders', user: 'app', password: process.env.DB_PASSWORD }),
  new Pool({ host: process.env.SHARD_2_HOST, database: 'orders', user: 'app', password: process.env.DB_PASSWORD }),
];

const NUM_SHARDS = shardPools.length;

// Consistent shard routing — same userId always maps to same shard
function getShardIndex(shardKey) {
  const hash = crypto.createHash('sha256').update(String(shardKey)).digest('hex');
  // Take first 8 hex chars as a number, mod by shard count
  return parseInt(hash.slice(0, 8), 16) % NUM_SHARDS;
}

function getShardPool(shardKey) {
  return shardPools[getShardIndex(shardKey)];
}

// ─── SINGLE-SHARD OPERATIONS (fast) ──────────────────────────────────────

// All orders for a user live on one shard — single shard query
async function getOrdersByUser(userId) {
  const pool = getShardPool(userId); // Always shard 0, 1, or 2 based on userId
  const result = await pool.query(
    'SELECT * FROM orders WHERE user_id = $1 ORDER BY created_at DESC',
    [userId]
  );
  return result.rows;
}

async function createOrder(order) {
  const pool = getShardPool(order.userId);
  const result = await pool.query(
    'INSERT INTO orders (user_id, amount, status) VALUES ($1, $2, $3) RETURNING *',
    [order.userId, order.amount, 'pending']
  );
  return result.rows[0];
}

// ─── CROSS-SHARD QUERY (scatter-gather, use sparingly) ───────────────────

// No shard key available — must query all shards and merge
async function getOrdersByStatus(status, limit = 100) {
  // Scatter: query all shards in parallel
  const shardResults = await Promise.all(
    shardPools.map(pool =>
      pool.query(
        'SELECT * FROM orders WHERE status = $1 ORDER BY created_at DESC LIMIT $2',
        [status, limit]
      ).then(r => r.rows)
    )
  );

  // Gather: merge and sort across shard results
  const allOrders = shardResults.flat();
  allOrders.sort((a, b) => new Date(b.created_at) - new Date(a.created_at));
  return allOrders.slice(0, limit); // Global top N after merge
}

Real World

How Discord shards messages and Instagram shards users

Discord: Shards message storage by channel ID. All messages in a channel live on one shard — a single channel query hits only one shard. Discord built Cassandra sharding on top of this pattern, with the channel ID as the partition key.

Instagram: Used a logical shard approach — instead of a fixed number of physical shards, they created 4096 logical shards mapped to physical nodes. When they add a physical server, they move logical shards to it. This avoids the N→N+1 rehashing problem: moving a logical shard to a new physical node is much less disruptive than rehashing all data.

MongoDB: Built-in sharding via the mongos router process. You pick a shard key per collection; mongos routes queries to the right shard or fans out for shard-key-free queries. MongoDB also supports hashed shard keys for even distribution.

The cross-shard JOIN problem: SQL JOINs across shards require fetching data from multiple shards and merging in the application layer — there is no magic. This is why shard key selection is so critical: your most common JOIN patterns should resolve to the same shard. If User ↔ Order JOINs are common, shard both by userId so user data and their orders are co-located.

Pros

  • Near-linear write scalability — add shards, add capacity
  • Dataset can grow beyond single-machine limits
  • Fault isolation — one shard failure affects only its data slice
  • Queries with shard key hit only one node (fast)

Cons

  • Cross-shard queries are expensive (scatter-gather)
  • No cross-shard ACID transactions
  • Resharding when shard count changes requires data migration
  • Hotspots if shard key has skewed distribution
  • Operational complexity — N databases to backup, monitor, failover

Key Takeaway

Sharding is the mechanism for database horizontal scaling — when one machine's capacity ceiling is not enough. The entire success of a sharding strategy depends on the shard key: it must distribute writes evenly and align with your most common read patterns. Avoid cross-shard queries wherever possible by co-locating related data on the same shard. The operational complexity of sharding is real — try vertical scaling, read replicas, and caching first. Shard when you've exhausted those options, not as a first move.
