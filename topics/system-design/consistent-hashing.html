Consistent Hashing

Distributes keys across nodes using a virtual ring so that when a node joins or leaves, only K/N keys need to be remapped (K = keys, N = nodes) — instead of remapping everything. The algorithm powering Cassandra, DynamoDB, and every distributed cache worth its salt.

Concept11 of 13
CategoryDistributed Systems
Used ByCassandra, DynamoDB, Memcached, Chord DHT
Invented ByKarger et al., MIT (1997)

The problem with naive hashing (hash(key) % N) is that changing N invalidates almost everything. Go from 4 nodes to 5 and hash(key) % 5 maps most keys to different nodes than hash(key) % 4. In a distributed cache, this means a cache miss on almost every key during scaling — a thundering herd against your database. Consistent hashing limits remapping to ~1/N of keys when you add or remove a node.

The intuition: Instead of hashing keys to a slot number (0 to N), hash both keys and nodes to positions on a ring (0 to 2³²-1). Each key is owned by the first node clockwise from its position. When a node is added or removed, only keys in the arc between the new node and its predecessor need to move.

The Algorithm

The hash ring — step by step

Hash ring — 0 to 2³² positions, nodes and keys placed by hash Ring (0 ──────────────────────────────────── 2³²) 0 │ ┌────┘────────────────────────────────────┐ │ Ring │ │ Node A @ 100 key "foo" @ 150 │ │ ↓ │ │ first node clockwise = Node A│ ✗ wrong direction │ │ │ key "foo" @ 150 ──▶ first node CW │ │ nodes on ring: A@100, B@300, C@600 │ │ 150 → next CW node after 150 = B@300 │ │ │ └─────────────────────────────────────────┘ Node removal — only keys in the vacated arc move Before: A@100 B@300 C@600 Keys 100–300 owned by B (keys: "foo"@150, "bar"@280) Remove B: Keys 100–300 now owned by C (next CW node) Only "foo" and "bar" move. All other keys unaffected. In naive hashing: removing B remaps hash(key) % 2 for ALL keys. Virtual nodes — fixes uneven distribution Instead of placing Node A once, place it 150 times around the ring: A-1@50, A-2@450, A-3@820, A-4@1200, ... B-1@200, B-2@600, B-3@950, ... Each node owns many small arcs → load distributes evenly.
Implementation

Consistent hash ring in Node.js

import crypto from 'crypto';

class ConsistentHashRing {
  constructor(virtualNodes = 150) {
    this.virtualNodes = virtualNodes; // replicas per physical node
    this.ring = new Map();            // position → nodeId
    this.sortedPositions = [];        // sorted list of positions for binary search
  }

  // Hash a string to a position on the ring (0 to 2³² - 1)
  _hash(key) {
    const buf = crypto.createHash('md5').update(key).digest();
    return buf.readUInt32BE(0); // First 4 bytes as unsigned 32-bit int
  }

  // Add a physical node — creates virtualNodes positions on the ring
  addNode(nodeId) {
    for (let i = 0; i < this.virtualNodes; i++) {
      const virtualKey = `${nodeId}#${i}`;
      const position   = this._hash(virtualKey);
      this.ring.set(position, nodeId);
    }
    this.sortedPositions = [...this.ring.keys()].sort((a, b) => a - b);
  }

  // Remove a physical node — removes all its virtual positions
  removeNode(nodeId) {
    for (let i = 0; i < this.virtualNodes; i++) {
      const position = this._hash(`${nodeId}#${i}`);
      this.ring.delete(position);
    }
    this.sortedPositions = [...this.ring.keys()].sort((a, b) => a - b);
  }

  // Get the node responsible for a key — binary search for next clockwise position
  getNode(key) {
    if (this.sortedPositions.length === 0) throw new Error('Ring is empty');

    const pos = this._hash(key);

    // Binary search for first position >= pos
    let lo = 0, hi = this.sortedPositions.length - 1;
    while (lo < hi) {
      const mid = (lo + hi) >> 1;
      if (this.sortedPositions[mid] < pos) lo = mid + 1;
      else hi = mid;
    }

    // Wrap around the ring if pos is beyond the last node
    const ringPos = this.sortedPositions[lo] >= pos
      ? this.sortedPositions[lo]
      : this.sortedPositions[0];

    return this.ring.get(ringPos);
  }

  // Get N nodes for a key (for replication) — walks clockwise for N distinct physical nodes
  getNodes(key, n) {
    const nodes = [];
    const seen  = new Set();
    const pos   = this._hash(key);
    const start = this.sortedPositions.findIndex(p => p >= pos) ?? 0;

    for (let i = 0; i < this.sortedPositions.length && nodes.length < n; i++) {
      const idx  = (start + i) % this.sortedPositions.length;
      const node = this.ring.get(this.sortedPositions[idx]);
      if (!seen.has(node)) {
        seen.add(node);
        nodes.push(node);
      }
    }
    return nodes;
  }
}

// Usage — distributed cache with consistent hashing
const ring = new ConsistentHashRing(150);
ring.addNode('cache-1'); // 10.0.0.1
ring.addNode('cache-2'); // 10.0.0.2
ring.addNode('cache-3'); // 10.0.0.3

const cacheClients = {
  'cache-1': createRedisClient('10.0.0.1'),
  'cache-2': createRedisClient('10.0.0.2'),
  'cache-3': createRedisClient('10.0.0.3'),
};

async function cacheGet(key) {
  const node   = ring.getNode(key);
  const client = cacheClients[node];
  return client.get(key);
}

async function cacheSet(key, value, ttl = 300) {
  const node   = ring.getNode(key);
  const client = cacheClients[node];
  return client.setEx(key, ttl, value);
}

// Scale out — add a 4th cache node
// Only ~25% of keys (1/4) will remap to the new node. Others stay put.
ring.addNode('cache-4');
cacheClients['cache-4'] = createRedisClient('10.0.0.4');

// For replication: write to 2 nodes for every key (N=2 replicas)
async function cacheSetReplicated(key, value, ttl = 300) {
  const nodes = ring.getNodes(key, 2); // 2 distinct physical nodes
  await Promise.all(nodes.map(node => cacheClients[node].setEx(key, ttl, value)));
}
Real World

Cassandra, DynamoDB, and Memcached

Cassandra: Uses consistent hashing natively. The partition key of every row is hashed to a token (position on the ring). Each node owns a range of tokens. Cassandra distributes replica copies of each row to the next N nodes clockwise (where N is the replication factor). When you add a node via nodetool decommission, Cassandra automatically transfers only the token ranges that move to the new node — not all data.

Amazon DynamoDB: Uses consistent hashing internally for partition management. Partition keys are hashed to determine which storage node holds the data. AWS manages the ring transparently — you never see it — but it's why DynamoDB scales seamlessly and why partition key design matters for even distribution.

Memcached (client-side): Unlike Redis Cluster (server-side sharding), Memcached is single-node. Client libraries (libmemcached, node-memcached) implement consistent hashing to distribute keys across a pool of Memcached nodes. This is exactly the pattern above — the client decides which node holds each key.

Virtual nodes solve the uneven distribution problem. Without virtual nodes, when a node is removed, all its keys go to its single successor — which may already be the busiest node. With 150 virtual nodes per physical node, the removed node's keys are spread across ~150 different successors, distributing the migration load evenly.

Pros

  • Minimal key remapping when nodes join/leave (1/N keys)
  • No central coordinator — each node knows the ring
  • Natural support for replication (walk clockwise for N nodes)
  • Incremental scaling — add one node at a time

Cons

  • Uneven distribution without virtual nodes (pure CH)
  • Ring state must be propagated to all nodes (gossip protocol overhead)
  • During ring transitions, some keys map to both old and new nodes (dual-read)
  • More complex than simple modulo hashing to implement correctly

Key Takeaway

Consistent hashing solves the scale-without-invalidation problem. Naive modulo hashing remaps almost everything when you add or remove a node. Consistent hashing limits remapping to roughly 1/N of keys per node change — by mapping both keys and nodes to a ring. Virtual nodes (150+ per physical node) fix the uneven load distribution problem. This algorithm is the foundation of Cassandra, DynamoDB, and every distributed cache that needs to scale incrementally.
