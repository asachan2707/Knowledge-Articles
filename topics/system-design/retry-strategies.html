Retry Strategies

How to safely retry failed operations without making things worse. The wrong retry strategy turns a momentary hiccup into a cascading failure. The right one makes your system resilient to transient faults without amplifying load on already-struggling downstream services.

CategoryResilience

Pair WithCircuit Breaker, Timeouts, Idempotency

Networks fail. Databases overload. Services restart. In a distributed system, transient failures are not exceptions — they are the norm. The question is not whether to retry, but how. Retry naively and you thundering-herd a struggling service into full outage. Retry intelligently and your system self-heals without the user ever seeing an error.

The core constraint: You can only safely retry operations that are idempotent. Retrying a non-idempotent operation (like POST /payments without an idempotency key) creates duplicates. Before designing a retry strategy, ensure the operation can be safely replayed — or add an idempotency layer first.

Retry Patterns

Four strategies — from simplest to most robust

Immediate retry — no delay between attempts attempt 1 ──▶ fail ──▶ attempt 2 ──▶ fail ──▶ attempt 3 ──▶ success t=0ms t=0ms t=0ms Good for: transient CPU spikes, lock contention. Bad for: overloaded services — hammers them harder at their worst moment. Fixed interval — wait N ms between each attempt attempt 1 ──▶ fail ──▶ wait 1s ──▶ attempt 2 ──▶ fail ──▶ wait 1s ──▶ attempt 3 ──▶ success Better, but 1000 clients all retrying at t=1s creates a synchronized spike. Exponential backoff — delay doubles each attempt attempt 1 ──▶ fail wait 1s (2⁰) attempt 2 ──▶ fail wait 2s (2¹) attempt 3 ──▶ fail wait 4s (2²) attempt 4 ──▶ success Reduces retry load over time. Still synchronised if clients started together. Exponential backoff + jitter — the production standard delay = Math.random() * Math.min(MAX_DELAY, BASE * Math.pow(2, attempt)) attempt 1: wait 347ms (random fraction of 1s) attempt 2: wait 1,821ms (random fraction of 2s) attempt 3: wait 3,412ms (random fraction of 4s) Desynchronises clients — no synchronized thundering herd.

Implementation

Production-grade retry utility in Node.js

// Full-featured retry utility with exponential backoff + full jitter
async function retry(fn, options = {}) {
  const {
    maxAttempts    = 3,
    baseDelayMs    = 500,
    maxDelayMs     = 30_000,
    deadlineMs     = null,       // Abort all retries after this many ms from first call
    retryIf        = (err) => true, // Predicate: should this error be retried?
    onRetry        = (err, attempt, delayMs) => {}, // Hook for logging/metrics
  } = options;

  const startedAt = Date.now();

  for (let attempt = 0; attempt < maxAttempts; attempt++) {
    try {
      return await fn();
    } catch (err) {
      const isLastAttempt = attempt === maxAttempts - 1;
      const deadlineExceeded = deadlineMs && (Date.now() - startedAt) >= deadlineMs;

      // Don't retry non-retryable errors (e.g. 400 Bad Request, 401 Unauthorized)
      if (!retryIf(err) || isLastAttempt || deadlineExceeded) {
        throw err;
      }

      // Full jitter: uniform random in [0, min(maxDelay, base * 2^attempt)]
      const cap       = Math.min(maxDelayMs, baseDelayMs * Math.pow(2, attempt));
      let   delayMs   = Math.random() * cap;

      // If we have a deadline, cap the delay so we don't overshoot it
      if (deadlineMs) {
        const remaining = deadlineMs - (Date.now() - startedAt);
        delayMs = Math.min(delayMs, remaining - 50); // 50ms safety margin
        if (delayMs <= 0) throw err;
      }

      onRetry(err, attempt + 1, delayMs);
      await new Promise(r => setTimeout(r, delayMs));
    }
  }
}

// Classify which errors are safe to retry
function isRetryableHttpError(err) {
  if (!err.status) return true; // Network error (no status) → always retryable
  // 429 Too Many Requests → retryable (back off and retry)
  // 500/502/503/504 → retryable (server-side transient error)
  // 400/401/403/404 → NOT retryable (client error — retrying won't help)
  return err.status === 429 || err.status >= 500;
}

// If the server sends a Retry-After header, honour it instead of computing backoff
async function fetchWithRetry(url, init = {}) {
  let attempt = 0;
  while (attempt < 5) {
    const resp = await fetch(url, init);
    if (resp.ok) return resp.json();

    if (resp.status === 429) {
      const retryAfterSec = parseInt(resp.headers.get('Retry-After') ?? '1');
      await new Promise(r => setTimeout(r, (retryAfterSec + Math.random()) * 1000));
      attempt++;
      continue;
    }

    if (resp.status >= 500) {
      const delay = Math.min(30_000, 500 * Math.pow(2, attempt)) * Math.random();
      await new Promise(r => setTimeout(r, delay));
      attempt++;
      continue;
    }

    throw new Error(`Non-retryable error: ${resp.status}`);
  }
  throw new Error('Max retries exceeded');
}
Real World

AWS SDK, gRPC, and the thundering herd problem

AWS SDK: Uses exponential backoff with full jitter out of the box. Every AWS SDK call that fails with a retryable error (throttling, service unavailable) automatically retries up to the configured max. The jitter is critical — without it, a fleet of Lambda functions all hitting the same DynamoDB table simultaneously after a brief outage would spike together and re-trigger the outage.

gRPC: Has built-in retry policy configuration in service config JSON. You specify: max attempts, initial backoff, max backoff, backoff multiplier, and which status codes are retryable. Importantly, gRPC retries only activate for hedging-eligible RPCs — the framework prevents retrying non-idempotent streaming calls.

The thundering herd: When 10,000 clients are all connected and a service restarts, they all try to reconnect at the same moment. Without jitter, this overwhelms the restarting service before it can become healthy. With jitter, reconnection attempts spread across a window (e.g. 0–30 seconds), letting the service recover gracefully.

Never retry 4xx errors (except 429). A 400 Bad Request means the client sent something malformed — retrying will get the same 400 every time. A 401 Unauthorized means credentials are wrong — retrying will fail every time and may trigger account lockout. Only retry 429 (rate limited) and 5xx (server-side transient errors).

Pros

  • Self-heals transient failures without user impact
  • Jitter prevents thundering herds during mass recovery
  • Deadline-aware retries prevent infinite wait for users
  • Dramatically reduces error rates for network-heavy services

Cons

  • Can amplify load on an already-overloaded service
  • Non-idempotent operations create duplicates without idempotency keys
  • Hides real bugs if misconfigured to retry non-retryable errors
  • Increases end-to-end latency during partial outages

Key Takeaway

Always use exponential backoff with full jitter for retries in production. The jitter is not optional — it is what prevents the thundering herd. Only retry idempotent operations, and only retry retryable errors (429 and 5xx, not 4xx). Use deadline-aware retries for user-facing requests so a retry storm does not block a user for 30 seconds. Pair retries with a circuit breaker so that when a downstream service is truly down, you stop retrying fast rather than slowly timing out on every attempt.
