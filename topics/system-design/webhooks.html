Webhooks

Reverse APIs: instead of your server polling for changes, the external service calls your server when something happens. Signature verification, reliable delivery, idempotent receivers — the design of a webhook system that doesn't lose events and doesn't process them twice.

Concept25 of 33
CategoryAPI Design
Used ByStripe, GitHub, Shopify, Twilio, PagerDuty
Pair WithEvent-Driven APIs (Concept 21), Idempotency (Concept 13)

Polling asks "did anything change?" every N seconds. Webhooks tell you "something changed" the moment it happens. For payment systems, repository events, and any state change that drives downstream actions, webhooks are the right model — lower latency, lower API cost, simpler logic. The design challenge is on the receiver side: every webhook will be retried on failure, so receivers must be idempotent.

Design

Webhook delivery lifecycle

Sender side: reliable delivery with retry Event occurs (payment.succeeded) │ ▼ Write to outbox table (atomic with business event) │ ▼ Background worker picks up outbox entry │ ├── POST https://receiver.com/webhooks ──▶ 200 OK → mark delivered │ ├── POST https://receiver.com/webhooks ──▶ timeout → retry after 1 min ├── POST https://receiver.com/webhooks ──▶ 503 → retry after 5 min ├── POST https://receiver.com/webhooks ──▶ 500 → retry after 1 hr └── POST https://receiver.com/webhooks ──▶ fail → mark failed, alert Receiver side: idempotent processing POST /webhooks { event_id: "evt_123", type: "payment.succeeded" } │ ├── Verify signature (reject if invalid) ├── Check timestamp (reject if > 5 min old) ├── Check if evt_123 already processed (idempotency check) │ → if yes: return 200 immediately (duplicate, already handled) │ → if no: process event, record evt_123 as processed └── Return 200 quickly, process async
Implementation

Webhook sender and receiver in Node.js

import crypto from 'crypto';
import express from 'express';

// ─── WEBHOOK SENDER ────────────────────────────────────────────────────────
async function sendWebhook(endpoint, eventType, payload, secret) {
  const eventId   = `evt_${crypto.randomBytes(12).toString('hex')}`;
  const timestamp = Math.floor(Date.now() / 1000).toString();
  const body      = JSON.stringify({ id: eventId, type: eventType, data: payload, created: timestamp });

  const signature = crypto
    .createHmac('sha256', secret)
    .update(`${timestamp}.${body}`)
    .digest('hex');

  const response = await fetch(endpoint, {
    method:  'POST',
    headers: {
      'Content-Type':       'application/json',
      'Webhook-ID':          eventId,
      'Webhook-Timestamp':   timestamp,
      'Webhook-Signature':  `v1,${signature}`,
    },
    body,
    signal: AbortSignal.timeout(15_000),
  });

  return response.ok;
}

// ─── WEBHOOK RECEIVER ──────────────────────────────────────────────────────
const app = express();

app.post('/webhooks', express.raw({ type: 'application/json' }), async (req, res) => {
  // 1. Verify signature
  const webhookId   = req.headers['webhook-id'];
  const timestamp   = req.headers['webhook-timestamp'];
  const sigHeader   = req.headers['webhook-signature'];

  if (!webhookId || !timestamp || !sigHeader) {
    return res.status(400).json({ error: 'Missing webhook headers' });
  }

  // Replay prevention: reject if timestamp > 5 minutes old
  if (Math.abs(Date.now() / 1000 - parseInt(timestamp)) > 300) {
    return res.status(400).json({ error: 'Timestamp out of tolerance' });
  }

  const body     = req.body.toString();
  const expected = crypto.createHmac('sha256', process.env.WEBHOOK_SECRET)
    .update(`${timestamp}.${body}`).digest('hex');
  const provided = sigHeader.replace('v1,', '');

  try {
    if (!crypto.timingSafeEqual(Buffer.from(provided, 'hex'), Buffer.from(expected, 'hex'))) {
      return res.status(401).json({ error: 'Invalid signature' });
    }
  } catch {
    return res.status(401).json({ error: 'Invalid signature format' });
  }

  // 2. Parse and validate shape
  const event = JSON.parse(body);

  // 3. Idempotency check — have we already processed this event?
  const exists = await db.query(
    'SELECT id FROM processed_webhook_events WHERE event_id = $1',
    [event.id]
  );
  if (exists.rows.length > 0) {
    return res.json({ received: true, duplicate: true }); // Already handled
  }

  // 4. Respond 200 IMMEDIATELY — do not block on processing
  res.json({ received: true });

  // 5. Process async (after response sent)
  setImmediate(async () => {
    try {
      // Record as processed first (idempotency key)
      await db.query(
        'INSERT INTO processed_webhook_events (event_id, event_type, processed_at) VALUES ($1, $2, NOW()) ON CONFLICT DO NOTHING',
        [event.id, event.type]
      );
      await processWebhookEvent(event);
    } catch (err) {
      // Log error — event was already recorded so won't double-process on next delivery attempt
      console.error({ msg: 'Webhook processing error', eventId: event.id, error: err.message });
    }
  });
});

async function processWebhookEvent(event) {
  switch (event.type) {
    case 'payment.succeeded':
      await fulfillOrder(event.data.orderId);
      break;
    case 'payment.failed':
      await notifyUserPaymentFailed(event.data.userId);
      break;
    default:
      console.log({ msg: 'Unknown webhook type', type: event.type });
  }
}
Real World

Stripe webhook design — the gold standard

Stripe: Retries failed deliveries over 3 days with exponential backoff (immediately, 1 min, 5 min, 30 min, 1 hr, 6 hr, 12 hr, 24 hr × 3). Each event has a stable ID (evt_xxx). Stripe recommends always fetching the canonical object from the API after receiving a webhook rather than trusting the payload data — the payload may be stale if retried after the object changed.

GitHub webhooks: Show full delivery history (request headers, body, response) in the repo settings. You can redeliver any past event manually. This makes debugging webhook receivers far easier than guessing at what was sent.

Design your receiver to handle out-of-order and duplicate events. Webhook retries mean you will receive the same event multiple times. Network delays mean events can arrive out of order — a payment.failed can arrive before payment.created. Use the event ID as an idempotency key, store it, and skip processing if already seen. Use timestamps from the event payload (not arrival time) to order events correctly.

Pros

  • Real-time notification — no polling latency
  • Lower API quota usage vs polling
  • Simple mental model: event happens → you get called

Cons

  • Receiver must be publicly accessible (HTTPS endpoint)
  • Duplicate delivery requires idempotent receivers
  • Order not guaranteed — receivers must handle out-of-order events
  • Receiver downtime = missed events unless sender retries for long enough

Key Takeaway

Webhooks are server-to-server push notifications over HTTP. Always verify the HMAC signature before processing any webhook payload — reject unsigned or replayed requests. Respond 200 immediately and process asynchronously — slow handlers get retried and create duplicates. Record each event ID after processing to prevent duplicate execution on retry. Design your entire handler assuming every event will be delivered at least twice and may arrive out of order. The outbox pattern on the sender side ensures no events are lost even if the sender crashes mid-delivery.
