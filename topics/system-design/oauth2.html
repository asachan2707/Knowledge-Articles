OAuth 2.0

The delegation protocol powering every "Sign in with Google" button. How Authorization Code + PKCE works step by step, why it exists, the difference between access and refresh tokens, and why you should almost never implement OAuth 2.0 from scratch.

CategorySecurity
StandardRFC 6749 (OAuth 2.0), RFC 7636 (PKCE), OpenID Connect
Pair WithAuthentication & Authorization (Concept 23)

OAuth 2.0 solves a specific problem: how can a third-party app access your Google Calendar without knowing your Google password? Before OAuth, you gave your password to the third party — which was insecure and unrevokable. OAuth replaces the password with a scoped, time-limited access token. The third party never sees your password. You can revoke their access without changing your password.

OAuth 2.0 is an authorization protocol, not an authentication protocol. An access token tells you what a client is allowed to do — not who the user is. OpenID Connect (OIDC) is the identity layer built on top of OAuth 2.0 that adds the "who are you" piece via an ID token. Most "Sign in with X" flows are OIDC, not raw OAuth.

The Flow

Authorization Code + PKCE — step by step

Roles Resource Owner = the user (has a Google account) Client = your app (wants to read the user's calendar) Authorization = Google (verifies identity, issues tokens) Server Resource Server = Google Calendar API (holds the data) Authorization Code + PKCE flow (for SPAs and mobile apps) 1. Client generates: code_verifier = random 43–128 char string (e.g. "dBjftJeZ4CVP-mB92K...") code_challenge = BASE64URL(SHA256(code_verifier)) 2. Client redirects user to Authorization Server: GET https://accounts.google.com/o/oauth2/auth? response_type=code &client_id=YOUR_CLIENT_ID &redirect_uri=https://yourapp.com/callback &scope=openid email calendar.readonly &state=random_csrf_token &code_challenge=BASE64URL(SHA256(verifier)) &code_challenge_method=S256 3. User logs in to Google, approves scopes. 4. Google redirects to your callback: https://yourapp.com/callback?code=AUTHZ_CODE&state=random_csrf_token 5. Client exchanges code for tokens: POST https://oauth2.googleapis.com/token { code: "AUTHZ_CODE", code_verifier: "dBjftJeZ4CVP-mB92K...", ← proves we initiated the flow grant_type: "authorization_code" } Response: { access_token: "ya29.xxx", ← use for API calls (15min – 1h) refresh_token: "1//xxx", ← exchange for new access token (days – weeks) id_token: "eyJ...", ← JWT with user identity (OIDC) expires_in: 3600 } 6. Call Calendar API with access token: GET https://www.googleapis.com/calendar/v3/events Authorization: Bearer ya29.xxx PKCE (Proof Key for Code Exchange) prevents authorization code interception attacks on mobile/SPA apps.
Implementation

OAuth 2.0 server-side flow in Node.js

import crypto from 'crypto';
import express from 'express';

const app = express();

// ─── INITIATE OAUTH FLOW ──────────────────────────────────────────────────
app.get('/auth/google', (req, res) => {
  // PKCE: generate verifier + challenge
  const codeVerifier   = crypto.randomBytes(32).toString('base64url');
  const codeChallenge  = crypto.createHash('sha256').update(codeVerifier).digest('base64url');
  const state          = crypto.randomBytes(16).toString('hex'); // CSRF protection

  // Store in session (httpOnly cookie) for later verification
  req.session.oauthState        = state;
  req.session.oauthCodeVerifier = codeVerifier;

  const params = new URLSearchParams({
    response_type:          'code',
    client_id:              process.env.GOOGLE_CLIENT_ID,
    redirect_uri:           process.env.GOOGLE_REDIRECT_URI,
    scope:                  'openid email profile',
    state,
    code_challenge:         codeChallenge,
    code_challenge_method:  'S256',
    access_type:            'offline', // Request refresh token
    prompt:                 'consent', // Force consent screen to always get refresh token
  });

  res.redirect(`https://accounts.google.com/o/oauth2/v2/auth?${params}`);
});

// ─── OAUTH CALLBACK ────────────────────────────────────────────────────────
app.get('/auth/google/callback', async (req, res) => {
  const { code, state, error } = req.query;

  if (error) return res.status(400).json({ error: `OAuth error: ${error}` });

  // CSRF check: state must match what we stored in session
  if (state !== req.session.oauthState) {
    return res.status(400).json({ error: 'State mismatch — possible CSRF attack' });
  }

  // Exchange authorization code for tokens
  const tokenResponse = await fetch('https://oauth2.googleapis.com/token', {
    method: 'POST',
    headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
    body: new URLSearchParams({
      code,
      client_id:     process.env.GOOGLE_CLIENT_ID,
      client_secret: process.env.GOOGLE_CLIENT_SECRET, // Backend only — never expose to frontend
      redirect_uri:  process.env.GOOGLE_REDIRECT_URI,
      grant_type:    'authorization_code',
      code_verifier: req.session.oauthCodeVerifier, // PKCE: prove we started the flow
    }),
  });

  if (!tokenResponse.ok) {
    return res.status(401).json({ error: 'Token exchange failed' });
  }

  const tokens = await tokenResponse.json();
  // tokens.access_token  — use for Google API calls
  // tokens.refresh_token — store securely, use to refresh access token
  // tokens.id_token      — JWT with user identity

  // Verify and decode ID token (OIDC)
  const idTokenPayload = verifyIdToken(tokens.id_token);

  // Find or create user in your database
  let user = await db.query('SELECT * FROM users WHERE google_id = $1', [idTokenPayload.sub]);
  if (!user.rows.length) {
    user = await db.query(
      'INSERT INTO users (google_id, email, name) VALUES ($1, $2, $3) RETURNING *',
      [idTokenPayload.sub, idTokenPayload.email, idTokenPayload.name]
    );
  }

  // Store refresh token encrypted — access token is short-lived and not worth storing
  if (tokens.refresh_token) {
    await db.query(
      'INSERT INTO oauth_credentials (user_id, provider, refresh_token) VALUES ($1, $2, $3) ON CONFLICT (user_id, provider) DO UPDATE SET refresh_token = $3',
      [user.rows[0].id, 'google', encrypt(tokens.refresh_token)]
    );
  }

  // Issue your own session / JWT for the user
  req.session.userId = user.rows[0].id;
  res.redirect('/dashboard');
});

// ─── USING REFRESH TOKEN ──────────────────────────────────────────────────
async function getValidGoogleAccessToken(userId) {
  const { rows } = await db.query(
    'SELECT refresh_token FROM oauth_credentials WHERE user_id = $1 AND provider = $2',
    [userId, 'google']
  );

  const refreshToken = decrypt(rows[0].refresh_token);
  const response = await fetch('https://oauth2.googleapis.com/token', {
    method: 'POST',
    headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
    body: new URLSearchParams({
      refresh_token: refreshToken,
      client_id:     process.env.GOOGLE_CLIENT_ID,
      client_secret: process.env.GOOGLE_CLIENT_SECRET,
      grant_type:    'refresh_token',
    }),
  });

  const { access_token, expires_in } = await response.json();
  return access_token; // Valid for expires_in seconds (typically 3600)
}

Never implement OAuth 2.0 from scratch. The spec has dozens of edge cases, security requirements (PKCE, state, token binding), and subtle vulnerabilities. Use a well-audited library (passport-oauth2, openid-client) or a managed identity provider (Auth0, Clerk, Supabase Auth, AWS Cognito). Implement the parts your provider doesn't handle — user creation, your own session, role assignment — but let the OAuth machinery be someone else's problem.

Key Takeaway

OAuth 2.0 delegates access without sharing passwords — users approve specific scopes, and third-party apps receive scoped, revokable tokens. The Authorization Code + PKCE flow is the standard for all public clients (SPAs, mobile). PKCE prevents authorization code interception; state prevents CSRF. Access tokens are short-lived (1h), refresh tokens are long-lived and must be stored encrypted. Use Client Credentials for machine-to-machine flows. Delegate to an identity provider (Auth0, Clerk, Google) rather than building the OAuth server yourself — the implementation complexity and security surface are not worth building in-house.
