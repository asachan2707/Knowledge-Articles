API Gateway

The single entry point for all client requests. Routes to the right service, enforces authentication, applies rate limits, and handles cross-cutting concerns — so individual services don't have to implement any of it.

Concept06 of 13
CategoryMicroservices Architecture
ToolsKong, AWS API Gateway, Nginx, Traefik, Express Gateway
Used ByNetflix, Amazon, Uber, every microservices org

Without an API gateway, every client — mobile app, web app, third-party partner — needs to know the address of every microservice. Every service needs to implement auth, rate limiting, and logging independently. When services move or are split, every client breaks. An API gateway centralizes all of this: clients call one address, the gateway handles everything else.

What it replaces: If you find yourself copying the same JWT validation middleware into every service, implementing rate limiting in five different languages, or having mobile clients call 8 different services to render one screen — you need an API gateway.

How It Works

Request pipeline — what happens on every request

Every inbound request flows through the pipeline: Client ──▶ API Gateway │ ├─ 1. TLS termination (decrypt HTTPS once, forward HTTP internally) ├─ 2. Authentication (validate JWT / API key) ├─ 3. Authorization (does this token have permission for this route?) ├─ 4. Rate limiting (has this client exceeded their quota?) ├─ 5. Request validation (required headers, payload size) ├─ 6. Routing (which upstream service handles this path?) ├─ 7. Request transformation (add headers, strip fields, protocol translation) │ ▼ Upstream Service (only sees pre-validated, pre-authenticated requests) │ ├─ 8. Response transformation (reshape, strip internal fields) ├─ 9. Response caching (cache and return on future identical requests) └─ 10. Logging / tracing (emit to observability stack) │ ▼ Client
Implementation

Building an API Gateway middleware pipeline in Node.js

import express from 'express';
import httpProxy from 'http-proxy-middleware';
import jwt from 'jsonwebtoken';
import { createClient } from 'redis';

const app = express();
const redis = createClient({ url: process.env.REDIS_URL });
await redis.connect();

// ─── ROUTE TABLE ─────────────────────────────────────────────────────────
// Maps URL prefixes to upstream services — config-driven routing
const routes = [
  { prefix: '/api/orders',   upstream: 'http://order-service:8080',   auth: true  },
  { prefix: '/api/products', upstream: 'http://product-service:8080',  auth: false },
  { prefix: '/api/users',    upstream: 'http://user-service:8080',     auth: true  },
];

// ─── MIDDLEWARE 1: Authentication ─────────────────────────────────────────
function authenticate(req, res, next) {
  const route = routes.find(r => req.path.startsWith(r.prefix));
  if (!route?.auth) return next(); // Public route — skip auth

  const authHeader = req.headers['authorization'];
  if (!authHeader?.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'Missing Authorization header' });
  }

  try {
    const payload = jwt.verify(authHeader.slice(7), process.env.JWT_SECRET);
    req.user = payload; // Attach decoded token for downstream middleware
    // Forward user identity to upstream services via header (not the raw token)
    req.headers['x-user-id']   = payload.sub;
    req.headers['x-user-roles'] = payload.roles?.join(',');
    delete req.headers['authorization']; // Strip token — upstream never sees it
    next();
  } catch (err) {
    return res.status(401).json({ error: 'Invalid or expired token' });
  }
}

// ─── MIDDLEWARE 2: Rate Limiting (sliding window in Redis) ────────────────
function rateLimiter({ maxRequests = 100, windowSeconds = 60 } = {}) {
  return async (req, res, next) => {
    const key = `ratelimit:${req.user?.sub || req.ip}`;
    const now = Date.now();
    const windowStart = now - windowSeconds * 1000;

    // Sliding window log using Redis sorted set
    const multi = redis.multi();
    multi.zRemRangeByScore(key, 0, windowStart);   // Remove old entries
    multi.zAdd(key, [{ score: now, value: now.toString() }]); // Add current
    multi.zCard(key);                               // Count requests in window
    multi.expire(key, windowSeconds);

    const results = await multi.exec();
    const requestCount = results[2];

    res.setHeader('X-RateLimit-Limit', maxRequests);
    res.setHeader('X-RateLimit-Remaining', Math.max(0, maxRequests - requestCount));

    if (requestCount > maxRequests) {
      return res.status(429).json({
        error: 'Rate limit exceeded',
        retryAfter: windowSeconds
      });
    }

    next();
  };
}

// ─── MIDDLEWARE 3: Request logging and tracing ────────────────────────────
function tracing(req, res, next) {
  const requestId = req.headers['x-request-id'] || crypto.randomUUID();
  req.headers['x-request-id'] = requestId;    // Propagate to upstream
  res.setHeader('x-request-id', requestId);   // Echo back to client

  const start = Date.now();
  res.on('finish', () => {
    console.log(JSON.stringify({
      requestId,
      method: req.method,
      path:   req.path,
      status: res.statusCode,
      durationMs: Date.now() - start,
      userId: req.user?.sub,
    }));
  });
  next();
}

// ─── WIRE THE PIPELINE ────────────────────────────────────────────────────
app.use(tracing);
app.use(authenticate);
app.use(rateLimiter({ maxRequests: 100, windowSeconds: 60 }));

// Dynamic routing based on prefix
for (const route of routes) {
  app.use(route.prefix, httpProxy.createProxyMiddleware({
    target:      route.upstream,
    changeOrigin: true,
    on: {
      error: (err, req, res) => {
        console.error(`Proxy error for ${route.upstream}:`, err.message);
        res.status(502).json({ error: 'Service temporarily unavailable' });
      }
    }
  }));
}

app.listen(3000);
Advanced Patterns

Backend For Frontend (BFF) and request aggregation

A standard API gateway routes requests 1:1 to upstreams. A BFF (Backend For Frontend) is a specialized gateway layer that aggregates multiple service calls into one response — tailored to a specific client type (mobile vs web vs partner API).

// BFF pattern: one client request → multiple upstream calls → single response
// Reduces mobile round trips from 5 to 1

app.get('/api/dashboard', authenticate, async (req, res) => {
  const userId = req.user.sub;

  // Fetch from 3 services in parallel — client pays for the slowest one, not all three
  const [profile, orders, recommendations] = await Promise.allSettled([
    fetch(`http://user-service:8080/users/${userId}`).then(r => r.json()),
    fetch(`http://order-service:8080/orders?userId=${userId}&limit=5`).then(r => r.json()),
    fetch(`http://recommendation-service:8080/recs/${userId}`).then(r => r.json()),
  ]);

  res.json({
    profile:         profile.status === 'fulfilled'         ? profile.value         : null,
    recentOrders:    orders.status === 'fulfilled'           ? orders.value          : [],
    recommendations: recommendations.status === 'fulfilled' ? recommendations.value : [],
  });
  // Mobile gets 1 network call instead of 3, with partial results if any service is down
});

Pros

  • Cross-cutting concerns (auth, rate limiting, logging) in one place
  • Services stay simple — no auth code in each service
  • Protocol translation (REST ↔ gRPC, HTTP ↔ WebSocket)
  • Canary deployments — route 5% of traffic to new service version
  • Client-facing API stable even as backend topology changes

Cons

  • Single point of failure — must be HA
  • Added latency — one more network hop
  • Becomes a bottleneck under extreme load
  • Gateway sprawl — don't let it become a monolith of business logic
  • Complex debugging — failures may be gateway or upstream
Don't put business logic in the gateway. The API gateway is for cross-cutting infrastructure concerns: auth, routing, rate limiting, protocol translation. When business rules start appearing in the gateway ("if user is premium, route to premium-service"), you're building a distributed monolith with an identity crisis. Keep logic in services.

Key Takeaway

An API gateway is the enforcer at the border of your system — it handles every concern that would otherwise be duplicated across every service. Authentication, rate limiting, logging, routing, and TLS termination all belong here. Keep business logic out of the gateway — it is infrastructure, not application code. Use the BFF pattern when clients need aggregated responses to reduce round trips, and always run the gateway in an HA configuration with health-check-based routing.
