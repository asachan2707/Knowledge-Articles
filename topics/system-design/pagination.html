Pagination

How APIs return large datasets in manageable chunks. Offset pagination is simple but breaks at scale. Cursor-based pagination is stable and performant. Keyset pagination is the production standard for large, frequently-updated datasets.

Concept28 of 33
CategoryAPI Design

Used ByTwitter timeline, Stripe list endpoints, GitHub API

Every list endpoint that can return more results than fit in a single response needs pagination. The naive approach — OFFSET and LIMIT — works for prototypes but has two serious problems at scale: it gets slower as the offset increases (the database must skip rows), and it produces incorrect results under concurrent writes (inserts and deletes shift rows between pages).

Strategies

Three pagination strategies compared

OFFSET pagination — simple, broken at scale Page 1: SELECT * FROM orders ORDER BY created_at DESC LIMIT 20 OFFSET 0 Page 2: SELECT * FROM orders ORDER BY created_at DESC LIMIT 20 OFFSET 20 Page N: SELECT * FROM orders ORDER BY created_at DESC LIMIT 20 OFFSET (N-1)*20 Problem 1 — PERFORMANCE: OFFSET 100000 means the DB reads 100,020 rows, discards the first 100,000. On a 10M row table: OFFSET 9,999,980 = full table scan. O(offset) cost. Problem 2 — STABILITY: User reads page 1 (rows 1-20). New order inserted at row 5. User reads page 2 (OFFSET 20): old row 20 is now row 21. Row 20 appears on both pages! Concurrent deletes cause the opposite: rows are silently skipped. Cursor (keyset) pagination — stable and O(log n) Page 1: SELECT * FROM orders ORDER BY created_at DESC, id DESC LIMIT 20 Last row: { created_at: '2024-01-15', id: 'ord_500' } Cursor: base64url({ createdAt: '2024-01-15', id: 'ord_500' }) Page 2: SELECT * FROM orders WHERE (created_at, id) < ('2024-01-15', 'ord_500') ORDER BY created_at DESC, id DESC LIMIT 20 Uses composite index on (created_at DESC, id DESC) → O(log n) regardless of page. New rows at the top don't affect pages you've already passed. Deleted rows don't cause skips in subsequent pages.
Implementation

Keyset pagination with stable cursors

import { z } from 'zod';
import { Pool } from 'pg';

const db = new Pool({ connectionString: process.env.DATABASE_URL });

// ─── CURSOR ENCODING ──────────────────────────────────────────────────────
// Opaque token — clients treat as a black box, never parse or construct
function encodeCursor(data) {
  return Buffer.from(JSON.stringify(data)).toString('base64url');
}

function decodeCursor(cursor) {
  try {
    return JSON.parse(Buffer.from(cursor, 'base64url').toString());
  } catch {
    throw new Error('Invalid cursor');
  }
}

// ─── LIST ORDERS — keyset pagination ──────────────────────────────────────
const QuerySchema = z.object({
  cursor: z.string().optional(),
  limit:  z.coerce.number().int().min(1).max(100).default(20),
});

app.get('/orders', async (req, res) => {
  const { cursor, limit } = QuerySchema.parse(req.query);

  let query, params;

  if (cursor) {
    const { createdAt, id } = decodeCursor(cursor);
    // Use the composite index: WHERE (created_at, id) < (lastCreatedAt, lastId)
    query  = `SELECT id, user_id, total, status, created_at
              FROM orders
              WHERE (created_at, id) < ($1::timestamptz, $2)
              ORDER BY created_at DESC, id DESC
              LIMIT $3`;
    params = [createdAt, id, limit + 1]; // +1 to detect if there's a next page
  } else {
    query  = `SELECT id, user_id, total, status, created_at
              FROM orders
              ORDER BY created_at DESC, id DESC
              LIMIT $1`;
    params = [limit + 1];
  }

  const rows = (await db.query(query, params)).rows;
  const hasMore = rows.length > limit;
  const items   = hasMore ? rows.slice(0, limit) : rows;

  const last       = items[items.length - 1];
  const nextCursor = hasMore && last
    ? encodeCursor({ createdAt: last.created_at, id: last.id })
    : null;

  res.json({
    items,
    pagination: {
      nextCursor,
      hasMore,
      count: items.length,
    },
  });
});

// ─── SEEK PAGINATION for random access (e.g. page 5 of 20) ───────────────
// When random page access is needed (not just "next page"), use seek pagination:
// Sort by an indexed column, use the last value as the filter boundary.
// Trade-off: no "page N" concept, only "items after X".

// ─── EXAMPLE: BIDIRECTIONAL PAGINATION (like Twitter timeline) ───────────
app.get('/timeline', async (req, res) => {
  const { beforeId, afterId, limit = 20 } = req.query;

  let query, params;

  if (beforeId) {
    // Load older posts (scroll down)
    query  = 'SELECT * FROM posts WHERE id < $1 ORDER BY id DESC LIMIT $2';
    params = [beforeId, limit];
  } else if (afterId) {
    // Load newer posts (pull to refresh)
    query  = 'SELECT * FROM posts WHERE id > $1 ORDER BY id ASC LIMIT $2';
    params = [afterId, limit];
  } else {
    // Initial load — most recent
    query  = 'SELECT * FROM posts ORDER BY id DESC LIMIT $1';
    params = [limit];
  }

  const rows = (await db.query(query, params)).rows;

  // For afterId, reverse to always return newest-first
  if (afterId) rows.reverse();

  res.json({
    items:   rows,
    oldestId: rows[rows.length - 1]?.id,
    newestId: rows[0]?.id,
  });
});

Real World

Stripe, GitHub, and the opaque cursor convention

Stripe: Every list endpoint (charges, invoices, customers) uses cursor pagination with starting_after and ending_before parameters. The cursor is just the ID of the last item in the previous page — opaque to the client. Stripe explicitly documents that the order of returned items is not guaranteed to be stable under concurrent API calls, so consumers should not depend on ordering across sessions.

GitHub REST API: Uses Link header pagination (RFC 5988) with opaque cursor-based URLs. Response headers contain rel="next" and rel="prev" links. GitHub explicitly tells developers to follow the Link header rather than constructing pagination URLs manually — because the internals of the cursor can change without notice.

Never expose raw OFFSET in your API. Once clients start using OFFSET as a page counter, they'll rely on the ordering being stable. Any insert or delete mid-dataset corrupts their pagination state silently. If you must support "page N" UX (an admin table with 50 pages), use OFFSET internally but cap the maximum offset at a value you're comfortable with (e.g. max 10,000 rows) to prevent full table scans.

Key Takeaway

Always use cursor-based (keyset) pagination for production list endpoints. OFFSET is convenient but produces incorrect results under concurrent writes and performs an O(offset) table scan. Cursor pagination uses a composite index for O(log n) performance at any page depth, and is stable — inserts and deletes don't cause rows to appear twice or be skipped. Encode cursors as opaque base64 tokens. Include hasMore and nextCursor in responses. Require a composite index on the sort columns — without it, keyset pagination degrades to a full table scan too.
