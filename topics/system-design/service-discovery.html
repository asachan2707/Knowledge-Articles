Service Discovery

Automatically tracks available service instances so components can locate and communicate with each other dynamically — without hardcoded IPs that break every time a pod restarts or scales.

Concept08 of 13
CategoryMicroservices Infrastructure
ToolsConsul, Kubernetes DNS, Eureka, etcd
Used ByEvery containerized / Kubernetes system

In a monolith, service calls are function calls — no addressing needed. In microservices, Service A needs to find Service B over the network. In a Kubernetes cluster, a pod's IP changes every time it restarts. You cannot hardcode IPs. Service discovery solves the address book problem: it tracks where every service is at all times, and makes that information available to callers.

Why Kubernetes DNS is service discovery: When you create a Kubernetes Service, kube-dns automatically registers my-service.namespace.svc.cluster.local. Pods call that DNS name; kube-proxy routes to a healthy pod behind it. You never manage IP addresses. This is service discovery built into the platform.

Two Models

Client-side vs server-side discovery

Server-Side Discovery (Kubernetes, AWS ECS) Service A ──▶ Load Balancer / DNS ──▶ Service B instance 1 │ queries registry, Service B instance 2 │ selects instance, Service B instance 3 │ forwards request │ Service A only knows the LB address. Discovery is transparent. Client-Side Discovery (Netflix Eureka + Ribbon) Service A ──▶ Service Registry ──▶ returns list of healthy B instances (Eureka / Consul) Service A ──▶ selects instance (round-robin, least-conn) Service A ──▶ Service B instance 2 (calls directly, no LB hop)
Server-Side (Kubernetes DNS)Client-Side (Eureka + Ribbon)
Who picks the instanceLoad balancer / kube-proxyThe calling service itself
Service awarenessNone — calls a name, LB decidesFull — service knows all instances
Load balancing logicIn the LB/kube-proxyIn the client (configurable per-service)
LatencyOne extra LB hopDirect call, no extra hop
ComplexityPlatform handles itEach service must implement discovery logic
Implementation

Service registry with Consul in Node.js

import Consul from 'consul';

const consul = new Consul({ host: process.env.CONSUL_HOST });

// ─── SERVICE REGISTRATION (on startup) ───────────────────────────────────
async function registerService() {
  const serviceId   = `order-service-${process.env.HOSTNAME}`;
  const servicePort = parseInt(process.env.PORT || '8080');

  await consul.agent.service.register({
    id:   serviceId,
    name: 'order-service',
    port: servicePort,
    address: process.env.POD_IP, // Injected by Kubernetes downward API

    // Health check — Consul polls this endpoint every 10s
    check: {
      http:     `http://${process.env.POD_IP}:${servicePort}/health`,
      interval: '10s',
      timeout:  '3s',
      // Deregister if unhealthy for 1 minute (prevents stale entries)
      deregisterCriticalServiceAfter: '1m'
    },

    // Tags enable filtering: route /v2/* to instances tagged 'v2'
    tags: ['v2', process.env.NODE_ENV],

    // Metadata for client-side routing decisions
    meta: {
      version:   '2.1.0',
      region:    process.env.AWS_REGION,
      startTime: new Date().toISOString()
    }
  });

  console.log(`Registered ${serviceId} with Consul`);

  // Deregister gracefully on shutdown
  process.on('SIGTERM', async () => {
    await consul.agent.service.deregister(serviceId);
    process.exit(0);
  });
}

// ─── SERVICE LOOKUP (in callers) ─────────────────────────────────────────
async function discoverService(serviceName) {
  // Returns only healthy instances (Consul filters out failed health checks)
  const services = await consul.health.service({
    service: serviceName,
    passing: true  // Only healthy instances
  });

  if (!services.length) {
    throw new Error(`No healthy instances of ${serviceName} found`);
  }

  return services.map(s => ({
    id:      s.Service.ID,
    address: s.Service.Address,
    port:    s.Service.Port,
    tags:    s.Service.Tags,
    meta:    s.Service.Meta
  }));
}

// Client-side load balancing over discovered instances
class ServiceClient {
  constructor(serviceName) {
    this.serviceName = serviceName;
    this.instances   = [];
    this.index       = 0;
    this.refreshInterval = 30000;
    this._refresh();
    setInterval(() => this._refresh(), this.refreshInterval);
  }

  async _refresh() {
    try {
      this.instances = await discoverService(this.serviceName);
    } catch (err) {
      console.error(`Failed to refresh ${this.serviceName} instances:`, err.message);
    }
  }

  // Round-robin selection over healthy instances
  next() {
    if (this.instances.length === 0) throw new Error(`No instances for ${this.serviceName}`);
    const instance = this.instances[this.index % this.instances.length];
    this.index = (this.index + 1) % this.instances.length;
    return `http://${instance.address}:${instance.port}`;
  }

  async call(path, options = {}) {
    const baseUrl = this.next();
    const response = await fetch(`${baseUrl}${path}`, options);
    if (!response.ok) throw new Error(`${this.serviceName} responded ${response.status}`);
    return response.json();
  }
}

Real World

How Kubernetes and Netflix handle this

Kubernetes: kube-dns + Services is service discovery baked into the platform. Every Service object creates a stable DNS name. kube-proxy on each node watches the Endpoints object and keeps iptables rules updated as pods come and go. Your services call http://payment-service:8080 and it just works — Kubernetes handles everything behind the name.

Netflix (pre-Kubernetes): Used Eureka (self-hosted, Java-based) for registration and Ribbon (client-side load balancer library) for selection. Every service embedded the Ribbon library. On startup, services registered with Eureka. Callers used Ribbon to query Eureka, get the instance list, and round-robin between healthy ones. This gave them fine-grained control over routing logic per service — useful for traffic shifting during canary deploys.

In 2026, Kubernetes DNS handles service discovery for the vast majority of teams. Consul or etcd are still valuable for multi-cluster, hybrid cloud, or when services run outside Kubernetes. If you're on Kubernetes, use its built-in service discovery — it's already solved.

Pros

  • No hardcoded IPs — services find each other dynamically
  • Automatic failover — unhealthy instances removed from registry
  • Enables blue/green and canary deployments via traffic routing tags
  • Zero-downtime scaling — new instances register, old ones deregister

Cons

  • Registry becomes critical infrastructure — must be HA
  • Health check latency — brief window where dead instance still in registry
  • Client-side discovery adds code complexity to each service
  • DNS caching can route to stale instances (use short TTLs)

Key Takeaway

Service discovery is the address book of your distributed system — it answers "where is Service B right now?" in a world where containers restart, scale, and move constantly. In Kubernetes, it's solved for you via DNS and kube-proxy. Outside Kubernetes, Consul is the standard choice. Client-side discovery gives more routing control at the cost of complexity in each service; server-side discovery is transparent but adds an LB hop. The health check is the most important piece — it ensures the registry only contains live, ready instances.
