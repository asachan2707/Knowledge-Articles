Consistency Models

What guarantees a distributed database makes about what you read after you write. Strong, eventual, causal, and read-your-writes consistency — each is a different contract between the database and its users, and each trades availability or latency for correctness.

Concept22 of 33
CategoryDistributed Systems
RelatedCAP Theorem (Concept 31), Sharding (Concept 09)
Key InsightConsistency is a spectrum, not a binary

In a single-node database, consistency is trivial — after you write, every subsequent read returns the new value. In a distributed database with replicas on multiple nodes, it is not. A write goes to the primary; replicas are updated asynchronously. If you read from a replica before the update arrives, you see old data. Consistency models define the rules: what are you guaranteed to see after a write, under what conditions?

Why it matters: A shopping cart that shows "1 item" immediately after adding something, but "0 items" when you navigate to checkout — that is eventual consistency biting a user. An account that lets you withdraw the same money twice because two nodes each saw a positive balance — that is a consistency model mismatch with a financial workload.

The Spectrum

Consistency models from strongest to weakest

Linearisability (strongest) — all operations appear instantaneous and in order Write x=2 completes at t=5ms All reads after t=5ms return x=2, regardless of which replica serves them. Feels like a single machine. Requires coordination — high latency. Example: single-leader PostgreSQL with synchronous replication, etcd, ZooKeeper. Read-your-writes — you always see your own writes User writes profile picture at t=10ms Next request from SAME USER always returns new picture. Other users MAY still see old picture (replica lag). Good enough for most user-facing apps. Achieved by routing reads to primary OR by attaching a session-scoped "read-after" timestamp to every request. Causal consistency — causally related operations are seen in order User A posts a comment (t=1) User B replies to it (t=2) — causally depends on A's comment Any replica serving reads sees: A's comment appears BEFORE B's reply. Unrelated operations (different posts) can arrive in any order. Used in: collaborative editing, social feed ordering. Eventual consistency (weakest guaranteeing convergence) Write x=2 on node 1. Node 2 still serves x=1 for up to N seconds of replication lag. Eventually (after lag), all nodes converge to x=2. Intermediate state: "split-brain" reads possible. Highest availability and lowest latency. Example: Cassandra, DynamoDB default.

Implementation

Read-your-writes in Node.js with PostgreSQL read replicas

import { Pool } from 'pg';

const primaryPool = new Pool({ host: process.env.DB_PRIMARY_HOST });
const replicaPool = new Pool({ host: process.env.DB_REPLICA_HOST });

// ─── READ-YOUR-WRITES via session tracking ────────────────────────────────
// After a write, tag the session with a LSN (Log Sequence Number).
// When reading, check if the replica has caught up to that LSN.
// If not, fall back to primary for this request.

async function writeWithLSN(query, params, session) {
  const result = await primaryPool.query(query, params);

  // PostgreSQL exposes current WAL position after a write
  const { rows } = await primaryPool.query('SELECT pg_current_wal_lsn()::text AS lsn');
  session.lastWriteLSN = rows[0].lsn; // Store in session (cookie, Redis, JWT claim)

  return result;
}

async function readWithConsistency(query, params, session) {
  if (!session.lastWriteLSN) {
    // No recent writes from this session — replica is safe
    return replicaPool.query(query, params);
  }

  // Check if replica has replayed up to our write position
  const { rows } = await replicaPool.query(
    `SELECT pg_last_wal_replay_lsn() >= $1::pg_lsn AS caught_up`,
    [session.lastWriteLSN]
  );

  if (rows[0].caught_up) {
    return replicaPool.query(query, params); // Replica is caught up — use it
  }

  // Replica is behind — read from primary to guarantee read-your-writes
  return primaryPool.query(query, params);
}

// ─── CASSANDRA: tunable consistency ──────────────────────────────────────
// Cassandra lets you specify consistency PER OPERATION.
// QUORUM = majority of replicas must respond (strong-ish)
// ONE = any single replica suffices (eventual, fastest)
// ALL = all replicas must respond (strongest, slowest)

// Node.js cassandra-driver example:
const { Client, types } = require('cassandra-driver');
const cassandra = new Client({ contactPoints: ['cassandra-1', 'cassandra-2'] });

async function writeOrder(order) {
  // Write with QUORUM consistency — majority of replicas acknowledge
  await cassandra.execute(
    'INSERT INTO orders (id, user_id, total) VALUES (?, ?, ?)',
    [order.id, order.userId, order.total],
    { consistency: types.consistencies.quorum }
  );
}

async function readOrderStrong(orderId) {
  // Read with QUORUM — majority must agree → read-your-writes guaranteed
  const result = await cassandra.execute(
    'SELECT * FROM orders WHERE id = ?',
    [orderId],
    { consistency: types.consistencies.quorum }
  );
  return result.rows[0];
}

async function readOrderEventual(orderId) {
  // Read with ONE — any replica responds → lowest latency, may be stale
  const result = await cassandra.execute(
    'SELECT * FROM orders WHERE id = ?',
    [orderId],
    { consistency: types.consistencies.one }
  );
  return result.rows[0];
}

// ─── CONFLICT RESOLUTION for eventual consistency ─────────────────────────
// When two nodes both accept writes to the same record (split brain),
// they diverge. Merging is application-specific.

// Cassandra: Last-Write-Wins (timestamp-based) — default, simplest
// CRDTs (Conflict-free Replicated Data Types): counters, sets, maps
//   that merge deterministically without coordination

// Example CRDT counter (add-only, no subtraction):
class GCounter {
  constructor(nodeId) {
    this.nodeId   = nodeId;
    this.counters = { [nodeId]: 0 };
  }

  increment() { this.counters[this.nodeId]++; }

  value() { return Object.values(this.counters).reduce((sum, v) => sum + v, 0); }

  merge(other) {
    // Take max of each node's counter — convergent under any ordering
    for (const [node, count] of Object.entries(other.counters)) {
      this.counters[node] = Math.max(this.counters[node] ?? 0, count);
    }
  }
}
Real World

How Amazon, Cassandra, and Google Spanner choose consistency

Amazon shopping cart: Uses eventual consistency intentionally. Items added to a cart by two browser tabs might briefly show different counts — one tab's writes haven't replicated yet. Amazon's engineers determined that the business cost of an incorrect cart count for a few milliseconds is lower than the latency cost of strong consistency for every cart operation, globally. The cart is eventually correct.

Google Spanner: Achieves external consistency (effectively linearisability) across a globally distributed database by using TrueTime — atomic clocks and GPS receivers in every Google data centre. Spanner knows the exact time a write committed and can determine before-after relationships across regions. This is unique to Google's infrastructure; most teams cannot replicate it.

Cassandra tunable consistency: The standard pattern is QUORUM reads + QUORUM writes (R + W > N where N=replication factor). With N=3 replicas, QUORUM=2. A write confirmed by 2 replicas + a read from 2 replicas guarantees at least 1 overlapping node — you always read your own writes. Trade stronger consistency for higher latency by increasing the quorum level.

Read-your-writes is the minimum most applications actually need. True linearisability is expensive and usually unnecessary. For social apps, e-commerce, and most user-facing systems, the important guarantee is: "after I submit my form, when I navigate to the next page, I see my submission." Routing reads to primary immediately after writes (or using session LSN tracking) achieves this with far lower cost than full linearisability.

Key Takeaway

Consistency is not binary — it is a spectrum from linearisability (every read sees the latest write, globally) to eventual consistency (all replicas converge eventually, but may diverge temporarily). Most applications need read-your-writes consistency, not full linearisability. Achieve it cheaply by routing reads to primary immediately after writes, or by tracking WAL position in the session. For globally distributed systems, Cassandra's tunable consistency (QUORUM reads + QUORUM writes) is the practical sweet spot. Choose the consistency level that matches the business requirement — the shopping cart can be eventually consistent; the bank account cannot.
