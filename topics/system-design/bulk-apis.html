Bulk APIs

Endpoints that accept and process multiple resources in a single round trip. The difference between 1000 HTTP calls and 1, partial failure semantics, cursor-based pagination for large result sets, and the database patterns that make bulk operations fast.

Concept20 of 33
CategoryAPI Design
PatternBatch write, multi-get, cursor pagination
Used ByStripe batch, SendGrid bulk, AWS batch operations

When a client needs to create 500 records, making 500 individual HTTP calls is not just slow — it is architecture-by-accident. Each call has connection overhead, TLS handshake amortisation, server-side processing overhead, and a round-trip time. Bulk APIs replace 500 calls with 1, reducing latency from minutes to seconds. The design challenge is handling partial failure: what happens when 498 of 500 succeed?

The N+1 problem in reverse: The classic N+1 query problem (1 query to get IDs, N queries to fetch each one) applies equally to APIs. A frontend loading 100 user profiles with 100 GET /users/:id calls has the same problem. Bulk APIs are the solution: one call, one response, all 100 users.

Patterns

Bulk write, multi-get, and cursor pagination

Without bulk: 500 individual creates t=0ms POST /products { item 1 } ──▶ 201 t=50ms POST /products { item 2 } ──▶ 201 ... t=25000ms POST /products { item 500 } ──▶ 201 Total: 25 seconds, 500 network round trips With bulk: 1 call t=0ms POST /products/bulk { items: [ item1...item500 ] } ──▶ 207 Multi-Status t=800ms Response: { "succeeded": 498, "failed": 2, "results": [ { "id": "prod_1", "status": 201 }, ... { "index": 302, "status": 400, "error": "Duplicate SKU" }, { "index": 487, "status": 422, "error": "Price must be positive" } ] } Total: 800ms, 1 network round trip Cursor pagination — stable across concurrent writes GET /products?limit=100 → { items: [...100 items], nextCursor: "prod_1234" } ← opaque cursor GET /products?limit=100&cursor=prod_1234 → { items: [...next 100], nextCursor: "prod_1334" } vs OFFSET pagination: OFFSET 100 on a 1M row table requires skipping 100 rows on every page — slow AND unstable (new inserts shift results).
Implementation

Bulk create with partial failure and cursor pagination in Node.js

import { z } from 'zod';
import { Pool } from 'pg';

const db = new Pool({ connectionString: process.env.DATABASE_URL });

// ─── BULK CREATE — partial failure response ───────────────────────────────
const BulkCreateProductSchema = z.object({
  items: z.array(z.object({
    sku:          z.string().min(1).max(50),
    name:         z.string().min(1).max(200),
    priceInCents: z.number().int().positive(),
    category:     z.string(),
  })).min(1).max(500), // Max 500 per request — prevents abuse
});

app.post('/products/bulk', async (req, res) => {
  const result = BulkCreateProductSchema.safeParse(req.body);
  if (!result.success) return res.status(400).json({ error: 'Validation failed', errors: result.error.issues });

  const { items } = result.data;
  const results   = [];
  let succeeded   = 0;
  let failed      = 0;

  // Option A: All-or-nothing (transactional bulk insert)
  // Use when partial success is worse than full failure (e.g. financial records)
  // client.query('BEGIN'); ... INSERT all; client.query('COMMIT');

  // Option B: Best-effort (process each, collect results)
  // Use when partial success is acceptable (e.g. product imports)
  for (let i = 0; i < items.length; i++) {
    const item = items[i];
    try {
      const row = await db.query(
        `INSERT INTO products (sku, name, price_cents, category)
         VALUES ($1, $2, $3, $4)
         ON CONFLICT (sku) DO NOTHING
         RETURNING id`,
        [item.sku, item.name, item.priceInCents, item.category]
      );

      if (row.rows.length === 0) {
        // ON CONFLICT DO NOTHING — duplicate SKU
        failed++;
        results.push({ index: i, status: 409, error: `SKU '${item.sku}' already exists` });
      } else {
        succeeded++;
        results.push({ index: i, status: 201, id: row.rows[0].id });
      }
    } catch (err) {
      failed++;
      results.push({ index: i, status: 422, error: err.message });
    }
  }

  // Option C: True batch INSERT — fastest but all-or-nothing
  // INSERT INTO products (sku, name) VALUES ($1,$2),($3,$4),...($N-1,$N)
  // Use for massive imports (100k+ rows) where individual error handling isn't needed

  const statusCode = failed === 0 ? 201 : succeeded === 0 ? 400 : 207; // 207 = Multi-Status
  res.status(statusCode).json({ succeeded, failed, total: items.length, results });
});

// ─── TRUE BATCH INSERT (high throughput) ─────────────────────────────────
async function batchInsertProducts(items) {
  // Build parameterised VALUES clause: ($1,$2,$3), ($4,$5,$6), ...
  const values  = [];
  const params  = [];
  items.forEach((item, i) => {
    const base = i * 3;
    values.push(`($${base + 1}, $${base + 2}, $${base + 3})`);
    params.push(item.sku, item.name, item.priceInCents);
  });

  await db.query(
    `INSERT INTO products (sku, name, price_cents) VALUES ${values.join(',')} ON CONFLICT (sku) DO NOTHING`,
    params
  );
}

// ─── CURSOR PAGINATION ────────────────────────────────────────────────────
// Cursor = opaque token encoding the last-seen row position
// Stable under concurrent inserts/deletes; O(1) database work per page

const PaginationSchema = z.object({
  cursor: z.string().optional(),
  limit:  z.coerce.number().int().min(1).max(100).default(20),
});

app.get('/products', async (req, res) => {
  const { cursor, limit } = PaginationSchema.parse(req.query);

  let decodedCursor = null;
  if (cursor) {
    try {
      decodedCursor = JSON.parse(Buffer.from(cursor, 'base64url').toString());
    } catch {
      return res.status(400).json({ error: 'Invalid cursor' });
    }
  }

  // Keyset pagination: WHERE (created_at, id) < (lastSeenCreatedAt, lastSeenId)
  // Uses a composite index on (created_at DESC, id DESC) — O(log n)
  const query = decodedCursor
    ? `SELECT id, sku, name, price_cents, created_at
       FROM products
       WHERE (created_at, id) < ($1, $2)
       ORDER BY created_at DESC, id DESC
       LIMIT $3`
    : `SELECT id, sku, name, price_cents, created_at
       FROM products
       ORDER BY created_at DESC, id DESC
       LIMIT $1`;

  const params = decodedCursor
    ? [decodedCursor.createdAt, decodedCursor.id, limit + 1]
    : [limit + 1];

  const rows    = (await db.query(query, params)).rows;
  const hasMore = rows.length > limit;
  const items   = hasMore ? rows.slice(0, limit) : rows;

  const lastItem    = items[items.length - 1];
  const nextCursor  = hasMore && lastItem
    ? Buffer.from(JSON.stringify({ createdAt: lastItem.created_at, id: lastItem.id })).toString('base64url')
    : null;

  res.json({ items, nextCursor, hasMore });
});

// ─── MULTI-GET: fetch many resources in one call ──────────────────────────
app.post('/products/multi-get', async (req, res) => {
  const { ids } = z.object({ ids: z.array(z.string().uuid()).max(100) }).parse(req.body);

  const result = await db.query(
    'SELECT * FROM products WHERE id = ANY($1)',
    [ids]
  );

  // Return results keyed by ID — client can look up by ID easily
  const byId = Object.fromEntries(result.rows.map(r => [r.id, r]));
  res.json({ products: byId, found: result.rows.length, requested: ids.length });
});
Real World

Stripe batch, SendGrid, and the cursor pagination standard

Stripe: Every list endpoint uses cursor-based pagination. GET /charges returns a has_more flag and a data[last].id that becomes the starting_after parameter for the next page. This works correctly even as new charges are created during pagination — OFFSET-based pagination would miss or duplicate charges created between page fetches.

Salesforce Bulk API: Designed for loading millions of records. Clients create a "job," upload CSV batches, and poll for completion. Salesforce processes up to 100 million records per 24-hour period. The key design: asynchronous processing with a status polling mechanism, not a synchronous response.

AWS batch operations: DynamoDB BatchGetItem fetches up to 100 items in one call. BatchWriteItem writes up to 25 items. Responses include an UnprocessedItems field with any items that failed — the client is responsible for retrying only the failed items. This partial-success-with-retry pattern is the standard for high-throughput bulk APIs.

Encode cursors as opaque tokens. Never expose ?page=3&offset=60 as your pagination mechanism — clients treat integers as arithmetic and add 1, or jump to arbitrary pages, creating unpredictable load. Encode the cursor as a base64 blob. Clients treat it as opaque, pass it back verbatim, and can't manipulate it. You can change the cursor internals without changing the API contract.

Pros

  • Orders-of-magnitude latency reduction (N calls → 1)
  • Reduces connection pool pressure on the database
  • Cursor pagination stable under concurrent writes — no missed/duplicate results
  • Partial failure responses let clients retry only failed items

Cons

  • Larger request payloads can hit body size limits
  • Partial failure responses require more client complexity
  • Batch size limits require client-side batching logic
  • Long-running bulk operations may need async processing + polling pattern

Key Takeaway

Bulk APIs replace N network round trips with 1. Design bulk endpoints with explicit size limits (max 500 per batch), partial failure responses (HTTP 207 Multi-Status with per-item status), and idempotency for safe retries. Use cursor-based (keyset) pagination instead of OFFSET — it is O(log n) at the database, stable under concurrent writes, and works correctly at scale. For list endpoints, always return an opaque nextCursor token rather than exposing raw OFFSET or page numbers to clients.
