Database Selection

Relational, document, wide-column, graph, time-series, and search engines — each is optimised for a different data shape and access pattern. The decision framework architects use to choose the right database for each job, and why most production systems use three or more database types simultaneously.

Concept33 of 33
CategoryData Architecture
Key PrincipleMatch data shape and access pattern to the database's data model
Anti-patternUsing one database type for everything

No single database is best for everything. PostgreSQL is excellent for transactional data with complex relationships. Redis is excellent for sub-millisecond lookups by key. Elasticsearch is excellent for full-text search across millions of documents. Cassandra is excellent for write-heavy time-series data at massive scale. Using PostgreSQL for everything is like using a hammer for every job — it works until you need a screwdriver, and then it really doesn't.

The decision framework: What is the shape of your data? What are your access patterns? How much data do you have and how fast is it growing? What are your consistency requirements? What are your latency SLAs? The answers determine the database type before you consider specific products.

Decision Framework

Choosing the right database — the architect's questions

Step 1: What is the primary access pattern? Lookup by primary key (userId, orderId) → key-value or document Complex queries with JOINs → relational Full-text search ("find all posts about cats") → search engine Traverse relationships (friend-of-friend) → graph Write millions of time-stamped metrics → time-series or wide-column Semantic similarity search (RAG, embedding) → vector DB Step 2: What are the write characteristics? Low write volume, complex read queries → PostgreSQL High write volume, simple read by key → Cassandra, DynamoDB Write-heavy + time-ordered (append only) → TimescaleDB, InfluxDB, Cassandra Burst writes, idempotent operations → DynamoDB (on-demand) Step 3: What are the consistency requirements? ACID transactions required → PostgreSQL, MySQL Eventual consistency acceptable → Cassandra, DynamoDB (default) Strong consistency globally → Spanner, CockroachDB Step 4: What is the schema evolution story? Schema changes frequently (startup, agile) → Document (MongoDB) or PostgreSQL (ALTER TABLE) Schema is stable and well-defined → Relational Schema-less, arbitrary fields → Document, key-value Common production combinations (polyglot persistence) PostgreSQL ← Source of truth: users, orders, transactions Redis ← Sessions, caching, rate limits, pub/sub Elasticsearch ← Full-text search, log aggregation S3 ← Files, images, large objects (Optional) Cassandra ← Activity feeds, time-series events (Optional) pgvector ← Semantic search if already on PostgreSQL
Implementation

Polyglot persistence — using the right store for each job

import { Pool } from 'pg';
import { createClient } from 'redis';
import { Client as ESClient } from '@elastic/elasticsearch';

const pg    = new Pool({ connectionString: process.env.DATABASE_URL });
const redis = createClient({ url: process.env.REDIS_URL });
const es    = new ESClient({ node: process.env.ELASTICSEARCH_URL });

await redis.connect();

// ─── PRODUCT CATALOG: PostgreSQL as source of truth ───────────────────────
async function createProduct(product) {
  const { rows } = await pg.query(
    'INSERT INTO products (sku, name, price_cents, description, category) VALUES ($1,$2,$3,$4,$5) RETURNING *',
    [product.sku, product.name, product.priceInCents, product.description, product.category]
  );

  // Write-through: sync to Elasticsearch for search
  await es.index({
    index: 'products',
    id:    rows[0].id,
    body:  { name: rows[0].name, description: rows[0].description, category: rows[0].category, price: rows[0].price_cents },
  });

  // Invalidate Redis cache for this product
  await redis.del(`product:${rows[0].id}`);

  return rows[0];
}

// ─── PRODUCT READ: Redis cache → PostgreSQL fallback ─────────────────────
async function getProduct(productId) {
  const cached = await redis.get(`product:${productId}`);
  if (cached) return JSON.parse(cached);

  const { rows } = await pg.query('SELECT * FROM products WHERE id = $1', [productId]);
  if (!rows.length) return null;

  await redis.setEx(`product:${productId}`, 300, JSON.stringify(rows[0]));
  return rows[0];
}

// ─── FULL-TEXT SEARCH: Elasticsearch ─────────────────────────────────────
async function searchProducts(query, filters = {}) {
  const must = [
    { multi_match: { query, fields: ['name^3', 'description', 'category'], fuzziness: 'AUTO' } },
  ];

  if (filters.category) must.push({ term: { category: filters.category } });
  if (filters.maxPrice) must.push({ range: { price: { lte: filters.maxPrice } } });

  const result = await es.search({
    index: 'products',
    body:  { query: { bool: { must } }, size: 20 },
  });

  return result.hits.hits.map(hit => ({ id: hit._id, score: hit._score, ...hit._source }));
}

// ─── USER ACTIVITY FEED: Cassandra or Redis Sorted Set ────────────────────
// Store activity events by timestamp — append-only, read by time range

// Redis sorted set approach (simpler for moderate scale)
async function addActivityEvent(userId, event) {
  const score  = Date.now();
  const member = JSON.stringify(event);
  await redis.zAdd(`activity:${userId}`, { score, value: member });
  await redis.zRemRangeByRank(`activity:${userId}`, 0, -1001); // Keep latest 1000
}

async function getActivityFeed(userId, limit = 20) {
  const members = await redis.zRevRange(`activity:${userId}`, 0, limit - 1, { withScores: true });
  return members.map(m => ({ ...JSON.parse(m.value), timestamp: m.score }));
}

// ─── SESSION STORAGE: Redis ───────────────────────────────────────────────
async function createSession(userId, metadata) {
  const sessionId = crypto.randomBytes(32).toString('hex');
  await redis.setEx(`session:${sessionId}`, 86400, JSON.stringify({ userId, ...metadata }));
  return sessionId;
}

async function getSession(sessionId) {
  const data = await redis.get(`session:${sessionId}`);
  return data ? JSON.parse(data) : null;
}
Real World

Instagram, Uber, and Discord's polyglot architecture

Instagram: PostgreSQL for core social graph (users, follows, posts). Cassandra for activity feeds and timelines (time-ordered, high write volume). Redis for session management and frequently-accessed counters. Elasticsearch for hashtag search. S3 for photos. Each store is chosen for its specific job — no single store handles all of Instagram's data.

Uber: MySQL/PostgreSQL for financial and transactional data. Cassandra for trip history and driver location history (append-heavy, geospatially keyed). Redis for real-time driver locations (ephemeral, sub-millisecond reads). Elasticsearch for log aggregation and driver search. Schemaless (Uber's in-house document DB on MySQL) for flexible service data.

Discord: Famous for migrating message storage from MongoDB to Cassandra (scaling issue) and later partially to ScyllaDB (Cassandra-compatible, written in C++ for lower latency). The key insight: messages are append-only, queried by channel ID + time range — a perfect Cassandra access pattern. The message schema is stable — no need for document flexibility.

Start with PostgreSQL. For the vast majority of applications, PostgreSQL handles everything — relations, JSON documents, full-text search (tsvector), geospatial (PostGIS), and with pgvector, even embeddings. Add specialised stores when you have a measured problem that PostgreSQL can't solve at your scale. The operational cost of managing multiple stores is real — don't pay it before you need to.
The polyglot consistency problem. When data lives in multiple stores (PostgreSQL + Elasticsearch + Redis), keeping them in sync is your problem. A write to PostgreSQL might fail before syncing to Elasticsearch — leaving stale search results. Use the outbox pattern: write to PostgreSQL first, sync to secondary stores asynchronously via CDC (change data capture with Debezium) or a background sync worker. PostgreSQL is always the source of truth.

Key Takeaway

Match the database to the data model and access pattern, not to familiarity or hype. Relational (PostgreSQL) for structured data with transactions and complex queries. Redis for sub-millisecond key lookups, sessions, and caches. Elasticsearch for full-text search. Cassandra for write-heavy time-series data at global scale. Vector DB for semantic search over embeddings. Most production systems are polyglot — PostgreSQL as source of truth, Redis for hot-path reads, Elasticsearch for search. Start with PostgreSQL and add specialised stores only when you have measured problems at scale — the operational cost of multiple stores is real and should not be paid prematurely.
