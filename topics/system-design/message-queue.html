Message Queue

Decouples producers from consumers by putting a durable buffer between them. Producers enqueue messages at their own pace; consumers process them asynchronously at theirs. The mechanism behind every reliable async workflow.

Concept04 of 13
CategoryAsync Architecture
ToolsRabbitMQ, AWS SQS, Kafka, BullMQ
Used ByStripe, Uber, Slack, every e-commerce platform

Without a message queue, Service A calls Service B directly. If B is slow, A waits. If B is down, A fails. If B is overwhelmed, A gets errors. A message queue breaks every one of these coupling points: A drops a message into the queue and immediately returns. B processes it whenever it can, at its own pace, and can be restarted without losing work.

The core property: Durability + decoupling. The queue persists messages to disk — if the consumer crashes, messages survive and are reprocessed. The producer never needs to know if the consumer is alive, slow, or being deployed. This is what makes queues so valuable for reliability engineering.

How It Works

Message lifecycle and delivery semantics

Message Queue — point-to-point, competing consumers Producer A ──▶ │ │ ──▶ Consumer 1 (processes msg 1) Producer B ──▶ │ Queue │ ──▶ Consumer 2 (processes msg 2) Producer C ──▶ │ [1][2][3][4] │ ──▶ Consumer 3 (processes msg 3) │ │ Each message delivered to EXACTLY ONE consumer (competing consumers pattern) Visibility timeout — in-flight message protection Consumer 1 picks msg ──▶ msg becomes invisible (30s visibility timeout) Consumer 1 processes ──▶ ACK ──▶ msg deleted from queue Consumer 1 crashes ──▶ timeout expires ──▶ msg becomes visible again ──▶ Consumer 2 picks it up ──▶ reprocessed safely

Delivery guarantees

GuaranteeMeaningExample systems
At-most-onceMessage may be lost, never duplicated. Fire and forget.UDP, SNS without SQS
At-least-onceMessage will definitely be delivered, but may be duplicated on failure/retry. Consumer must be idempotent.SQS, RabbitMQ, Kafka
Exactly-onceDelivered once and only once. Hard to implement, expensive. Usually achieved via idempotency at the consumer, not the queue.Kafka transactions, SQS FIFO + deduplication
Exactly-once is a lie at the infrastructure level. Every real system is at-least-once, and you make it effectively exactly-once by building idempotent consumers. Design your consumers to handle duplicate messages safely — check if work was already done before doing it.
Implementation

Producer and consumer in Node.js

// Using BullMQ — Redis-backed job queue for Node.js
import { Queue, Worker } from 'bullmq';
import { createClient } from 'redis';

const connection = { host: process.env.REDIS_HOST, port: 6379 };

// ─── PRODUCER ─────────────────────────────────────────────────────────────
const emailQueue = new Queue('email-notifications', { connection });

// Enqueue a job — producer returns immediately, does not wait for processing
async function sendWelcomeEmail(userId, email) {
  await emailQueue.add(
    'welcome',
    { userId, email, template: 'welcome-v2' },
    {
      attempts: 3,                    // retry up to 3 times on failure
      backoff: { type: 'exponential', delay: 2000 }, // 2s, 4s, 8s
      removeOnComplete: 100,          // keep last 100 completed jobs for debugging
      removeOnFail: 500,              // keep last 500 failed jobs for investigation
    }
  );
}

// ─── CONSUMER (Worker) ────────────────────────────────────────────────────
const emailWorker = new Worker(
  'email-notifications',
  async (job) => {
    const { userId, email, template } = job.data;

    // Idempotency check — don't send the same email twice
    const alreadySent = await db.query(
      'SELECT 1 FROM sent_emails WHERE user_id=$1 AND template=$2',
      [userId, template]
    );
    if (alreadySent.rows.length > 0) {
      console.log(`Email already sent for ${userId}, skipping`);
      return { skipped: true }; // idempotent — safe to call multiple times
    }

    await emailService.send({ to: email, template });

    // Record that we sent it
    await db.query(
      'INSERT INTO sent_emails (user_id, template, sent_at) VALUES ($1, $2, NOW())',
      [userId, template]
    );

    return { delivered: true };
  },
  {
    connection,
    concurrency: 5,  // process 5 jobs simultaneously per worker instance
  }
);

emailWorker.on('completed', (job, result) => {
  console.log(`Job ${job.id} done:`, result);
});

emailWorker.on('failed', (job, err) => {
  console.error(`Job ${job.id} failed after ${job.attemptsMade} attempts:`, err.message);
  // After all retries exhausted, job moves to 'failed' set
  // Alert engineering team, investigate root cause
});

// ─── DEAD LETTER QUEUE ───────────────────────────────────────────────────
// Messages that fail all retry attempts go to a DLQ for manual inspection
// BullMQ keeps them in the 'failed' set — you can inspect and retry them manually

async function requeueFailedJobs() {
  const failedJobs = await emailQueue.getFailed(0, 100);
  for (const job of failedJobs) {
    await job.retry(); // manually requeue after investigation/fix
  }
}

Dead Letter Queue (DLQ) — the safety net

When a message fails all its retry attempts, it is moved to a Dead Letter Queue — a separate queue for failed messages. This prevents bad messages from blocking the main queue forever. Ops teams monitor DLQs for patterns: if 100 messages fail with the same error, that's a bug to fix, not random noise.

Real World

How Stripe and Uber use message queues

Stripe: Every payment event (charge.succeeded, charge.failed, invoice.paid) is published to an internal queue. Downstream services subscribe to relevant events — fraud detection, email receipts, Sigma analytics, webhook delivery. The payment processing core is decoupled from everything that happens after a payment. If the email service is down, payments still work; emails retry when it recovers.

Uber: Driver location updates from millions of drivers arrive continuously. A queue buffers these updates before they flow into the dispatch system, the mapping system, and the ETA calculation service. Without the queue, a spike in driver updates would directly overwhelm the dispatch database.

Pros

  • Producer/consumer fully decoupled — deploy independently
  • Natural traffic shaping — queue absorbs spikes
  • Built-in retry with backoff for transient failures
  • Dead letter queue for failure investigation
  • Consumer can scale independently based on queue depth

Cons

  • Eventual (not immediate) processing — not for real-time requirements
  • Message ordering not guaranteed (unless FIFO queue)
  • At-least-once delivery means idempotent consumers required
  • Another system to operate and monitor
  • Debugging async failures is harder than synchronous calls

Key Takeaway

A message queue is a durability and decoupling primitive. It lets producers and consumers operate at different speeds, scale independently, and fail independently without losing work. The price is asynchrony — processing is delayed, not immediate. Design consumers to be idempotent (safe to run twice), implement a dead letter queue for failures, and size your consumer fleet to the queue depth metric, not to the producer rate.
