Publish-Subscribe (Pub/Sub)

A messaging pattern where publishers emit events to a topic, and all subscribers receive a copy of every event — without the publisher knowing who is listening. Enables fan-out, event-driven architectures, and adding new consumers without changing producers.

Concept05 of 13
CategoryEvent-Driven Architecture
ToolsKafka, AWS SNS, Redis Pub/Sub, Google Pub/Sub
Used BySlack, LinkedIn, Twitter, every event-driven system

In a message queue, one message goes to one consumer. In pub/sub, one message goes to all subscribers. When a user places an order, ten different systems might care: inventory needs to reserve stock, analytics needs to record revenue, email needs to send confirmation, fraud detection needs to score the transaction. Pub/sub lets the "order placed" event flow to all ten simultaneously — without the order service knowing about any of them.

The key benefit: open/closed principle at the infrastructure level. You can add a new subscriber (new system, new use case) without touching the publisher. The order service doesn't need a deploy when you add a new downstream consumer. This is what makes event-driven architectures so extensible.

How It Works

Fan-out mechanics

Pub/Sub — one event, many consumers (fan-out) Order Service │ publishes: { event: "order.placed", orderId: "ORD-4521", amount: 99 } ▼ Topic: orders.placed │ ├──▶ Inventory Service (reserve stock) ├──▶ Email Service (send confirmation) ├──▶ Fraud Detection (score transaction) ├──▶ Analytics Service (record revenue) └──▶ Mobile Push Service (notify user) All receive the same event. Each processes independently. Order Service has zero knowledge of downstream consumers.

Topic vs Queue — the mechanical difference

In a queue, messages are consumed — one consumer takes ownership, processes, acknowledges, and the message is gone. In a topic, messages are broadcast — each subscription gets its own copy. Kafka implements this with consumer groups: all consumers in the same group compete (queue semantics); consumers in different groups each get all messages (pub/sub semantics).

Implementation

Redis Pub/Sub and Kafka — two flavors

// ── REDIS PUB/SUB — fast, ephemeral (no persistence) ──────────────────────
// Use for: real-time notifications, live chat, collaborative features
// Do NOT use for: critical business events that must not be lost

import { createClient } from 'redis';

// Publisher
const publisher = createClient({ url: process.env.REDIS_URL });
await publisher.connect();

async function publishOrderEvent(orderId, amount, userId) {
  const event = JSON.stringify({
    event:     'order.placed',
    orderId,
    amount,
    userId,
    timestamp: Date.now()
  });
  // Publish to channel — all current subscribers receive it
  // If no subscribers are connected at this moment, message is LOST (fire-and-forget)
  await publisher.publish('orders:events', event);
}

// Subscriber — runs in a separate process/service
const subscriber = createClient({ url: process.env.REDIS_URL });
await subscriber.connect();

await subscriber.subscribe('orders:events', (message) => {
  const event = JSON.parse(message);
  if (event.event === 'order.placed') {
    // Each service subscribes and handles what it cares about
    inventoryService.reserveStock(event.orderId);
  }
});

// ── KAFKA — durable, replayable, high-throughput ──────────────────────────
// Use for: audit logs, event sourcing, stream processing, high-volume events
// Kafka retains messages for a configurable period (default: 7 days)
// Consumers can replay from any offset — even events from yesterday

import { Kafka } from 'kafkajs';

const kafka = new Kafka({
  clientId: 'order-service',
  brokers:  [process.env.KAFKA_BROKER]
});

// Producer (Order Service)
const producer = kafka.producer();
await producer.connect();

async function publishOrderPlaced(order) {
  await producer.send({
    topic: 'orders.placed',
    messages: [{
      key:   order.orderId,          // partition key — same orderId → same partition → ordered
      value: JSON.stringify(order),
      headers: { eventType: 'order.placed', version: '2' }
    }]
  });
}

// Consumer (Inventory Service) — consumer GROUP determines semantics
const inventoryConsumer = kafka.consumer({ groupId: 'inventory-service' });
await inventoryConsumer.connect();
await inventoryConsumer.subscribe({ topic: 'orders.placed', fromBeginning: false });

await inventoryConsumer.run({
  eachMessage: async ({ topic, partition, message }) => {
    const order = JSON.parse(message.value.toString());

    // Idempotency check — Kafka delivers at-least-once
    const processed = await db.query(
      'SELECT 1 FROM processed_events WHERE event_id=$1 AND consumer=$2',
      [order.orderId, 'inventory-service']
    );
    if (processed.rows.length > 0) return; // Already processed

    await inventoryService.reserveStock(order);

    // Mark as processed
    await db.query(
      'INSERT INTO processed_events (event_id, consumer) VALUES ($1, $2)',
      [order.orderId, 'inventory-service']
    );
  }
});

// Email Service consumer — SEPARATE consumer group, gets ALL the same messages
const emailConsumer = kafka.consumer({ groupId: 'email-service' });
// This consumer group receives every message independently of the inventory group
Redis Pub/Sub vs Kafka: Redis is ephemeral — if no subscriber is connected when a message is published, the message is gone. Kafka is durable — messages are written to disk and retained for days. Use Redis for real-time features where losing a few messages is acceptable (live cursor positions, chat typing indicators). Use Kafka for business events that must never be lost (payments, orders, audit logs).
Event Schema Design

Designing events that age well

// A well-designed event — includes all context consumers might need
// so they don't need to call back to the publisher for more data

const orderPlacedEvent = {
  // Identity
  eventId:    'evt_01HXYZ...',        // unique event ID for idempotency
  eventType:  'order.placed',
  eventVersion: 2,                    // schema version for backwards compatibility
  timestamp:  '2026-06-10T09:00:00Z',

  // Correlation — for tracing across services
  correlationId: 'req_abc123',
  causationId:   'evt_prev456',       // what event caused this one

  // Payload — enough context for consumers to act without calling back
  data: {
    orderId:   'ORD-4521',
    userId:    'USR-789',
    amount:    9900,                  // in cents — avoid floating point
    currency:  'USD',
    items: [
      { productId: 'P-01', name: 'Blue Hoodie', qty: 1, price: 9900 }
    ],
    shippingAddress: { city: 'San Francisco', country: 'US' }
  }
};

// Version your event schema — consumers must handle old and new versions
function handleOrderPlaced(event) {
  if (event.eventVersion === 1) {
    // v1 used 'totalAmount' not 'amount' — backwards compat
    event.data.amount = event.data.totalAmount;
  }
  // process event.data.amount
}

Pros

  • Complete decoupling — publisher has zero knowledge of consumers
  • Fan-out to N consumers with no code change to publisher
  • Natural audit log — every event is a historical record
  • Enables event sourcing and CQRS architectures
  • Consumers can replay missed events (Kafka)

Cons

  • Eventual consistency — consumers lag behind real-time
  • Schema evolution is a shared responsibility — breaking changes break consumers
  • Harder to debug than synchronous calls — trace through multiple services
  • Redis Pub/Sub: no durability — messages lost if no subscriber connected
  • Ordering guarantees limited to same partition/key

Key Takeaway

Pub/Sub is the extensibility pattern for distributed systems — publish an event once, have any number of services consume it without modifying the publisher. The key design decisions are durability (ephemeral Redis vs durable Kafka), schema evolution (version your events and handle old schemas in consumers), and idempotency (Kafka delivers at-least-once; your consumers must handle duplicates). For critical business events, always use a durable broker — never use fire-and-forget for things like payments or inventory changes.
