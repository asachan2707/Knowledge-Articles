API Versioning

How to evolve an API without breaking existing clients. URL versioning, header versioning, and content negotiation — the trade-offs of each approach, what constitutes a breaking change, and how Stripe maintains hundreds of simultaneous API versions without a spaghetti codebase.

CategoryAPI Design
Key tensionDeveloper experience vs implementation simplicity
Used ByStripe, Twilio, GitHub, AWS

APIs are contracts. When a client builds on your API, they depend on its shape — field names, response structures, error formats. Changing these without notice breaks clients. API versioning is the mechanism that lets you evolve the API while honouring existing clients' contracts.

Breaking vs non-breaking changes: Adding a new optional field to a response is non-breaking — existing clients ignore unknown fields. Removing a field, renaming a field, changing a field's type, or changing the semantics of an existing field is a breaking change. Introduce a new version for breaking changes; additive changes can ship in the current version.

Strategies

URL versioning, header versioning, and content negotiation

URL versioning — most common, most visible GET /v1/users/123 GET /v2/users/123 Pros: visible in URLs, easy to test in browser, simple to route. Cons: version in URL is technically not RESTful (URL should identify resource, not its representation). Header versioning — cleaner URLs GET /users/123 Accept-version: v2 Pros: URL stays clean, resource-oriented. Cons: harder to test (must set header), not visible when sharing URLs. Date-based versioning (Stripe pattern) GET /v1/charges Stripe-Version: 2024-09-30 The endpoint stays /v1, but the response shape is determined by the date. Each new API account gets the latest date as default. Old accounts are "pinned" to their date — see no breaking changes. Stripe has 100+ active API version dates, each slightly different. Content negotiation — RFC-compliant GET /users/123 Accept: application/vnd.myapi.v2+json Verbose. Rarely used in practice outside media/hypermedia APIs.

Implementation

URL versioning with version-specific transforms

import express from 'express';

const app = express();

// ─── URL VERSIONING — separate routers per version ────────────────────────
const v1 = express.Router();
const v2 = express.Router();

// v1: user has firstName + lastName as separate fields
v1.get('/users/:id', async (req, res) => {
  const user = await userService.findById(req.params.id);
  res.json({
    id:         user.id,
    firstName:  user.name.split(' ')[0],
    lastName:   user.name.split(' ').slice(1).join(' '),
    email:      user.email,
  });
});

// v2: user has single name field + added roles array
v2.get('/users/:id', async (req, res) => {
  const user = await userService.findById(req.params.id);
  res.json({
    id:    user.id,
    name:  user.name,       // Breaking change: firstName/lastName → name
    email: user.email,
    roles: user.roles ?? [], // Additive: new optional field
  });
});

app.use('/v1', v1);
app.use('/v2', v2);
// /v2 is also the default for unversioned paths
app.use('/', v2);

// ─── HEADER-BASED VERSIONING ────────────────────────────────────────────
function versionMiddleware(req, res, next) {
  const version = req.headers['api-version'] ?? '2';
  req.apiVersion = parseInt(version);
  next();
}

app.use(versionMiddleware);

app.get('/users/:id', async (req, res) => {
  const user = await userService.findById(req.params.id);

  if (req.apiVersion === 1) {
    return res.json({ id: user.id, firstName: user.name.split(' ')[0], email: user.email });
  }
  return res.json({ id: user.id, name: user.name, email: user.email, roles: user.roles });
});

// ─── DEPRECATION HEADERS ──────────────────────────────────────────────────
// Always signal deprecated versions via response headers
// so clients can migrate before the sunset date.
function deprecationMiddleware(version, sunsetDate) {
  return (req, res, next) => {
    if (req.path.startsWith(`/v${version}`)) {
      res.setHeader('Deprecation', `true`);
      res.setHeader('Sunset', new Date(sunsetDate).toUTCString());
      res.setHeader('Link', `; rel="successor-version"`);
    }
    next();
  };
}

// /v1 is deprecated, sunsets 2027-01-01
app.use(deprecationMiddleware(1, '2027-01-01'));

// ─── WHAT COUNTS AS BREAKING ──────────────────────────────────────────────
const BREAKING_CHANGES = [
  'Removing a field',
  'Renaming a field',
  'Changing a field type (string → number)',
  'Changing HTTP method (GET → POST)',
  'Changing status code semantics (200 → 201)',
  'Adding a required field to a request body',
  'Changing authentication requirements',
  'Changing error response shape',
];

const NON_BREAKING_CHANGES = [
  'Adding a new optional response field',
  'Adding a new endpoint',
  'Adding a new optional request parameter with a default',
  'Expanding an enum (adding new valid values)',
  'Relaxing validation (accepting values previously rejected)',
];
Real World

How Stripe manages 100+ API versions

Stripe's approach: Every account has an API version date. New accounts default to the latest. Existing accounts are pinned to the date they joined. When Stripe makes a breaking change, they add a new version date. All requests include the account's pinned version in the Stripe-Version header. Internally, Stripe maintains a chain of "version gates" — each gate transforms the response for requests at that version date or older.

Twilio: Uses date-based versioning (2010-04-01, 2023-12-01) for major version milestones, combined with strict backward compatibility within a version. Any change that would break existing integrations requires a new version date.

GitHub: Uses header versioning: X-GitHub-Api-Version: 2022-11-28. GitHub commits to supporting API versions for at least 24 months before sunset.

Version early, version consistently. It is far harder to introduce versioning after your API already has clients. Design your routing to support versions from day one, even if you only have v1. When the time comes for v2, the infrastructure is already in place. Add Deprecation and Sunset headers to old versions months before removal to give clients time to migrate.

Key Takeaway

API versioning is how you evolve an API without breaking clients. URL versioning (/v2/) is the most practical for public APIs — visible, easy to route, easy to document. Non-breaking additions (new optional fields, new endpoints) can go into the current version. Breaking changes (removing or renaming fields, changing types) require a new version. Signal deprecation via response headers (Deprecation, Sunset) with at least 6–12 months of notice. Stripe's date-based versioning is the gold standard for APIs with many long-lived clients — each client stays pinned to the version they tested against.
