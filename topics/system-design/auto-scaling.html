Auto Scaling

Automatically adds or removes compute resources based on defined metrics, matching capacity to actual demand in real time. The mechanism that makes cloud economics work — pay only for what you use, never under-provision for peaks.

Concept12 of 13
CategoryCloud Infrastructure
PlatformsAWS ASG, Kubernetes HPA/VPA/KEDA, GCP MIG
Used ByEvery cloud-native production system

Before auto scaling, capacity planning meant choosing how many servers to run 24/7 based on peak expected load. If you got it wrong — too few during a viral moment, too many at 3am — you either had an outage or burned money on idle instances. Auto scaling makes capacity a dynamic function of actual demand, not a static guess.

The two sides of auto scaling: Scale-out (adding capacity) protects availability under load. Scale-in (removing capacity) controls cost during off-peak. The hard part is scale-in — remove instances too eagerly and you get flapping; too conservatively and you waste money. Most teams get scale-out right and over-provision on scale-in.

Scaling Types

Reactive vs proactive vs scheduled

Reactive (metric-based) — the default

Watches a metric (CPU, memory, request count, queue depth) and scales when thresholds are crossed. Simple and works well for gradual traffic changes. Problem: there is always a lag between the metric exceeding the threshold and new capacity being ready — typically 1–3 minutes for new EC2 instances, 30s for new Kubernetes pods.

Predictive (scheduled) — for known patterns

Pre-scale before expected load — e.g., add 50% capacity every morning at 8am, scale back at 11pm. Works for predictable workloads with daily/weekly patterns. Combine with reactive for the unpredictable spike on top of the predictable baseline.

Proactive (ML-based) — AWS Predictive Scaling

AWS trains a model on your historical metrics and pre-provisions capacity before the predicted demand arrives. Eliminates the cold-start lag by having instances warm and ready when load increases. Available in AWS EC2 Auto Scaling Groups with Predictive Scaling policy.

Scale-out decision flow (AWS Auto Scaling Group) CloudWatch metric: CPU = 85% (threshold: 70%) │ Scale-out policy triggered │ Launch new EC2 instance ├─ AMI bake (if custom): 2–5 min ├─ Instance launch: 1–2 min ├─ App startup: 30s–2 min └─ Health check pass: 30s │ Instance added to load balancer target group Total cold start: 2–8 minutes Kubernetes Pod (much faster — image already pulled) HPA: CPU = 85%, desired replicas = 3 → scale to 5 ├─ Pod scheduled: ~5s ├─ Container start: 5–30s └─ Readiness probe: 10–30s Total: ~30–60 seconds
Implementation

Kubernetes HPA and custom metrics with KEDA

// Kubernetes HPA — scale based on CPU/memory (YAML as reference)
// apiVersion: autoscaling/v2
// kind: HorizontalPodAutoscaler
// spec:
//   minReplicas: 2
//   maxReplicas: 20
//   metrics:
//   - type: Resource
//     resource:
//       name: cpu
//       target:
//         type: Utilization
//         averageUtilization: 70   # scale when avg CPU > 70%

// But CPU is often the wrong metric. Scale on what actually matters:
// queue depth, requests per second, latency p99

// KEDA (Kubernetes Event-Driven Autoscaler) — scale on ANY metric
// Scale-to-zero when queue is empty, scale-up when messages queue up

// KEDA ScaledObject (conceptual Node.js representation of what KEDA does)
const kedaScaledObject = {
  kind: 'ScaledObject',
  spec: {
    scaleTargetRef: { name: 'email-worker' },
    minReplicaCount: 0,    // Scale to ZERO when queue is empty (save cost)
    maxReplicaCount: 50,   // Scale up to 50 pods under load
    triggers: [
      {
        type: 'aws-sqs-queue',
        metadata: {
          queueURL:            process.env.SQS_QUEUE_URL,
          awsRegion:           'us-east-1',
          targetQueueLength:   '100',  // 1 pod per 100 messages in queue
          scaleOnInFlight:     'true', // count in-flight messages too
        }
      }
    ],
    // Scale-in is conservative — wait before removing pods
    // This prevents flapping on bursty queues
    cooldownPeriod: 300,  // 5 minutes before scaling in
  }
};

// Application-level health check — critical for auto scaling
// ASG and Kubernetes use this to know when an instance is ready to receive traffic
import express from 'express';
const app = express();

let isReady  = false;
let isAlive  = true;

// Startup: warm connections, load models, etc.
async function startup() {
  await db.connect();
  await cache.connect();
  // ... other initialization
  isReady = true;
  console.log('Service ready');
}

// Liveness: is the process alive? (if false, Kubernetes restarts the pod)
app.get('/health/live', (req, res) => {
  if (!isAlive) return res.status(503).json({ status: 'unhealthy' });
  res.json({ status: 'alive' });
});

// Readiness: is the process ready to accept traffic? (if false, LB removes from rotation)
// This prevents a just-started pod from getting traffic before it's warmed up
app.get('/health/ready', (req, res) => {
  if (!isReady) return res.status(503).json({ status: 'not ready' });
  res.json({ status: 'ready' });
});

// Graceful shutdown — drain in-flight requests before pod is terminated
process.on('SIGTERM', async () => {
  isReady = false;     // Stop accepting new traffic from LB
  await new Promise(r => setTimeout(r, 15000)); // 15s drain window for in-flight requests
  await db.end();
  process.exit(0);
});

startup();
Scaling Metrics

Choosing the right scaling metric

MetricGood forProblem
CPU %CPU-bound workloadsI/O-bound workloads use low CPU but are at capacity. Misleading for async services.
Memory %Memory-intensive servicesMemory leaks cause scale-out instead of restart. Must combine with liveness probe.
Requests/secondAPI servicesRequest rate doesn't capture request cost — 1 heavy request vs 1000 light ones
Request latency (p99)Latency-sensitive APIsLatency can spike from DB issues, not capacity — can cause unnecessary scale-out
Queue depthWorker/consumer servicesQueue depth can be high from slow consumers, not just backlog
Custom business metricDomain-specific (e.g. active rides, orders/min)Requires custom CloudWatch or Prometheus metric emission
Scale on the metric closest to what actually hurts. Scaling an order-processing worker on CPU is indirect — CPU goes up because queue depth is high. Scale directly on queue depth: 1 worker pod per N messages. Cut out the indirect correlation and your scaling is faster and more accurate.

Pros

  • Capacity matches actual demand — no over-provisioning
  • Cost efficiency — pay for compute used, not peak capacity 24/7
  • Handles unexpected traffic spikes automatically
  • Scale-to-zero for event-driven workers (KEDA) eliminates idle cost

Cons

  • Cold start latency — new instances aren't instant
  • Flapping — oscillates if thresholds too tight or cooldown too short
  • Stateful services can't scale as easily (session state, open connections)
  • Cost surprises if scale-in policy is too conservative

Key Takeaway

Auto scaling is how you make your system's capacity dynamic rather than static. The key design decisions are: what metric to scale on (CPU is a proxy; queue depth or latency p99 is often more direct), what the cold-start latency is (containers: 30s; VMs: 2–8 min — design your thresholds around this), and how to handle scale-in safely (cooldown periods prevent flapping; graceful shutdown prevents dropped requests). For event-driven workers, KEDA's scale-to-zero is the most cost-efficient pattern — no messages, no pods, no cost.
