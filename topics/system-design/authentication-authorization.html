Authentication & Authorization

Authentication (who are you?) and authorization (what can you do?) are the two security questions every API must answer. JWT structure, stateless vs stateful sessions, RBAC vs ABAC permission models, and the security pitfalls lurking in common implementations.

Concept23 of 33
CategorySecurity
Pair WithOAuth 2.0 (Concept 24), API Gateway (Concept 06)
FrameworksPassport.js, Auth0, Clerk, AWS Cognito

Authentication answers "who is this?" Authorization answers "are they allowed to do this?" They are separate concerns that are often conflated. A valid JWT proves identity. An authorization layer decides whether that identity can read resource X or write resource Y. Mixing them — stuffing permissions into the token and trusting them without re-checking — leads to privilege escalation and stale permission bugs.

The key distinction: A user who logs in successfully (authn passes) should still be denied if they try to read someone else's order (authz fails). Never assume that a successfully authenticated user is authorised for everything. Always check authorisation explicitly for every resource access.

Implementation

JWT auth middleware + RBAC and ABAC authorization

import jwt from 'jsonwebtoken';
import jwksClient from 'jwks-rsa';

// ─── JWT AUTHENTICATION MIDDLEWARE ────────────────────────────────────────
// Verify token signature using RS256 + JWKS (public key from auth server)
// This pattern works with Auth0, Clerk, AWS Cognito, and any OIDC provider

const jwks = jwksClient({ jwksUri: process.env.JWKS_URI }); // e.g. https://auth.example.com/.well-known/jwks.json

function getSigningKey(header, callback) {
  jwks.getSigningKey(header.kid, (err, key) => {
    if (err) return callback(err);
    callback(null, key.getPublicKey());
  });
}

async function authenticate(req, res, next) {
  const authHeader = req.headers.authorization;
  if (!authHeader?.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'Missing or malformed Authorization header' });
  }

  const token = authHeader.slice(7);
  try {
    const payload = await new Promise((resolve, reject) => {
      jwt.verify(token, getSigningKey, {
        algorithms:     ['RS256'],         // CRITICAL: whitelist only your algorithm
        audience:       process.env.JWT_AUDIENCE,  // e.g. 'https://api.example.com'
        issuer:         process.env.JWT_ISSUER,    // e.g. 'https://auth.example.com'
        clockTolerance: 30,                // 30s clock skew tolerance
      }, (err, decoded) => err ? reject(err) : resolve(decoded));
    });

    req.user = {
      id:    payload.sub,
      email: payload.email,
      roles: payload.roles ?? [],
    };
    next();
  } catch (err) {
    if (err.name === 'TokenExpiredError') return res.status(401).json({ error: 'Token expired' });
    if (err.name === 'JsonWebTokenError')  return res.status(401).json({ error: 'Invalid token' });
    next(err);
  }
}

// ─── RBAC: Role-Based Access Control ─────────────────────────────────────
// Roles are coarse-grained: admin, editor, viewer
// Simple, fast, but can't express "user can only edit THEIR OWN resources"

function requireRole(...roles) {
  return (req, res, next) => {
    const hasRole = req.user.roles.some(r => roles.includes(r));
    if (!hasRole) return res.status(403).json({ error: 'Forbidden — insufficient role' });
    next();
  };
}

app.delete('/users/:id', authenticate, requireRole('admin'), async (req, res) => {
  await userService.delete(req.params.id);
  res.status(204).end();
});

// ─── ABAC: Attribute-Based Access Control ─────────────────────────────────
// More expressive: decisions based on attributes of user, resource, and environment.
// "A user can edit an order IF order.userId === user.id OR user.roles includes 'admin'"

async function canEditOrder(user, orderId) {
  const order = await db.query('SELECT user_id, status FROM orders WHERE id = $1', [orderId]);
  if (!order.rows.length) return false;
  if (order.rows[0].status === 'shipped') return false; // No edits after shipping

  return user.id === order.rows[0].user_id  // Owner can edit
      || user.roles.includes('admin');        // Admins can always edit
}

app.put('/orders/:id', authenticate, async (req, res) => {
  const allowed = await canEditOrder(req.user, req.params.id);
  if (!allowed) return res.status(403).json({ error: 'Forbidden' });

  // Proceed with update
  const updated = await orderService.update(req.params.id, req.body);
  res.json(updated);
});

// ─── TOKEN REFRESH FLOW ────────────────────────────────────────────────────
// Access token: short-lived (15 min – 1 hour). Sent on every request.
// Refresh token: long-lived (days – weeks). Used only to get new access tokens.
// Stored in httpOnly cookie (not localStorage) to prevent XSS theft.

app.post('/auth/refresh', async (req, res) => {
  const refreshToken = req.cookies.refresh_token; // httpOnly cookie
  if (!refreshToken) return res.status(401).json({ error: 'No refresh token' });

  // Validate refresh token against DB (allows revocation)
  const session = await db.query(
    'SELECT user_id, expires_at FROM sessions WHERE refresh_token = $1',
    [refreshToken]
  );

  if (!session.rows.length || new Date(session.rows[0].expires_at) < new Date()) {
    res.clearCookie('refresh_token');
    return res.status(401).json({ error: 'Refresh token expired or revoked' });
  }

  const user        = await userService.findById(session.rows[0].user_id);
  const accessToken = jwt.sign(
    { sub: user.id, email: user.email, roles: user.roles },
    process.env.JWT_PRIVATE_KEY,
    { algorithm: 'RS256', expiresIn: '15m' }
  );

  res.json({ accessToken });
});
Real World

Common security pitfalls in auth implementations

PitfallWhat goes wrongFix
alg:none attackAttacker strips signature, sets alg=none, claims any identityWhitelist algorithms: { algorithms: ['RS256'] }
JWT in localStorageXSS attack reads token, impersonates user indefinitelyStore access token in memory; refresh token in httpOnly cookie
No token expiryCompromised token usable foreverShort-lived access tokens (15m), refresh token rotation
Permissions in JWT trusted blindlyRole changes not reflected until token expiresLook up current permissions from DB on sensitive operations
Missing resource-level checksUser accesses other users' resources by changing ID in URLABAC check: verify requester owns the resource
Weak JWT secret (HS256)Brute-force or dictionary attacks recover secretUse RS256 with 2048+ bit key, or switch to Auth0/Clerk
Never implement auth from scratch for production. Use a battle-tested library (jsonwebtoken for verification, Passport.js for strategy composition) or delegate entirely to an identity provider (Auth0, Clerk, AWS Cognito). The attack surface for auth is enormous and the failure modes are catastrophic. Your competitive advantage is in your product, not your auth stack.

Key Takeaway

Authentication and authorization are separate layers — verify identity first, then check permissions. Use JWTs with RS256 (asymmetric keys), explicit algorithm whitelisting, and short expiry (15m–1h). Store refresh tokens in httpOnly cookies, not localStorage. Implement RBAC for coarse-grained roles and ABAC for resource-level ownership checks. Never trust permissions embedded in a token for sensitive operations — check the current state from the database. And unless you have a security team reviewing it: use an identity provider rather than building auth yourself.
