Event-Driven APIs

APIs that notify consumers when something happens, rather than making consumers poll to check if something changed. Webhooks, Server-Sent Events, and async request-reply patterns — the right tool depends on who initiates, how real-time it must be, and whether the consumer can receive connections.

PatternsWebhooks, SSE, async request-reply, outbox
Used ByStripe, GitHub, Twilio, Slack, Shopify

Polling is the default when you don't know better: check every 5 seconds whether the order shipped, whether the payment cleared, whether the import job finished. It wastes 99% of requests, adds artificial latency, and scales linearly with the number of polling clients. Event-driven APIs invert this: the system tells you when something happens, not the other way around.

Three approaches: Webhooks (server calls your server), SSE (server streams to your browser), async request-reply (you fire a request, get a job ID, receive a callback when done). Each solves a different problem. Choosing the wrong one creates miserable developer experience.

Patterns

Webhook vs SSE vs async request-reply

Webhook: server calls client's endpoint when event occurs Stripe ──▶ POST https://your-server.com/webhooks/stripe { "type": "payment_intent.succeeded", "data": {...} } ← your server returns 200 OK within 30s Requirements: your server must have a public HTTPS URL, be available 24/7. Good for: server-to-server notifications (payment events, repo push events). SSE: server streams events to browser over persistent HTTP connection Browser ──▶ GET /events (long-lived connection, EventSource API) Server ──▶ data: {"type":"order.updated","status":"shipped"}\n\n Server ──▶ data: {"type":"message.new","text":"Hello"}\n\n One-directional: server → browser. Works through proxies. Reconnects automatically. Good for: live dashboards, job progress, notifications to browser clients. Async request-reply: fire job, get notified when done Client ──▶ POST /reports/generate { filters: {...} } ← 202 Accepted { "jobId": "job_abc123" } [server processes in background: 30 seconds] Server ──▶ POST https://client.com/callbacks (webhook) { "jobId": "job_abc123", "status": "completed", "downloadUrl": "..." } OR: client polls GET /reports/job_abc123 until status=completed

Implementation

Reliable webhook delivery with outbox pattern + SSE

import express from 'express';
import crypto from 'crypto';
import { Pool } from 'pg';

const db  = new Pool({ connectionString: process.env.DATABASE_URL });
const app = express();

// ─── WEBHOOK DELIVERY (reliable, via outbox pattern) ─────────────────────
// Problem: if you call the webhook inside a transaction and the webhook fails,
// do you roll back the DB write? Separate concerns with an outbox table.

// In the business transaction, write to outbox table (atomic with the business data)
async function processOrder(orderData, webhookUrl) {
  const client = await db.connect();
  try {
    await client.query('BEGIN');

    const order = await client.query(
      'INSERT INTO orders (user_id, total) VALUES ($1, $2) RETURNING id',
      [orderData.userId, orderData.total]
    );

    // Write webhook delivery task to outbox — same transaction as the order
    await client.query(
      `INSERT INTO webhook_outbox (event_type, payload, endpoint_url, status)
       VALUES ($1, $2, $3, 'pending')`,
      ['order.created', JSON.stringify({ orderId: order.rows[0].id, ...orderData }), webhookUrl]
    );

    await client.query('COMMIT');
    return order.rows[0];
  } catch (err) {
    await client.query('ROLLBACK');
    throw err;
  } finally {
    client.release();
  }
}

// Background worker: poll outbox table, deliver webhooks with retry
async function processWebhookOutbox() {
  while (true) {
    const { rows: pending } = await db.query(
      `SELECT * FROM webhook_outbox
       WHERE status = 'pending' AND next_attempt_at <= NOW()
       ORDER BY created_at
       LIMIT 10
       FOR UPDATE SKIP LOCKED`  // SKIP LOCKED prevents multiple workers grabbing same row
    );

    for (const item of pending) {
      await deliverWebhook(item);
    }

    await new Promise(r => setTimeout(r, 1000)); // Poll every 1s
  }
}

async function deliverWebhook(item) {
  const secret    = process.env.WEBHOOK_SECRET;
  const timestamp = Math.floor(Date.now() / 1000).toString();
  const body      = JSON.stringify(item.payload);

  // HMAC-SHA256 signature — receiver validates to prevent spoofing
  const signature = crypto.createHmac('sha256', secret)
    .update(`${timestamp}.${body}`)
    .digest('hex');

  try {
    const response = await fetch(item.endpoint_url, {
      method:  'POST',
      headers: {
        'Content-Type':       'application/json',
        'X-Webhook-Signature': `t=${timestamp},v1=${signature}`,
        'X-Webhook-ID':        item.id,
      },
      body,
      signal: AbortSignal.timeout(15_000), // 15s delivery timeout
    });

    if (response.ok) {
      await db.query("UPDATE webhook_outbox SET status='delivered', delivered_at=NOW() WHERE id=$1", [item.id]);
    } else {
      await scheduleRetry(item, `HTTP ${response.status}`);
    }
  } catch (err) {
    await scheduleRetry(item, err.message);
  }
}

async function scheduleRetry(item, errorMessage) {
  const RETRY_DELAYS = [60, 300, 3600, 86400]; // 1min, 5min, 1hr, 24hr
  const attempt = (item.attempt_count ?? 0) + 1;

  if (attempt >= RETRY_DELAYS.length) {
    await db.query("UPDATE webhook_outbox SET status='failed', error=$1 WHERE id=$2", [errorMessage, item.id]);
    return;
  }

  const nextAttempt = new Date(Date.now() + RETRY_DELAYS[attempt - 1] * 1000);
  await db.query(
    "UPDATE webhook_outbox SET attempt_count=$1, next_attempt_at=$2, last_error=$3 WHERE id=$4",
    [attempt, nextAttempt, errorMessage, item.id]
  );
}

// ─── WEBHOOK RECEIVER (validate signature) ────────────────────────────────
app.post('/webhooks/inbound', express.raw({ type: '*/*' }), (req, res) => {
  const sig       = req.headers['x-webhook-signature'];
  const [tPart, vPart] = (sig ?? '').split(',');
  const timestamp = tPart?.replace('t=', '');
  const provided  = vPart?.replace('v1=', '');

  // Replay attack prevention: reject if timestamp > 5 minutes old
  if (Math.abs(Date.now() / 1000 - parseInt(timestamp)) > 300) {
    return res.status(400).json({ error: 'Timestamp too old — possible replay attack' });
  }

  const expected = crypto.createHmac('sha256', process.env.WEBHOOK_SECRET)
    .update(`${timestamp}.${req.body.toString()}`)
    .digest('hex');

  if (!crypto.timingSafeEqual(Buffer.from(provided, 'hex'), Buffer.from(expected, 'hex'))) {
    return res.status(401).json({ error: 'Invalid signature' });
  }

  const event = JSON.parse(req.body.toString());
  processWebhookEvent(event);
  res.json({ received: true }); // Respond 200 fast — do processing async
});

// ─── SERVER-SENT EVENTS ────────────────────────────────────────────────────
app.get('/events', async (req, res) => {
  res.setHeader('Content-Type', 'text/event-stream');
  res.setHeader('Cache-Control', 'no-cache');
  res.setHeader('Connection', 'keep-alive');
  res.flushHeaders(); // Send headers immediately to establish stream

  const userId     = req.user.id;
  const sendEvent  = (type, data) => res.write(`event: ${type}\ndata: ${JSON.stringify(data)}\n\n`);

  // Subscribe to Redis pub/sub for this user's events
  const sub = redis.duplicate();
  await sub.connect();
  await sub.subscribe(`user:${userId}:events`, (message) => {
    const event = JSON.parse(message);
    sendEvent(event.type, event.data);
  });

  // Send initial heartbeat
  sendEvent('connected', { userId });

  // Clean up on disconnect
  req.on('close', async () => {
    await sub.unsubscribe();
    await sub.disconnect();
  });
});
Real World

Stripe webhooks, GitHub events, and the outbox pattern

Stripe webhooks: Deliver events with exponential backoff retries for up to 3 days. Each delivery includes a Stripe-Signature header — an HMAC-SHA256 computed from the payload and a timestamp using your signing secret. Stripe's dashboard shows every delivery attempt, the response received, and lets you manually retry. The timestamp in the signature prevents replay attacks: reject events older than 5 minutes.

GitHub webhooks: Configured per-repo or per-org. Every push, PR, issue, deployment gets sent to your URL. GitHub includes an X-Hub-Signature-256 header. Failed deliveries show up in the repo settings "Recent Deliveries" tab with full request/response details — invaluable for debugging receiver code.

The outbox pattern: Writing a webhook event in the same database transaction as the business data guarantees exactly-once delivery of the notification. Without it, you can crash between writing to the DB and calling the webhook — the event is never delivered. With it, a separate worker picks up the outbox entry even after a crash and delivers it.

Return 200 immediately, process async. Webhook senders expect a 200 response within their timeout (typically 5–30 seconds). If your handler does slow processing before responding, you'll time out and the sender will retry — creating duplicates. Respond 200 first, then process the event asynchronously. Design your handler to be idempotent so retries are safe.

Key Takeaway

Event-driven APIs eliminate polling waste by notifying consumers when state changes rather than making them ask. Use webhooks for server-to-server notifications, SSE for server-to-browser streams, and async request-reply for long-running jobs. For reliable webhook delivery, use the outbox pattern — write the delivery task transactionally with the business data, then deliver asynchronously with retry. Always validate webhook signatures to prevent spoofing and check timestamps to prevent replay attacks. Respond 200 immediately and process async to prevent delivery timeouts.
