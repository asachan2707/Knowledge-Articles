Content Delivery Network (CDN)

A geographically distributed network of edge servers that caches your static assets close to your users. A user in Tokyo fetching from a CDN PoP in Tokyo gets sub-10ms response times instead of the 150ms round trip to your US-East origin.

CategoryInfrastructure / Performance
Used ByEvery web-scale product
ProvidersCloudflare, AWS CloudFront, Fastly, Akamai

Physics is the enemy. The speed of light through fiber is roughly 200,000 km/s — a round trip from Sydney to New York is about 160ms of pure physics, before any server processing. A CDN sidesteps physics by putting a copy of your content in Sydney, cutting that round trip to under 5ms.

CDN vs server-side cache: A CDN is cache at the network edge — geographically distributed and HTTP-protocol-aware. A server-side cache (Redis) is in-memory storage in your data center. They solve different problems: CDN cuts geographic latency; server-side cache cuts database latency. Most production systems use both.

How It Works

Request flow — origin pull and cache fill

First request (cache MISS at edge) User (Tokyo) ──▶ CDN PoP Tokyo ──▶ MISS │ └──▶ Origin (US-East) ──▶ 150ms round trip │ Response + Cache-Control headers │ ◀────────────┘ Store in edge cache User ◀──────────────────┘ Subsequent requests (cache HIT at edge) User (Tokyo) ──▶ CDN PoP Tokyo ──▶ HIT ──▶ User ~3ms, no origin contact

The CDN respects your origin's Cache-Control headers to decide TTL. A Cache-Control: max-age=86400 header tells the CDN to cache the response for 24 hours before re-validating with the origin. Once cached, thousands of requests to that edge PoP are served from memory — the origin never sees them.

Push CDN vs Pull CDN

Pull CDN: You don't pre-load anything. When a user requests a file that isn't cached at the edge, the CDN pulls it from your origin, caches it, and serves it. Subsequent requests are served from cache. This is the default model (CloudFront, Cloudflare). Simple to set up; the first user after a cache miss pays the origin latency.

Push CDN: You explicitly upload assets to the CDN before any user requests them. Use for large static files (firmware updates, video files) where you want guaranteed availability at every PoP before the launch event. More operational overhead but no cold-start penalty.

Cache Headers

HTTP headers that control CDN behavior

// Express.js — controlling what the CDN caches and for how long
import express from 'express';
const app = express();

// Static assets — cache aggressively at CDN (1 year), use content hash for versioning
app.use('/static', express.static('public', {
  setHeaders(res, path) {
    // Hashed filenames (bundle.a3f9b2.js) can be cached forever
    // The hash changes when content changes — old URL = old content, always
    res.setHeader('Cache-Control', 'public, max-age=31536000, immutable');
  }
}));

// API responses — don't cache at CDN (dynamic, user-specific)
app.get('/api/user/profile', (req, res) => {
  res.setHeader('Cache-Control', 'private, no-store');
  res.json(userProfile);
});

// Public but dynamic — cache short-term, allow stale while revalidating
app.get('/api/products', (req, res) => {
  res.setHeader('Cache-Control', 'public, max-age=60, stale-while-revalidate=300');
  // CDN serves cached version for 60s, then revalidates in background
  // Users never wait for revalidation — they always get a fast cached response
  res.json(products);
});

// Conditional responses — support ETags for bandwidth-efficient revalidation
app.get('/api/feed', async (req, res) => {
  const feed = await getFeed();
  const etag = generateEtag(feed);

  if (req.headers['if-none-match'] === etag) {
    return res.status(304).end(); // Not Modified — CDN keeps its cached copy
  }

  res.setHeader('ETag', etag);
  res.setHeader('Cache-Control', 'public, max-age=30');
  res.json(feed);
});

// Purge the CDN cache programmatically (CloudFront example)
import { CloudFrontClient, CreateInvalidationCommand } from '@aws-sdk/client-cloudfront';

async function purgeCache(paths) {
  const client = new CloudFrontClient({ region: 'us-east-1' });
  await client.send(new CreateInvalidationCommand({
    DistributionId: process.env.CLOUDFRONT_DISTRIBUTION_ID,
    InvalidationBatch: {
      Paths: { Quantity: paths.length, Items: paths },
      CallerReference: Date.now().toString()
    }
  }));
}
Edge Computing

CDN is evolving into edge computing

Modern CDNs (Cloudflare Workers, AWS Lambda@Edge, Fastly Compute) let you run code at the edge, not just cache static files. This unlocks patterns that previously required origin round trips:

  • A/B testing at the edge: Route 10% of users to variant B by modifying the response at the edge — no origin call, no latency
  • Auth at the edge: Validate JWTs at the CDN layer and reject unauthorized requests before they reach your origin, reducing attack surface
  • Personalization: Rewrite HTML at the edge based on cookies (geo, language, plan tier) — different responses per user without sacrificing cache efficiency
  • Image optimization: Resize, convert to WebP, and compress images on the fly at the edge based on request headers (Accept, viewport width)
// Cloudflare Worker — JWT validation at the edge (never reaches origin if invalid)
export default {
  async fetch(request, env) {
    const url = new URL(request.url);

    // Only protect API routes, pass static assets through
    if (!url.pathname.startsWith('/api/')) {
      return fetch(request);
    }

    const authHeader = request.headers.get('Authorization');
    if (!authHeader?.startsWith('Bearer ')) {
      return new Response('Unauthorized', { status: 401 });
    }

    const token = authHeader.slice(7);
    try {
      // Verify JWT using Web Crypto API (available at edge)
      await verifyJWT(token, env.JWT_SECRET);
      return fetch(request); // Valid — pass to origin
    } catch {
      return new Response('Forbidden', { status: 403 });
    }
  }
};

Pros

  • Drastic latency reduction for geographically distributed users
  • Massive origin offload (95%+ of static requests never reach origin)
  • DDoS protection — CDN absorbs attack traffic at the edge
  • Bandwidth cost reduction (CDN egress cheaper than origin egress)
  • Edge compute enables auth/routing without origin round trips

Cons

  • Stale content — CDN serves old versions until TTL expires or invalidated
  • Cache invalidation propagation delay (purges take seconds to minutes)
  • Not suitable for personalized or authenticated responses without edge compute
  • Debugging is harder — add X-Cache: HIT/MISS headers
  • Cost at very high egress volume
Common Mistakes

What breaks in production

  • Caching API responses that include user data: If /api/dashboard returns user-specific data and gets cached, user A gets user B's data. Always set Cache-Control: private, no-store on authenticated endpoints.
  • Not versioning static assets: If you deploy new JS without changing the filename, CDN keeps serving old JS. Use content-hashed filenames (app.[hash].js) so every deploy gets a new URL.
  • Setting max-age too high on mutable URLs: Cache-Control: max-age=86400 on /logo.png means users see the old logo for up to 24 hours after you change it. Either use immutable + hashed filenames, or use short TTLs on mutable URLs.
  • Forgetting the Vary header: If your API returns different responses based on Accept-Language, set Vary: Accept-Language so the CDN caches one version per language, not one version for all.

Key Takeaway

A CDN is geography-aware caching. It solves the physics problem of global latency by putting copies of your content where your users are. The key to getting it right is your cache headers: use content-hashed filenames with long TTLs for static assets, short TTLs with stale-while-revalidate for public API responses, and never cache private or authenticated responses. Modern CDNs are also edge compute platforms — validating auth, routing, and personalizing responses without any origin round trip.
