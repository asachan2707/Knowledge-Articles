Concurrency Control

When two clients read the same record, modify it, and write back simultaneously, the second write overwrites the first — silently. Concurrency control is the set of mechanisms that prevent this: optimistic locking, pessimistic locking, and MVCC. The choice shapes your system's throughput, latency, and correctness under load.

In a single-threaded world, reads and writes happen one at a time — no races. In every real production system, multiple clients read the same data, compute changes, and write back. Without control, two concurrent updates can both read version N, independently compute changes, and both write version N+1 — but only the last write survives, silently destroying the first. This is the lost update problem, and it is more common than most developers think.

The classic lost update: Two users both open a shared document at the same time. User A changes title. User B changes body. Both clients read {title: "Old", body: "Old"}, each modify one field, and write back. Whoever writes last wins the whole record. One change is silently lost — no error, no conflict notification.

Strategies

Optimistic vs Pessimistic locking — and MVCC

Optimistic locking — check version at write time, retry on conflict READ: SELECT * FROM inventory WHERE id = 1 → { id:1, quantity:100, version:5 } Client A: quantity = 100 - 3 = 97 Client B: quantity = 100 - 10 = 90 Client A: UPDATE inventory SET quantity=97, version=6 WHERE id=1 AND version=5 → 1 row updated Client B: UPDATE inventory SET quantity=90, version=6 WHERE id=1 AND version=5 → 0 rows updated (version now 6) Client B retries: re-reads { quantity:97, version:6 }, computes 97-10=87, writes version=7 → success Pessimistic locking — lock the row before reading BEGIN; SELECT * FROM inventory WHERE id = 1 FOR UPDATE; -- row is LOCKED -- Client B's SELECT ... FOR UPDATE blocks here until Client A commits UPDATE inventory SET quantity = quantity - 3 WHERE id = 1; COMMIT; -- lock released Lower concurrency, simpler logic — one writer at a time.

Implementation

Optimistic locking and distributed CAS in Node.js

import { Pool } from 'pg';
import { createClient } from 'redis';

const db    = new Pool({ connectionString: process.env.DATABASE_URL });
const redis = createClient({ url: process.env.REDIS_URL });
await redis.connect();

// ─── OPTIMISTIC LOCKING — version field ──────────────────────────────────
// Schema: products(id, name, stock, version INT DEFAULT 0)

async function decrementStock(productId, quantity) {
  const MAX_RETRIES = 3;

  for (let attempt = 0; attempt < MAX_RETRIES; attempt++) {
    // 1. Read current state
    const { rows } = await db.query(
      'SELECT stock, version FROM products WHERE id = $1',
      [productId]
    );
    if (!rows.length) throw new Error('Product not found');
    const { stock, version } = rows[0];

    if (stock < quantity) throw new Error('Insufficient stock');

    // 2. Attempt write, gated on the version we read
    const result = await db.query(
      `UPDATE products
       SET stock = stock - $1, version = version + 1
       WHERE id = $2 AND version = $3`,
      [quantity, productId, version]
    );

    if (result.rowCount === 1) return { success: true, newStock: stock - quantity };

    // Version changed between our read and write — another client updated first
    // Retry with fresh read
    if (attempt === MAX_RETRIES - 1) throw new Error('Conflict: too many concurrent updates');
    // Small delay before retry to reduce contention
    await new Promise(r => setTimeout(r, Math.random() * 50));
  }
}

// ─── PESSIMISTIC LOCKING — SELECT FOR UPDATE ─────────────────────────────
// Use when conflict rate is HIGH and retries are expensive
async function transferBalance(fromId, toId, amount) {
  const client = await db.connect();
  try {
    await client.query('BEGIN');

    // Lock BOTH rows in consistent order (lower ID first) to prevent deadlocks
    const [id1, id2] = [fromId, toId].sort((a, b) => a - b);
    const { rows } = await client.query(
      'SELECT id, balance FROM accounts WHERE id = ANY($1) FOR UPDATE',
      [[id1, id2]]
    );

    const from = rows.find(r => r.id === fromId);
    const to   = rows.find(r => r.id === toId);

    if (from.balance < amount) throw new Error('Insufficient funds');

    await client.query('UPDATE accounts SET balance = balance - $1 WHERE id = $2', [amount, fromId]);
    await client.query('UPDATE accounts SET balance = balance + $1 WHERE id = $2', [amount, toId]);

    await client.query('COMMIT');
    return { success: true };
  } catch (err) {
    await client.query('ROLLBACK');
    throw err;
  } finally {
    client.release();
  }
}

// ─── REDIS COMPARE-AND-SWAP (WATCH + MULTI/EXEC) ─────────────────────────
// Distributed optimistic lock: watch a key, abort transaction if it changes
async function redisCompareAndSwap(key, expectedValue, newValue) {
  const MAX_RETRIES = 5;

  for (let attempt = 0; attempt < MAX_RETRIES; attempt++) {
    await redis.watch(key);                // Watch for changes

    const current = await redis.get(key);
    if (current !== expectedValue) {
      await redis.unwatch();
      return { success: false, reason: 'value_changed' };
    }

    const result = await redis
      .multi()
      .set(key, newValue)
      .exec();                             // null if key changed since WATCH

    if (result !== null) return { success: true };
    // else: conflict, retry
  }
  throw new Error('CAS failed after max retries');
}

// ─── HTTP ETag-based optimistic locking (REST APIs) ──────────────────────
// GET /resources/123 → ETag: "abc123"
// PUT /resources/123, If-Match: "abc123" → 200 OK (no conflict)
//                                        → 412 Precondition Failed (conflict)
app.get('/resources/:id', async (req, res) => {
  const resource = await db.query('SELECT *, version FROM resources WHERE id = $1', [req.params.id]);
  const etag     = `"${resource.rows[0].version}"`;
  res.setHeader('ETag', etag);
  res.json(resource.rows[0]);
});

app.put('/resources/:id', async (req, res) => {
  const ifMatch = req.headers['if-match'];
  if (!ifMatch) return res.status(428).json({ error: 'If-Match header required' });

  const expectedVersion = parseInt(ifMatch.replace(/"/g, ''));
  const result = await db.query(
    'UPDATE resources SET data = $1, version = version + 1 WHERE id = $2 AND version = $3 RETURNING version',
    [req.body, req.params.id, expectedVersion]
  );

  if (result.rowCount === 0) {
    return res.status(412).json({ error: 'Precondition Failed — resource was modified by another client' });
  }

  res.json({ version: result.rows[0].version });
});
Real World

PostgreSQL MVCC, DynamoDB conditional writes, and ETags

PostgreSQL MVCC: Every UPDATE creates a new row version rather than overwriting in place. Readers see a consistent snapshot from the start of their transaction — they never block on writers and writers never block on readers. The vacuum process periodically cleans up old row versions (called "dead tuples"). This is why high-write PostgreSQL tables need regular vacuuming or you get table bloat.

DynamoDB conditional writes: Every DynamoDB PutItem and UpdateItem can include a ConditionExpression. attribute_exists(version) AND version = :expected is exactly optimistic locking. If the condition fails, DynamoDB returns ConditionalCheckFailedException with zero cost — the write was rejected, not applied. Stripe uses this pattern for idempotency key deduplication.

HTTP ETags: The Web's built-in optimistic locking. GET returns an ETag header (a hash of the resource). PUT with If-Match: <etag> only applies if the server's current ETag matches — otherwise returns 412 Precondition Failed. This is the correct way to handle concurrent edits to REST resources.

Never hold a pessimistic lock across a network call. BEGIN; SELECT FOR UPDATE; [call external API for 3 seconds]; UPDATE; COMMIT. The row is locked for 3 seconds. Under concurrency, this causes lock queues that cascade into timeouts. Keep transactions as short as possible — do all I/O outside the transaction, then enter BEGIN and do only the database work.

Optimistic pros

  • High concurrency — no blocking between readers and writers
  • No deadlocks possible
  • Works across distributed systems (no central lock manager)
  • Simple to implement with a version/etag column

Optimistic cons

  • Conflict retries increase latency under high contention
  • Can starve clients under very high write contention
  • Requires application-level retry logic
  • Not suitable for operations where retrying has side effects

Key Takeaway

Use optimistic locking (version field + CAS on write) as the default — it provides high concurrency with no blocking. Only switch to pessimistic locking (SELECT FOR UPDATE) when conflict rate is genuinely high and retry cost is unacceptable (e.g. financial balance transfers). Never hold a pessimistic lock across a network call. Expose ETag headers on REST resources so clients can do conditional updates correctly. For distributed systems without a shared database, compare-and-swap in Redis provides the same semantics across service boundaries.
