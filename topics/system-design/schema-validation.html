Schema Validation

Validating request and response shapes at API boundaries using Zod, JSON Schema, or OpenAPI. The practice that prevents malformed input from propagating deep into your system, generates documentation for free, and makes runtime type errors disappear at the boundary where they should be caught.

Concept19 of 33
CategoryAPI Design
LibrariesZod, Joi, AJV, typebox, OpenAPI/Swagger
RuleValidate at every trust boundary — never inside the domain

Every API endpoint is a trust boundary. Beyond that boundary lies untrusted input — from users, third-party webhooks, or other services. Without validation, a missing required field becomes a null reference error three function calls deep in your business logic. With schema validation at the boundary, malformed input produces a clear 400 Bad Request immediately, the error message is useful, and your domain logic never sees invalid data.

Schema-first design: Define the contract (schema) before writing the implementation. This forces you to think about the API shape up front, generates automatic documentation, enables mock servers during frontend development, and makes breaking changes visible in diffs.

Validation Layers

Where to validate — and what each layer catches

Request lifecycle — validation at each trust boundary Internet │ ▼ API Gateway ← request size, rate limit, auth token present (structural) │ ▼ Route handler ← schema validation (Zod/AJV): types, required fields, formats │ → 400 Bad Request with field-level error details if invalid ▼ Business logic ← domain validation: "account balance must be positive AFTER transfer" │ → 422 Unprocessable Entity if business rule violated ▼ Database ← constraint validation (NOT NULL, UNIQUE, FK) → last resort — should not be your primary validation layer Rule: catch errors at the earliest possible layer. Schema validation catches format errors before they touch domain logic.
Implementation

Schema-first API with Zod in Node.js

import { z } from 'zod';
import express from 'express';

// ─── DEFINE SCHEMAS (the contract) ───────────────────────────────────────
// Define once, use for: request validation, response shaping, TypeScript types, docs

const CreateOrderSchema = z.object({
  userId:          z.string().uuid('userId must be a valid UUID'),
  items: z.array(z.object({
    productId:   z.string().uuid(),
    quantity:    z.number().int().min(1).max(100),
    priceInCents: z.number().int().positive(), // Store money as integers, not floats
  })).min(1, 'Order must contain at least one item').max(50),
  shippingAddress: z.object({
    line1:       z.string().min(1).max(200),
    city:        z.string().min(1).max(100),
    country:     z.string().length(2, 'Use ISO 3166-1 alpha-2 country codes'),
    postalCode:  z.string().regex(/^[A-Z0-9 -]{3,10}$/, 'Invalid postal code format'),
  }),
  couponCode: z.string().regex(/^[A-Z0-9]{6,12}$/).optional(),
});

// TypeScript type inferred from schema — single source of truth
// type CreateOrderInput = z.infer;

const OrderResponseSchema = z.object({
  orderId:     z.string().uuid(),
  status:      z.enum(['pending', 'confirmed', 'shipped', 'delivered', 'cancelled']),
  totalCents:  z.number().int().positive(),
  createdAt:   z.string().datetime(),
});

// ─── VALIDATION MIDDLEWARE ─────────────────────────────────────────────────
function validateBody(schema) {
  return (req, res, next) => {
    const result = schema.safeParse(req.body);

    if (!result.success) {
      // Zod provides field-level error paths — return them to the client
      const errors = result.error.issues.map(issue => ({
        field:   issue.path.join('.'),
        message: issue.message,
        code:    issue.code,
      }));

      return res.status(400).json({
        error:   'Validation failed',
        errors,
      });
    }

    req.validatedBody = result.data; // Only use validated data downstream
    next();
  };
}

function validateQuery(schema) {
  return (req, res, next) => {
    const result = schema.safeParse(req.query);
    if (!result.success) {
      return res.status(400).json({ error: 'Invalid query parameters', errors: result.error.issues });
    }
    req.validatedQuery = result.data;
    next();
  };
}

// ─── ROUTES ────────────────────────────────────────────────────────────────
const app = express();
app.use(express.json({ limit: '100kb' })); // Reject oversized bodies early

app.post('/orders', validateBody(CreateOrderSchema), async (req, res) => {
  const { userId, items, shippingAddress, couponCode } = req.validatedBody; // Guaranteed valid shape

  // Business logic validation (not schema validation — this uses domain knowledge)
  const total = items.reduce((sum, item) => sum + item.priceInCents * item.quantity, 0);
  if (total > 100_000_00) { // $100,000 max order
    return res.status(422).json({ error: 'Order total exceeds maximum allowed value' });
  }

  const order = await orderService.createOrder({ userId, items, shippingAddress, couponCode });

  // Validate response shape too — prevents leaking sensitive fields
  const response = OrderResponseSchema.parse(order);
  res.status(201).json(response);
});

// ─── PAGINATION QUERY VALIDATION ──────────────────────────────────────────
const PaginationSchema = z.object({
  cursor:   z.string().optional(),
  limit:    z.coerce.number().int().min(1).max(100).default(20), // coerce: "20" → 20
  sortBy:   z.enum(['createdAt', 'totalCents', 'status']).default('createdAt'),
  sortDir:  z.enum(['asc', 'desc']).default('desc'),
});

app.get('/orders', validateQuery(PaginationSchema), async (req, res) => {
  const { cursor, limit, sortBy, sortDir } = req.validatedQuery;
  // limit is guaranteed to be a number 1-100, not a string or undefined
  const orders = await orderService.listOrders({ cursor, limit, sortBy, sortDir });
  res.json(orders);
});

// ─── WEBHOOK PAYLOAD VALIDATION ────────────────────────────────────────────
// Always validate incoming webhook payloads from external services
const StripeWebhookSchema = z.object({
  id:      z.string().startsWith('evt_'),
  type:    z.string(),
  data:    z.object({ object: z.record(z.unknown()) }),
  created: z.number().int(),
});

app.post('/webhooks/stripe', express.raw({ type: 'application/json' }), (req, res) => {
  // 1. Verify signature FIRST — reject anything without valid sig
  const sig = req.headers['stripe-signature'];
  let event;
  try {
    event = stripe.webhooks.constructEvent(req.body, sig, process.env.STRIPE_WEBHOOK_SECRET);
  } catch {
    return res.status(400).json({ error: 'Invalid signature' });
  }

  // 2. Validate shape — even from Stripe (external trust boundary)
  const result = StripeWebhookSchema.safeParse(event);
  if (!result.success) return res.status(400).json({ error: 'Unexpected event shape' });

  handleStripeEvent(result.data);
  res.json({ received: true });
});
Real World

OpenAPI, contract-first design, and runtime type safety

OpenAPI (Swagger): Define your API in a YAML/JSON schema file first. From the schema, generate: server stubs, client SDKs, interactive documentation, mock servers for frontend dev, and validation middleware. Teams like Stripe publish machine-readable OpenAPI specs for their entire API — clients generate SDKs in any language from the spec rather than hand-writing them.

tRPC (TypeScript): Goes further than Zod — the server's router is the schema. TypeScript infers types from the server procedures to the client, with zero serialization step. If the server procedure signature changes, the TypeScript client fails to compile before any test runs. This is schema validation taken to its logical end: the type system enforces the contract.

GraphQL schema validation: Every GraphQL server validates queries against the schema before execution. A query requesting a non-existent field returns an error immediately without reaching any resolver. Schema introspection lets clients know exactly what fields and types are available — self-documenting by design.

Never trust data that crossed a trust boundary — even from your own services. A service calling your API is an external caller from your API's perspective. Validate incoming data from microservices the same way you validate user input. Internal attackers, bugs, and schema drift all inject invalid data through internal channels. Trust the schema, not the caller.

Pros

  • Invalid input caught at the boundary — never reaches domain logic
  • Field-level error messages guide API consumers to fix their requests
  • Schema is living documentation — always in sync with behaviour
  • TypeScript inference from Zod eliminates runtime type errors in handlers

Cons

  • Schema maintenance cost as API evolves (can be automated with code-gen)
  • Overly strict schemas reject valid edge cases (e.g. international postal codes)
  • Performance: Zod adds ~0.5–2ms per request for complex schemas (acceptable)
  • Schema drift if code and schema are maintained separately

Key Takeaway

Validate at every trust boundary and nowhere else. Use Zod (or equivalent) to define schemas that are the single source of truth for: request validation, response shaping, TypeScript types, and documentation. Return field-level errors on validation failure so API consumers can fix their requests without guessing. Never use unvalidated input in business logic — access only req.validatedBody, never req.body after validation middleware runs. Schema-first design, where the contract is defined before implementation, produces better APIs than implementation-first design every time.
