AI Agent Fundamentals: Context Engineering and Reliable Agent Design

Building an AI agent that works in demos is easy. Building one that works reliably in production — under real user pressure, with real data, across edge cases — requires understanding how the reasoning engine actually works, and how to engineer the context it operates within. This guide covers agent fundamentals, hybrid reasoning, context engineering, and the best practices that separate reliable agents from unpredictable ones.

Level Beginner to Intermediate

We have graduated from prompt engineering to context engineering. A single well-crafted prompt is no longer the unit of work — the entire system around the agent is. How you define its domains, write its instructions, scope its actions, and control what it can see and do determines whether your agent is reliable or just clever.

Core idea: Agent reliability is an architecture problem, not a prompt problem. The most common failure mode — an agent that behaves well in testing and unpredictably in production — is almost always caused by insufficient structure around the agent, not insufficient capability inside the model.

Fundamentals

What is an AI agent?

An AI agent is software that uses generative AI to make decisions about what to do next and how to do it. Unlike a simple chatbot that returns a fixed response, an agent can understand a user's intent, autonomously reason to determine what actions it needs to take, identify what data is required, and then act — with or without human intervention — to achieve a goal.

Key capabilities that define an agent

  • Dynamic adaptation — agents respond to the specific situation, environment, and information available at the time of each request, rather than following a fixed script.
  • Autonomous or guided operation — agents can act without human intervention or be configured to require human approval at defined steps.
  • Planning and reasoning — agents can break a goal into sub-steps, decide the sequence of actions, and adapt when an action produces an unexpected result.
  • Action execution — agents can call external systems, retrieve data, update records, trigger workflows, and send notifications — not just generate text.

How agents differ from traditional automation

Traditional automation (rules-based workflows, decision trees, scripted chatbots) follows a pre-defined path. Every branch is explicitly coded. It is reliable within its design scope but brittle outside it — any input the designer didn't anticipate either fails or falls through to a generic fallback.

An AI agent is different because it can handle inputs the designer never anticipated, reason about novel situations, and make judgment calls. The trade-off is that this flexibility comes with probabilistic behaviour — the same input can, in principle, produce different outputs on different runs.

The practical solution is hybrid reasoning: combining LLM-driven flexibility with deterministic, rules-based control in the same system. Deterministic automation and agentic AI do not have to be an either/or choice — they work together.

Traditional automationPure LLM agentHybrid agent
Handles novel inputs No — fails or falls through Yes — but unpredictably Yes — within defined guardrails
Follows business rules reliably Yes — always No — instructions can be ignored Yes — rules are enforced deterministically
Adapts to context No Yes Yes
Predictable outcomes High Low High for critical paths, flexible elsewhere
Choosing the Right Tool

Prompts vs. agents — when to use each

Agents are not the right tool for every task. Understanding the boundary between a prompt template and a full agent saves significant development complexity.

When a prompt template is sufficient

A prompt template is a single-turn, stateless interaction with a language model. It receives input, runs one generation, and returns output. It has no memory between calls, makes no decisions, and takes no actions. Use a prompt template when:

  • The business context is well understood and stable
  • The desired output follows a predictable, repeatable format
  • No evaluation of different paths or autonomous decisions is needed
  • No persistent state or context across multiple turns is required
  • The task is a single transformation: summarise this, classify that, draft this email

Good prompt template use cases:

  • Generating a structured record summary (e.g., summarise case notes)
  • Drafting templated email responses from input data
  • Extracting structured fields from unstructured text
  • Sentiment classification of a support ticket
  • Suggesting next best actions based on a fixed set of criteria

When you need a full agent

An agent is appropriate when the task requires multi-step reasoning, dynamic decision-making, or interaction with external systems based on evolving context. Use an agent when:

  • The environment or relevant context can change mid-conversation
  • Multiple decision branches are possible depending on retrieved data
  • The AI needs to replace or assist human judgment across a sequence of steps
  • External systems must be queried or updated as part of handling the request
  • The task requires memory of what happened earlier in the conversation

Good agent use cases:

  • Triaging and routing incoming support cases based on priority, topic, and customer history
  • Resolving booking issues by checking availability systems and applying policy rules
  • Managing multi-step workflows with approval gates and feedback loops
  • Scheduling appointments based on real-time availability and customer preferences
  • Detecting events and proactively triggering resolution actions through API calls

Rule of thumb: If the process is static and every step is known in advance, build it as a deterministic workflow — not an AI agent. Agents add value where flexibility, judgment, or dynamic context is genuinely required. Using an agent for a purely linear, rule-based task adds complexity without adding capability.

Context Engineering

From prompt engineering to context engineering

Prompt engineering — the practice of crafting individual prompts to produce better outputs — is a starting point, not a destination. Context engineering is the successor: a more holistic practice of designing the entire system of information, structure, and constraints that an agent operates within.

Where prompt engineering asks "how do I phrase this instruction?", context engineering asks "what should the agent know, what should it be allowed to do, and how should the system enforce boundaries around both?"

The three levers of context engineering

Well-designed agents are built by controlling three distinct elements, each serving a different function in shaping agent behaviour:

LeverWhat it controlsDeterministic or probabilistic?
1. Domains (specialist scopes) What the agent is allowed to handle; which "department" handles which type of request Deterministic — the routing logic is defined explicitly
2. Instructions How the agent handles conversations within a domain — tone, flow, sequencing, what to ask Probabilistic — guides but does not guarantee behaviour
3. Actions and rules What the agent can do; mandatory logic that must execute before, during, or after a task Deterministic — coded logic runs exactly as specified

The common failure mode in agent design is putting everything into instructions — trying to express business rules, mandatory sequences, and tone guidance all in natural language. Natural language instructions are probabilistic: the model reads them as suggestions, not guarantees. Critical business logic must be moved into deterministic structures (coded rules, action sequences, explicit conditions) — not left to the model's interpretation.

Lever 1 · Domains

Domains: defining what each agent scope handles

A domain (sometimes called a subagent, topic, or specialist scope depending on the platform) is a bounded area of responsibility within an agent. Think of domains as specialised departments — each with its own expertise, available actions, and instructions. When a user sends a message, the agent's routing logic first determines which domain should handle it before any action is taken.

Why domains matter

Without domains, a single agent must handle every type of request with one set of instructions and actions. As complexity grows, the instruction set becomes overloaded, overlapping, and increasingly unreliable. Domains solve this by decomposing the agent into focused, non-overlapping scopes — each optimised for a specific job.

Best practices for defining domains

  • Give each domain a clear, specific name. The name is the primary signal used by routing logic to match incoming requests. Vague names produce misroutes. "Handle order questions and issues" is vague. "Answer questions about order status or repair policies" is precise.
  • Write a descriptive purpose statement. The description should explicitly state what activities the domain covers — not just the topic area. This is the instruction the routing engine uses for classification, not human documentation.
  • Prevent scope overlap. Two domains with similar descriptions will produce routing ambiguity. Define the exact boundary between adjacent domains explicitly: "Domain A handles billing questions; Domain B handles account access questions — it does not handle billing."
  • Use explicit transitions. When the conversation needs to move from one domain to another (e.g., from an order domain to a returns domain), use coded transition logic rather than relying on the model to figure out when to hand off. Explicit beats implied.
Domain descriptionProblemBetter version
"Handle order questions and issues" Too vague — routing engine can't reliably classify intent "Answer questions about order status, estimated delivery dates, and repair policies for existing orders"
"Help with login problems" Doesn't specify what the domain actually does "Help customers who cannot log in by resetting passwords or looking up usernames — does not handle account billing or subscription changes"
"Customer service domain" Covers everything — provides no routing signal at all Split into discrete domains: Billing, Order Status, Returns, Technical Support
Lever 2 · Instructions

Instructions: guiding conversation behaviour within a domain

Instructions direct how an agent handles conversations within a domain. They guide action selection, set conversation patterns, define what information to collect before acting, and provide business context that shapes the model's responses.

Instructions are probabilistic — the model treats them as strong guidance, not guaranteed execution. This is both their strength (they produce natural, flexible conversation) and their limitation (they cannot reliably enforce mandatory business rules). Never put critical compliance requirements or mandatory sequencing in instructions — those belong in coded logic.

What instructions are for

  • Defining the tone, persona, and communication style for a domain
  • Specifying what information to gather from the user before taking an action
  • Setting conversation patterns — when to ask a follow-up, when to summarise, when to confirm
  • Providing business context the model needs to make good decisions ("Promo codes are discount codes; Registration codes are warranty activation codes — treat them differently")
  • Guiding action selection between similar options ("Use the Email Reset action if the user has email access; use the Security Question action if they do not")

Best practices for writing instructions

  • Define your terms explicitly. Do not assume the model knows what domain-specific terms mean. Define them inline: "A 'chargeback' in this context means a disputed transaction initiated by the customer's bank — it is different from a refund initiated by our support team."
  • Be specific about sequences. If steps must happen in a certain order, state the full sequence in one instruction block. Do not scatter ordering constraints across multiple instructions — the model may not synthesise them correctly.
  • Use consistent terminology. Refer to actions by their exact names throughout all instructions. Calling the same action "the Knowledge action," "the knowledge lookup," and "the FAQ search" in different instructions will confuse action selection.
  • Be intentional with absolutes. Words like "always" and "never" have disproportionate weight in how the model prioritises instructions. Use them deliberately and sparingly.
  • Provide decision criteria for similar options. When the agent must choose between two similar actions, give it the exact criteria: "Use Action A when X is true; use Action B when Y is true." Ambiguous choice points produce inconsistent behaviour.
Weak instructionStrong instructionWhy it's better
"Get the customer's order details" "If a customer asks about order status, offer all lookup options — email address or order ID — before running the order lookup action" Specifies what to offer and when, using language that matches the action name
"Help with device issues" "Before using the Knowledge action, clarify the device type (iOS or Android) — the knowledge base has separate articles for each" Gives clear pre-action data collection instruction with the reason why
"Use knowledge for product questions" "Identify the specific product name first. Then use the Knowledge action with the exact product name as the search query" Provides a clear two-step sequence with the exact input to pass to the action
"Check if customers need help" "After providing a shipping status update, always ask if the customer needs anything else related to their current order before closing the conversation" Specific about when, what to ask, and the scope of the follow-up
Lever 3 · Actions

Actions: what the agent can actually do

Actions are the mechanisms through which an agent interacts with the world beyond the conversation — retrieving data, updating records, triggering workflows, calling APIs, and running business logic. The reasoning engine reviews available actions by their names, descriptions, and defined inputs to decide which action to call for a given situation.

Types of actions

Action typeWhat it doesBest used for
Knowledge retrieval Searches a knowledge base or document index and returns relevant chunks (RAG) Answering questions from internal documentation, knowledge articles, FAQs
Data lookup Queries a database, CRM, or API for specific record data Looking up order status, account details, inventory, pricing
Data update Writes to a system — creates, modifies, or deletes records Updating shipping addresses, creating support tickets, processing refunds
Workflow trigger Initiates an automated process or handoff Routing to a human agent, triggering an approval flow, sending a notification
External API call Connects to third-party systems via REST, GraphQL, or webhooks Checking carrier tracking, verifying identity, querying partner systems
Generative action Calls an LLM with a specific prompt template to produce structured output Drafting response emails, generating summaries, producing structured reports

Best practices for designing actions

  • Always check if a standard action exists before building a custom one. Most agent platforms provide pre-built actions for common operations. Custom actions add maintenance burden — use them only when no standard action covers the need.
  • Design actions for reusability. An action that can serve multiple domains is more valuable than one built for a single narrow case. Generic, well-parameterised actions reduce total action count and simplify the agent's decision space.
  • Write action descriptions that match user language. The reasoning engine uses the action name and description to decide when to call it. Write these in the language your users use — not internal system names or technical identifiers.
  • Define action inputs explicitly. Specify exactly what information the action needs. Vague input definitions produce either failed calls (the agent doesn't know what to pass) or hallucinated inputs (the agent invents values it doesn't have).
  • Use coded logic for mandatory pre- and post-action steps. If an identity check must always run before a reset action, or a confirmation must always be sent after an update, encode those as deterministic rules — not instructions the model might follow.

Instructions vs. actions — a critical boundary

Use actions and coded rules for control

Critical business rules, mandatory sequencing, complex calculations, compliance requirements, and data validation should live in coded actions or deterministic logic — not in natural language instructions. These must execute exactly as specified, every time.

Use instructions for behaviour

Tone, persona, conversation flow, what to ask next, how to handle ambiguity, when to follow up — these are good candidates for instructions. They shape the agent's voice and guide its conversational choices, but they do not replace coded enforcement of non-negotiable rules.

How Agents Reason

How the agent reasoning engine works

Understanding the internal loop an agent runs on every request is essential for designing reliable agents. Each turn, the reasoning engine follows a consistent pattern — regardless of the underlying LLM or platform.

StepActivityDescription
1 Receive request The agent receives the user's message along with the current conversation history and session context.
2–3 Classify domain The engine matches the message to the most appropriate domain based on the domain's name and description. This is where clear, non-overlapping domain definitions pay off — ambiguous descriptions produce misroutes.
4–5 Execute scripted logic and assemble context Any deterministic rules defined for this domain (mandatory pre-steps, identity checks, data fetches that must always run) are executed before the model sees the conversation. The available actions and instructions are assembled into the context window.
6 Send to LLM The assembled prompt — domain scope, instructions, available actions, conversation history, and any data collected so far — is sent to the language model.
7 LLM decides: respond or act The model evaluates all available information and decides whether to: run an action to retrieve or update data; ask the user for more information; or respond directly with an answer.
8–9 Execute action (if needed) If an action is selected, the engine calls it with the appropriate inputs and collects the results. Inputs are derived from the conversation context — anything the agent cannot confidently supply should be explicitly requested from the user first.
10 Run post-action logic Any deterministic rules defined to run after an action (confirmations, logging, handoffs) are executed. This is the second place where coded logic can enforce mandatory behaviour.
11 Action loop The engine re-evaluates the new information returned by the action and decides again: run another action, ask for more information, or respond. This loop continues until the agent has enough information to respond.
12 Grounding check and response Before sending the final response, the engine verifies: the answer is based on retrieved data or defined instructions (not generated from memory); it stays within the domain's scope; it follows the guidelines set in the domain's instructions. The grounded response is sent to the user.

What this means for agent design

  • Steps 2–3 depend entirely on domain quality. If domain descriptions are vague or overlapping, requests will be misrouted before any instructions or actions are consulted. Domain definition is the first place to debug routing failures.
  • Steps 4–5 are your deterministic control points. Pre-action scripted logic runs before the LLM sees anything — this is where to enforce mandatory steps that must always execute.
  • Step 7 is probabilistic by design. The model's decision to call an action vs. respond directly is influenced by action descriptions, instructions, and conversation context. Poorly described actions will be under-called or over-called.
  • Step 12 is your last defence against hallucination. The grounding check should enforce that the response derives from retrieved data or defined instructions — not from the model's training memory.
Hybrid Reasoning

Hybrid reasoning: combining deterministic and probabilistic control

The most reliable agent architectures use a "deterministic sandwich" pattern: coded logic handles the beginning and end of every interaction, while the LLM handles the flexible, natural-language middle portion. This gives you the best of both worlds — the precision of rule-based systems where it matters most, and the adaptability of language models where flexibility adds value.

Levels of determinism

Not all agent behaviours need the same level of reliability. A useful mental model is to think of a dial from fully deterministic to fully probabilistic, and consciously choose where each part of your agent sits:

Behaviour typeRecommended approachWhy
Identity verification before sensitive actions Fully deterministic (coded rule) Must run every time, exactly as specified — cannot be optional or probabilistic
Mandatory data collection before an action Deterministic (scripted pre-step) The action will fail or produce wrong results without required inputs — do not leave this to the model to decide
Routing between domains Deterministic classification with explicit handoff logic Misroutes are invisible to users and hard to debug — explicit transitions remove ambiguity
Conversation tone and style Probabilistic (instructions) Natural language is the right tool for guiding personality and voice — no coded equivalent exists
Action sequencing when order matters Deterministic (coded sequence) If Action B must always follow Action A, code the dependency — do not instruct the model to remember the order
Handling ambiguous edge cases Probabilistic (LLM with clear fallback instructions) Ambiguity is exactly where LLM flexibility adds value — define the fallback clearly and let the model handle the variety

The doom loop problem: When too much logic is stuffed into natural language instructions, the prompt becomes so long and internally contradictory that the model struggles to follow it reliably. This "prompt doom loop" is the primary symptom of an agent that behaves well in simple tests but fails under realistic load. The fix is to move business rules out of instructions and into coded logic — not to write better instructions.

Reliable Context

Reliable context: filters, scoping, and citations

Reliable agents require thoughtful control over what the model can see, what it can do, and how its answers are traceable. Three mechanisms provide this: filters, scoped actions, and citations.

Filters — controlling what the model sees

Filters are system-level controls that restrict which domains, actions, or retrieved content are available on a given turn of the conversation. They operate before the model makes any decisions — constraining the decision space rather than instructing the model to constrain itself.

  • Domain filters — make certain domains available only when specific conditions are met (e.g., a "Premium Support" domain is only active for customers with an active support contract).
  • Action filters — restrict which actions are callable based on session state (e.g., the "Process Refund" action is only available after identity has been verified).
  • Retrieval filters — scope knowledge retrieval to the relevant subset of the knowledge base (e.g., only retrieve mortgage documents when the user is asking about home loans — not credit card or investment content).
  • Static filters — conditions fixed at design time (e.g., only retrieve published articles).
  • Dynamic filters — conditions resolved at runtime from the conversation context (e.g., filter retrieval to the user's product family, determined from the session data).

Scoped actions — principle of least privilege

An agent should only have access to the actions it needs for the domain it is currently handling. Exposing all actions to all domains increases the surface area for misuse and makes action selection less reliable — the model must choose from a larger set of options, increasing the chance of calling the wrong one. Apply the same principle as access control in security: grant minimum necessary access per context.

Citations — making answers verifiable

Citations connect the agent's response back to the specific source data that grounded it. They serve three purposes:

  • Trust — users who can see the source of an answer trust it more. Users who cannot, shouldn't.
  • Verifiability — a cited response can be audited. Users can check whether the source actually says what the agent claims.
  • Debuggability — when an answer is wrong, a citation immediately identifies which source document is the problem. This is a content governance issue — fixable at the source — not just an AI problem.

Always point citations to content the end user can access. A citation to an internal document the user cannot open creates friction and distrust rather than transparency.

Best Practices Summary

Context engineering best practices checklist

Domains

  • Give each domain a specific, descriptive name that matches the language of user requests.
  • Write domain descriptions that explicitly state the scope of activities — what the domain handles and what it does not.
  • Eliminate scope overlap between adjacent domains; define the boundary explicitly in each description.
  • Use deterministic transition logic to move users between domains — do not rely on the model to infer when a handoff is needed.

Instructions

  • Use instructions for tone, conversation flow, and guidance — not for mandatory business rules.
  • Define all domain-specific terms inline — do not assume the model has the necessary context.
  • State step sequences explicitly and in one place — do not scatter ordering constraints.
  • Use consistent action names throughout all instructions.
  • Provide explicit decision criteria when the agent must choose between similar options.
  • Use "always" and "never" deliberately and sparingly — they have outsized weight.

Actions

  • Use standard actions before building custom ones.
  • Design actions to be reusable across domains where possible.
  • Write action names and descriptions in user language — not system or technical identifiers.
  • Define all action inputs explicitly — vague inputs produce failed or hallucinated calls.
  • Move mandatory pre- and post-action steps into deterministic logic, not instructions.
  • Apply domain-scoped action filtering — expose only the actions a domain actually needs.

Reliability and testing

  • Apply the deterministic sandwich: coded logic at the start and end of every interaction; LLM handles the flexible middle.
  • Move any instruction that uses words like "must," "always," "required," or "mandatory" into coded logic — instructions cannot guarantee these behaviours.
  • Validate agent behaviour in a test environment with a representative set of real-world queries before deployment.
  • Monitor agent performance in production and use observed failures to improve domain descriptions, instructions, and actions — not just to re-tune prompts.
  • Apply citations to all production agents. Point only to content the end user can access.
  • Review and update domain descriptions and instructions whenever the underlying business process changes. Stale instructions produce stale behaviour.
Common Questions

FAQ on AI agent fundamentals

  • What is hybrid reasoning in the context of AI agents?

    Hybrid reasoning is an approach to agent orchestration that combines deterministic, rule-based logic with LLM-driven intelligence in the same system. It lets builders dial up or dial down AI autonomy depending on how much reliability vs. flexibility a given task requires. Critical business rules execute as coded logic; conversational flexibility is handled by the model.

  • What is the difference between a domain (subagent) and an action?

    A domain is a bounded area of responsibility — it defines what the agent handles and sets the context (instructions, available actions) for that type of request. An action is a specific mechanism — a data lookup, API call, or workflow trigger — that the agent uses to perform a task within a domain. Domains are about scope; actions are about capability.

  • When should I use instructions vs. coded logic?

    Use instructions for behaviour: tone, persona, conversation flow, what to ask next, how to handle ambiguity. Use coded logic for control: mandatory sequencing, business rules, compliance requirements, identity checks, and any step that must execute exactly as specified every time. If the instruction contains the word "must," "always," or "required," it belongs in code, not natural language.

  • What is context engineering and how does it differ from prompt engineering?

    Prompt engineering asks "how do I phrase this instruction to get a better output?" Context engineering asks "what should the agent know, what should it be allowed to do, and how does the system enforce those boundaries?" Context engineering encompasses the design of domains, instructions, actions, filters, and deterministic rules — the entire operating environment of the agent, not just the text of a single prompt.

  • Why does my agent work in testing but fail in production?

    The most common causes are: (1) test queries were too similar and didn't cover the range of real user language; (2) production content has drifted from what was indexed during testing; (3) instructions that appeared to work in simple tests are inconsistently followed under more complex, realistic conversations. The fix is usually to move critical logic from instructions into coded rules, improve domain scoping, and build a representative benchmark of production-style queries to test against on every change.

  • Do all AI agents need a retrieval (RAG) layer?

    No — but most production agents benefit from one. If the agent's answers are based entirely on fixed instructions and coded logic (e.g., a booking agent that only queries a calendar API), RAG may not be needed. If the agent needs to answer questions from a changing knowledge base — policies, product documentation, support history — RAG is the right mechanism to keep those answers accurate and up to date without retraining the model.

Key Takeaway

Reliable AI agents are built through context engineering — the deliberate design of domains, instructions, actions, filters, and deterministic rules that define the entire operating environment of the agent. The most common failure in agent design is treating instructions as a substitute for structure: writing longer, more detailed natural language prompts instead of moving business logic into coded rules. Instructions guide behaviour; code enforces it. Design with that distinction in mind and your agents will be as predictable as the business processes they represent.
