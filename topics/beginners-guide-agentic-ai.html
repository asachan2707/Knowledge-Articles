The Beginner's Guide to Agentic AI: From Zero to Your First AI Agent

What separates a chatbot from an agent, how the agent loop works, what tool calling actually does at the code level, and how to build your first working agent in under 50 lines of Python.

SourcesHugging Face Agents Course, Anthropic building-effective-agents
Frameworks coveredsmolagents, LangChain, LangGraph, raw API
LevelBeginner — no prior AI engineering experience needed

A chatbot answers and waits. An agent acts, observes the result, decides what to do next, and keeps going until a goal is met. That one difference — the loop — is what makes agentic AI capable of completing multi-step tasks that no single prompt could handle.

Core idea: An agent is a loop wrapped around an LLM. The LLM reasons about what to do; the agent framework executes the action; the result feeds back into the LLM's context. Repeat until done.

The spectrum of agency

Not everything called an "agent" is the same. The autonomy spectrum runs from simple processors (LLM output → fixed action) all the way to multi-agent systems where one LLM orchestrates others. Most production deployments sit at levels 2–3 — routing and tool calling — not at full autonomous agency. Start at level 2. Add complexity only when it demonstrably helps.

The agent loop

Think → Act → Observe — the ReAct pattern

The core loop has three phases that cycle continuously:

  1. Think (Reason): the LLM receives the current context — system instructions, conversation history, tool definitions, previous results — and reasons about the next action. It may write this reasoning explicitly as a scratchpad before deciding.
  2. Act: the model emits a tool call — a structured JSON object naming a tool and providing arguments. The agent framework (not the model) executes the function.
  3. Observe: the tool's return value is added to the context as an observation message. The loop repeats. When the LLM produces a final answer instead of a tool call, the loop terminates.

This is the ReAct pattern (Reason + Act), from the 2022 paper of the same name. Every major agent framework — LangChain, LangGraph, smolagents, AutoGPT — implements a variation of this loop.

How tool calling works at the code level

A tool is a Python function with a name, a description, and typed arguments. The description is what the LLM reads to decide when and how to call the tool — write it as if briefing a junior engineer. 

from smolagents import tool

@tool
def calculator(a: int, b: int) -> int:
    """
    Multiply two integers together.
    Use when the user needs a numerical calculation.
    Args:
        a: first integer
        b: second integer
    """
    return a * b

The LLM never runs this function. It outputs structured text saying it wants to call calculator with specific arguments. Your agent framework parses that output and calls the actual Python function, then feeds the return value back to the LLM as the next observation.

Your first agent

A minimal working agent in ~30 lines

from smolagents import CodeAgent, DuckDuckGoSearchTool, HfApiModel

# 1. Choose an LLM
model = HfApiModel(model_id="meta-llama/Meta-Llama-3.1-8B-Instruct")

# 2. Give it a tool
tools = [DuckDuckGoSearchTool()]

# 3. Create the agent
agent = CodeAgent(tools=tools, model=model)

# 4. Run it
result = agent.run(
    "What is the current population of Tokyo, and how does it compare to Paris?"
)
print(result)

What happens internally: the agent sends the question + tool schema to the LLM → the LLM emits a search call → the agent executes the search → the result goes back to the LLM → the LLM either calls another tool or writes a final answer.

Frameworks compared

Which framework to use when

  • smolagents (Hugging Face): minimal, beginner-friendly, best for learning the fundamentals without framework magic obscuring what is happening.
  • LangChain: most popular, fastest to get a basic agent running across many LLM providers. Richest ecosystem of pre-built tools and integrations.
  • LangGraph: more control over the loop — supports branching, cycles, and complex state machines. Better for production agents that need precise flow control.
  • LlamaIndex: strongest on retrieval and RAG-focused agents. Best choice for document Q&A and knowledge-base agents.
  • Raw API: Anthropic's own recommendation for production — "Start simple. Add framework complexity only when it demonstrably improves outcomes." Direct API calls are more debuggable and less surprising than framework abstractions.

Common workflow patterns beyond the basic loop

  • Prompt chaining: output of step N feeds step N+1 — for tasks with a fixed sequence of sub-problems.
  • Routing: LLM classifies input and directs to a specialist handler — good for support bots covering multiple domains.
  • Parallelisation: run subtasks simultaneously — dramatically faster for independent sub-problems.
  • Orchestrator-Workers: a central LLM breaks work into tasks and delegates to worker LLMs — scales to complex research or code generation tasks.
What beginners get wrong

The three mistakes to avoid early

  • Over-agentic design: most production tasks only need level 2–3 autonomy, not a fully autonomous agent. Start with the simplest design that could work.
  • No stopping conditions: always set a maximum step limit. An agent without a stopping condition will loop indefinitely on ambiguous tasks or LLM errors.
  • Vague tool descriptions: the LLM reads your tool descriptions at inference time. Vague descriptions ("get data") cause wrong tool selection. Precise descriptions ("search the web and return the top 3 results as a list of dicts with title and url keys") get correct, reliable invocations.

Key Takeaway

An agent is a loop: the LLM thinks, calls a tool, observes the result, and repeats until the task is done. Start with a single tool in smolagents or raw API calls. Add tools and complexity only when you have demonstrated you need them. Write tool descriptions as if briefing a junior engineer — that description is literally what the model reads to decide whether and how to use the tool.
