Most people use Claude the way they use a search engine — ask a question, get an answer. The techniques here flip that dynamic. Instead of asking Claude to give you information, you ask Claude to put you through a structured thinking process. The output is not just an answer — it is a trained intuition for reasoning about that domain.

Core idea: The best prompts don't ask Claude to tell you things. They ask Claude to put you through a process. Socratic questioning, steelmanning, first-principles decomposition — these are processes, not answers. Prompts that trigger processes produce 10x more learning than prompts that request summaries.

The Aristotle Prompt

The prompt that transforms Claude into a structured thinking partner

Aristotle developed a systematic approach to reasoning: define terms, identify categories, examine causes, consider counterarguments, and derive principles. The Aristotle Prompt structures Claude's response around these classical logic layers.

The Aristotle Prompt Template
Analyze [topic/concept/decision] using the Aristotle framework: 1. DEFINITION — Define the core concept precisely. What does it actually mean? What are the boundaries (what it is NOT)? 2. CATEGORIES — What are the main types or variants? How do they differ from each other? 3. FOUR CAUSES (Aristotle's causal framework): - Material cause: What is it made of / composed of? - Formal cause: What is its structure or pattern? - Efficient cause: What produced it / what mechanism creates it? - Final cause: What is it FOR? What purpose does it serve? 4. DIALECTIC — What is the strongest argument FOR this and the strongest argument AGAINST it? (Steelman both sides) 5. SYNTHESIS — What principles can we derive that are TRUE across different contexts? Topic: [Your topic here]
Use case: Analyzing a technical architecture decision, understanding a complex concept, or preparing for a debate or presentation on a topic.

Aristotle Prompt example: analyzing "microservices"

// Prompt: "Analyze microservices using the Aristotle framework"
// Expected output structure (abbreviated):

/*
DEFINITION:
Microservices: an architectural style where a single application is built
as a suite of small, independently deployable services, each running in
its own process and communicating via lightweight mechanisms.
NOT microservices: a monolith split at the file level; services that share
a database; "services" that can only be deployed together.

CATEGORIES:
1. Domain-driven microservices (aligned to business capabilities)
2. Data ownership microservices (each owns its data store)
3. Event-driven microservices (communicate via message bus)

FOUR CAUSES:
Material: Individual services (code, containers), inter-service communication
  (HTTP/gRPC/events), infrastructure (Kubernetes, service mesh)
Formal: Bounded contexts, single responsibility, independent deployability
Efficient: Scaling pain in monoliths, team autonomy needs, independent release cycles
Final: Enable teams to deploy independently, scale parts not wholes, limit blast radius

DIALECTIC:
FOR: Independent scaling, team autonomy, technology diversity, fault isolation
AGAINST: Network complexity, distributed systems fallacies, operational overhead,
  data consistency challenges, debugging nightmares

SYNTHESIS:
- Microservices trade operational complexity for team/deployment autonomy
- The trade is worth it when teams > services (not service-per-developer)
- Always start with a monolith; extract services when you've identified seams
*/
6 master prompts for learning

Frameworks that accelerate deep understanding of any subject

Prompt 1: The Feynman Explainer

Forces Claude to explain a concept at multiple levels of depth, exposing where complexity is genuinely necessary vs where it is jargon.

Explain [concept] using the Feynman technique: Step 1: Explain it as if I'm 12 years old — no jargon, just analogy and intuition. Step 2: Explain it as if I'm a competent developer who knows nothing about this specific domain. Step 3: Explain where Step 1's analogy breaks down — what does the simplification miss? Step 4: Give me the one question a deep expert would ask that would expose shallow understanding. Concept: [Your concept]
Best for: Learning new technical concepts, preparing to teach something, identifying gaps in your understanding.

Prompt 2: The Constraint Map

Maps the constraint landscape of a decision space — what's fixed, what's flexible, what's unknown. Essential for architecture and design decisions.

I need to understand the constraint landscape for [decision/problem]. Map it as follows: 1. HARD CONSTRAINTS — what cannot change, no matter what? 2. SOFT CONSTRAINTS — what has a cost to change but could change? 3. ASSUMPTIONS — what am I treating as a constraint that might not be? 4. UNKNOWN CONSTRAINTS — what constraints might exist that I haven't discovered yet? 5. CONSTRAINT INTERACTIONS — which constraints conflict with each other? Problem: [Your problem]
Best for: Architecture decisions, project planning, any time you feel "boxed in" by requirements.

Prompt 3: The Failure Pre-Mortem

Project failure before it happens. Forces you to think about failure modes before committing to an approach.

I'm planning to [action/project/decision]. Conduct a pre-mortem: 1. Imagine it's 6 months from now and this has completely failed. What are the 5 most likely causes of failure? 2. For each cause: how early would the warning signs appear, and what would they look like? 3. For each cause: what is one preventive action I could take NOW? 4. Which failure mode is most likely given my current situation? 5. What's the one question I'm probably not asking that I should be? Plan: [Your plan]
Best for: Technology choices, project kick-offs, major decisions with long lead times.

Prompt 4: The Socratic Deep Dive

Claude asks you the questions instead of answering them. Forces active recall and surfaces your own gaps.

I want to deeply understand [topic]. Instead of explaining it to me, use the Socratic method: - Ask me one question at a time that probes my understanding - When I answer, either ask a follow-up that deepens it, or correct a misconception if one exists - Continue for 10 rounds - After round 10, give me a map of what I understand well vs where I have gaps Start with your first question. Do not explain anything until I've answered. Topic: [Your topic]
Best for: Active learning sessions, exam prep, understanding you need to build rather than receive.

Prompt 5: The Mental Model Extractor

Extracts the underlying mental models from any domain so you can apply them outside the domain they came from.

I want to extract transferable mental models from [domain/field/book]. For each key mental model: 1. Name and one-sentence description 2. The core insight (what does this model reveal that's non-obvious?) 3. The original domain where it was developed 4. Two domains OUTSIDE the original where this model applies 5. The failure mode — when does this mental model lead you astray? Extract the top 5 mental models from: [Your domain]
Best for: After finishing a book, understanding a new field, building cross-domain thinking.

Prompt 6: The First-Principles Rebuilder

Strips a concept down to its foundational axioms, then rebuilds it from scratch. Elon Musk's approach applied to any learning domain.

Apply first-principles thinking to [topic]: Step 1: List every assumption most people make about this topic (at least 5). Step 2: For each assumption — is it actually true, or is it convention? Step 3: Strip away all assumptions. What do we know for certain as foundational facts? Step 4: Rebuild an understanding from only those foundational facts. Step 5: What does the rebuilt understanding tell us that the conventional wisdom misses? Topic: [Your topic]
Best for: Challenging orthodoxy in your field, preparing contrarian arguments, truly original thinking about a domain.
How to combine them

Chaining prompts for maximum depth

// Example: learning distributed systems from scratch using prompt chaining
// Session 1 (Feynman): Get intuition for the core concepts
// Session 2 (Socratic): Test and expose gaps
// Session 3 (Aristotle): Deep-structure the most confusing concept (CAP theorem)
// Session 4 (First-Principles): Rebuild CAP theorem from scratch
// Session 5 (Mental Model Extractor): Extract models that transfer to your domain
// Session 6 (Pre-Mortem): Apply to your actual architecture decision

// The key insight: each prompt builds on the prior session's understanding.
// Don't use these as standalone one-offs — use them as a structured learning curriculum.

// Practical pattern for a new topic:
const learningCurriculum = async (topic) => {
  const step1 = `Feynman: Explain ${topic} at three levels + expose the simplification.`;
  const step2 = `Socratic: Quiz me on ${topic} for 10 rounds, then map my gaps.`;
  const step3 = `Aristotle: Structure the hardest concept from ${topic} using all four causes.`;
  const step4 = `Pre-Mortem: I'm about to apply ${topic} to my project — what will go wrong?`;
  // Each step feeds the next — reference prior outputs in later prompts
};

Key Takeaway

The best Claude prompts are process prompts, not answer prompts. The Aristotle framework structures reasoning around definition, categories, causes, dialectic, and synthesis — turning any topic into a thorough analysis. The six learning frameworks (Feynman, Constraint Map, Pre-Mortem, Socratic, Mental Model Extractor, First-Principles Rebuilder) each target a different type of understanding gap. Use them in sequence for a new domain; use them individually when you hit a specific knowledge wall.
