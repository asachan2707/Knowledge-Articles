Stop Memorizing Design Patterns: Use This Decision Tree Instead

Design patterns are solutions to recurring structural problems — but the hard part was never remembering 23 GoF patterns. It was knowing which problem you have. This guide teaches the decision logic: a set of diagnostic questions that lead you to the right pattern based on the shape of your actual problem, with JavaScript implementations for each.

Patterns covered16 most-used GoF + modern JS patterns
LevelIntermediate JavaScript developer

The reason design patterns feel hard to apply is that most resources teach them forwards: "here is the Observer pattern, here is how it works." But you encounter problems backwards: "I have this messy coupling problem — which pattern fixes it?" This guide reverses the flow. Start with the problem shape and follow the decision tree to the pattern.

Core idea: Three diagnostic questions unlock 90% of pattern selection: (1) Is the problem about object creation, structure, or behaviour? (2) Do you want to decouple, extend, or abstract? (3) Is the relationship one-to-one or one-to-many?

The decision tree

Step 1: What category is your problem?

What is the nature of the problem?
→ Creating objects is complex or inconsistent → Creational patterns (Factory, Builder, Singleton, Prototype)
→ Classes/objects need to be composed but coupling is messy → Structural patterns (Adapter, Decorator, Facade, Proxy, Composite)
→ Objects need to communicate but I want to keep them decoupled → Behavioural patterns (Observer, Strategy, Command, Chain, Iterator, State)
Creational patterns

When object creation itself is the problem

Creational
Use when: you need to create objects without specifying exact class

Factory Method

A function/method that returns objects. Callers don't know (or care) which concrete type they get.

Creational
Use when: constructing complex objects step by step

Builder

Separate construction from representation. Great for objects with many optional fields.

Creational
Use when: exactly one instance must exist globally

Singleton

Lazy-initialise once, return same instance. Overused — prefer dependency injection in most cases.

// Factory Method — create the right logger without knowing which type
function createLogger(type) {
  switch (type) {
    case 'console': return new ConsoleLogger();
    case 'file':    return new FileLogger('./app.log');
    case 'remote':  return new RemoteLogger(process.env.LOG_ENDPOINT);
    default:        throw new Error(`Unknown logger type: ${type}`);
  }
}

// Callers just ask for a logger — don't care about the type
const logger = createLogger(process.env.LOG_TARGET ?? 'console');
logger.info('Application started');

// Builder — construct a complex query object step by step
class QueryBuilder {
  #table = ''; #conditions = []; #cols = ['*']; #limit = null; #orderBy = null;

  from(table)        { this.#table = table;         return this; }
  select(...cols)    { this.#cols = cols;            return this; }
  where(cond)        { this.#conditions.push(cond);  return this; }
  limit(n)           { this.#limit = n;              return this; }
  orderBy(col, dir)  { this.#orderBy = `${col} ${dir ?? 'ASC'}`; return this; }

  build() {
    let q = `SELECT ${this.#cols.join(', ')} FROM ${this.#table}`;
    if (this.#conditions.length) q += ` WHERE ${this.#conditions.join(' AND ')}`;
    if (this.#orderBy) q += ` ORDER BY ${this.#orderBy}`;
    if (this.#limit)   q += ` LIMIT ${this.#limit}`;
    return q;
  }
}

const query = new QueryBuilder()
  .from('orders')
  .select('id', 'user_id', 'total')
  .where('status = \'active\'')
  .where('total > 100')
  .orderBy('created_at', 'DESC')
  .limit(50)
  .build();
// SELECT id, user_id, total FROM orders WHERE status = 'active' AND total > 100 ORDER BY created_at DESC LIMIT 50
Structural patterns

When composition and coupling are the problem

Structural
Use when: two incompatible interfaces need to work together

Adapter

Wrap an existing class to match the interface a consumer expects. Common when integrating third-party libraries.

Structural
Use when: you want to add behaviour to objects without subclassing

Decorator

Wrap an object and intercept its methods. Stacks cleanly — add logging, caching, timing without modifying the original.

Structural
Use when: you want to simplify a complex subsystem

Facade

One clean interface over a complex system. Hides the complexity of many interacting classes behind a single entry point.

Structural
Use when: you need to control access to an object (lazy load, logging, security)

Proxy

Same interface as the real object but adds a layer of control. Used for lazy loading, caching, and access control.

// Decorator — add logging to any service without touching it
class LoggedService {
  #inner;
  #logger;

  constructor(inner, logger = console) {
    this.#inner  = inner;
    this.#logger = logger;
    // Proxy all method calls
    return new Proxy(this, {
      get(target, prop) {
        if (typeof target.#inner[prop] !== 'function') return target.#inner[prop];
        return async (...args) => {
          const start = performance.now();
          try {
            const result = await target.#inner[prop](...args);
            target.#logger.info(`${prop} completed in ${(performance.now()-start).toFixed(1)}ms`);
            return result;
          } catch (err) {
            target.#logger.error(`${prop} failed:`, err.message);
            throw err;
          }
        };
      }
    });
  }
}

// Usage — wrap any service
const userService       = new UserService(db);
const loggedUserService = new LoggedService(userService);
await loggedUserService.createUser({ email: 'a@b.com' }); // logs timing automatically

// Facade — hide 5 microservice calls behind one method
class CheckoutFacade {
  constructor(inventoryService, paymentService, orderService, emailService, analyticsService) {
    this.inventory  = inventoryService;
    this.payment    = paymentService;
    this.orders     = orderService;
    this.email      = emailService;
    this.analytics  = analyticsService;
  }

  async placeOrder(userId, cartItems, paymentMethod) {
    await this.inventory.reserve(cartItems);
    const charge = await this.payment.charge(userId, cartItems, paymentMethod);
    const order  = await this.orders.create({ userId, cartItems, chargeId: charge.id });
    await Promise.all([
      this.email.sendConfirmation(userId, order),
      this.analytics.trackPurchase(userId, order),
    ]);
    return order;
  }
}
// Caller sees one method — not five services
Behavioural patterns

When communication and decoupling are the problem

Behavioural
Use when: one change should notify many others without tight coupling

Observer / EventEmitter

Publisher emits events; subscribers react. Node.js EventEmitter is the canonical implementation.

Behavioural
Use when: you want to swap algorithms at runtime

Strategy

Encapsulate algorithms behind a common interface and pass the right one in. Great for payment methods, sort algorithms, pricing rules.

Behavioural
Use when: requests should pass through a series of handlers

Chain of Responsibility

Each handler decides: process and stop, or pass to the next. Express middleware is exactly this pattern.

Behavioural
Use when: an object's behaviour changes based on internal state

State

Encode each state as an object or value. Transitions are explicit. Eliminates sprawling if/switch blocks on status fields.

Behavioural
Use when: you want to queue, undo, or log operations

Command

Encapsulate an operation as an object with execute/undo. Enables undo history, queueing, and logging.

// Strategy — swap pricing algorithms at runtime
class PricingEngine {
  #strategy;

  constructor(strategy) { this.#strategy = strategy; }
  setStrategy(strategy) { this.#strategy = strategy; }
  calculate(order)      { return this.#strategy(order); }
}

const strategies = {
  standard:   order => order.items.reduce((s, i) => s + i.price, 0),
  premium:    order => order.items.reduce((s, i) => s + i.price, 0) * 0.9,     // 10% discount
  enterprise: order => order.items.reduce((s, i) => s + i.price * 0.75, 0),    // 25% discount
  tiered:     order => {
    const subtotal = order.items.reduce((s, i) => s + i.price, 0);
    return subtotal > 1000 ? subtotal * 0.85 : subtotal > 500 ? subtotal * 0.92 : subtotal;
  },
};

const engine = new PricingEngine(strategies.standard);
engine.setStrategy(strategies[user.tier ?? 'standard']);
const total = engine.calculate(order);

// State — order lifecycle
const ORDER_TRANSITIONS = {
  pending:    ['processing', 'cancelled'],
  processing: ['shipped', 'cancelled'],
  shipped:    ['delivered', 'returned'],
  delivered:  ['returned'],
  cancelled:  [],
  returned:   [],
};

class Order {
  #status = 'pending';

  transition(newStatus) {
    const allowed = ORDER_TRANSITIONS[this.#status];
    if (!allowed.includes(newStatus))
      throw new Error(`Invalid transition: ${this.#status} → ${newStatus}`);
    this.#status = newStatus;
    return this;
  }

  get status() { return this.#status; }
}

const o = new Order();
o.transition('processing').transition('shipped'); // OK
// o.transition('pending');  // throws: Invalid transition shipped → pending
Modern JS patterns

Patterns that emerged from JavaScript specifically

// Module pattern — private state via closures (pre-ES6 class alternative)
const counter = (() => {
  let count = 0; // private
  return {
    increment: () => ++count,
    decrement: () => --count,
    value:     () => count,
  };
})();

// Middleware pattern (Express-style Chain of Responsibility)
function compose(...middlewares) {
  return (ctx) => {
    function dispatch(i) {
      if (i >= middlewares.length) return Promise.resolve();
      const fn = middlewares[i];
      return fn(ctx, () => dispatch(i + 1));
    }
    return dispatch(0);
  };
}

const pipeline = compose(
  async (ctx, next) => { ctx.start = Date.now(); await next(); ctx.duration = Date.now() - ctx.start; },
  async (ctx, next) => { if (!ctx.user) throw new Error('Unauthorized'); await next(); },
  async (ctx, next) => { ctx.result = await db.query(ctx.sql); await next(); },
);

// Mixin pattern — horizontal reuse without inheritance chains
const Serializable = Base => class extends Base {
  toJSON()              { return JSON.stringify(this); }
  static fromJSON(json) { return Object.assign(new this(), JSON.parse(json)); }
};

const Timestamped = Base => class extends Base {
  constructor(...args) { super(...args); this.createdAt = new Date(); }
};

class User extends Timestamped(Serializable(EventEmitter)) {
  constructor(data) { super(); Object.assign(this, data); }
}

const u = new User({ id: 1, name: 'Alice' });
console.log(u.toJSON()); // inherited from Serializable
console.log(u.createdAt); // inherited from Timestamped

Key Takeaway

Pattern selection starts with a diagnostic: is the problem about creation, structure, or behaviour? Then narrow by intent: decouple? extend without modifying? swap algorithms? React to changes? The answer usually points to one of five or six patterns. Memorising all 23 is far less useful than knowing these diagnostic questions deeply — and having working JavaScript implementations you can reach for immediately.
