How to Build Claude Skills 2.0 Better than 99% of People

Claude Skills 2.0 introduced a Skill-Creator, progressive disclosure, and built-in evals. The 1% who write skills that actually trigger correctly and produce reliable output use those tools to test instead of guess.

Focus SKILL.md, evals, A/B testing, progressive disclosure

Claude Skills are reusable instruction packs — a folder with a SKILL.md and optional scripts and assets — that teach Claude to behave consistently for a specific task. Skills 2.0 took the format from a clever trick to a real engineering surface, with three features that change how serious teams build them.

Core idea: what separates the top 1% is not creativity — it is the discipline to test. Anthropic itself found bugs in 5 of 6 of their own skills when they ran evals against them.

Why Skills 2.0 changed the game

The first wave of Claude Skills felt like elaborate prompts. Skills 2.0 added a Skill-Creator (a meta-skill that drafts and iterates skills for you), progressive disclosure so context windows stay clean even with dozens of skills installed, and a built-in eval system that tests skills the way you would test code.

That shift turned skills into something you can ship, version, and improve — not something you write once and hope works.

The two skill types

Capability uplift vs encoded preferences

1. Capability uplift

Teaches Claude something it cannot do well by default — your proprietary testing framework, an internal DSL, or a specialized output format. The skill carries domain knowledge the base model does not have.

2. Encoded preferences

Sequences what Claude already knows into your specific workflow — your writing style, your folder conventions, your release process. The skill carries your taste, not new knowledge.

Most failed skills mix the two. The best skills pick one and commit.

Anatomy of SKILL.md

The two parts that matter most

A SKILL.md file has two critical parts. Get either wrong and the skill fails silently.

YAML frontmatter — loaded at startup

  • name — the skill's identifier and trigger.
  • description — when to use it. This is what Claude reads to decide whether to invoke the skill.
  • Optional: triggers, allowed-tools, context injection settings.

Main content — loaded only when triggered

  • Keep it under ~500 lines — past that, the skill becomes its own context problem.
  • Step-by-step instructions, not vague guidance.
  • Concrete examples (few-shot patterns anchor format and tone).
  • Use progressive disclosure: link to large reference files instead of inlining them.
The 1% trick: evals

Test your skills like you test your code

Skills 2.0 ships a built-in eval system. It is the single biggest separator between skills that look good and skills that work.

  • Run test prompts against your skill and measure pass rate.
  • Detect regressions when you tweak the instructions.
  • Spot when the skill is no longer needed because the base model improved.
  • Blind A/B test the skill against raw Claude — sometimes raw is better.
  • Tune trigger accuracy — false fires waste tokens, missed fires waste the skill.

When Anthropic ran evals on its own bundled skills, it found problems in 5 of 6. If they need evals, so do you.

A working SKILL.md

What a tight skill actually looks like

---
name: release-tweet
description: Convert a raw changelog into a 5-bullet release tweet
  in our brand voice. Use when input mentions "changelog",
  "release notes", or a list of merged PRs.
allowed-tools: []
---

# Release Tweet Composer

## Output format
- Exactly 5 bullets, each starting with a verb.
- 240 character total cap.
- No emojis. No marketing superlatives.
- Always end with the release version in parentheses.

## Refuse if
- The input has no version number.
- The input has no shipped features.
- The release is internal-only.

## Example
Input: "v2.4.0 — Added bulk import, fixed CSV parsing, ..."
Output:
- Add bulk import for CSV files
- Fix parsing on quoted fields
- ...
(v2.4.0)

Notice what is missing: cleverness. The skill does one task, with rules, with examples, with a refusal condition. That is exactly why it works.

Progressive disclosure

Why skills don't blow up your context

Progressive disclosure means the skill's frontmatter is loaded into Claude's awareness at all times, but the full body and any reference files only load when the skill is actually triggered. That is what makes 50+ installed skills feasible — only the active one occupies context.

Practically: keep frontmatter descriptions tight and unambiguous, push verbose details into the body, and push large references into linked files.

Common questions

FAQ for skill authors

  • How do I know my skill is auto-triggering correctly?

    Run the eval suite with prompts that should fire it and prompts that should not. The trigger accuracy metric is the number to optimize.

  • Is the Skill-Creator a substitute for writing my own?

    For first drafts, yes. For production-quality skills, no — it gives you a starting point you then have to scope, constrain, and eval.

  • When should I delete a skill?

    When evals show it does not beat raw Claude on the same input. Deleting unhelpful skills is part of maintaining the suite.

Key Takeaway

Most people write a Claude Skill once, declare victory, and never test it. The 1% writes a tight SKILL.md, scopes the trigger, runs evals, A/B tests against raw Claude, and deletes skills that stop earning their seat. That discipline — not creativity — is what produces skills that actually work.
