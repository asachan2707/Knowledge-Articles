Anthropic Quietly Made Half My MCP Servers Obsolete

A walkthrough of the 6-Skill setup that replaced a stack of MCP servers — and the honest read on where MCP still wins, where Skills now win, and how the ecosystem is splitting into two tracks.

Portal Medium · Multiple sources
Authors Industry analysis · 2026
Published 2026
Focus MCP vs Skills, infra trade-offs, agentic tool surfaces

Skills did not kill MCP. But for a large class of MCP servers — the ones built mostly to load library docs, coding conventions, or version-specific guidance into context — Skills are the better fit. The infra tax of running a long-lived MCP endpoint just to deliver knowledge stopped making sense the moment the same content could ride inside a portable, file-based skill.

Core idea: MCP is for acting on systems. Skills are for teaching agents. Most "knowledge" MCP servers were quietly doing the second job and paying for the first.

Why this shift caught people off guard

MCP servers required infrastructure: a long-lived endpoint, identity and policy management, discovery, rate limits, and logging. That cost was acceptable when MCP looked like the only way to extend agents. Skills arrived without any of it — published as folders, distributed through the product surface, loaded incrementally — and silently took over the use cases where infrastructure was overkill.

Why MCP servers are being replaced

Two structural problems Skills fix

1. Infrastructure overhead

MCP needs a service. That means hosting, auth, discovery, monitoring, and a maintenance owner. Skills are files: zero infrastructure, version them in git, ship them through the agent's UI.

2. Token inefficiency

MCP tools often dump large JSON payloads into context whether the agent needs the data or not. Even Anthropic has acknowledged this. Skills load incrementally — only the active skill's body is in context.

The 6-skill replacement set

What replaced the obsolete servers

The article describes a real swap: six skills that absorbed the work of multiple MCP servers. The exact contents vary by stack, but the pattern is consistent — skills replace the "deliver knowledge to the agent" jobs, MCP keeps the "act on a real system" jobs.

  • Library / SDK skill — replaces docs-fetching MCP servers with a packaged reference and usage patterns.
  • Conventions skill — replaces "company style" MCP servers; ships repo norms inside a SKILL.md.
  • Code review skill — replaces lint-rule MCP servers; carries the rubric and examples.
  • Test scaffolding skill — replaces test-template MCPs; emits the right shape for the framework in use.
  • Release notes skill — replaces a dedicated changelog MCP; encodes voice and structure.
  • Onboarding skill — replaces a knowledge-base MCP; describes the project's mental model.

What does not get replaced: anything talking to a real system that needs entitlements, audit logs, or rate limits — databases, CRMs, internal APIs. That is still MCP territory.

Where MCP still wins

The line between Skills and MCP

  • Acting on real systems — production databases, APIs, internal services with permissions.
  • Audit trails — when every tool call must be traceable for compliance.
  • Live data — when the agent must see real, current state, not packaged knowledge.
  • Cross-team standardization — when many teams must hit the same backend with the same policies.

The shorthand from the analysis: MCP is "slow, heavy, correct." Skills are "fast, packaged, discoverable." Both belong in a serious agent stack — just stop using MCP for jobs Skills now do better.

How to decide

A simple test for each MCP server you maintain

Ask three questions about every MCP server in your setup:

  • Does it act on a real, live system, or does it deliver knowledge?
  • Does it need entitlements, rate limits, or audit logs?
  • Would the same value fit inside a SKILL.md plus reference files?

If the answer to the third question is yes and the first two are no — you are running infrastructure to do a file's job.

Common questions

FAQ for teams rethinking their stack

  • Is MCP dead?

    No. It is narrowing. Enterprise systems requiring entitlements and audit trails still need MCP. The "fetch docs into context" servers are the ones being replaced.

  • Can a skill call an MCP server?

    Yes — that is increasingly the recommended pattern. Skills carry the knowledge of how and when; MCP carries the actual capability to act.

  • What about discovery and governance?

    Skills are easier to discover (they show up in the agent UI) but harder to govern centrally. Most teams treat them like internal libraries: review, version, and publish via a shared repo.

Key Takeaway

The MCP-vs-Skills debate is settling into a clear split: Skills for teaching agents; MCP for acting on systems. Many teams' MCP catalogs were quietly doing the wrong job and paying infra cost for the privilege. A 6-skill set that replaces a rack of "knowledge" MCP servers is not a hack — it is the natural shape of the ecosystem in 2026.
