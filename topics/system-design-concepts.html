33 System Design Concepts Every Developer Should Know

From load balancing and consistent hashing to OAuth 2.0, WebSockets, CAP theorem, and observability — these are the patterns, protocols, and trade-offs every distributed system is built from. Master the mechanics behind each one and any architecture diagram becomes readable.

System design mastery is not about memorising every technology — it's about internalising the patterns that appear again and again. These 33 concepts span infrastructure reliability, API design, resilience engineering, data architecture, and real-time communication. Learn what each one solves, how it breaks at scale, and when to reach for it, and you will be able to reason about any architecture you encounter.

Load Balancing

Distributes incoming traffic across multiple servers so no single node is overwhelmed. The foundation of every horizontally scalable system.

Concept 02

Caching

Stores frequently accessed data in fast memory to reduce database load and response latency. One of the highest-leverage performance tools available.

Performance
Concept 03

Content Delivery Network (CDN)

Distributes static assets across geographically distributed edge servers so users download from the nearest location, cutting latency globally.

Infrastructure
Concept 04

Message Queue

Decouples producers from consumers — producers enqueue messages, consumers process them asynchronously at their own pace. Enables reliable async workflows.

Async Architecture
Concept 05

Publish-Subscribe

Multiple consumers each receive a copy of every message published to a topic. Powers real-time fan-out, notifications, and event-driven architectures.

Concept 06

API Gateway

Single entry point for all client requests — handles routing, auth, rate limiting, and protocol translation before requests reach any upstream service.

Architecture
Concept 07

Circuit Breaker

Monitors downstream service calls and automatically stops attempts when failure rate exceeds a threshold, preventing cascading failures across the system.

Resilience
Concept 08

Service Discovery

Automatically tracks available service instances so components can locate and communicate dynamically as pods scale up, restart, or move across hosts.

Concept 09

Sharding

Splits a large dataset horizontally across multiple database nodes using a shard key. Enables scale far beyond what a single machine can hold.

Data Architecture
Concept 10

Rate Limiting

Controls how many requests a client can make in a time window, protecting services from overload, abuse, and the noisy-neighbour problem.

API Design
Concept 11

Consistent Hashing

Distributes data across nodes so that only a minimal fraction of keys must be remapped when a node joins or leaves. Powers distributed caches and storage rings.

Distributed Systems
Concept 12

Auto Scaling

Automatically adds or removes compute resources based on defined metrics, matching capacity to actual demand in real time to balance cost and availability.

Cloud Infrastructure
Concept 13

Idempotency

Designing operations so that executing them multiple times produces the same result as once. The key to safe retries and reliable distributed systems.

API Design
Concept 14

Retry Strategies

How to safely retry failed operations — immediate, fixed, exponential backoff with jitter, and deadline-aware retries. The difference between making a system resilient and making it a DDoS against your own downstream.

Resilience
Concept 15

Timeouts

Connection, read, and deadline timeouts prevent slow or hung downstream services from blocking your threads forever. Setting them correctly is one of the most impactful reliability decisions in any service.

Resilience
Concept 16

Optimistic locking (version fields, compare-and-swap), pessimistic locking (SELECT FOR UPDATE), and MVCC — the mechanisms that prevent lost updates when multiple clients write the same record simultaneously.
