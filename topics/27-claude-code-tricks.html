27 Tricks to Level Up Claude Code in 16 Minutes

27 concrete tricks for getting dramatically more out of Claude Code — from session management and CLAUDE.md patterns to keyboard shortcuts, multi-file workflows, cost optimisation, and the hooks/skills/MCP patterns most developers miss. Each trick has a one-paragraph explanation and, where relevant, the exact code or config.

Format27 actionable tricks grouped by category
Time~35 seconds per trick to read and apply
Category A — Session & Context (tricks 1–6)
#1

Start every session with a grounding prompt

Type "Read CLAUDE.md and tell me the current state of the project" before doing anything. This forces Claude to orient before it starts generating. Saves you from 30 minutes of drift.

#2

Use /clear to reset when context degrades

When Claude starts contradicting itself or misunderstanding your codebase, /clear resets the conversation. Then re-ground with a targeted context prompt instead of retyping everything.

#3

Give Claude a map before asking it to navigate

"Before you start, read these three files: src/lib/logger.ts, src/db/index.ts, and DESIGN.md." Explicit context loading beats hoping Claude will find the right files on its own.

#4

Name your constraints upfront, not after the fact

Put your hard constraints in the first message: "implement this without adding new dependencies, without modifying the public API, and with the existing error handling pattern." Correcting after generation wastes tokens.

#5

Use "continue" to extend long generations

If Claude stops mid-generation, type "continue" — Claude will resume exactly where it left off. More reliable than asking it to regenerate the whole response.

#6

Keep a NOTES.md in your project root

Scratch pad for mid-session decisions, URLs, and context you want Claude to have but don't want in CLAUDE.md permanently. Start sessions with "read NOTES.md if it exists."

Category B — CLAUDE.md Patterns (tricks 7–12)
#7

Put your three most-violated rules at the top

Claude weights the beginning of the system prompt more heavily. Don't bury your most important constraints. "NEVER use console.log" should be line 2, not line 78.

#8

Use imperative stops, not advisory should

"STOP if tests fail" is followed more reliably than "you should run tests." "NEVER import directly from node_modules" beats "try to use the barrel export." The word STOP is Claude Code's hard brake.

#9

CLAUDE.local.md for personal preferences

Add CLAUDE.local.md to .gitignore. Put your personal style preferences there (verbose comments, specific editor format, etc). Shared CLAUDE.md stays team-neutral. Personal one stays out of the repo.

#10

Sub-directory CLAUDE.md files for local context

Claude Code reads CLAUDE.md from the root AND from the current sub-directory. Put API-specific rules in src/api/CLAUDE.md and database-specific rules in src/db/CLAUDE.md. They only activate in their directory.

#11

Pointer beats content — link, don't paste

"See DESIGN.md for architectural decisions" beats pasting 200 lines of DESIGN.md into CLAUDE.md. Pointers keep CLAUDE.md short; Claude reads the linked file when it needs the context.

#12

Add "known failure patterns" section

Keep a running list of mistakes Claude has made on your project: "Never import from '../utils' — always use '@/utils' alias." Every time Claude makes a new repeatable mistake, add it here.

Category C — Workflow Tricks (tricks 13–18)
#13

Ask for a plan before asking for code

"Before writing any code, describe the approach you'll take and list the files you'll modify." Reviews the plan, approve it, then say "implement it." You catch architectural mistakes before they become 200 lines.

#14

One task at a time, not a feature list

"Add authentication to the API" is too large. "Add JWT verification middleware to src/middleware/auth.ts that validates the Authorization header" is one task. Smaller scopes produce better, more reviewable output.

#15

Ask Claude to write the test first

Describe the behavior you want, ask for a failing test, review the test, then ask Claude to implement the code to make it pass. This forces both you and Claude to think about behavior before implementation.

#16

Use "explain this code before changing it"

Before asking Claude to modify a file, ask "explain what this function does and what calls it." This surfaces Claude's understanding (and any misunderstandings) before they become incorrect edits.

#17

Ask Claude to list assumptions it's making

After Claude proposes an approach: "what assumptions are you making about the codebase that you haven't verified?" This surfaces the most common source of wrong implementations — unverified assumptions about types, APIs, or patterns.

#18

Diff review: "what did you change and why?"

After a large edit: "list every file you modified, what you changed in each, and the reason." This generates your PR description AND catches any accidental changes you didn't ask for.

Category D — Hooks, Skills & MCP (tricks 19–22)
#19

Pre-tool hook to guard risky operations

Add a pre-tool hook that logs every tool call to a file. Two benefits: debugging (replay what Claude did) and a review log. Takes 10 minutes to set up, saves hours when something goes wrong.

#20

Post-tool hook to run tests automatically

Run tests automatically after every file write using a post-tool hook. Claude sees the test output and can fix failures immediately, without you manually running tests after every change.

#21

Create a /review skill for consistent code review

Write a REVIEW.md skill that always performs the same checks: security (injection, auth), error handling, test coverage, API contract. Run it as /review on any diff. Consistent review every time.

#22

MCP filesystem server for project file access

Add the filesystem MCP server scoped to your project root. Claude can then read and write files as tools instead of as bash commands — more structured, more auditable, and works without shell access.

# .claude/hooks/post-write.sh — run tests after every file write
#!/bin/bash
FILE="$1"
if [[ "$FILE" == *.ts ]] || [[ "$FILE" == *.js ]]; then
  echo "Running tests for changed files..."
  npx jest --passWithNoTests --findRelatedTests "$FILE" 2>&1
fi
Category E — Cost & Quality (tricks 23–27)
#23

Use extended thinking for hard architectural questions

For architecture decisions and complex debugging: add "think through this carefully before responding" or enable extended thinking mode. The extra reasoning tokens pay for themselves in answer quality on hard problems.

#24

Haiku for cheap tasks, Sonnet for the rest

In your own API usage (not Claude Code UI): use Haiku for classification, routing, and simple extraction. Haiku is ~20x cheaper than Opus and equally good at mechanical tasks. Only use Opus for the hardest reasoning tasks.

#25

Compress your system prompt quarterly

Every token in your CLAUDE.md is paid on every session. A 500-token CLAUDE.md costs 5x more than a 100-token one at the same usage. Audit it every quarter: delete constraints Claude now follows without being told.

#26

Ask Claude to score its own confidence

"On a scale of 1-10, how confident are you that this implementation is correct? What would you check to verify?" Scores below 7 are a signal to add verification before shipping. This works better than asking "is this correct?"

#27

Build a "second opinion" pattern for critical code

For security-critical or high-risk code: start a fresh session (no prior context) and ask Claude to review the implementation critically. A fresh-context review catches things that the generating context would rationalise away.

// Trick #27 in practice: second opinion via a fresh API call
import Anthropic from '@anthropic-ai/sdk';
const client = new Anthropic();

async function secondOpinion(codeToReview, context = '') {
  // Deliberately start fresh — no prior conversation history
  const response = await client.messages.create({
    model: 'claude-sonnet-4-6',
    max_tokens: 2048,
    system: `You are a critical code reviewer. Your job is to find problems, not to approve.
Be adversarial. Look for: security vulnerabilities, missing error handling, race conditions,
incorrect logic, missing edge cases, violations of stated constraints.
Do NOT give generic "looks good" feedback. Find specific issues.`,
    messages: [{
      role: 'user',
      content: `Context: ${context}\n\nCode to review:\n\`\`\`\n${codeToReview}\n\`\`\`\n\nFind every problem you can.`,
    }],
  });
  return response.content[0].text;
}

Key Takeaway

The highest-leverage Claude Code habits: always ground the session before writing code (#1), put your hardest constraints at the top of CLAUDE.md with imperative stops (#7, #8), ask for a plan before asking for code (#13), and use a fresh-context second opinion for security-critical code (#27). The hooks and skills tricks (#19–22) pay off most on teams — consistent automated checks that don't depend on anyone remembering to run them.
