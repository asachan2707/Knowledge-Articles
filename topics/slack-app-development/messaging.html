Messaging in Slack

The full posting surface: chat.postMessage, chat.postEphemeral, updates, deletes, scheduled messages, threads, reactions, and incoming webhooks. This page covers each method, the formatting model, and the patterns that keep your bot's chatter useful.

Posting is the most common thing a Slack app does. Choose the right method for the right situation and your app feels native. Choose the wrong one and your app is a notification spam machine.

Default: use chat.postMessage with blocks for anything user-facing. Use chat.postEphemeral for confirmations to one user. Reply in-thread when you're following up on an existing message.

The core methods

  • chat.postMessage — post a new message to a channel/DM/thread.
  • chat.postEphemeral — visible to one user only; not persisted.
  • chat.update — replace an existing message (you must own it).
  • chat.delete — remove an existing message.
  • chat.scheduleMessage — post at a future time (up to 120 days out).
  • reactions.add — add an emoji reaction to a message.
  • Incoming Webhooks — single-channel one-way URL; useful for static notifications.

The formatting model

A message can carry text (plain text or mrkdwn) and/or blocks (Block Kit JSON). Best practice: always send a text fallback (used by notifications, accessibility, search) and rich blocks for the in-Slack render. 

await client.chat.postMessage({
  channel: 'C1',
  text: 'Build #421 succeeded.',  // notification + accessibility fallback
  blocks: [
    {
      type: 'section',
      text: { type: 'mrkdwn', text: '*Build #421 succeeded* · main · 2m 14s' },
    },
    {
      type: 'actions',
      elements: [
        { type: 'button', text: { type: 'plain_text', text: 'Open' }, url: 'https://ci/...' },
      ],
    },
  ],
});

mrkdwn vs. plain_text

  • mrkdwn — Slack's flavored markdown. Supports *bold*, _italic_, ~strike~, `code`, > quote, <https://...|label>, <@U123> mentions, <#C1|name> channels.
  • plain_text — no formatting. Required for headers, button labels, input placeholders, modal titles.

Threads

To post a reply in a thread, set thread_ts to the parent message's ts. To "broadcast" the reply back into the channel, also set reply_broadcast: true (use sparingly). 

// Reply in a thread
await client.chat.postMessage({
  channel: parent.channel,
  thread_ts: parent.ts,
  text: 'Done!',
});

Updating & deleting

chat.update needs the original channel + ts. Common pattern: post a "running…" message, then update it with the result, then update again with a final state — never spawning a new message per state change. 

const { ts } = await client.chat.postMessage({ channel, text: 'Running…' });
await doWork();
await client.chat.update({ channel, ts, text: 'Done ✅' });

Scheduling, reminders, files

  • Scheduled messages: chat.scheduleMessage with a future post_at Unix timestamp. List with chat.scheduledMessages.list; delete with chat.deleteScheduledMessage.
  • Files: use files.uploadV2 for file uploads (the v1 endpoint is deprecated).
  • Reactions: reactions.add / reactions.remove — useful for "I saw it" acknowledgements without a reply.

Rate limits

Slack rate-limits per-method per-team. chat.postMessage is roughly 1 message per channel per second sustained, with bursting allowed. The @slack/web-api client handles 429s and retries with backoff if you set retryConfig: { retries: 3 }.

Anti-patterns to avoid

  • Posting one message per channel in a loop (use scheduled or bundled messages).
  • Updating a message every second to show progress (poll less, or use a single final update).
  • Sending public messages for personal confirmations.
  • Skipping the text fallback when using blocks.

Key takeaways

  • postMessage + blocks for rich UI; postEphemeral for personal confirmations.
  • Always send a text fallback alongside blocks.
  • Threads keep channels tidy — use thread_ts.
  • Update existing messages instead of spawning new ones for state changes.
