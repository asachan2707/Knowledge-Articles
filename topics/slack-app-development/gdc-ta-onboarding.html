Slack GDC TA Team Onboarding Guide

The internal team-specific overlay on top of the public Slack docs. Covers repo conventions, deployment patterns, on-call rotation, sandbox workspaces, and the team's standard Bolt template. Required reading before your first PR.

TrackSlack core · Internal
GoalMatch team conventions on day one
Time~60 min read
AudienceNew hires & rotators on the GDC TA team

The public Slack docs tell you what the platform does. The GDC TA onboarding doc tells you how this team uses it: which repo a new app goes in, which template to fork, which secrets manager holds your tokens, how to get a sandbox workspace, and how deployments + on-call work.

Why it deserves its own page: shortcuts you'll only learn from teammates. Skipping this doc costs you a week of "wait, where does config go?" questions in PR review.

What to look for as you read

  • Repo conventions — which monorepo or per-app repo, how features are structured, naming rules.
  • Standard Bolt template — the team's blessed starting point with auth, logging, error handling already wired.
  • Secrets management — where tokens live (Secrets Manager / SSM / vault), how to request access.
  • Sandbox workspace — how to get one for development; never test against the prod workspace.
  • OAuth client credentials — how to get the team's shared client_id/secret for distributed apps.
  • Deployment — CI/CD pipeline, environments (dev/stage/prod), promotion rules.
  • Observability — log destination (Datadog/Splunk), dashboards, alert routes.
  • On-call — rotation, escalation, runbooks for common failures.
  • Code review expectations — required reviewers, lint/test gates, PR template.

Questions to ask in your first 1:1

  1. Which Slack workspace is the official sandbox for our team's apps?
  2. Where do I request access to the team's secrets manager?
  3. What's the canonical Bolt template repo?
  4. Who is the on-call this week and what's the escalation path?
  5. Are there any apps currently in flight I shouldn't touch yet?

Key takeaways

  • Read the doc end-to-end before opening your first PR.
  • Note any conflict between team conventions and public docs — the team conventions win in this codebase.
  • Bookmark the sandbox workspace, secrets path, and on-call rotation page.
