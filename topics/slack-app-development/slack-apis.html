Slack APIs — Web API & Events API

Slack exposes two main API surfaces: the Web API (you call Slack) and the Events API (Slack calls you). Add Socket Mode as a third transport for the inbound side. This page covers all three and the patterns that keep them reliable.

Almost every method Bolt exposes is a thin wrapper over one of these APIs. Knowing the raw shapes — request, response, error codes — makes you faster at debugging and unblocks you when you outgrow the SDK.

Two directions: Web API (HTTP request → JSON response, ~250 methods). Events API (Slack POSTs JSON to your URL, or pushes via Socket Mode WebSocket).

Web API — anatomy

All endpoints are https://slack.com/api/<method>, take POST with application/json or application/x-www-form-urlencoded, and return { ok: boolean, error?: string, ... }. Authentication is via the Authorization: Bearer xoxb-… header. 

// Raw fetch
const res = await fetch('https://slack.com/api/chat.postMessage', {
  method: 'POST',
  headers: {
    Authorization: `Bearer ${process.env.SLACK_BOT_TOKEN}`,
    'Content-Type': 'application/json; charset=utf-8',
  },
  body: JSON.stringify({ channel: 'C1', text: 'hi' }),
});
const data = await res.json();
if (!data.ok) throw new Error(data.error);

// Or, the typed SDK client (preferred)
import { WebClient } from '@slack/web-api';
const slack = new WebClient(process.env.SLACK_BOT_TOKEN);
await slack.chat.postMessage({ channel: 'C1', text: 'hi' });

Method families to know

  • chat.* — postMessage, postEphemeral, update, delete, scheduleMessage.
  • conversations.* — list, info, history, members, replies, open, join, invite, archive.
  • users.* — info, list, lookupByEmail, profile.get/set.
  • views.* — open, push, update, publish (for App Home).
  • files.* — uploadV2, info, list, delete.
  • reactions.* — add, remove, get.
  • oauth.v2.* — access (token exchange), exchange (for refresh tokens).
  • auth.* — test (verify a token), revoke.
  • admin.* — Enterprise Grid admin actions (separate scope set).

Pagination — the cursor pattern

List endpoints return a response_metadata.next_cursor field. Pass it back as cursor until it's empty. Always set limit (max 200) explicitly. 

let cursor;
const channels = [];
do {
  const page = await slack.conversations.list({ limit: 200, cursor, exclude_archived: true });
  channels.push(...page.channels);
  cursor = page.response_metadata?.next_cursor;
} while (cursor);

// Or use the SDK's helper:
for await (const page of slack.paginate('conversations.list', { limit: 200 })) {
  channels.push(...page.channels);
}

Events API — what you receive

Events are bot-scope events your app subscribed to in the manifest. Each delivery is a JSON POST to your event URL containing: 

{
  "token": "...",                 // verification token (deprecated, prefer signing)
  "team_id": "T1",
  "api_app_id": "A1",
  "event": {
    "type": "message",
    "user": "U1",
    "text": "hello",
    "ts": "1715000000.000100",
    "channel": "C1",
    "event_ts": "1715000000.000100"
  },
  "type": "event_callback",
  "event_id": "Ev123",
  "event_time": 1715000000
}

You must respond with 200 OK within 3 seconds — same rule as interactivity. Slack retries failed deliveries with backoff, so your handler must be idempotent.

URL verification challenge

When you set or change your event URL, Slack sends a one-time { "type": "url_verification", "challenge": "…" }. You must echo the challenge back. Bolt handles this for you; for a custom receiver you must do it.

Socket Mode — WebSocket transport

Instead of receiving HTTP POSTs, your app opens a WebSocket to Slack and receives events as frames. No public URL required — perfect for local dev, behind-the-firewall apps, and small internal tools. Requires an app-level token (xapp-…) with the connections:write scope. 

import { App } from '@slack/bolt';

const app = new App({
  token:    process.env.SLACK_BOT_TOKEN,
  appToken: process.env.SLACK_APP_TOKEN,   // xapp-...
  socketMode: true,
});

await app.start();
console.log('Socket Mode connected');

Errors & rate limits

  • Common errors: not_authed, missing_scope, channel_not_found, not_in_channel, cant_invite_self, rate_limited.
  • Rate limits: per-method per-team. Tier 1 (1/min), Tier 2 (20/min), Tier 3 (50+/min), Tier 4 (100+/min). chat.postMessage is roughly 1/channel/sec.
  • 429 handling: the WebClient retries by default. Customize with retryConfig.

Idempotency — events fire twice

Slack will retry events if you don't ack within 3 seconds. Some events (especially message) can also fan out unexpectedly. Always dedupe by event_id (cache for 24 hours in Redis or Mongo with TTL).

Key takeaways

  • Web API = you → Slack. Events API / Socket Mode = Slack → you.
  • Use the WebClient; it handles retries, pagination helpers, types.
  • Always ack events fast and dedupe by event_id.
  • Socket Mode for dev / firewalled; HTTP for production distributed apps.
