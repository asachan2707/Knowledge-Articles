Redis — In-Memory Data Structures

Redis is a single-threaded in-memory data structure server. The Net Ninja playlist walks strings, hashes, lists, sets, sorted sets, streams, and pub/sub. This page maps each one to a concrete Slack-app pattern.

Redis isn't just a cache — it's a Swiss army datastore. Once you understand sorted sets and streams, half of what you'd otherwise build with a custom DB schema becomes a one-line Redis command.

Mental model: Redis is a single-threaded server (commands run atomically, in order) that holds named typed values in memory. Keys are strings; values can be strings, hashes, lists, sets, sorted sets, streams, bitmaps, or HyperLogLogs.

The data types you'll use

  • String — key/value cache, counters (INCR), TTL-flagged tokens.
  • Hash — small object stored at one key (HSET user:U1 name "Amy" tz "UTC").
  • List — FIFO/LIFO queue (LPUSH/RPOP).
  • Set — uniqueness, dedup, set algebra.
  • Sorted Set (ZSET) — leaderboards, time-windowed counters, priority queues.
  • Stream — append-only log with consumer groups (Kafka-lite).
  • Bitmap — boolean flags by user ID at huge scale.
  • HyperLogLog — approximate cardinality (unique-counts) in 12KB.

Slack-app recipes

Cache users.info

// Save with 5-minute TTL
await redis.setex(`user:${id}`, 300, JSON.stringify(user));

// Read
const cached = await redis.get(`user:${id}`);
return cached ? JSON.parse(cached) : null;

Event dedup

// SETNX returns 1 if key was set, 0 if it already existed
const isNew = await redis.set(`event:${eventId}`, '1', 'NX', 'EX', 86400);
if (isNew !== 'OK') return; // already processed

Per-user rate limit (token bucket)

const key = `rate:${userId}`;
const count = await redis.incr(key);
if (count === 1) await redis.expire(key, 60); // 1-minute window
if (count > 30) throw new Error('rate_limited');

Distributed lock (one process at a time)

const lock = await redis.set('lock:standup-cron', '1', 'NX', 'EX', 30);
if (lock !== 'OK') return; // someone else has it
try { await runStandupCron(); }
finally { await redis.del('lock:standup-cron'); }

Priority job queue

// Add a job with priority = run-at timestamp
await redis.zadd('jobs', Date.now() + delayMs, JSON.stringify(job));

// Worker pops the next-due job
const [[job, ts]] = await redis.zpopmin('jobs', 1);
if (Number(ts) > Date.now()) {
  await redis.zadd('jobs', Number(ts), job); // not yet
}

Operational essentials

  • Persistence: RDB snapshots (point-in-time) + AOF (write-ahead log). Use both.
  • Eviction: set maxmemory + maxmemory-policy (usually allkeys-lru for cache).
  • Replication: primary + replicas; managed services give this out of the box.
  • Cluster: for > ~25 GB; uses key-slot hashing across nodes.
  • Pipelining: batch commands in one round-trip — huge throughput gain.
  • Lua scripts (or MULTI/EXEC) for atomic multi-command operations.

When NOT to use Redis

  • As your only durable store — use Postgres/Mongo for source-of-truth data.
  • For data that doesn't fit in memory — Redis is RAM-bound.
  • For complex queries — Redis is key/value-shaped, not relational.

Key takeaways

  • Redis is the default Slack-app sidekick: cache, dedup, rate limit, lock, queue.
  • Always set TTLs and a maxmemory-policy.
  • Use SETNX/NX for dedup & locks; INCR+EXPIRE for rate limits.
  • Don't make Redis your source of truth — pair it with a durable DB.
