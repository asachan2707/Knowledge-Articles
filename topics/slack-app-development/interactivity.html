Interactivity in Slack

Interactivity covers everything users do with your app: slash commands, shortcuts, buttons, select menus, modals, and Home tab interactions. The 3-second ack rule and the response_url mechanism are the two ideas that govern all of it.

TrackSlack core
GoalWire up commands, buttons, modals correctly
Time~45 min read
Prereq forBlock Kit, Bolt patterns, App Home

Every interactive thing — a button click, a slash command, a modal submission — sends a payload to your app. Your app must respond within 3 seconds (with an empty ack if needed) and then optionally do follow-up work using either the Web API or the response_url Slack provides in the payload.

Two rules: 1) ack() within 3 seconds, always. 2) Use response_url for follow-up replies to slash commands and message buttons; use views.update/views.push for modal flows.

The interactive surface

  • Slash commands — global verbs typed in any channel (/standup).
  • Global shortcuts — invoked from the lightning-bolt menu, no channel context.
  • Message shortcuts — invoked from a specific message's "More actions" menu.
  • Block actions — buttons, select menus, datepickers in a message or App Home.
  • View submissions — modal "Submit" button.
  • View closed — modal closed without submitting.
  • Home opened — user opens your App Home tab.

The 3-second ack rule

Slack expects a 200 OK within 3 seconds. If you don't ack, the user sees an error like "/standup failed with the error: operation_timeout." Bolt's ack() sends the 200 immediately; then you do the slow work. 

app.command('/standup', async ({ command, ack, client }) => {
  await ack();                              // 1. acknowledge fast
  await someSlowWork();                     // 2. then do work (any duration)
  await client.chat.postMessage({           // 3. notify when done
    channel: command.channel_id,
    text: 'Standup created!',
  });
});

response_url — the back-channel

For slash commands and block actions, the payload includes a response_url you can POST to up to 5 times within 30 minutes. This is how you respond to a slash command without needing the bot's posting scope or a channel ID. 

// Inside a slash command handler:
await ack();

// Send an ephemeral follow-up via response_url (no chat:write needed)
await fetch(command.response_url, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    response_type: 'ephemeral',  // or 'in_channel'
    text: 'Working on it…',
  }),
});

Modals (views) — open / push / update

  • views.open — open a brand new modal in response to an interaction (needs trigger_id, valid for ~3s).
  • views.push — stack another modal on top of the current one.
  • views.update — replace the current modal's content (e.g., on validation error).
  • views.publish — render a view to App Home.
app.command('/feedback', async ({ command, ack, client }) => {
  await ack();
  await client.views.open({
    trigger_id: command.trigger_id,
    view: {
      type: 'modal',
      callback_id: 'feedback_modal',
      title: { type: 'plain_text', text: 'Send feedback' },
      submit: { type: 'plain_text', text: 'Send' },
      blocks: [
        {
          type: 'input',
          block_id: 'b_msg',
          label: { type: 'plain_text', text: 'Your message' },
          element: { type: 'plain_text_input', action_id: 'a_msg', multiline: true },
        },
      ],
    },
  });
});

app.view('feedback_modal', async ({ ack, body, view, client }) => {
  await ack();
  const message = view.state.values.b_msg.a_msg.value;
  await client.chat.postMessage({ channel: 'C_FEEDBACK', text: `From <@${body.user.id}>: ${message}` });
});

Validation in modals

To show inline errors on a modal submission, ack with { response_action: 'errors', errors: { block_id: 'message' } } instead of a bare ack(). 

app.view('feedback_modal', async ({ ack, view }) => {
  const text = view.state.values.b_msg.a_msg.value;
  if (text.length < 5) {
    return ack({
      response_action: 'errors',
      errors: { b_msg: 'Please write at least 5 characters.' },
    });
  }
  await ack();
});

Block actions — every button click

A button (or select / datepicker / checkbox) carries an action_id. Bolt routes each click to the matching app.action(id, …) handler. 

app.action('approve_button', async ({ ack, body, client }) => {
  await ack();
  await client.chat.update({
    channel: body.channel.id,
    ts: body.message.ts,
    text: 'Approved',
    blocks: [{ type: 'section', text: { type: 'mrkdwn', text: `Approved by <@${body.user.id}> ✅` } }],
  });
});

Patterns & pitfalls

  • Trigger IDs expire fast (~3s) — don't await slow work before opening a modal.
  • action_id collisions — make them unique and namespaced (review:approve).
  • Confirmation dialogs — every destructive action gets a confirm on the button.
  • Error responses — let users know via ephemeral or modal errors, never silent failure.
  • Idempotency — buttons can be double-clicked; check + dedupe by message ts + action id.

Key takeaways

  • Ack within 3 seconds. Always.
  • Use response_url for slash command follow-ups; views.* for modals.
  • Validate modal submissions with response_action: 'errors'.
  • Namespace your action_ids and confirm destructive actions.
