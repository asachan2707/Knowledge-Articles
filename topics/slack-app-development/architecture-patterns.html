Top 5 Architecture Patterns

Layered, microservices, event-driven, microkernel, and pipe-and-filter — the five patterns most teams pick from. For a Slack app, the practical mix is usually layered inside an event-driven shell.

GoalPick the right shape for the right scale

Every architecture pattern is an answer to the question "where do the seams go?" Picking the wrong shape early forces expensive rewrites; picking the wrong shape too early (overengineering) wastes the same time. This page is a quick map.

1 · Layered (n-tier)

Code organized into stacked layers — typically presentation → application → domain → data. Each layer talks only to the one below.

Pros: easy to learn, easy to test, low ceremony. Cons: can ossify into a "lasagna" with too many layers.

Slack fit: command/view handler → service → repository → DB. The default for any Bolt app under ~50k LOC.

2 · Microservices

Independent services, each owning one bounded context, communicating over HTTP/gRPC/queues.

Pros: independent deploys; scale services independently; tech freedom per service.

Cons: distributed system tax — networking, observability, deployment cost multiplied.

Slack fit: only when one feature dominates load (e.g., a heavy analytics service). Most Slack apps stay a monolith and split off a worker or two.

3 · Event-driven

Components communicate by emitting and consuming events on a broker (Kafka, SQS, Redis Streams).

Pros: loose coupling; great for async workflows; easy to add consumers.

Cons: harder to reason about end-to-end; eventual consistency surprises.

Slack fit: the Events API is event-driven. For your own internal pipelines, an event bus pays off when you need fan-out (e.g., a "PR opened" event triggering both notification and metrics).

4 · Microkernel (plugin)

A tiny core + a set of plugins that extend it. The core knows how to load plugins; plugins know nothing about each other.

Pros: ideal for products that vary per customer (dev tools, IDEs, ETL platforms).

Cons: the plugin API itself becomes the hardest thing to design.

Slack fit: rare. A customer-extensible Slack app could load per-workspace plugins this way.

5 · Pipe-and-filter

Data flows through a series of independent transformations, each one consuming from one pipe and writing to the next.

Pros: simple, composable, easy to parallelize.

Cons: not a great fit for interactive systems; better for batch / streaming pipelines.

Slack fit: a digest pipeline (fetch → enrich → format → post) is a textbook pipe-and-filter, even if it's just three function calls.

How a Slack app evolves

  1. Phase 1 (script): one file, layered loosely. 300 LOC.
  2. Phase 2 (real app): layered with feature folders. command → service → repo. 5–20k LOC.
  3. Phase 3 (heavy work): add a queue + worker process. Mostly still a monolith.
  4. Phase 4 (multi-team): split off the heaviest service (analytics, billing). Microservices show up here.

Anti-patterns

  • Microservices on day one — the network calls between services dwarf any benefit until you've earned the complexity.
  • Layer obsession — ten layers between the request and the DB. Keep it to four.
  • Event-driven without idempotency — duplicate events break the world. Always dedupe.
  • "Hexagonal-clean-DDD-everything" upfront — start simple; extract patterns when pain demands them.

Key takeaways

  • Default to layered + feature folders for a Bolt app.
  • Add an event bus only when fan-out becomes painful.
  • Defer microservices until you have measured pain that justifies them.
  • Patterns are tools to solve specific pains, not goals to achieve.
