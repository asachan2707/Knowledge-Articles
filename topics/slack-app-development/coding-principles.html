10 Coding Principles

A short, opinionated set of rules of thumb that act as code-review tie-breakers. They aren't theorems — they're heuristics that prevent the boring class of bugs and make code easy to come back to in three months.

Senior engineers don't have a magic intuition. They have a small set of internalized principles that they apply automatically. Here are the ten the video covers, with how each one shows up in a Slack-app codebase.

1 · DRY — Don't Repeat Yourself

Each piece of knowledge has one canonical home. But: beware premature DRY — two functions that look similar may have different change reasons. Wait for the third repetition.

2 · KISS — Keep It Simple, Stupid

Pick the boring solution that works. A 3-line if-statement beats a clever Strategy pattern for two cases.

3 · YAGNI — You Aren't Gonna Need It

Don't build for hypothetical future requirements. Add the abstraction when the second concrete use case shows up.

4 · Fail Fast

Validate inputs at the boundary. Throw or return early. Never let bad state propagate three functions deep before crashing. 

// good: fail fast at the boundary
app.command('/report', async ({ command, ack, client }) => {
  await ack();
  const { project, since } = parseArgs(command.text);
  if (!project) return client.chat.postEphemeral({ ..., text: 'Missing project arg' });
  if (!since)   return client.chat.postEphemeral({ ..., text: 'Missing since arg' });
  // ...rest of handler proceeds with valid inputs
});

5 · Separation of Concerns

Each module has one concern: parsing, business logic, persistence, presentation. Keep Slack payload shaping out of your business services.

6 · Principle of Least Astonishment

A function should do what its name suggests, with no surprises. save() doesn't send emails. postMessage() doesn't write to the database.

7 · Composition over Inheritance

Build behavior by combining small functions/objects, not by extending a base class through three levels. Inheritance is rigid; composition is reconfigurable.

8 · Program to an Interface

Take parameters as the smallest interface you need ({ chat: { postMessage } }), not the concrete WebClient. Easier to swap, easier to mock.

9 · Prefer Immutability

Don't mutate inputs. Use spread/structuredClone when building new objects. Block Kit composition is a textbook case — never mutate a shared blocks array. 

// bad: mutates the shared template
const message = baseTemplate;
message.blocks.push(extraBlock);

// good: creates a new object
const message = { ...baseTemplate, blocks: [...baseTemplate.blocks, extraBlock] };

10 · Explicit over Implicit

Magic strings, hidden globals, "the framework will figure it out" — all sources of bugs that are hard to find. Pass dependencies in. Name things.

Code review checklist (use this on yourself)

  • Did I fail fast on bad inputs?
  • Is there a third repetition that justifies extracting a function?
  • Could a junior engineer read this in 6 months?
  • Did I add an abstraction I don't have two real callers for?
  • Did I leave a TODO in production code? (Make it a ticket instead.)

Key takeaways

  • These are tie-breakers, not commandments. Apply them after the code works.
  • Fail fast + explicit + immutable removes 80% of "weird Slack bug" tickets.
  • Wait for the third repetition before DRYing — the wrong abstraction is worse than duplication.
