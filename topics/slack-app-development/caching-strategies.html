Caching Strategies & Pitfalls

Cache placement, eviction policies, and four classic patterns — plus the pitfalls that make caches dangerous: stale data, thundering herds, and cache stampedes. For Slack apps, caching turns slow Web API workflows into instant ones.

A cache is just a fast copy of slow data. The hard part isn't the copy — it's deciding when the copy is wrong and what to do about it.

Default for Slack apps: cache-aside in Redis with explicit TTL, invalidate on writes, and never cache errors or 404s.

Where caches live

  • Browser — HTTP Cache-Control headers; doesn't apply to Slack server apps.
  • CDN / edge — for HTML/static assets; not for Slack API responses.
  • Application memory — fastest; lost on restart; not shared across replicas.
  • Distributed cache (Redis, Memcached) — shared across processes; survives a restart.
  • Database — query results / materialized views; for expensive aggregates.

The four classic patterns

Cache-aside (lazy loading)

Application reads the cache. On miss, it loads from the source, writes to the cache, returns the value. Most common, simplest, and the default. 

async function getUser(id) {
  const cached = await redis.get(`user:${id}`);
  if (cached) return JSON.parse(cached);
  const user = (await slack.users.info({ user: id })).user;
  await redis.setex(`user:${id}`, 300, JSON.stringify(user));   // 5-min TTL
  return user;
}

Read-through

Application reads only from the cache. The cache itself loads on miss. Requires a cache-library that supports it (e.g., a wrapped client). Simpler app code; cache becomes a smarter layer.

Write-through

Every write updates both cache and source synchronously. Cache stays consistent; writes are slower.

Write-behind (write-back)

App writes to cache; cache writes to source asynchronously. Fast writes; risk of data loss if the cache crashes before flush. Rarely used in Slack apps.

Eviction policies

  • TTL — entry expires after N seconds. Always set one in production.
  • LRU — evict least recently used when full. Default for Redis when configured.
  • LFU — evict least frequently used. Better for items that are accessed irregularly but matter a lot.
  • FIFO — evict in insertion order. Rare.

Pitfalls — and how to avoid them

Stale data

The cache shows a value the source has changed. Fix: TTL short enough that staleness is tolerable, or invalidate on write.

Thundering herd

A popular cache key expires; 100 requests miss simultaneously and all hit the source. Fix: probabilistic early refresh, or a single-flight lock.

Cache stampede

Same as thundering herd but worse — multiple keys expire together (e.g., on full restart). Fix: jittered TTLs (300 + random(0..60)).

Cache penetration

Repeated requests for keys that don't exist hit the source every time. Fix: cache the "not found" with a short TTL, or use a Bloom filter.

Caching errors

Caching a 500 means every subsequent request gets the same 500 until TTL. Fix: only cache successful responses.

Inconsistent on writes

You updated the source but forgot the cache. Fix: invalidate (delete the cache key) on write, then let the next read repopulate.

Slack-app caches that matter

  • users.info — cache for 5–15 min. Profile rarely changes.
  • conversations.list — cache for 1–5 min when you only need the channel set.
  • Workspace settings — cache forever; invalidate on write from your settings UI.
  • Rate-limit token buckets — Redis INCR with TTL.
  • Event dedup — Redis SETNX with 24h TTL keyed by event_id.

Key takeaways

  • Cache-aside + TTL + invalidate-on-write covers 90% of needs.
  • Never cache errors. Sometimes cache "not found" with a short TTL.
  • Jitter your TTLs to avoid stampedes.
  • Slack rate limits make caching users.info and channel lists almost mandatory.
