AWS & Cloud Computing for Slack Apps

The AWS surface every backend engineer is expected to know — compute, storage, databases, networking, IAM, observability — and the typical "first deploy" architectures for a Bolt for JS Slack app on AWS.

GoalPick a sane first-deploy architecture

You don't need to know all 200+ AWS services. You need a working map of the dozen or so that come up in real architecture discussions, and a default deploy shape for a Slack app on AWS.

The dozen services to know

Compute

  • EC2 — virtual machines. Use when you need full control or run heavy stateful workloads.
  • ECS / Fargate — managed containers. Fargate runs containers without you running EC2.
  • Lambda — serverless functions. Pay per ms; cold-starts; great for low-traffic apps.
  • EKS — managed Kubernetes. Only when you have real k8s expertise and need it.

Storage

  • S3 — object storage. For files, backups, logs, static sites.
  • EBS — block storage attached to EC2.
  • EFS — shared file system; rarely needed for Slack apps.

Databases

  • RDS (Postgres / MySQL) — managed relational DB. Default for installation store + audit log.
  • DynamoDB — managed NoSQL key-value; pay-per-request mode is great for low traffic.
  • ElastiCache (Redis) — managed Redis. Default for caching, dedup, locks.
  • OpenSearch — full-text and analytics; only for search-heavy features.

Networking

  • VPC — private network for your resources.
  • ALB / NLB — load balancers in front of ECS / EC2.
  • Route 53 — DNS.
  • CloudFront — CDN; only if you serve static assets at scale.
  • API Gateway — front door for Lambda / HTTP.

Identity, secrets, observability

  • IAM — roles & policies. Principle: least privilege.
  • Secrets Manager / SSM Parameter Store — store Slack tokens, signing secrets, DB credentials. Never .env on a server.
  • CloudWatch — logs, metrics, alarms.
  • X-Ray — distributed tracing.

Three first-deploy shapes for a Bolt app

Shape A — ECS Fargate (recommended default)

Internet → Route 53 → ALB → ECS Fargate (2 Bolt containers)
                                  ├─ RDS Postgres (installations, audit)
                                  ├─ ElastiCache Redis (cache, dedup, locks)
                                  └─ Secrets Manager (tokens, signing secret)

Long-lived processes, easy logging, simple model. Best if your app is HTTP-based and modestly active.

Shape B — Lambda + API Gateway (event-only)

Slack → API Gateway → Lambda (single Bolt handler, AwsLambdaReceiver)
                            ├─ DynamoDB (installations)
                            └─ Secrets Manager

No idle cost; cold-starts can hurt the 3-second ack. Good for low-traffic internal apps.

Shape C — Socket Mode on Fargate (no public URL)

ECS Fargate (1+ Bolt container, Socket Mode)
              ├─ RDS Postgres
              ├─ ElastiCache Redis
              └─ Secrets Manager

No ALB, no public ingress. Great for behind-the-firewall apps and dev sandboxes.

IAM in one screen

  • Identity — users, roles, federated identities.
  • Policy — JSON document granting actions on resources.
  • Trust policy — who can assume a role.
  • Principle of least privilege — start with no permissions; grant exactly what the app needs.

Your ECS task should run with an IAM Task Role that grants only: read on the specific Secrets Manager entries, write to its CloudWatch log group, and connect to its RDS / ElastiCache. Never use long-lived AWS access keys in env vars.

Cost mental model

  • Fargate (2 × 0.5 vCPU 1 GB): ~$30/mo.
  • RDS Postgres (db.t4g.micro): ~$15/mo.
  • ElastiCache Redis (cache.t4g.micro): ~$13/mo.
  • ALB: ~$20/mo.
  • Secrets Manager: $0.40 / secret / mo.

A small Slack app running on AWS is realistically < $100/mo until it grows.

Key takeaways

  • ECS Fargate + RDS + ElastiCache is the safe default for an HTTP Bolt app.
  • Lambda only if your app is genuinely event-driven and tolerates cold-starts.
  • IAM Task Roles + Secrets Manager — never long-lived keys, never .env in prod.
  • You don't need 200 services; you need 12 used well.
