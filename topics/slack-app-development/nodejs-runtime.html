Node.js Runtime

Node.js is the JavaScript runtime your Slack app runs on. Programming with Mosh's "Node in 1 Hour" covers the runtime, modules, npm, and the file system. This page focuses on the bits a Bolt for JS app actually uses.

Prereq forExpress, Bolt, deployment

Node.js is V8 (the JavaScript engine inside Chrome) repackaged to run on a server. It gives you a non-blocking I/O model, a module system, a package manager (npm), and a standard library for files, networking, crypto, and processes. A Bolt Slack app is just a Node process that listens for HTTP webhooks (or holds a WebSocket open in Socket Mode).

Mental model: Node = single-threaded event loop + libuv for I/O + V8 for executing your JS. Long synchronous work blocks everything. Push CPU-heavy work to worker threads or background queues.

Core modules every Slack app touches

  • http / https — under the hood of Express and Bolt's HTTP receiver.
  • process — environment variables (process.env.SLACK_BOT_TOKEN), exit codes, signals.
  • fs / fs/promises — reading config, writing logs.
  • crypto — verifying Slack request signatures (HMAC-SHA256).
  • events — the EventEmitter pattern Bolt builds on.
  • util — promisify, inspect for debugging.

npm essentials

npm (Node Package Manager) is how you install dependencies and run scripts. The two files you'll edit constantly are package.json (declared deps + scripts) and package-lock.json (resolved versions — commit it!). 

# Init a new app
npm init -y

# Install Bolt + dotenv
npm install @slack/bolt dotenv

# Add a TypeScript dev dep
npm install -D typescript @types/node

# Run a defined script
npm run dev

A minimal package.json for a Bolt app

{
  "name": "my-slack-app",
  "version": "1.0.0",
  "type": "module",
  "scripts": {
    "dev": "node --watch src/app.js",
    "start": "node src/app.js"
  },
  "dependencies": {
    "@slack/bolt": "^3.21.0",
    "dotenv": "^16.4.0"
  }
}

The event loop (the part Slack devs must know)

Node's event loop has phases: timers → pending callbacks → poll (I/O) → check (setImmediate) → close. Microtasks (resolved promises) run between every phase. The practical rule for Slack: if a function takes more than ~50ms of pure CPU work, the event loop stops processing other requests during that time — including your ack().

Heavy CPU work (PDF generation, image processing, big regexes) belongs in:

  • A background queue (Bull / BullMQ on Redis, or SQS) consumed by a worker.
  • A worker thread (node:worker_threads) for in-process parallelism.
  • A separate microservice when it's hot enough to deserve its own scaling.

Modules: CommonJS vs. ESM

Older Node code uses require()/module.exports (CommonJS). New code uses import/export (ESM) with "type": "module" in package.json and .js file extensions inside imports. Bolt for JS supports both; pick one and stick with it.

Environment & secrets

A Slack app has at least four secrets: SLACK_BOT_TOKEN, SLACK_SIGNING_SECRET, SLACK_CLIENT_ID, SLACK_CLIENT_SECRET (plus an app-level token for Socket Mode). Locally, store them in a .env file and load with dotenv. In production, use AWS Secrets Manager / SSM, never .env on a server. 

import 'dotenv/config';
import { App } from '@slack/bolt';

const app = new App({
  token: process.env.SLACK_BOT_TOKEN,
  signingSecret: process.env.SLACK_SIGNING_SECRET,
  port: Number(process.env.PORT) || 3000,
});

await app.start();
console.log('⚡ Bolt app is running');

Practice plan

  1. Build a tiny HTTP server using only the built-in http module.
  2. Refactor it to Express (next prereq) and feel the difference.
  3. Write a script that loads a .env file and prints a redacted version.
  4. Wrap a CPU-heavy function in a worker thread and prove the event loop stays responsive.

Key takeaways

  • Node = single-threaded event loop. Don't block it.
  • npm scripts are your dev/test/build/deploy entry points.
  • ESM modules + strict TypeScript is the modern default.
  • Secrets live in env vars locally and a real secrets manager in production.
