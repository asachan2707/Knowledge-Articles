SOLID Principles

The five SOLID principles — Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion — are the language we use to describe well-structured object code. They show up the moment your Slack app has more than one command.

SOLID isn't a set of rules to grade code against. It's a vocabulary that lets reviewers say "this class has two reasons to change" or "this hard-codes the dependency" instead of gesturing at a pile of code and saying "this feels off."

Practical use: when reviewing your own Slack app PR, ask "what would I need to change here to add a second slash command?" If the answer is "rewrite half the module," SOLID is the toolkit for fixing that.

S — Single Responsibility

A class/module should have one reason to change.

Bad: a handler that does everything

app.command('/standup', async ({ command, ack, client }) => {
  await ack();
  // 1. parse input
  const project = command.text.split(' ')[0];
  // 2. fetch data
  const items = await fetch(`https://jira/api/${project}`).then(r => r.json());
  // 3. render UI
  const blocks = items.map(i => ({ type: 'section', text: { type: 'mrkdwn', text: i.title } }));
  // 4. post
  await client.chat.postMessage({ channel: command.channel_id, blocks });
});

Better: split responsibilities

// 1. parser.ts
export const parseStandupArgs = (text) => ({ project: text.split(' ')[0] });

// 2. jira-service.ts
export const fetchStandupItems = (project) => fetch(`https://jira/api/${project}`).then(r => r.json());

// 3. blocks.ts
export const standupBlocks = (items) => items.map(i => ({ type: 'section', text: { type: 'mrkdwn', text: i.title } }));

// 4. command.ts (the only Slack-aware file)
app.command('/standup', async ({ command, ack, client }) => {
  await ack();
  const { project } = parseStandupArgs(command.text);
  const items = await fetchStandupItems(project);
  await client.chat.postMessage({ channel: command.channel_id, blocks: standupBlocks(items) });
});

O — Open/Closed

Modules should be open for extension, closed for modification.

For a Slack app, this often means: instead of growing one big handleAction switch statement, register many small app.action(id, handler) entries. Adding a new button means adding a new file, not editing the dispatcher.

L — Liskov Substitution

Subtypes must be usable wherever their base type is expected, without surprising the caller.

The classic Slack-app sin: an InstallationStore implementation that throws on delete "because we never delete." Now no caller can rely on the interface. Either implement it correctly or document the limitation in the type.

I — Interface Segregation

Many small interfaces beat one fat interface. Don't force a class to implement methods it doesn't need.

Example: instead of one SlackClient that has every Web API method, your service can take just a { chat: { postMessage } } shape. Easy to mock in tests, easy to replace.

D — Dependency Inversion

High-level modules should not depend on low-level modules. Both should depend on abstractions.

In a Slack app: pass the Web API client into your service rather than importing the global one. Same for the database. Your service no longer "knows" about Slack or Mongo — it just uses the interfaces it was handed. 

// service depends on an interface, not a concrete client
export class StandupService {
  constructor(private slack: { chat: { postMessage: Function } },
              private repo: { save: Function }) {}
  async run(channel, items) {
    await this.repo.save({ channel, items });
    await this.slack.chat.postMessage({ channel, text: '...' });
  }
}

// in tests, pass mocks. in app, pass real clients. no code change in the service.

SOLID smell test

Run through these questions before merging:

  • If I add a second feature, do I edit existing code or just add new files? (O)
  • Does this module have more than one reason to change? (S)
  • Can I unit-test this without instantiating Slack/DB clients? (D)
  • Does any subclass make the parent's contract weaker? (L)
  • Does any consumer have to implement methods it doesn't need? (I)

Key takeaways

  • SOLID is a code-review vocabulary, not a grade.
  • Pass dependencies in (DI) — your services become unit-testable for free.
  • Many small action handlers > one big dispatcher (OCP).
  • Keep Slack-aware code in command/view/event files; pure logic everywhere else.
