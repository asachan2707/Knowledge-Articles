SQL Fundamentals

Even when your Slack app starts on Mongo, SQL eventually shows up — for installation history, audit trails, billing records, and analytics. The full SQL playlist covers schema design, queries, joins, indexes, and transactions. This page is the senior cheat sheet over it.

SQL is two skills: writing queries (SELECT, joins, aggregates) and designing schemas (normalization, keys, indexes). Both matter equally — a beautiful schema with bad queries is slow, and a great query against a bad schema is unmaintainable.

Where SQL fits in a Slack app: Postgres is the safe default for the installation store, audit log, billing, and any data that must survive a node restart with zero loss. Mongo for variable-shape state, Postgres for relational truth.

The query toolbox

-- 1. SELECT + WHERE
SELECT id, team_id, scopes
FROM   installations
WHERE  enterprise_id IS NULL
  AND  installed_at > NOW() - INTERVAL '7 days';

-- 2. JOIN
SELECT i.team_id, COUNT(e.id) AS event_count
FROM   installations i
LEFT JOIN events e ON e.team_id = i.team_id
GROUP BY i.team_id
ORDER BY event_count DESC;

-- 3. Window function
SELECT team_id,
       installed_at,
       ROW_NUMBER() OVER (PARTITION BY team_id ORDER BY installed_at DESC) AS rn
FROM   installations;

-- 4. CTE
WITH active AS (
  SELECT team_id FROM installations WHERE uninstalled_at IS NULL
)
SELECT * FROM active LIMIT 100;

Joins — the four shapes

  • INNER JOIN — rows that match in both tables.
  • LEFT JOIN — every row from the left, NULLs where the right is missing.
  • RIGHT JOIN — same, mirrored.
  • FULL OUTER JOIN — every row from both sides.

90% of the time you want INNER or LEFT. The trap: LEFT JOIN ... WHERE right.col = ... silently turns into an INNER join. Push the predicate into the ON clause to keep the LEFT semantics.

Schema design checklist

  1. Normalize first — third normal form by default. Denormalize only with a measured reason.
  2. Pick keys deliberately — surrogate BIGSERIAL/UUID primary keys; natural keys as UNIQUE indexes.
  3. Foreign keys with ON DELETE — pick CASCADE / SET NULL / RESTRICT consciously.
  4. Constraints over comments — NOT NULL, CHECK, UNIQUE beat documentation.
  5. Index every JOIN and WHERE column — and verify with EXPLAIN ANALYZE.

Indexes that matter

  • B-tree — default; equality + range.
  • Composite — leftmost-prefix matters: index on (team_id, ts) serves WHERE team_id = ? and WHERE team_id = ? AND ts > ? but not WHERE ts > ? alone.
  • Partial — index only the rows you query: CREATE INDEX ... WHERE uninstalled_at IS NULL.
  • GIN/GiST — full-text, JSONB containment, ranges.

Transactions & isolation

Wrap multi-statement writes in a transaction. Default isolation in Postgres is READ COMMITTED; for "balance" / counter logic, use REPEATABLE READ or SERIALIZABLE. For bookkeeping, prefer UPDATE ... RETURNING over read-then-write so the database does the locking. 

BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;

Reading EXPLAIN

The single most useful skill: read a query plan. Look for:

  • Sequential Scan on a big table = missing index.
  • Hash Join on huge inputs with no index = missing FK index.
  • Sort with disk = work_mem too small or missing covering index.
EXPLAIN (ANALYZE, BUFFERS) 
SELECT * FROM events WHERE team_id = 'T1' ORDER BY ts DESC LIMIT 50;

Slack app data that belongs in SQL

  • OAuth installation history (one row per install/uninstall, never overwritten).
  • Audit log of admin actions and event-API deliveries (with retention TTL).
  • Billing / metering data — must be transactional and exact.
  • Workspace ↔ user ↔ role relationships in B2B apps.

Key takeaways

  • Postgres is the safe default for any Slack data that must be exact and durable.
  • Index every JOIN/WHERE column; read EXPLAIN before you "optimize" anything.
  • Use transactions for multi-row writes; pick the right isolation level deliberately.
  • Constraints in the database beat comments in the codebase.
