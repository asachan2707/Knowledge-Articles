Reference Example App

A working example app is the fastest way to internalize Bolt and the Slack platform. Read it end to end once: how it boots, how it registers handlers, how it persists installations, how it's structured. Then clone it and try changing one feature.

Prereq forBuilding your own app

Docs explain concepts; sample apps show shape. Once you've watched the prereq videos and skimmed the Slack docs, the example app is where it all comes together — and where you start to develop real intuition for the framework.

How to read it: in passes. First pass for the file structure and entry point. Second pass to trace one slash command from request to response. Third pass to trace the OAuth install flow. After three passes you'll have a real mental model.

Pass 1 — file structure

Open the repo and answer:

  • Where is the entry point (app.ts, index.ts)?
  • Where are config & secrets read from?
  • How are features organized — flat or by feature folder?
  • Where is the installation store implementation?
  • Where is the database / Redis client initialized?
  • Where are tests?

Pass 2 — trace one slash command

Pick a slash command in the app. Follow:

  1. Where is app.command(...) registered? (probably in a feature module)
  2. Which middleware runs before it? (logging, auth, rate-limit)
  3. What does the handler call? (a service function, the Web API client, a modal open)
  4. If it opens a modal, where is the corresponding app.view(...)?
  5. Where does the result get persisted, if anywhere?

Pass 3 — trace the OAuth install flow

  1. What URL does a user hit to start install? (/slack/install or similar)
  2. Who generates the install URL? (Bolt's InstallProvider)
  3. What scopes are requested?
  4. What happens at the redirect URI? (storeInstallation is called)
  5. Where do the tokens land? (DB column / collection — encrypted?)
  6. Where is fetchInstallation used at request time?

Modify one feature

The fastest learning step: change something small.

  • Add a new field to a modal and surface it in the resulting message.
  • Add a confirmation dialog to a destructive button.
  • Replace one Web API call with a slightly different one (e.g., post to thread instead of channel).
  • Add a unit test for one handler using jest.fn() for ack + client.

Notes worth taking

  • Naming conventions for action_id, callback_id, block_id.
  • How errors are caught and reported (Sentry? app.error? per-handler try/catch?).
  • Where retries / idempotency keys are tracked (Redis? a DB table?).
  • Logging format (JSON? plain text? PII redaction?).

Key takeaways

  • Reading is not understanding — change something to actually learn it.
  • Trace one slash command and the OAuth flow end-to-end before you build your own.
  • Capture conventions (action_id format, error handling) — your team's app will follow them.
