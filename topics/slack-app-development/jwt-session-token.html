JWT & Session vs. Token Auth

JWTs are signed, self-contained tokens used for stateless authentication. Sessions are server-side state pointed at by a cookie. Two videos cover both; this page focuses on which to pick and why — including for Slack-app companion dashboards.

Slack apps usually have a small companion web surface — an admin dashboard, a privacy page, maybe a settings UI. That surface needs its own auth. The choice between sessions and JWTs shapes everything from logout behavior to multi-instance scaling.

One-line rule: sessions for first-party browser apps where you control the backend; JWT for SPAs and service-to-service. Slack itself uses neither for its own webhook verification — it uses HMAC over the raw body.

What a JWT actually is

Three base64url-encoded parts, dot-separated: 

HEADER       . PAYLOAD                            . SIGNATURE
{"alg":"HS256","typ":"JWT"}
              {"sub":"U1","exp":1715000000,"iat":1714996400,"role":"admin"}
                                                  HMAC_SHA256(base64(header) + "." + base64(payload), secret)

Anyone can read the payload (it's just base64). The signature stops anyone without the secret from changing it. Never put secrets in the payload — it's not encrypted.

Sessions — the classic model

The server creates a session row (in Redis or a DB), sends back a cookie with the session ID. On every request, the server looks up the session by ID.

Pros:

  • Easy revocation — delete the row.
  • Small cookie payload.
  • Server can store arbitrary state per session.

Cons:

  • Requires a store reachable from every request handler.
  • Cross-domain (e.g., SPA → API on different host) is harder; needs CORS + cookie tricks.

JWT — the stateless model

The server signs a JWT containing claims (user id, expiry, role). The client stores it (usually in memory, or an httpOnly cookie). On every request, the server verifies the signature and reads the claims — no DB lookup needed.

Pros:

  • Stateless — any service that knows the secret can verify.
  • Great for service-to-service and SPAs across domains.

Cons:

  • Hard to revoke before expiry (need a denylist or short TTL + refresh tokens).
  • Bigger payload sent with every request.
  • Easy to misuse (storing in localStorage = XSS risk).

JWT pitfalls (the famous ones)

  • Trusting alg: never accept alg: "none"; pin the expected alg in your verifier.
  • HS256 secret leaks: shared secret = anyone with it can mint tokens. Use RS256/ES256 with public-key verification when possible.
  • No expiry: always set exp. 15 min for access tokens; refresh tokens for longer sessions.
  • Storing in localStorage: any XSS can steal it. Prefer httpOnly cookies with SameSite=Strict.
  • Treating JWT as encryption: JWS = signed, not encrypted. Use JWE if you need encryption.

What Slack uses

Slack does not use JWTs for webhook auth. Each request from Slack carries X-Slack-Signature, an HMAC-SHA256 of the raw body + timestamp. You verify with your signing secret. This is simpler and avoids JWT pitfalls.

For Slack OAuth, the result of installation is a token (xoxb-...), which is opaque to you — Slack doesn't expose the structure. You store it and send it as Authorization: Bearer xoxb-... on Web API calls.

Recipe — companion dashboard for a Slack app

  1. "Sign in with Slack" via OAuth user-scope. Get the user's Slack ID + workspace.
  2. Issue a short-lived JWT (15 min) with { sub: userId, team: teamId, role } + a refresh token (7 days, stored in httpOnly cookie).
  3. SPA puts the JWT in an Authorization header on API calls.
  4. API verifies signature + expiry on every request — no DB hit.
  5. Refresh endpoint rotates the JWT using the refresh cookie.

Key takeaways

  • Sessions for classic server-rendered apps; JWT for SPAs and service-to-service.
  • Always set exp, pin the algorithm, never accept alg: none.
  • Store JWTs in httpOnly cookies, not localStorage.
  • For Slack itself, signature verification (HMAC over raw body) — not JWT.
