Authentication & OAuth

Slack auth has three pillars: OAuth 2.0 for installation, scopes for least-privilege permissions, and signed requests for verifying inbound webhooks. Get any of them wrong and your app is either insecure or broken — usually both.

TrackSlack core
GoalSecurely install, persist, and verify
Time~60 min read
Prereq forDistributed apps, multi-workspace, production

OAuth is how a workspace says "yes, this app can act on my behalf with these specific permissions." Signing secrets are how Slack proves that a webhook really came from Slack. Both are non-negotiable for a production app.

Three rules to live by: 1) Verify every inbound request signature. 2) Encrypt tokens at rest. 3) Ask for the smallest scope set that works.

  │  user approves → Slack redirects to your /oauth/callback?code=...

  │

  ▼
Slack returns { access_token: 'xoxb-...', team: { id }, scope, ... }
  │
  ▼
You persist installation to your installation store
  │
  ▼
Redirect user back into Slack (deep link to your app)

Token types & when to use each

  • Bot token (xoxb-…) — most common. Your app acts as the bot user. Use for posting messages, opening modals, listing channels the bot is in.
  • User token (xoxp-…) — your app acts as the installing user. Use only when bot scopes can't do the job (e.g., reading DMs of the user, certain admin actions).
  • App-level token (xapp-…) — used for Socket Mode and certain admin endpoints. Generated separately, never user-scoped.
  • Refresh tokens — opt-in via token rotation. Bot tokens become short-lived (12 h) and you exchange a refresh token for a new bot token before expiry.

Request signing — the mandatory check

Every request Slack sends to your app includes X-Slack-Request-Timestamp and X-Slack-Signature. You must verify the signature using your signing secret, or attackers can forge requests. 

import crypto from 'node:crypto';

function verifySlackRequest(req, signingSecret) {
  const timestamp = req.headers['x-slack-request-timestamp'];
  const signature = req.headers['x-slack-signature'];

  // 1. Reject anything older than 5 minutes (replay protection)
  if (Math.abs(Date.now() / 1000 - Number(timestamp)) > 60 * 5) return false;

  // 2. Compute the expected signature
  const baseString = `v0:${timestamp}:${req.rawBody}`;
  const hmac = crypto.createHmac('sha256', signingSecret).update(baseString).digest('hex');
  const expected = `v0=${hmac}`;

  // 3. Constant-time compare
  return crypto.timingSafeEqual(Buffer.from(expected), Buffer.from(signature));
}

Bolt does this for you. If you ever build a custom receiver, this is the code.

The installation store

For distributed apps you must persist installations. Bolt's InstallProvider accepts a custom InstallationStore with three methods: 

const installationStore = {
  storeInstallation: async (installation) => {
    // upsert keyed by enterpriseId + teamId
    await db.installations.upsert({
      enterpriseId: installation.enterprise?.id ?? null,
      teamId: installation.team?.id ?? null,
      installation: encrypt(installation),  // encrypt at rest
    });
  },
  fetchInstallation: async (query) => {
    const row = await db.installations.findOne({
      enterpriseId: query.enterpriseId ?? null,
      teamId: query.teamId ?? null,
    });
    return decrypt(row.installation);
  },
  deleteInstallation: async (query) => {
    await db.installations.delete(query);
  },
};

Scopes — declare deliberately

Slack uses granular bot scopes. Each scope grants one capability. Examples:

  • chat:write — post messages as the bot.
  • commands — receive slash command invocations.
  • app_mentions:read — see @bot mentions.
  • channels:history — read messages in public channels the bot is in.
  • users:read — read user profiles.
  • files:write — upload and manage files.

When you add a scope after install, every existing workspace must re-consent. Plan scope changes carefully.

Common auth bugs

  • Body parsed too early — Express's json() consumed the raw body and signature verification fails. Fix: parse only after Bolt, or use verify hook to capture raw body.
  • Tokens stored in plaintext — encrypt at rest. Slack tokens are credentials.
  • No replay protection — always reject requests with an old timestamp.
  • One global token in distributed apps — silently uses Workspace A's token in Workspace B. Always look up by team/enterprise.
  • Token rotation ignored — if you opt in, you must persist refresh tokens and rotate before expiry.

Key takeaways

  • OAuth v2 = code exchange → persisted installation per workspace.
  • Verify every signature. Reject old timestamps. Constant-time compare.
  • Encrypt tokens at rest. Treat them like passwords.
  • Smallest scope set that works; expand only when forced.
