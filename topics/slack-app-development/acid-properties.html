ACID Properties

Atomicity, Consistency, Isolation, Durability — the four guarantees a transactional database gives you. Knowing what each one really means is the difference between a Slack app that loses installations on a crash and one that doesn't.

GoalReason about correctness under failure

Slack tie-inInstallation store, billing, audit log

ACID is the contract a transactional database honors when you wrap operations in a transaction. Most NoSQL stores give you a weaker set of guarantees — they trade some ACID for scale, and you need to know what you're trading.

A — Atomicity

A transaction either entirely succeeds or entirely fails. No partial application, ever.

Why it matters for Slack: when a workspace installs your app, you need to insert the bot token, the user mapping, and an audit row in one transaction. Without atomicity, a crash between two writes leaves your app in a corrupted half-installed state. 

BEGIN;
INSERT INTO installations (team_id, bot_token) VALUES ('T1', 'xoxb-...');
INSERT INTO audit_log     (event, team_id)     VALUES ('install', 'T1');
COMMIT;     -- both rows land, or neither

C — Consistency

The database moves from one valid state to another. Constraints (foreign keys, CHECK, UNIQUE) hold before and after every transaction.

Why it matters for Slack: a UNIQUE constraint on (team_id, enterprise_id) stops two parallel installs from creating duplicate rows. The DB enforces it; your app doesn't have to.

I — Isolation

Concurrent transactions don't see each other's intermediate state. The four standard isolation levels:

  • READ UNCOMMITTED — sees other transactions' uncommitted writes ("dirty reads"). Almost never used.
  • READ COMMITTED — sees only committed data. Default in Postgres.
  • REPEATABLE READ — same row reads the same value within a transaction. Default in MySQL.
  • SERIALIZABLE — transactions appear to run sequentially. Strongest, slowest.

Why it matters for Slack: when two clicks of the same "Approve" button race, you need either an UPDATE … WHERE status='pending' guard or higher isolation, or one of them silently overwrites the other.

D — Durability

Once a transaction commits, the result survives crashes, power loss, restarts. Implemented by write-ahead logs flushed to disk before COMMIT returns.

Why it matters for Slack: "I installed your app" must be durable. If the install row is lost on a database restart, your event-API webhooks start failing for that workspace. Always run with WAL/fsync enabled in production; never skip fsync for "performance".

NoSQL realities

Document and KV stores often offer tunable consistency:

  • MongoDB: single-document writes are atomic; multi-document transactions exist (since 4.0) but are slower.
  • DynamoDB: single-item writes are atomic; transactions across items are explicit (TransactWriteItems).
  • Cassandra: tunable consistency per query (ONE, QUORUM, ALL).
  • Redis: commands are atomic; MULTI/EXEC + Lua for multi-step atomicity.

Practical Slack-app rules

  • OAuth installation, billing, audit log → SQL with transactions. Don't compromise.
  • Per-workspace settings, message templates → Mongo or Postgres JSONB; either works.
  • Counters and rate limits → Redis; not durable, but you don't need to recover them after a crash.
  • For racey writes (approve/reject), use UPDATE ... WHERE status='pending' and check rowcount.

Key takeaways

  • ACID is what you give up when you choose "fast" over "safe" — be conscious about it.
  • Wrap multi-row writes in transactions. Single-row writes are already atomic.
  • Use guarded UPDATEs (WHERE status='pending') to avoid race conditions on Slack actions.
  • Never disable durability ("fsync off") for production data.
