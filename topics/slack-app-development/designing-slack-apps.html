Designing Slack Apps

Slack's design guidelines push you to think of an app as a collaborator in a conversation, not a website embedded inside Slack. This page distills the heuristics into concrete rules: when to DM, when to channel-post, when to open a modal, and how to write tone that fits.

Bad Slack apps shout, interrupt, and force you into modals to do what could have been one message. Good ones blend in: they post when relevant, ephemeral when only one person should see, in-thread when the context exists, and use the App Home tab for state instead of spamming channels.

One sentence: minimum interruption, maximum context. Every UI choice should be the smallest one that gets the job done.

The hierarchy of interruption

From quietest to loudest:

  1. App Home tab — passive, user-initiated; great for dashboards, settings, history.
  2. Ephemeral message — visible only to one user, no notification, no persistence.
  3. DM from your bot — gentle nudge to one person, persists, notifies.
  4. In-thread reply — additive to an existing conversation.
  5. Channel message — broadcast to everyone in the channel.
  6. Modal — blocks the user's flow; only when input is required.

Always pick the lowest level that works.

Tone of voice

  • Concise. One sentence is usually enough. Two is the limit.
  • Plain. No marketing words. No exclamation marks unless something genuinely worked.
  • Useful. Every message contains a fact, an action, or a link — never a status update.
  • Polite, not chatty. "Done. View the report →" beats "🎉 Yay! Your report is ready! Click here!"

Design heuristics

1. Prefer messages over modals

Modals are interruptions. If the user only needs to confirm or pick from a small list, do it inline with a Block Kit message + button.

2. Prefer ephemeral over public

A confirmation only the requester needs to see should be ephemeral. Public confirmations turn channels into status pages.

3. Prefer threads over reply-noise

If your app needs to follow up, reply in the thread of the original message. Don't create a new channel-level message.

4. Use App Home for state

Settings, history, configuration, dashboards — all belong in App Home. Don't make users dig through old DMs to find them.

5. Make destructive actions confirmable

Anything that deletes, sends, or charges money should require a confirmation step. Block Kit confirmation dialogs exist for this — use them.

Surface design quick guide

Need                                            → Surface
─────────────────────────────────────────────────────────────────
"Did this work?" (one user)                     → ephemeral message
"This happened, here is the link"               → channel message (with unfurl)
"Please pick a date / fill a form"              → modal
"Show settings I've configured"                 → App Home tab
"Continue an ongoing conversation"              → thread reply
"Confirm a destructive action"                  → confirm dialog on a button

Naming & commands

  • Slash commands should be verbs: /standup, /report, /approve — not nouns.
  • One slash command per app is plenty. Multiple sub-actions belong in a modal launched from the command.
  • App display name should be short, lowercase-friendly, and unique.

Accessibility & localization

  • Every image must have alt_text. Block Kit enforces this.
  • Use plain text fallbacks (text field) for screen readers and notification previews.
  • Respect the user's locale (users.info) for date and number formatting.

Key takeaways

  • App Home for state. Ephemerals for confirmations. Modals only when forced.
  • Tone: concise, plain, useful. No marketing voice.
  • Threads over channel noise.
  • Always confirm destructive actions.
