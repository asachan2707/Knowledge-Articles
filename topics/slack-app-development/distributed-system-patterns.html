Top 7 Distributed System Patterns

Ambassador, circuit breaker, CQRS, event sourcing, leader election, publisher–subscriber, and sharding. Even at modest scale, your Slack app will touch at least three of them. This page maps each pattern to a concrete Slack-app problem.

Distributed-system patterns are names for problems other people already solved. You will re-discover them painfully if you don't learn them up front.

1 · Ambassador

A small proxy in front of an external service, handling cross-cutting concerns (retries, metrics, circuit breaking, auth). Often runs as a sidecar.

Slack fit: wrap your Slack WebClient in a thin "Ambassador" that adds tracing, retries, and rate-limit awareness — every Web API call goes through it.

2 · Circuit Breaker

When a downstream service fails too often, "open the circuit" — short-circuit calls and fail fast for a cooldown period. Prevents cascading failures.

Slack fit: wrap outbound calls (Jira, GitHub, your own API) in a circuit breaker (opossum, cockatiel). When Jira is down, your slash command fails with a friendly "service unavailable" instead of timing out.

import { circuitBreaker, ConsecutiveBreaker, retry, ExponentialBackoff } from 'cockatiel';

const breaker = circuitBreaker(retry({ maxAttempts: 3, backoff: new ExponentialBackoff() }), {
  halfOpenAfter: 10_000,
  breaker: new ConsecutiveBreaker(5),
});

const result = await breaker.execute(() => fetch('https://jira/api/...'));

3 · CQRS (Command Query Responsibility Segregation)

Reads and writes go through separate models. Writes go to the canonical store; reads go to a denormalized read store optimized for query.

Slack fit: heavy reporting dashboards over Slack event data — write events to Postgres, project them into a denormalized read table for fast dashboard queries.

4 · Event Sourcing

Persist every state change as an event. Rebuild current state by replaying events. Gives you an audit log for free and time-travel debugging.

Slack fit: a workflow app where every transition (request → approve → fulfill) becomes an event, and the current state is a projection. Excellent audit story.

5 · Leader Election

One node in a cluster is elected leader; only the leader runs certain tasks. Implementations: ZooKeeper, etcd, Redis Redlock, or a simple DB-backed lease.

Slack fit: running a daily standup cron — only one Bolt replica should fire it. Use a Redis lease (SET lock:cron 1 NX EX 90) renewed every 30s.

6 · Publisher–Subscriber

Publishers emit events to a topic; subscribers receive copies independently. Loose coupling between producers and consumers.

Slack fit: a "user submitted feedback" event published to a topic; subscribers handle storage, Slack notification, and analytics independently. Add a new subscriber later without touching the publisher.

7 · Sharding

Split data across nodes by a shard key (user id, region, hash). Each shard handles a portion of the load.

Slack fit: Enterprise-Grid scale apps shard by enterprise_id + team_id. Even at smaller scale, shard background queues by team to keep one noisy workspace from blocking others.

The two foundational properties to bake in

Idempotency

The same operation applied twice has the same effect as once. Critical for Slack — Events API retries, button double-clicks, webhook redelivery. Use idempotency keys (event_id, request_id) and dedup with Redis.

Backpressure

When consumers are slower than producers, push back instead of falling over. Bounded queues, rate limiting, 429 responses with Retry-After.

Key takeaways

  • Circuit breakers + idempotency are the two patterns every Slack app needs first.
  • Leader election (Redis lease) for any cron / scheduled task across replicas.
  • Pub/sub when you need fan-out without coupling producer to consumers.
  • Don't introduce a pattern unless you have a measured pain it solves.
