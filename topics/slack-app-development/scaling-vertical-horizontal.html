Vertical vs. Horizontal Scaling

Vertical scaling = bigger box. Horizontal scaling = more boxes. The choice has cost, reliability, and architectural implications. For a Slack app this is the moment you decide between "one Bolt process behind a load balancer" and "many Bolt instances with shared state".

Scaling exists because traffic grows, and a single box has limits. The question is never "vertical or horizontal" — it's "what's the cheapest path to the next 10× of load while keeping reliability acceptable?"

Default for Slack apps: design stateless from day one. Push installation tokens, dedup keys, and per-team state to Mongo / Postgres / Redis. Then horizontal scaling is just "add more containers."

Vertical scaling — bigger box

How: upgrade the instance class — more CPU, more RAM, faster disks.

Pros:

  • Zero application changes.
  • No coordination cost (single process owns all state).
  • Cheap to operate at small scale.

Cons:

  • Hard ceiling — eventually no bigger box exists.
  • Single point of failure: machine down = service down.
  • Cost curve is non-linear; the biggest instance types are exponentially priced.

Horizontal scaling — more boxes

How: run multiple instances behind a load balancer; each handles a portion of traffic.

Pros:

  • Effectively unbounded ceiling.
  • One node failing doesn't take down the service.
  • Linear-ish cost.

Cons:

  • Requires stateless processes (or sticky sessions, or a shared state store).
  • Coordination cost: leader election, distributed locks, consistency tradeoffs.
  • Higher operational complexity (deployments, observability, log aggregation).

Slack-app implications

Stateless Bolt processes

Bolt is a Node process. Make it stateless by:

  • Storing installations in a shared DB (not a local Map).
  • Storing event-dedup keys in Redis with TTL.
  • Avoiding in-process counters / queues; use Redis or SQS.

Socket Mode caveat

Each Socket Mode connection is held by one process. If you run 3 Bolt replicas in Socket Mode, Slack will round-robin events across them — fine for stateless handlers, problematic if your handler assumed it had local state.

Background work

Once you're horizontal, push slow work to a queue (BullMQ, SQS) consumed by separate worker processes. Web tier handles ack + light work; workers do the rest.

The third option: scale up until it breaks, then scale out

Pragmatic teams scale vertically first (it's free in engineering time), then go horizontal when:

  • The biggest instance can't keep up.
  • Reliability becomes a real concern (single-node SLA isn't enough).
  • Deployment risk requires zero-downtime rollouts.

Cost mental model

A 4-vCPU container at $40/mo serving 1× traffic. To handle 10× traffic:

  • Vertical: one 64-vCPU box at ~$1,600/mo.
  • Horizontal: ten 4-vCPU containers at ~$400/mo (often cheaper, plus redundancy).

The tradeoff: horizontal needs the engineering investment in statelessness up front.

Key takeaways

  • Design Slack apps stateless from day one — pay the small upfront cost, unlock horizontal scaling forever.
  • Vertical scaling is fine for early days; horizontal becomes mandatory at modest scale.
  • Push state into Mongo/Postgres/Redis; keep the Bolt process disposable.
  • Background queues let your web tier stay fast while heavy work runs elsewhere.
