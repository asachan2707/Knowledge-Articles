MongoDB — Document Database for Slack Apps

MongoDB is the document database most JS-stack tutorials reach for. Net Ninja's "Complete MongoDB Tutorial" playlist walks you from CRUD through Mongoose. This page focuses on the data models and operations a Slack app actually needs: installations, settings, audit logs.

Mongo stores JSON-like documents in collections. There's no enforced schema by default, so modeling discipline matters more, not less. For Slack apps, Mongo shines for variable-shape per-workspace settings, OAuth installation records, and high-cardinality event logs.

Decision rule: use Mongo for documents that vary in shape across tenants (per-workspace settings, audit events). Use SQL for things with strict relations (users → roles, orders → line items).

Concepts the playlist covers

  • Documents & collections: BSON, _id, ObjectId.
  • CRUD: insertOne/insertMany, find/findOne, updateOne/updateMany, deleteOne/deleteMany.
  • Query operators: $eq, $gt, $in, $and/$or, $exists, $regex.
  • Update operators: $set, $unset, $inc, $push, $pull, $addToSet.
  • Indexes: single, compound, partial, TTL, text — and explain('executionStats').
  • Aggregation pipelines: $match → $group → $sort → $project → $lookup.
  • Mongoose: schemas, models, validation, middleware (pre/post hooks), populate, virtuals.

A real Slack installation store

The official InstallationStore for distributed Slack apps maps cleanly to a Mongo collection. Each document holds an enterprise / team install, the tokens, and metadata. 

// installations collection schema
{
  _id: ObjectId,
  enterpriseId: 'E1' | null,
  teamId: 'T1',
  userId: 'U1',           // installer
  bot: {
    token: 'xoxb-...',     // encrypt at rest!
    userId: 'UBOT',
    scopes: ['chat:write', 'commands'],
  },
  appId: 'A1',
  installedAt: ISODate('2026-05-11T...'),
  isEnterpriseInstall: false,
}

// indexes
db.installations.createIndex({ teamId: 1, enterpriseId: 1 }, { unique: true });
db.installations.createIndex({ installedAt: -1 });

Mongoose vs. native driver

The official mongodb driver is the most direct path. Mongoose adds schema validation, hooks, and population (Mongo's pseudo-join). For a Slack app, Mongoose is usually worth it — schemas catch shape drift before it bites. 

import mongoose, { Schema } from 'mongoose';

const InstallationSchema = new Schema({
  teamId: { type: String, required: true, index: true },
  enterpriseId: { type: String, default: null },
  bot: {
    token: { type: String, required: true },
    scopes: [String],
  },
  installedAt: { type: Date, default: Date.now },
});

InstallationSchema.index({ teamId: 1, enterpriseId: 1 }, { unique: true });

export const Installation = mongoose.model('Installation', InstallationSchema);

// Usage
await Installation.create({ teamId: 'T1', bot: { token: 'xoxb-...', scopes: ['chat:write'] } });
const found = await Installation.findOne({ teamId: 'T1' });

The two modeling decisions

1. Embed vs. reference

Embed when the child data is owned by the parent and small/bounded (e.g., a user's notification settings). Reference when the child is large, shared, or grows unbounded (e.g., audit events per workspace).

2. Indexes you cannot skip

  • Every field you filter on in a hot query path.
  • Compound indexes that match your most common filter+sort.
  • TTL indexes for ephemeral data (event dedup keys, OTP tokens).
  • Unique indexes for natural keys (teamId + enterpriseId).

Aggregation: a practical example

"Top 10 channels by message volume in the last 7 days":

db.events.aggregate([
  { $match: { type: 'message', ts: { $gt: new Date(Date.now() - 7*24*60*60*1000) } } },
  { $group: { _id: '$channelId', count: { $sum: 1 } } },
  { $sort: { count: -1 } },
  { $limit: 10 },
]);

Operational essentials

  • Connection pooling: the driver does it; one client per process is enough.
  • Retries: Mongo retryable writes are on by default; rely on them.
  • Backups: Atlas continuous backup or mongodump to S3 daily.
  • Encryption at rest: Atlas does it; verify in your provider docs.
  • Token storage: Slack tokens are credentials — encrypt before storing.

Key takeaways

  • Mongo fits Slack apps for installations, settings, and event logs.
  • Define schemas with Mongoose to catch shape drift early.
  • Index every field you filter on; explain() any slow query.
  • Encrypt Slack tokens at rest; always have backup + restore tested.
