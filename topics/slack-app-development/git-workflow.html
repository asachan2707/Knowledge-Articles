Git Workflow + Merge vs. Rebase

Git is the substrate every other engineering practice rests on. Three short videos cover the mental model, daily workflow, and the merge vs. rebase tradeoff. This page summarizes the ones you'll use every day on a Slack-app team.

TrackCommon foundations
GoalMove confidently in real shared repos
Time~90 min total (3 videos)
Slack tie-inEvery PR you open against the team's repo

Git is two things: a content-addressable store of snapshots (commits) and a set of pointers (branches, tags, HEAD) into it. Once that mental model clicks, every command makes sense — rebase, cherry-pick, reset, even reflog.

One sentence: commits are immutable; branches are movable pointers; HEAD is "where I am". Everything else is a verb that moves pointers around.

The daily verbs

git clone <url>                          # bring the repo down
git checkout -b feature/standup           # branch from current HEAD
# ...edit files...
git status                                # what changed
git diff                                  # what changed (line by line)
git add -p                                # stage hunks (review as you go)
git commit -m "feat(standup): add /standup command"
git push -u origin feature/standup        # push and set upstream
# ...open PR...

git fetch origin                          # update local refs (no merge)
git rebase origin/main                    # replay my commits onto latest main
# ...resolve any conflicts, edit files, then:
git add <files>
git rebase --continue
git push --force-with-lease               # safe force-push of rebased branch

Merge vs. rebase

Merge

git merge feature on main creates a merge commit that joins two histories. The full branching graph is preserved.

Pros: non-destructive; preserves what really happened.

Cons: messy history with many parallel lines; git log gets noisy.

Rebase

git rebase main on feature replays your branch's commits on top of the latest main, producing a linear history.

Pros: clean linear history; easy to bisect; great for review.

Cons: rewrites commit hashes; never rebase a branch others are based on without coordination.

The team rule

Most modern teams: rebase your feature branch onto main before opening / updating a PR; merge with --no-ff into main on PR approval (or squash-merge for tiny PRs).

Force pushing safely

After a rebase, your local commits have new hashes. To push, you need a force push. git push --force-with-lease is the safe version: it refuses to overwrite if someone else pushed to your branch since you last fetched. 

# Safe — refuses if remote moved
git push --force-with-lease

# Dangerous — never do this on shared branches
git push --force

Commit hygiene

  • One logical change per commit. Refactors and features in separate commits.
  • Conventional commits — feat:, fix:, refactor:, docs:, chore:. Tooling can generate changelogs automatically.
  • Imperative mood. "Add /standup command" not "Added /standup command".
  • Body explains why. The diff shows what; the commit body explains the reasoning.

Recovering from disasters

  • Bad commit: git revert <sha> — adds a new commit that undoes it.
  • Wrong branch: git stash → git checkout right-branch → git stash pop.
  • Lost commits: git reflog shows every move HEAD made; reset to a previous state.
  • Broken merge: git merge --abort.
  • Broken rebase: git rebase --abort.

Slack-app PR norms

  • Small, atomic PRs — one feature at a time. Reviewers will actually read them.
  • Run lint + tests locally before pushing; let CI confirm.
  • Update the PR description with the Slack-app behavior change (screenshots of message UI help).
  • Rebase onto main right before merge — keeps history linear in main.

Key takeaways

  • Commits are immutable; branches are pointers. Everything else is a verb on pointers.
  • Rebase your feature branch onto main; merge into main on approval.
  • Always --force-with-lease, never plain --force.
  • Small atomic commits + conventional messages = easy PR review and great git log.
