Slack Foundations

The orientation pass before any code. Slack Foundations covers what Slack is as a platform, the workspace / channel / message hierarchy, the difference between user and bot identities, and the developer mental model every Bolt app builds on.

Before you write a single line of Bolt code, you need a stable mental model of the Slack platform: what an "app" is, where it lives, what it can see, and how it talks. Most "weird Slack bug" stories trace back to a misunderstanding of one of these primitives.

Mental model: a Slack app is an identity (a bot user) installed into a workspace, granted a set of scopes, that reacts to events and user actions and can act via the Web API.

The platform hierarchy

  • Enterprise Grid — a collection of workspaces sharing org-level admin and SSO. Apps can install at the org or workspace level.
  • Workspace (Team) — the unit a Slack app installs into. Identified by team_id.
  • Channel — a conversation: public (C…), private (G…), DM (D…), or group DM (G…).
  • Conversation — Slack's umbrella term for any of the above; the modern API uses conversations.* uniformly.
  • Message — identified by channel + ts (timestamp). Threads are messages with a thread_ts.
  • User — a real human (U…) with profile, timezone, and roles.
  • Bot user — your app's identity inside a workspace (UBOT… or B…).

Identities & tokens

  • Bot token (xoxb-…) — your app posts as the bot user. The most common token your code holds.
  • User token (xoxp-…) — your app acts on behalf of an installing user. Use only when bot scopes can't do the job.
  • App-level token (xapp-…) — used for Socket Mode and certain admin endpoints.
  • Refresh + rotation — newer apps support token rotation; if enabled, you must persist refresh tokens and rotate before expiry.

Scopes — the permission model

A scope is a permission your app declares it needs. Slack uses granular bot scopes (e.g., chat:write, commands, channels:history). The principle is simple: ask only for what you need, today. Adding a scope later is fine; removing one breaks installs.

  • chat:write — post messages as the bot.
  • commands — define and receive slash commands.
  • app_mentions:read — see @your-app mentions.
  • channels:history — read messages in public channels you are in.
  • users:read — read profile of users.

The two communication directions

Slack apps are bidirectional: Slack → your app via the Events API or Socket Mode, and your app → Slack via the Web API. Designing a feature usually means deciding which event triggers it and which API methods carry out the response.

  • Inbound: events (message posted, app mentioned, reaction added), interactivity payloads (button click, modal submitted, slash command).
  • Outbound: chat.postMessage, views.open, conversations.list, etc.

Surfaces a Slack app can render UI in

  • Messages in channels and DMs (visible to all participants).
  • Ephemeral messages (visible to one user only, never persisted).
  • Modals opened from buttons / shortcuts / commands.
  • App Home tab (your app's own dashboard inside Slack).
  • Notifications via chat.postMessage with unfurl_links.

Distribution: internal vs. distributed

An internal app installs only in your own workspace; you can use a single bot token. A distributed app supports OAuth installation by any workspace, and you must persist installations to a store (Mongo, Postgres, DynamoDB) keyed by team_id + enterprise_id.

Key takeaways

  • App = identity + scopes + handlers. Internalize this triplet.
  • Tokens are credentials — encrypt at rest, rotate when supported.
  • Two directions: events come in via webhooks/Socket Mode; you push out via the Web API.
  • Decide internal vs. distributed early — it changes your storage model.
