JavaScript Fundamentals

The base layer for every Slack app on the JS stack. Programming with Mosh's "JavaScript Tutorial for Beginners" lands the language essentials in a single sitting; this page condenses what to watch for and how each piece shows up in real Bolt code.

JavaScript runs in browsers and on Node.js servers. Every Slack app you'll build — slash commands, modals, event handlers — is JavaScript (or its typed superset, TypeScript) executing inside a Node process. Before any framework, you need fluency in the core language: types, control flow, functions, scope, the DOM (in passing), the event loop, and async.

Why start here: the Slack docs and Bolt examples assume you can already read and write modern JavaScript without thinking about it. Spending one hour on the fundamentals saves you ten when you're debugging an event handler.

What the video actually covers

  • Variables & types: let/const/var, primitives (string, number, boolean, null, undefined, symbol, bigint), and reference types (object, array).
  • Operators: arithmetic, comparison, logical, ternary, optional chaining (?.), nullish coalescing (??).
  • Control flow: if/else, switch, for, for-of, for-in, while, do-while, break/continue.
  • Functions: declarations vs. expressions vs. arrow, parameters, default values, rest parameters.
  • Objects & arrays: property access, methods, destructuring, spread, common array methods (map, filter, reduce, find, some, every).
  • Scope & closures: block scope, lexical scope, the closure pattern.
  • The DOM: document.querySelector, events, light browser interactivity.
  • Async basics: callbacks → promises → async/await.

Three concepts that trip up Slack beginners

1. this binding

In a regular function, this depends on how the function is called. In an arrow function, this is captured from the surrounding scope. This matters in Bolt listeners — most examples use arrow functions exactly because they don't rebind this.

2. The event loop & async

JavaScript is single-threaded but non-blocking. Long synchronous work blocks the event loop and will make your Slack app miss the 3-second ack window. Always await I/O, never block.

Block kit: bad vs. good

// BAD - blocks the event loop, will time out
app.command('/report', async ({ command, ack, client }) => {
  const data = doExpensiveSyncWork(); // 5 seconds
  await ack(); // too late!
});

// GOOD - ack first, then do work
app.command('/report', async ({ command, ack, client }) => {
  await ack();
  const data = await doWorkAsync(); // can take any time
  await client.chat.postMessage({ channel: command.channel_id, text: data });
});

3. Equality (== vs. ===)

Always use === (strict equality). The double-equals coerces types and produces surprising results (0 == '' == false). The only common exception is x == null as a quick check for both null and undefined.

How this shows up in a Slack app

Look at any Bolt for JS example and you will see destructuring ({ ack, body, client }), arrow functions, async/await, optional chaining on payloads (body.user?.id), and array methods on Slack data ((await client.conversations.list()).channels.filter(c => !c.is_archived)). Every one of those patterns is plain JavaScript — the framework doesn't add new syntax.

Practice plan after the video

  1. Open a Node REPL (node) and try every primitive operator until it feels natural.
  2. Write a small script that fetches data with fetch() and uses map/filter/reduce on the response.
  3. Build a tiny todo list in plain HTML + JS to feel the DOM and event listeners.
  4. Write a function that uses async/await with proper try/catch.

Key takeaways

  • Use const by default, let only when you mutate, var never.
  • Always ===. Always async/await over raw promise chains.
  • Destructuring + spread are how every modern JS API is shaped.
  • The event loop is non-negotiable Slack knowledge — block it and your handler dies.
