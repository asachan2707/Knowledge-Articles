Data Structures — The Everyday Set

You don't need to be a competitive programmer to build Slack apps, but you should fluently know the cost of common operations on the structures you reach for daily: arrays, hash maps, sets, queues, stacks, trees, heaps, tries, and graphs.

Most Slack-app performance bugs aren't algorithmic — they're picking the wrong structure for the job and then writing an O(n) loop where an O(1) lookup would do. This page is the cheat sheet of "what to reach for when".

Plus two specials: Trie (prefix lookup, autocomplete) and Graph (relationships, recommendations, dependency walks).

JavaScript built-ins

  • Array — index access O(1), push/pop O(1), shift/unshift O(n).
  • Map — true hash map; preserves insertion order; any key type.
  • Set — unique values; supports iteration order; perfect for dedup.
  • WeakMap/WeakSet — keys held weakly; great for caches keyed by object identity.
  • No native heap — use heap-js or implement with an array.

Slack-app applications

  • Event dedup: Set<event_id> in Redis with TTL — keep last 24h to ignore retries.
  • User cache: Map<userId, profile> with TTL to reduce users.info calls.
  • Rate-limit budget: token bucket with a Map<teamId, count>.
  • Background work queue: Redis list (FIFO) consumed by workers.
  • Top-K active channels: heap, kept at size K.
  • Thread tree walks: queue/stack for BFS/DFS over conversations.replies.

Big-O traps

  • arr.includes(x) in a loop = O(n²). Convert to a Set.
  • arr.indexOf(x) repeatedly = same trap.
  • JSON.parse/stringify on a hot path is silently O(n) of the structure.
  • Spread on huge arrays creates copies — [...a, ...b] is O(a+b) every call.

Key takeaways

  • Hash map and set are the right answer to most "is this in a list" questions.
  • Queue for FIFO work, heap for top-K and scheduling.
  • Use Map/Set over plain objects/arrays when keys aren't strings or order matters.
  • Profile before "optimizing" — most slowness is the wrong structure, not the wrong language.
