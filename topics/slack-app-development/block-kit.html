Block Kit — Slack's UI Framework

Block Kit is Slack's declarative UI framework. Messages, modals, and Home tabs are JSON arrays of blocks composed of elements. This page walks the block types, element types, validation rules, and the Block Kit Builder workflow.

TrackSlack core
GoalCompose any Slack UI fluently
Time~45 min read + practice
Prereq forModals, App Home, rich messages

Block Kit replaced the older "attachments" API. A message is now an array of blocks; each block has a type and (often) a set of nested elements. The result renders identically across desktop, mobile, and notification surfaces.

Workflow: design in Block Kit Builder first → paste the JSON → wire in dynamic data → ship. Don't write block JSON from scratch in your editor; the Builder is faster and catches errors immediately.

Block types

  • section — text + optional accessory (button, image, select). The workhorse.
  • divider — horizontal line. Use sparingly.
  • header — large bold text. plain_text only.
  • context — small text + small images. Footer/metadata.
  • actions — a row of interactive elements (buttons, selects, etc.).
  • input — only valid in modals/Home; one editable field.
  • image — an image with alt text.
  • file — embed a file uploaded to Slack.
  • video — inline video player.
  • rich_text — read-only rich text (mostly for representing existing message content).

Element types

  • button — primary action element. Has style: default, primary, danger.
  • static_select / multi_static_select — pick from a static list.
  • users_select, channels_select, conversations_select — Slack-aware pickers.
  • external_select — options loaded from your endpoint.
  • datepicker, timepicker, datetimepicker.
  • plain_text_input (single + multiline), number_input, email_input, url_input.
  • checkboxes, radio_buttons.
  • overflow — three-dot menu.
  • image as element (e.g., section accessory).

Worked example — an approval message

const approvalMessage = {
  channel: 'C1',
  text: 'PTO request from @amy needs your approval',  // notification fallback
  blocks: [
    {
      type: 'header',
      text: { type: 'plain_text', text: 'PTO Request' },
    },
    {
      type: 'section',
      fields: [
        { type: 'mrkdwn', text: '*From:* <@U_AMY>' },
        { type: 'mrkdwn', text: '*Dates:* May 20 – May 24' },
        { type: 'mrkdwn', text: '*Reason:* Vacation' },
        { type: 'mrkdwn', text: '*Days:* 5' },
      ],
    },
    {
      type: 'actions',
      block_id: 'approval_actions',
      elements: [
        {
          type: 'button',
          action_id: 'approve_pto',
          text: { type: 'plain_text', text: 'Approve' },
          style: 'primary',
          value: 'pto:42',
        },
        {
          type: 'button',
          action_id: 'reject_pto',
          text: { type: 'plain_text', text: 'Reject' },
          style: 'danger',
          value: 'pto:42',
          confirm: {
            title:   { type: 'plain_text', text: 'Are you sure?' },
            text:    { type: 'plain_text', text: 'This will reject the PTO request.' },
            confirm: { type: 'plain_text', text: 'Reject' },
            deny:    { type: 'plain_text', text: 'Cancel' },
          },
        },
      ],
    },
    {
      type: 'context',
      elements: [{ type: 'mrkdwn', text: 'Submitted 2026-05-11 · ID 42' }],
    },
  ],
};

The validation rules that catch you

  • 50 blocks max per message. 100 in modals/App Home.
  • Header text max 150 chars, plain_text only.
  • section text max 3000 chars; sections support up to 10 fields.
  • actions — up to 25 elements per row.
  • input blocks require unique block_id and unique action_id on the inner element.
  • buttons need plain_text labels (no markdown).
  • Images require alt_text.

Reading modal state

Modal submissions arrive as view.state.values: an object keyed by block_id, then by action_id, then a per-element value. 

app.view('feedback_modal', async ({ ack, view }) => {
  await ack();
  const v = view.state.values;
  const message    = v.b_msg.a_msg.value;                    // plain_text_input
  const reason     = v.b_reason.a_reason.selected_option.value; // static_select
  const dueDate    = v.b_due.a_due.selected_date;            // datepicker
  const ccUserIds  = v.b_cc.a_cc.selected_users;             // multi_users_select
});

Composition patterns

  • Builder functions — write small functions that return blocks. Compose with spread.
  • Snapshot tests — Block Kit JSON is stable structured output, perfect for Jest snapshots.
  • Block Kit Builder permalink — when designing, share the Builder URL with PMs/designers for review.
  • Localization — wrap every plain_text string in your i18n function.

Key takeaways

  • Always design in Block Kit Builder first.
  • Send a text fallback for notifications + accessibility.
  • Memorize the validation limits — they bite at runtime, not compile time.
  • Write small builder functions that return blocks; snapshot-test them.
