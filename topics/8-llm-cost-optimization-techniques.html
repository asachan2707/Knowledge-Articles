8 LLM Cost Optimization Techniques: How to Cut API Spend by Up to 70%

Eight concrete techniques — from prompt caching and model routing to output length control and batching — that visually-explained reduce LLM API spend by 40–70% without degrading quality.

Published Apr 2026

LLM API costs follow a simple but counterintuitive structure: you pay for every token the model reads and every token it writes, on every request. Most teams optimise only the obvious dimension (shorter prompts) and miss the 70% of spend driven by context repetition, overqualified models, and uncontrolled output length. The eight techniques below address all the layers.

Cost structure primer: Input tokens ≈ 3× cheaper than output tokens on most providers. Cached input tokens ≈ 10× cheaper than uncached. This changes the priority order for optimisation — context management matters more than prompt length, and output control matters most of all.

Story · The $4,000 month that became $1,200

A startup running a customer support AI hit a $4,000/month API bill at 50k conversations. After applying prompt caching (static system prompt), routing simple queries to Haiku, and adding output length constraints — the same workload cost $1,200. The product behaviour was unchanged. Three configuration changes cut the bill by 70%.

Technique 1

Prompt caching (the highest-leverage single technique)

If your system prompt is more than a few hundred tokens and is the same across most requests — cache it. Cached input tokens cost roughly 10% of uncached tokens on Claude. For a 2,000-token system prompt sent with every request:

  • Without caching: 2,000 × $0.000003 × 1M requests = $6,000
  • With caching: 2,000 × $0.0000003 × 1M requests = $600

Implementation: add "cache_control": {"type": "ephemeral"} to the last block of your system prompt. The cache persists for 5 minutes (Claude); refreshing it before expiry with a keepalive request eliminates re-cache overhead for high-traffic applications. 

{
  "system": [
    {
      "type": "text",
      "text": "You are a helpful assistant for Acme Corp. [2000 tokens of context]",
      "cache_control": { "type": "ephemeral" }
    }
  ]
}
Technique 2

Model routing (match task complexity to model tier)

The cost difference between model tiers is not marginal — it is 10–20×:

  • Haiku 4.5: ~$0.80 / 1M input tokens — classification, extraction, simple Q&A
  • Sonnet 4.6: ~$3.00 / 1M input tokens — standard engineering tasks, most conversations
  • Opus 4.8: ~$15.00 / 1M input tokens — complex reasoning, novel problem-solving, high-stakes decisions

A routing classifier (itself a fast, cheap model call) categorises each request and dispatches to the appropriate tier. The classifier overhead is negligible — a Haiku call to classify costs $0.0001 and saves $0.01–$0.15 per mis-routed request. 

def route_query(query: str) -> str:
    classification = haiku.classify(
        query,
        categories=["simple_faq", "standard_task", "complex_reasoning"]
    )
    return {
        "simple_faq": "claude-haiku-4-5",
        "standard_task": "claude-sonnet-4-6",
        "complex_reasoning": "claude-opus-4-8"
    }[classification]
Technique 3

Output length control

Output tokens cost 3× more than input tokens and are the most predictable source of waste — models generate until they reach a natural stopping point, which is often longer than necessary.

Three controls in order of effectiveness

  • max_tokens parameter: Hard cap on response length. Set it based on your measured 95th-percentile of useful response length for each use case. Measure first; guess second.
  • Format instructions: "Reply in 2-3 sentences maximum" or "Output only the JSON object, no explanation" in the system prompt. Models follow explicit length instructions reliably.
  • Structured output: JSON Schema-constrained outputs are consistently shorter than free-text responses conveying equivalent information. Use structured outputs wherever the consuming code already needs to parse the response.
Technique 4

Context window management

For multi-turn conversations, the full conversation history is re-sent with every request. A 20-turn conversation where each turn averages 500 tokens sends 10,000 tokens of history on every request — even though the last 2–3 turns are almost always sufficient for the model to answer correctly.

Strategies in ascending cost savings

  • Sliding window: Keep only the last N turns in context. Easiest to implement; loses long-term session context.
  • Progressive summarisation: Summarise older turns into a compressed summary and keep recent turns verbatim. Preserves relevant context while reducing token count by 60–80%.
  • Semantic retrieval: Store all turns as embeddings; retrieve only the most relevant turns for each request. Near-lossless context at O(1) token count per turn.
Technique 5

Batch API for async workloads

Most LLM providers offer a batch API that processes requests asynchronously (within 24 hours) at 50% of the synchronous price. If you have workloads that don't need immediate responses — nightly report generation, bulk classification, embedding refreshes, offline content processing — use batch mode exclusively. 

from anthropic import Anthropic

client = Anthropic()

# Batch: 50% cheaper, 24hr SLA
batch = client.messages.batches.create(
    requests=[
        {"custom_id": f"req_{i}", "params": {"model": "claude-haiku-4-5", "messages": [...]}}
        for i, item in enumerate(bulk_items)
    ]
)
# Poll for completion; retrieve results batch
print(f"Batch ID: {batch.id}  Status: {batch.processing_status}")
Technique 6

RAG instead of context stuffing

The instinct when building a knowledge-base application is to put all the relevant documents in the system prompt. At scale, this is prohibitively expensive — and the model's performance degrades when the context is packed with documents only partially relevant to the query.

Retrieval-Augmented Generation (RAG) replaces context stuffing with retrieval: embed the knowledge base, retrieve only the top-K relevant chunks for each query, and inject only those chunks into the context. Typical result: 80% context token reduction with equal or better accuracy because the injected context is more targeted.

The cost of the embedding + retrieval step is typically $0.0001 per query — negligible against the token savings.

Technique 7

Response caching for repeated queries

Many production LLM applications receive a small set of high-frequency queries. Caching the responses to these queries eliminates API calls entirely for cache hits. Implementation:

  • Exact match caching: Hash the full request (system prompt + user message) and cache the response in Redis with a TTL. Trivial to implement; effective for FAQ-style applications where users repeat common queries.
  • Semantic caching: Embed the user query and check for high-similarity prior queries in a vector store. Cache hit if similarity exceeds threshold (typically 0.95). Handles paraphrases and spelling variants.

Real-world cache hit rates for customer support applications: 30–60%. That fraction of requests costs zero API tokens.

Technique 8

Fine-tuning to reduce prompt overhead

When a system prompt contains more than 500 tokens of instructions that apply to every request — persona, output format, constraints, examples — fine-tuning bakes those instructions into the model weights. Post-fine-tuning, the system prompt can be reduced to a few sentences.

The break-even calculation: fine-tuning cost (one-time) vs. token savings per request × request volume. For high-volume applications (1M+ requests/month) with large system prompts (1000+ tokens), fine-tuning pays back the one-time cost in days. For low-volume applications, it rarely makes sense.

Apply in priority order: Prompt caching → model routing → output length control → context window management → batch API → RAG → response caching → fine-tuning. Each technique has diminishing complexity. Start with the first three before considering the rest.

Common questions

FAQ

  • Which technique has the best ROI for a new application?

    Prompt caching + model routing. Prompt caching requires one line of configuration and reduces input token costs by 90% for repeated system prompts. Model routing requires a classifier but reduces per-request model cost by 75–90% for simple queries. Together they address the two largest cost drivers before you've written 50 lines of optimisation code.

  • Does context window management affect output quality?

    Sliding window does — it loses long-term context. Progressive summarisation is nearly lossless for most use cases because the summary captures the key decisions and established facts. Semantic retrieval is lossless by design but adds latency. For a chatbot, progressive summarisation is the standard starting point.

  • Is the 70% cost reduction claim realistic for all applications?

    70% is achievable when starting from zero optimisation and applying all eight techniques. The specific savings depend heavily on application structure: a simple FAQ bot gets more from response caching; a complex agentic application gets more from model routing and context management. Measure first — find your largest cost driver and optimise that specifically.

Key Takeaway

LLM API costs are driven by input token repetition (system prompt, conversation history), over-qualified model selection, and uncontrolled output length. Prompt caching, model routing, and output length control address these three root causes and together deliver 40–60% cost reduction with minimal engineering investment. Apply in that order before reaching for the more complex techniques.
