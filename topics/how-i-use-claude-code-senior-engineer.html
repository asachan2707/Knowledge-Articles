How I Actually Use Claude Code as a Senior Engineer

Not the idealised workflow — the actual daily usage: what kinds of tasks go to Claude, which stay manual, how the collaboration is structured, and what a senior engineer's Claude Code setup looks like after 12+ months of real use.

Published May 2026

Senior engineers use Claude Code differently than beginners — not because they know more advanced prompts, but because they have better mental models of where AI adds the most leverage. The difference is task routing: knowing which work benefits most from AI acceleration and which requires your judgment, experience, and domain context that Claude doesn't have.

Core idea: Claude Code multiplies engineering capacity most when it handles the work that is high-volume, well-specified, or pattern-following. Architectural decisions, ambiguous requirements, and cross-system tradeoffs are not delegated — they are informed by Claude but owned by the engineer.

Story · The 10x is not about typing speed

The naive version of "AI makes you 10x more productive" is that you type prompts and Claude types code instead of you. That is not what the productivity gain looks like in practice. The real gain: you spend your cognitive budget on the hard problems (architecture, design tradeoffs, ambiguous requirements) and Claude handles the high-volume execution work. The ratio of thinking to typing shifts dramatically. That is where the leverage is.

Task routing

What goes to Claude, what stays manual

High-leverage tasks for Claude

  • Boilerplate-heavy implementation: CRUD endpoints, serialisers, migration files, test scaffolding — anything that is high-volume and pattern-following. Claude is faster and equally correct.
  • Refactoring to a known pattern: "Migrate all these API calls from the old client to the new one," "Add error handling to every function in this file that matches this pattern." Clear transformation rules, many instances — perfect for Claude.
  • Writing tests for existing code: Given a function, Claude writes comprehensive unit tests faster than any human. Especially valuable for legacy code with no coverage.
  • Translating specs into code: A well-defined spec (API contract, data schema, acceptance criteria) produces high-quality implementation from Claude. The spec quality determines the output quality.
  • Code review for a specific check: "Does this code have any SQL injection vulnerabilities?" "Are there any race conditions in this concurrent code?" Targeted review questions produce better results than open-ended review.
  • Documentation: Docstrings, README updates, inline comments for non-obvious logic — Claude is thorough and consistent here in ways humans rarely are.

Tasks that stay manual

  • Architectural decisions: Choosing between architectural patterns requires context about team skills, operational complexity, existing infrastructure, and organisational constraints that Claude does not have. Get Claude's analysis, but own the decision.
  • Ambiguous requirements: When the product requirement is genuinely unclear, implementing it creates the wrong thing efficiently. Clarify the requirement before starting.
  • Performance-critical code: Claude produces correct code; it rarely produces optimal code for a specific performance envelope without deep, system-specific context. Write the hot path yourself, use Claude for the surrounding infrastructure.
  • Security-sensitive paths: Authentication, authorisation, cryptography — verify every line personally. Claude code review after implementation, not instead of personal review.
The CLAUDE.md setup

What a production CLAUDE.md looks like

After 12 months of iteration, the effective CLAUDE.md is shorter than you expect — under 150 lines — and focused entirely on things Claude cannot infer from the code: 

# Project conventions

## Architecture
- API layer in /src/api — controllers only, no business logic
- Business logic in /src/services — no direct DB access
- Data access in /src/repositories — all DB queries here
- Never import across these layers except in this order

## Patterns
- Errors: always throw typed errors from /src/errors/types.ts
- Async: always async/await, never raw Promises
- Tests: one describe block per function, BDD-style naming ("it should...")

## Constraints
- No new external npm dependencies without discussion
- Logging uses the wrapper at /src/utils/logger.ts, never console.log
- All user-facing strings go through the i18n helper, never raw strings

## Known gotchas
- The `users` table has a soft-delete field (deleted_at) — always filter it
- Redis keys expire after 1 hour — account for cache misses
- The payment processor sandbox rejects card numbers starting with 4111

This is the information a capable new team member would need on their first day. Everything else — architecture patterns, code style — Claude infers from the existing codebase when given relevant file context.

Session workflow

How a typical day looks

Morning: plan-first, then delegate

For any ticket that involves more than one file: start with /plan or a natural-language planning prompt — "Here's the ticket. What files need to change and in what order?" Review the plan. If it's wrong, correct it before Claude writes a single line. This is the most leveraged minute in the workflow.

During implementation: bounded sessions

One session per logical chunk of work. /clear at each task boundary. For longer tasks, /compact when context grows large rather than letting it drift. The session cost and quality both improve with this discipline.

Before commit: targeted review

Every diff gets a targeted review pass before committing — either /code-review or a specific question: "Is there any way this function could fail silently?" The review is not a wholesale re-read; it is a specific check for the failure modes most likely given what was just written.

PR description: Claude writes it, engineer approves it

"Write a PR description for this diff" takes 10 seconds and produces a description that would take 5 minutes to write manually. Review it for accuracy; it occasionally gets the "why" wrong when the context isn't fully in the session.

Where senior experience shows

The judgment calls that make AI collaboration effective

The difference between a senior and junior engineer using Claude Code is not the tool use — it is the judgment exercised around it:

  • Knowing when to stop delegating: Recognising that the current task is ambiguous enough that Claude's output will require more rework than writing it from scratch.
  • Reading the plan before approving it: Juniors often say "looks good" to a Claude plan they haven't actually read. The plan is where errors are cheap; the implementation is where they're expensive.
  • Providing the right context rather than hoping Claude figures it out: Experienced engineers know which parts of the codebase are non-obvious. They reference those specifically, rather than letting Claude discover them (or not) on its own.
  • Verifying before shipping: Every non-trivial Claude-generated function gets tested manually before the PR opens, regardless of what the code review output says.
Common questions

FAQ

  • Does using Claude Code make you a worse engineer over time?

    The concern is valid but the outcome depends entirely on how you use it. If Claude writes all the boilerplate and you spend the freed time on architectural thinking, design tradeoffs, and system reasoning — you get better. If you outsource all the code without reading or understanding it, you are building a dependency. The rule: understand everything you ship, even if you didn't write it.

  • How much does a senior engineer's daily Claude Code usage actually cost?

    With disciplined session hygiene (bounded sessions, /clear between tasks, CLAUDE.md under 150 lines) — roughly $15-30/day for an active coding day on Claude Max, or equivalent API spend. Without hygiene, the same workload can cost 3–5x more. The Max subscription ($100/month) covers most developers who don't run continuous large-scale agents.

  • How do you handle code reviews when a lot of the code is AI-generated?

    The same as code written by a junior engineer: read it, understand it, verify the logic, test the edge cases. "AI wrote it" is not a quality signal in either direction. Some AI-generated code is excellent; some has subtle bugs in the exact areas where the model lacks context. Your review is the quality gate regardless of the source.

Key Takeaway

Senior engineers get leverage from Claude Code by routing the right tasks to it — high-volume, well-specified, pattern-following work — while keeping architectural decisions, ambiguous requirements, and security-sensitive code under personal ownership. The CLAUDE.md holds project-specific context; sessions stay bounded; every output gets reviewed. The productivity gain is real; the engineering judgment required to realise it is still entirely human.
