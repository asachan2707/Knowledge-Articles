URL Shortener: Low-Level Design

This page turns the architecture into concrete building blocks: APIs, schemas, entities, validation rules, cache keys, short-code strategies, and redirect behavior.

Low-level design is where a clean high-level diagram either becomes implementable or falls apart. The goal is to define the data and request contracts so clearly that multiple engineers can build the service without conflicting assumptions.

Key generation

Short-code generation options

Option 1: Global sequence + Base62 encoding

This is simple and predictable. Generate a monotonically increasing numeric ID, then encode it in Base62. This works well when a central or partitioned sequence service is acceptable.

Option 2: Preallocated ID ranges per node

This reduces central bottlenecks. A coordinator assigns ranges to nodes, and each node generates codes within its own range.

Option 3: Randomized collision-checked tokens

Simpler to start with but less attractive at large scale because collision checks add cost and unpredictability.

Senior recommendation: choose sequence or range allocation when predictable uniqueness and operational clarity matter more than token randomness.

Storage model

Example schema

CREATE TABLE short_links (
  short_code VARCHAR(16) PRIMARY KEY,
  long_url TEXT NOT NULL,
  created_by VARCHAR(64),
  tenant_id VARCHAR(64),
  domain VARCHAR(128) NOT NULL,
  is_custom_alias BOOLEAN NOT NULL DEFAULT FALSE,
  status VARCHAR(16) NOT NULL,
  created_at TIMESTAMP NOT NULL,
  expires_at TIMESTAMP NULL
);

A relational database is often sufficient early because lookup by short_code is straightforward, constraints are helpful, and operational maturity is high. NoSQL becomes more attractive when scale, regional distribution, or write patterns justify it.

Redirect path

Lookup and cache behavior

  1. Normalize and validate the incoming short code.
  2. Check cache using a key like link:{short_code}.
  3. If cached, verify status and expiration, then issue redirect.
  4. If not cached, fetch durable storage, evaluate status, then cache the result.
  5. Publish click analytics asynchronously.

Cache value should include target URL, status, expiration metadata, and possibly domain rules so the redirect service does not need extra database reads for common cases.

Validation rules

Low-level checks that prevent bad production behavior

  • Reject malformed or unsupported URLs.
  • Block disallowed schemes such as unsafe protocol usage.
  • Protect custom aliases with uniqueness checks and reserved-word rules.
  • Support soft-delete or disabled status instead of immediate destructive deletion.
  • Validate expiration semantics and time-zone handling consistently.

Key Takeaway

Good low-level design for a URL shortener means more than defining tables and endpoints. It means making the redirect lookup predictable, the short-code strategy reliable, and every validation rule explicit enough to avoid ambiguity across the engineering team.
