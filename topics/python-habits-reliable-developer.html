Reliability rarely comes from clever code. It comes from boring habits forged by getting burned in production — logging discipline, timeouts, validation at boundaries, and visible failures.

Story · The cron job that died at 3 AM

The author describes the classic incident: a scheduled job stops running. No alert fires. No exception is raised loudly. Someone notices a missing dashboard the next morning, 11 hours later.

The fix is not a smarter algorithm. The fix is structured logs, a health check, a timeout, and an alert. Boring, durable habits — not heroics.

Detailed breakdown

Key insights and why they matter

1) Stop trusting print() — log structurally

Use logging with levels, context fields (request_id, job_id), and JSON output in production. print() loses meaning under load.

2) Every external call needs a timeout

No bare requests.get(url). Always pass timeout=(connect, read). Same for DB clients, queues, and HTTP libraries. Hanging clients are how services die silently.

3) Validate inputs at the boundary

Pydantic or similar at the edge of your service. Bad data should fail fast and loud, not corrupt downstream state.

4) Idempotency keys for retries

Any operation that can be retried needs an idempotency key, or you will eventually double-charge / double-send / double-write.

5) Make failure visible

Health endpoints, alerts on error-rate spikes, and dashboards on the four golden signals (latency, traffic, errors, saturation).

6) Tests for failure paths, not just happy paths

For every happy-path test, write at least one for malformed input, slow dependency, or partial outage.

Practical learning

How to apply this in real work

  • Audit your service: find every network call without a timeout and fix it this sprint.
  • Replace ad-hoc print debugging with structured logs + correlation IDs.
  • Add a /health endpoint that checks real dependencies, not just liveness.
  • Write one chaos-style test per critical path (kill DB, slow API, malformed payload).
Common questions

FAQ

  • Is this overkill for a side project?

    If users depend on it, no. Adding observability later, after an incident, is always more expensive.

  • Where should I start?

    Two changes pay off fastest: structured logging with correlation IDs, and timeouts on every external call.

  • What about retries — always good?

    Only with idempotency and backoff. Naive retries amplify outages instead of healing them.
