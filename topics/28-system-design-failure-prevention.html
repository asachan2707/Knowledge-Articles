28 Core System Design Concepts, Explained Through the Failures They Prevent

Every system design concept exists because a real system failed without it. This guide pairs each concept with the failure it prevents, a mental model for when to apply it, and a concrete JavaScript implementation. Covers infrastructure, data, resilience, API design, and distributed systems.

LevelIntermediate → Advanced
FocusFailure-first mental models + Node.js patterns

System design is failure prevention engineering. Each concept in this guide answers one question: "what breaks without this?" Learn that first, then the solution clicks into place naturally.

Core idea: Don't memorise patterns. Memorise the failure each pattern prevents. In an interview or a production incident, you'll reach for the right tool because you recognise the failure mode, not because you recalled a definition.

Infrastructure

The foundation concepts

#01

Load Balancer

Prevents: single server overload + single point of failure

Distributes traffic across server pool. Round-robin for stateless services. Sticky sessions when you must route a user to the same server. Health checks remove dead servers automatically.

#02

Horizontal vs Vertical Scaling

Prevents: hitting hardware ceiling

Vertical = bigger machine (fast, has a ceiling). Horizontal = more machines (no ceiling, requires statelessness). Prefer horizontal for web tiers. Vertical for managed databases (simpler).

#03

CDN (Content Delivery Network)

Prevents: high latency for geographically distant users + origin overload

Caches static assets at edge nodes near users. Cloudflare, AWS CloudFront, Fastly. Cache-Control headers control TTL. Invalidate on deploy.

#04

Auto-scaling

Prevents: paying for idle capacity and crashing under traffic spikes

Rules-based (CPU > 70% → add instance) or predictive (scale up before Monday morning peak). Always set a max to prevent runaway cost.

Caching

Keeping data close

// Redis caching pattern with cache-aside
import { createClient } from 'redis';
import postgres from 'postgres';

const redis = createClient({ url: process.env.REDIS_URL });
const sql   = postgres(process.env.DATABASE_URL);
await redis.connect();

async function getUserById(userId) {
  const key = `user:${userId}`;

  // 1. Try cache first
  const cached = await redis.get(key);
  if (cached) return JSON.parse(cached);

  // 2. Cache miss → hit DB
  const [user] = await sql`SELECT * FROM users WHERE id = ${userId}`;
  if (!user) return null;

  // 3. Populate cache with TTL
  await redis.setEx(key, 300, JSON.stringify(user)); // 5 min TTL
  return user;
}

// Invalidate on write
async function updateUser(userId, data) {
  await sql`UPDATE users SET ${sql(data)} WHERE id = ${userId}`;
  await redis.del(`user:${userId}`); // evict stale cache
}
#05

Cache-Aside (Lazy Loading)

Prevents: caching data that is never read (wasteful)

App checks cache; on miss, reads DB and writes to cache. Most common pattern. Cache is eventual — stale reads possible during the TTL window.

#06

Write-Through Cache

Prevents: stale cache reads after writes

Write to cache AND DB in the same operation. Always consistent. Higher write latency. Use when reads must always be fresh.

#07

Cache Eviction Policies

Prevents: cache filling with useless data

LRU (least recently used) — default for most caches. LFU (least frequently used) — better for hot/cold patterns. TTL — time-based eviction regardless of usage.

Database patterns

Data at scale

// Read replica routing — send reads to replicas, writes to primary
class DatabaseRouter {
  #primary;
  #replicas;
  #rr = 0; // round-robin counter

  constructor(primaryUrl, replicaUrls) {
    this.#primary  = postgres(primaryUrl);
    this.#replicas = replicaUrls.map(url => postgres(url));
  }

  // All writes MUST go to primary for consistency
  write(query) { return this.#primary.unsafe(query); }

  // Reads round-robin across replicas
  read(query) {
    if (this.#replicas.length === 0) return this.#primary.unsafe(query);
    const replica = this.#replicas[this.#rr % this.#replicas.length];
    this.#rr++;
    return replica.unsafe(query);
  }
}
#08

Read Replicas

Prevents: read query load overloading the primary database

Replicas get async copies of all writes. Route SELECT queries to replicas. Accepts slight replication lag (~10–100ms) in exchange for horizontal read scale.

#09

Database Sharding

Prevents: write throughput ceiling on a single DB server

Split rows across multiple DB instances by a shard key (user_id % N). Each shard owns its subset. Cross-shard queries become expensive — choose the shard key to minimise them.

#10

Database Indexing

Prevents: full table scans on large tables (O(n) → O(log n))

Index every column used in WHERE, JOIN ON, or ORDER BY. B-tree for range queries. Hash for equality. Don't over-index — writes pay the penalty.

#11

SQL vs NoSQL

Prevents: choosing the wrong data model and paying for years

SQL: relational data, strong consistency, complex queries. NoSQL: flexible schema, horizontal write scale, simple access patterns. Most apps start SQL — switch only when you hit a specific wall.

#12

CAP Theorem

Prevents: believing you can have perfect consistency AND availability AND partition tolerance

Pick 2: Consistency (all nodes see same data), Availability (always responds), Partition Tolerance (works despite network splits). In practice: CP (ZooKeeper, HBase) or AP (Cassandra, DynamoDB). CA is only possible without partitions — not realistic in distributed systems.

Resilience patterns

Staying up when things break

// Circuit breaker implementation
class CircuitBreaker {
  #state = 'CLOSED'; // CLOSED → OPEN → HALF_OPEN
  #failures = 0;
  #lastFailTime = 0;
  #threshold;
  #resetTimeout;

  constructor({ threshold = 5, resetTimeout = 30_000 } = {}) {
    this.#threshold     = threshold;
    this.#resetTimeout  = resetTimeout;
  }

  async call(fn) {
    if (this.#state === 'OPEN') {
      if (Date.now() - this.#lastFailTime > this.#resetTimeout) {
        this.#state = 'HALF_OPEN'; // test if service recovered
      } else {
        throw new Error('Circuit OPEN — request rejected fast');
      }
    }

    try {
      const result = await fn();
      if (this.#state === 'HALF_OPEN') {
        this.#reset(); // service recovered
      }
      return result;
    } catch (err) {
      this.#record(err);
      throw err;
    }
  }

  #record(err) {
    this.#failures++;
    this.#lastFailTime = Date.now();
    if (this.#failures >= this.#threshold) {
      this.#state = 'OPEN';
      console.warn(`Circuit OPEN after ${this.#failures} failures`);
    }
  }

  #reset() {
    this.#state    = 'CLOSED';
    this.#failures = 0;
  }
}

const breaker = new CircuitBreaker({ threshold: 5, resetTimeout: 30_000 });
const data = await breaker.call(() => fetch('https://external-api.com/data').then(r => r.json()));
#13

Circuit Breaker

Prevents: cascading failures when a downstream service is slow or down

Three states: CLOSED (normal), OPEN (failing fast without calling service), HALF_OPEN (testing recovery). Protects the upstream service from overloading the already-struggling downstream.

#14

Retry with Exponential Backoff

Prevents: thundering herd when retries amplify load on a recovering service

Wait 1s, 2s, 4s, 8s... between retries. Add jitter (random ±20%) to prevent all clients retrying simultaneously. Cap at max retries (3–5) to avoid infinite loops.

#15

Bulkhead

Prevents: one slow consumer exhausting all connection pool slots

Isolate resource pools per downstream dependency. If Service A's pool is full, Service B's pool is unaffected. Named after ship watertight compartments.

#16

Timeout

Prevents: requests hanging indefinitely and exhausting thread/connection pools

Every outbound call needs a timeout. Rule of thumb: P99 latency × 3 is a reasonable start. Use read timeouts AND connection timeouts separately.

// Retry with exponential backoff + jitter
async function withRetry(fn, maxAttempts = 4, baseDelayMs = 500) {
  for (let attempt = 1; attempt <= maxAttempts; attempt++) {
    try {
      return await fn();
    } catch (err) {
      if (attempt === maxAttempts) throw err;
      const delay = baseDelayMs * 2 ** (attempt - 1);
      const jitter = delay * (0.8 + Math.random() * 0.4); // ±20% jitter
      console.log(`Attempt ${attempt} failed. Retrying in ${Math.round(jitter)}ms...`);
      await new Promise(r => setTimeout(r, jitter));
    }
  }
}
Messaging & async patterns

Decoupling for scale

#17

Message Queue

Prevents: slow jobs blocking request threads; producer overwhelming consumer

Producer publishes; consumer processes at its own pace. BullMQ (Redis-backed), SQS, RabbitMQ. Enables horizontal scaling of workers independently of API servers.

#18

Pub/Sub

Prevents: tight coupling between services that need to react to the same event

Publisher emits events; multiple subscribers react independently. Kafka, Redis Pub/Sub, AWS SNS. Decouples services — adding a new subscriber doesn't change the publisher.

#19

Event Sourcing

Prevents: losing audit history and difficulty replaying state

Store events (what happened), not current state (what is). Replay events to reconstruct state at any point in time. Enables CQRS, audit logs, and time-travel debugging.

#20

Idempotency

Prevents: duplicate charges, double sends, or corrupted state from retried requests

Same operation called N times = same result as called once. Implement via idempotency keys (UUID per operation, stored in DB). Essential for payments, emails, and any at-least-once delivery system.

// BullMQ job queue pattern
import { Queue, Worker } from 'bullmq';
import { createClient } from 'redis';

const connection = { host: 'localhost', port: 6379 };
const emailQueue = new Queue('emails', { connection });

// Producer — API handler
async function sendWelcomeEmail(userId) {
  await emailQueue.add('welcome', { userId }, {
    jobId: `welcome-${userId}`,  // idempotency: same userId won't queue twice
    attempts: 3,
    backoff: { type: 'exponential', delay: 2000 },
  });
}

// Consumer — separate worker process
const worker = new Worker('emails', async (job) => {
  const { userId } = job.data;
  const user = await getUserById(userId);
  await sendEmail({ to: user.email, template: 'welcome' });
  console.log(`Welcome email sent to ${user.email}`);
}, { connection, concurrency: 10 });
API design patterns

Designing APIs that age well

#21

Rate Limiting

Prevents: one client overwhelming the service; cost abuse; DDoS amplification

Token bucket or sliding window algorithms. Per-user and per-IP limits. Return 429 with Retry-After header. Implement in Redis for distributed rate limiting across nodes.

#22

API Gateway

Prevents: every microservice reimplementing auth, rate limiting, logging, and routing

Single entry point that handles cross-cutting concerns centrally. Kong, AWS API Gateway, Nginx. Routes to the right microservice. Terminates TLS. Enforces auth before requests reach services.

#23

API Versioning

Prevents: breaking existing clients when you evolve the API

URL versioning (/v1/users) is the simplest. Header versioning (Accept: application/vnd.api+json;version=2) is cleaner. Maintain at least one previous major version.

#24

Pagination

Prevents: returning millions of rows in a single response (OOM, timeout)

Offset pagination is simple but has skew under concurrent inserts. Cursor pagination (next_cursor token) is stable and scales. Always cap max page size server-side.

// Redis sliding window rate limiter
import { createClient } from 'redis';

const redis = createClient();

async function rateLimit(identifier, limit = 100, windowMs = 60_000) {
  const key = `ratelimit:${identifier}:${Math.floor(Date.now() / windowMs)}`;

  const [count] = await redis
    .multi()
    .incr(key)
    .expire(key, Math.ceil(windowMs / 1000) + 1)
    .exec();

  const remaining = Math.max(0, limit - count);
  const allowed = count <= limit;

  return { allowed, remaining, limit };
}

// Express middleware
function rateLimitMiddleware(limit = 100, windowMs = 60_000) {
  return async (req, res, next) => {
    const id = req.user?.id ?? req.ip;
    const { allowed, remaining } = await rateLimit(id, limit, windowMs);

    res.setHeader('X-RateLimit-Limit', limit);
    res.setHeader('X-RateLimit-Remaining', remaining);

    if (!allowed) return res.status(429).json({ error: 'Too Many Requests' });
    next();
  };
}
Distributed systems

The hard problems of distribution

#25

Consistent Hashing

Prevents: massive cache invalidation when a node is added or removed

Place nodes and keys on a hash ring. Adding a node only moves ~1/N of keys. Used by Redis Cluster, Cassandra, Memcached sharding. Virtual nodes (vnodes) smooth out uneven distribution.

#26

Leader Election

Prevents: split-brain: two nodes both believing they are the write leader

One node elected leader; others are followers. Raft and Paxos are the consensus algorithms. ZooKeeper, etcd, and Consul handle this in managed form. Kafka uses KRaft (Kafka Raft) internally.

#27

Service Discovery

Prevents: hardcoded service addresses breaking when containers restart with new IPs

Services register themselves on startup; clients query the registry. Consul, Kubernetes DNS, AWS Cloud Map. Health checks remove dead registrations automatically.

#28

Observability (Logs + Metrics + Traces)

Prevents: debugging production incidents with no information

Logs = what happened. Metrics = how fast/how many. Traces = the path through the system. The three pillars. OpenTelemetry is the standard for instrumenting all three in one pass.

// OpenTelemetry tracing setup for Node.js
import { NodeSDK } from '@opentelemetry/sdk-node';
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http';
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node';

const sdk = new NodeSDK({
  traceExporter: new OTLPTraceExporter({ url: 'http://localhost:4318/v1/traces' }),
  instrumentations: [getNodeAutoInstrumentations()], // auto-instruments http, express, pg, redis
});
sdk.start();

// Manual span for custom operations
import { trace } from '@opentelemetry/api';
const tracer = trace.getTracer('my-service');

async function processOrder(orderId) {
  const span = tracer.startSpan('process-order');
  span.setAttribute('order.id', orderId);

  try {
    const order = await fetchOrder(orderId);
    span.setAttribute('order.amount', order.total);
    await chargePayment(order);
    span.setStatus({ code: SpanStatusCode.OK });
    return order;
  } catch (err) {
    span.recordException(err);
    span.setStatus({ code: SpanStatusCode.ERROR });
    throw err;
  } finally {
    span.end();
  }
}

Key Takeaway

Every system design concept is a battle-tested response to a documented class of failure. Learn the failure first — cascading failures, hot-shard contention, replication lag, thundering herds — and the concept that prevents it becomes intuitive rather than arbitrary. The 28 patterns above cover the failures that every production system will eventually encounter at scale.
