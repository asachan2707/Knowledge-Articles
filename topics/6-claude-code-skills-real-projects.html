The 6 Claude Code Skills I Would Use for Real Projects

Not a list of every skill that exists — a curated shortlist of the six that survive contact with actual production work, why each one earns its place, and how to configure them to stop being a novelty and start being leverage.

Portal Internal · Curated
Focus Production-grade Claude Code skill selection and configuration
Published Jun 2026

The Claude Code skill ecosystem has more than 100 community skills and a growing list of official ones. Most are impressive in demos, useless in practice. The filter is simple: does it save meaningful time on work I do every week? Six skills pass that test. The others are worth knowing about but not worth loading by default.

Core idea: Skill selection is a configuration discipline, not a discovery game. Load fewer skills, use each one deeply, and reload your configuration when a new model ships.

Story · What skill overload looks like

After adding a dozen skills to a CLAUDE.md file, the session startup becomes slow, the model gets confused about which skill applies to which task, and context burns faster. Stripping back to six — with a CLAUDE.md entry for each explaining exactly when to invoke it — cut that friction entirely.

The six

Skills that earn their place in real projects

1. /code-review — the one you run before every PR

Runs a structured code review across correctness, security, performance, and readability dimensions. The key is using it at two effort levels: /code-review --effort=low before committing (catches obvious bugs, 30 seconds) and /code-review --effort=high before opening a PR (deeper cross-file analysis, 3–5 minutes). The high-effort pass with --fix applies the findings automatically.

2. /frontend-design — for UI work that doesn't look AI-generated

Guides Claude away from generic template choices toward deliberate design decisions: typography hierarchy, spacing system, colour semantics, component composition. Without it, Claude defaults to visually safe but characterless outputs. With it, you get a system prompt tuned to push toward craft. Most useful when building UI from scratch or doing significant visual refactors.

3. /simplify — refactoring pass without scope creep

Reviews changed code for reuse, simplification, and efficiency opportunities, then applies them. Critically, it is quality-only — it does not hunt for bugs or introduce new features. This makes it safe to run after any significant implementation: it cleans up without expanding scope. Pairs well with running it immediately after a feature is working but before committing.

4. /init — project bootstrapping in one command

Scans a new codebase and generates a CLAUDE.md file tailored to what it finds — tech stack, conventions, known patterns, tooling. Saves the 30-minute manual process of writing a CLAUDE.md from scratch. Re-run it quarterly to keep the file accurate as the project evolves.

5. /deep-research — when you need cited, multi-source answers

Runs a fan-out web search, fetches multiple sources, adversarially verifies claims, and returns a cited report. The key use case is architectural decisions: "what is the current best practice for rate limiting at the edge in 2026?" gets you a researched answer in 5 minutes instead of 45. Don't use it for questions that a single documentation page answers — the overhead isn't worth it.

6. /run — verifying that the thing actually works

Launches the project, runs it, and checks that a specific change behaves correctly in the live app. The gap between "tests pass" and "feature works" is wider than most developers admit. This skill bridges it by actually opening the app, navigating to the relevant UI, and confirming behaviour — not just checking that the code compiles.

Configuration

How to wire these into your workflow

CLAUDE.md entry pattern

Each skill needs a short entry in your CLAUDE.md explaining when Claude should proactively invoke it. Without this, you remember to use the skill only when you type the slash command. With it, Claude surfaces the skill at the right moment automatically. 

## Skills
- Run /code-review --effort=low before any commit touching more than 3 files.
- Run /simplify after any feature implementation before committing.
- Run /frontend-design when generating new UI components or layouts from scratch.
- Run /run after any bug fix to confirm the fix works in the live app.

Skill load strategy

  • Load only the skills you use at least once per week. Everything else adds context overhead without value.
  • Re-evaluate your skill set when a new Claude model ships. Capabilities that required a skill in the previous model sometimes become native behaviour in the next.
  • Per-project CLAUDE.md beats a global one for skill configuration. Your backend API project needs different skills than your frontend dashboard.
Common questions

FAQ

  • What about community skills — are any worth adding?

    A few. The best community skills are narrow and well-tested: skills for specific frameworks (e.g., Rails conventions, FastAPI patterns) or specific workflows (e.g., changelog generation). Avoid skills that overlap with what these six already do — duplication confuses the model about which one to invoke.

  • Can I build a custom skill that isn't available officially?

    Yes. A SKILL.md file is a markdown document with a YAML frontmatter block and an instruction body. If you have a workflow you run more than once a week, it is worth codifying as a skill. The investment is 30–60 minutes; the return is every subsequent execution.

  • Do skills work in the IDE extensions (VS Code, JetBrains)?

    Yes. Skills load from the same CLAUDE.md and skills registry regardless of whether you are using the CLI, the desktop app, or an IDE extension. The slash command interface is consistent across surfaces.

Key Takeaway

Six well-configured skills that match your actual workflow beat thirty skills you invoke randomly. Load less, configure precisely, and revisit the list every time a new model ships.
