Stop Designing REST APIs Like a Mid-Level Dev: 4 Advanced Patterns Senior Engineers Use Instead

The four API design patterns that separate senior engineers from mid-level ones — not syntax, but structural decisions around resource modelling, versioning, evolvability, and client performance that prevent costly migrations later.

Portal Internal · Curated
Focus REST API advanced patterns: resource design, versioning, HATEOAS, batch/async operations
Published May 2026

Mid-level REST API design is correct but rigid: resources mapped directly to database tables, versions bumped with every breaking change, pagination bolted on afterwards, and error responses that give clients no information about what to do next. Senior API design is also correct — but built to evolve, survive client diversity, and remain usable as the system scales. The four patterns below are where that gap lives.

Core idea: The cost of a bad API decision is not immediate — it appears eighteen months later when you need to change something, and the change is now a migration that breaks every existing client. Senior API design front-loads that thinking.

Story · The versioning trap

The pattern is nearly universal: a team builds /v1/users, ships it, then realises the response shape needs to change. So they build /v2/users. Then /v3. Two years later they are maintaining four parallel implementations of the same endpoint, with mobile clients still pinned to /v1 that cannot be force-upgraded. This is not a versioning problem — it is a resource design and evolvability problem that versioning was expected to solve.

Pattern 1 — Resource design

Design resources around client needs, not database shape

The mid-level instinct is to expose a REST resource for every database table: GET /orders, GET /line-items, GET /products. This is correct from a REST purity standpoint and actively harmful for client performance and evolvability.

The N+1 API problem

A mobile client building an order summary screen needs the order, its line items, and the product name and image for each item. With table-mapped resources, that requires:

  1. GET /orders/42 — 1 request
  2. GET /orders/42/line-items — 1 request
  3. GET /products/{id} for each line item — N requests

That is N+2 round trips on a mobile network. A senior engineer asks: what does the client actually need, and can we deliver it in one response?

Compound documents and sparse fieldsets

The solution is not collapsing everything into one giant response — it is designing resources that serve use cases. A GET /orders/42?include=line_items,products&fields[products]=id,name,image_url request returns the order with embedded line items and lightweight product stubs — one round trip, exactly the data the screen needs.

This is the JSON:API pattern: compound documents (embed related resources) and sparse fieldsets (request only needed fields). It keeps the API RESTful while serving clients efficiently.

Intent-based resources for writes

Write endpoints that map to intents, not state changes: POST /orders/42/confirm, POST /orders/42/cancel, POST /orders/42/ship are clearer, safer, and more evolvable than PATCH /orders/42 {status: "confirmed"}}. Intent-based endpoints can enforce business rules at the route level and produce domain events naturally.

Pattern 2 — Evolvability without versioning

Additive changes and Postel's Law

Most breaking API changes are not actually breaking — they are just not designed to be additive. Senior engineers design APIs to evolve without major versions by following a strict discipline: only add, never remove or rename.

Additive-only changes

  • Add fields, never remove: Mark deprecated fields with a deprecated_at header or field annotation and keep them in responses for 12+ months.
  • Add optional parameters, never make existing ones required: Clients that don't send a parameter should get a reasonable default, not a 400.
  • Add new endpoints for new behaviour: If the semantics of an existing endpoint need to change, add a new endpoint and deprecate the old one.

Tolerant reader, conservative writer

Postel's Law applied to APIs: be conservative in what your API sends and liberal in what it accepts. Clients following the inverse (strict parsers, flexible generators) break on every new field. Design clients that ignore unknown fields — this alone eliminates most "breaking change" situations.

When versioning is actually required

Version the API when the semantic contract changes fundamentally — a different authentication model, a completely different resource structure, a protocol change. That is maybe once every few years for a stable API. Every other change should be backwards-compatible. If your team is versioning monthly, the problem is not the versioning — it is the upfront design.

Pattern 3 — HATEOAS and discoverability

Links, not hardcoded paths

Hypermedia As The Engine Of Application State (HATEOAS) is the REST constraint that most APIs skip. It means responses include links to related actions — so clients discover what they can do next from the response, rather than hardcoding paths. 

{
  "id": 42,
  "status": "pending",
  "total": 59.99,
  "_links": {
    "self":    { "href": "/orders/42" },
    "confirm": { "href": "/orders/42/confirm", "method": "POST" },
    "cancel":  { "href": "/orders/42/cancel",  "method": "POST" },
    "items":   { "href": "/orders/42/line-items" }
  }
}

The practical payoff: when URLs change (and they do), clients that follow links from responses update automatically. When new actions become available based on state, they appear in the links — the client's UI can enable/disable buttons based on link presence without hardcoding business logic about which states allow which actions.

Pragmatic HATEOAS

Full HATEOAS adoption is a high bar. The pragmatic minimum: include a _links object with self and the primary actions available for this resource in its current state. That 20% of the pattern delivers 80% of the benefit.

Pattern 4 — Batch and async operations

Designing for operations that take longer than an HTTP request

Mid-level API design treats every operation as synchronous: request in, response out, done. Senior API design recognises two classes of operation that this breaks: batch operations (one request affecting many resources) and long-running operations (processing time exceeds connection timeout).

Batch endpoints

Instead of requiring 500 individual DELETE /items/{id} calls to clear a cart, provide: POST /items/batch-delete with an array of IDs in the body. Return a bulk result: how many succeeded, which failed and why. 

// Request
POST /items/batch-delete
{ "ids": [1, 2, 3, 4, 5] }

// Response
{
  "succeeded": [1, 3, 4, 5],
  "failed": [
    { "id": 2, "reason": "item_locked", "message": "Item is in a pending order" }
  ]
}

Batch endpoints require partial success semantics — returning both what worked and what failed rather than all-or-nothing, because client code needs to handle partial failures.

Async operations with job resources

For operations that take more than a few seconds, return a job resource immediately and let the client poll: 

POST /reports/generate
→ 202 Accepted
{
  "job_id": "rpt_abc123",
  "_links": {
    "status": { "href": "/jobs/rpt_abc123" },
    "cancel":  { "href": "/jobs/rpt_abc123", "method": "DELETE" }
  }
}

GET /jobs/rpt_abc123
→ { "status": "running", "progress": 0.4, "eta_seconds": 12 }

GET /jobs/rpt_abc123
→ {
    "status": "complete",
    "_links": { "result": { "href": "/reports/rpt_abc123" } }
  }

The 202 Accepted + job resource pattern keeps HTTP semantics clean, gives clients visibility into progress, and enables cancellation. Add a webhook option for clients that prefer push over poll.

Common questions

FAQ

  • Should I use GraphQL instead to avoid these problems?

    GraphQL solves the N+1 API problem and sparse fieldsets elegantly — but introduces complexity in caching, rate limiting, and query security (cost analysis, depth limiting). For a public API or a single team consuming it, REST with these patterns works well. For multiple internal teams with diverse data needs, GraphQL is worth the complexity. Both require the same underlying thinking about intent-based mutations and async operations.

  • How do I convince my team to adopt HATEOAS when it adds response size?

    Start with intent-based write endpoints and the job resource pattern — these have obvious, immediate value that teams can see. HATEOAS on read responses is a harder sell. Frame it as: "links make our URL structure a private implementation detail" — when a senior engineer on the team has had to fix a client because a URL changed, that argument lands.

  • What is the most common mistake teams make when adding versioning?

    Versioning the whole API instead of a specific resource. When one endpoint's contract needs to change, bumping from /v1 to /v2 requires migrating every client across every endpoint simultaneously. Version individual resources (or use content negotiation headers) to limit the migration surface to what actually changed.

Key Takeaway

The four patterns that separate senior API design from mid-level API design: resources built around client use cases (not tables), additive-only changes that avoid versioning debt, HATEOAS links that decouple clients from URL structure, and async job resources for long-running operations. The investment pays off 18 months later when the system needs to change and the change doesn't break everything.
