Concept 1

Server Components React 19 — zero client JavaScript

React Server Components (RSC) run on the server — they can access databases, filesystems, and secrets directly, and their output is serialized HTML + React tree. They ship zero JavaScript to the browser. The client/server split is now at the component level, not the page level.

Old model: pages are client or server SSR page → runs on server, hydrates with ALL component JS on client CSR page → runs entirely in browser, no server rendering SPA route → client JS only, requires fetch after load React 19 model: components are client or server Server Component → runs on server, zero JS sent to browser Client Component → runs in browser (interactive, hooks, event handlers) Shared boundary → parent is server, child is client — data passes as props 
// ─── Server Component (default in Next.js App Router) ────────────────────
// No 'use client' → Server Component
// Can: await, access DB, read filesystem, use server secrets
// Cannot: useState, useEffect, event handlers

// app/products/[id]/page.tsx
export default async function ProductPage({ params }: { params: Promise<{ id: string }> }) {
  const { id } = await params;

  // Direct DB access — no API round trip, no loading state
  const product = await prisma.product.findUnique({
    where: { id },
    include: { reviews: { take: 5, orderBy: { createdAt: 'desc' } } },
  });

  if (!product) return notFound();

  return (
    <div>
      <h1>{product.name}</h1>
      <p>${(product.priceCents / 100).toFixed(2)}</p>
      <p>{product.description}</p>

      {/* Server-rendered review list — zero JS */}
      <ReviewList reviews={product.reviews} />

      {/* Client component for interactivity */}
      <AddToCartButton productId={id} priceCents={product.priceCents} />
    </div>
  );
}

// ─── Client Component — marked with 'use client' directive ───────────────
// components/AddToCartButton.tsx
'use client';

function AddToCartButton({ productId, priceCents }: { productId: string; priceCents: number }) {
  const { addItem }   = useCartStore();
  const [added, setAdded] = useState(false);

  const handleAdd = () => {
    addItem({ id: productId, priceCents });
    setAdded(true);
    setTimeout(() => setAdded(false), 2000);
  };

  return (
    <button onClick={handleAdd}>
      {added ? '✓ Added!' : 'Add to Cart'}
    </button>
  );
}
The biggest mental shift: In RSC, your component tree can span the server/client boundary. A server component can render a client component as a child, passing server-fetched data as props. The client component receives the data but doesn't need to fetch it — it was already there.
Concept 2

Server Actions React 19 — form handling without API routes

Server Actions are async functions marked with 'use server' that run on the server but can be called from client components. They replace the need for dedicated API route handlers for form submissions.

// ─── Server Action definition ─────────────────────────────────────────────
// actions/product.ts
'use server';

import { revalidatePath } from 'next/cache';
import { z } from 'zod';

const CreateProductSchema = z.object({
  name:        z.string().min(1),
  priceCents:  z.number().int().positive(),
  description: z.string(),
});

export async function createProduct(formData: FormData) {
  const session = await getServerSession();
  if (!session?.user.isAdmin) throw new Error('Unauthorized');

  const parsed = CreateProductSchema.safeParse({
    name:        formData.get('name'),
    priceCents:  Number(formData.get('priceCents')),
    description: formData.get('description'),
  });

  if (!parsed.success) {
    return { error: parsed.error.flatten().fieldErrors };
  }

  const product = await prisma.product.create({ data: parsed.data });

  revalidatePath('/products');  // invalidate Next.js cache for this path
  redirect(`/products/${product.id}`);  // server-side redirect
}

// ─── Using Server Action directly in a form ───────────────────────────────
// (no 'use client' needed — form action runs on server)
function CreateProductForm() {
  return (
    <form action={createProduct}>
      <input name="name" placeholder="Product name" required />
      <input name="priceCents" type="number" placeholder="Price in cents" />
      <textarea name="description"></textarea>
      <button type="submit">Create Product</button>
    </form>
  );
}

// ─── Server Action from a Client Component ────────────────────────────────
'use client';
import { createProduct } from '@/actions/product';

function CreateProductFormClient() {
  const [state, formAction, isPending] = useActionState(createProduct, null);

  return (
    <form action={formAction}>
      <input name="name" />
      {state?.error?.name && <p className="error">{state.error.name[0]}</p>}
      <button type="submit" disabled={isPending}>
        {isPending ? 'Creating...' : 'Create Product'}
      </button>
    </form>
  );
}
Concept 3

useActionState React 19 — form state management

useActionState (previously useFormState) manages the state returned by a Server Action, plus a pending state — eliminating the manual loading/error/data tri-state pattern for forms.

'use client';
import { useActionState } from 'react';
import { loginAction } from '@/actions/auth';

// ─── loginAction (server action) returns ActionResult ─────────────────────
type ActionResult = { error: string } | { success: true; redirect: string } | null;

function LoginForm() {
  const [state, formAction, isPending] = useActionState<ActionResult, FormData>(
    loginAction,
    null  // initial state
  );

  return (
    <form action={formAction}>
      <div>
        <label htmlFor="email">Email</label>
        <input type="email" id="email" name="email" required />
      </div>

      <div>
        <label htmlFor="password">Password</label>
        <input type="password" id="password" name="password" required />
      </div>

      {state && 'error' in state && (
        <div role="alert" className="error">{state.error}</div>
      )}

      <button type="submit" disabled={isPending}>
        {isPending ? 'Logging in...' : 'Log In'}
      </button>
    </form>
  );
}

// ─── useFormStatus — read pending state inside a form ─────────────────────
import { useFormStatus } from 'react-dom';

// This component must be INSIDE a <form> — it reads the parent form's status
function SubmitButton() {
  const { pending } = useFormStatus();
  return (
    <button type="submit" disabled={pending}>
      {pending ? <Spinner /> : 'Submit'}
    </button>
  );
}
Concept 4

use() API React 19 — promises and context in render

The use() hook can unwrap promises and context values during rendering. Unlike other hooks, it can be called conditionally and inside loops — a fundamental change to React's hook rules.

import { use, Suspense } from 'react';

// ─── use() with a promise ─────────────────────────────────────────────────
// Pass a Promise to use() — React suspends until it resolves
// The component must be wrapped in Suspense

function UserProfile({ userPromise }: { userPromise: Promise<User> }) {
  const user = use(userPromise);  // suspends until resolved
  return <h1>{user.name}</h1>;
}

// Parent creates the promise and passes it down
// (creates it at render time so Suspense can catch it)
function UserPage({ userId }: { userId: string }) {
  const userPromise = fetchUser(userId);  // Promise starts immediately

  return (
    <Suspense fallback={<Skeleton />}>
      <UserProfile userPromise={userPromise} />
    </Suspense>
  );
}

// ─── use() with context ───────────────────────────────────────────────────
// use() is an alternative to useContext() — it CAN be used conditionally
function UserInfo({ showEmail }: { showEmail: boolean }) {
  if (!showEmail) return null;

  // useContext cannot be called conditionally — use() can
  const user = use(UserContext);
  return <p>{user?.email}</p>;
}

// ─── Passing server data to client components via use() ───────────────────
// Server Component passes a streaming Promise — client receives it via use()
// This enables streaming data from server to client without waterfalls

// Server Component
async function ProductPage() {
  // Both start fetching in parallel
  const productPromise  = fetchProduct(id);
  const reviewsPromise  = fetchReviews(id);

  return (
    <>
      <Suspense fallback={<ProductSkeleton />}>
        <ProductDetails productPromise={productPromise} />
      </Suspense>

      <Suspense fallback={<ReviewsSkeleton />}>
        <ReviewList reviewsPromise={reviewsPromise} />
      </Suspense>
    </>
  );
  // Product and reviews stream in independently — no waterfall
}

Edge Rendering — compute close to the user

Edge functions run in data centers distributed globally — the same code that runs in Node.js runs 20–50ms from any user instead of 200ms+ from a single-region server. Critical for authentication middleware and personalized pages.

// ─── Next.js Edge Middleware (runs on every request at the CDN edge) ───────
// middleware.ts (runs at Cloudflare/Vercel edge — not Node.js)
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';

export async function middleware(request: NextRequest) {
  const { pathname } = request.nextUrl;

  // Auth check at the edge — before Next.js even starts rendering
  if (pathname.startsWith('/dashboard')) {
    const token = request.cookies.get('session')?.value;

    if (!token) {
      return NextResponse.redirect(new URL('/login', request.url));
    }

    // Verify JWT at the edge (using Web Crypto — no Node.js available)
    const isValid = await verifyToken(token);
    if (!isValid) {
      const res = NextResponse.redirect(new URL('/login', request.url));
      res.cookies.delete('session');
      return res;
    }
  }

  // A/B testing at the edge — zero latency
  if (pathname === '/') {
    const bucket = request.cookies.get('ab-bucket')?.value ?? (Math.random() < 0.5 ? 'a' : 'b');
    const res = NextResponse.rewrite(new URL(`/variants/${bucket}`, request.url));
    res.cookies.set('ab-bucket', bucket, { maxAge: 60 * 60 * 24 * 30 });
    return res;
  }

  return NextResponse.next();
}

export const config = {
  matcher: ['/dashboard/:path*', '/'],
};

// ─── Edge API Route ───────────────────────────────────────────────────────
// app/api/personalize/route.ts
export const runtime = 'edge';

export async function GET(request: Request) {
  const { searchParams } = new URL(request.url);
  const userId = searchParams.get('userId');

  // Runs at edge — use global Web APIs, not Node.js
  const userData = await fetch(`https://profile-service.internal/users/${userId}`, {
    headers: { Authorization: `Bearer ${process.env.INTERNAL_TOKEN}` },
  });

  return Response.json(await userData.json());
}
Edge vs Node.js runtime: Edge runs Web APIs (fetch, Response, Request, crypto, URL) but NOT Node.js APIs (fs, path, Buffer, process). Most business logic works. Database drivers like Prisma require connection poolers (PgBouncer, Neon serverless) because WebSockets, not TCP, are available at the edge.

Agentic Apps — streaming AI responses in React

Agentic apps build on React's streaming primitives to handle AI responses that arrive incrementally. The Vercel AI SDK provides the React-native patterns for this — streaming text, structured outputs, and tool call rendering.

// npm install ai @ai-sdk/openai
// app/api/chat/route.ts (Next.js App Router)
import { streamText } from 'ai';
import { openai } from '@ai-sdk/openai';

export async function POST(req: Request) {
  const { messages } = await req.json();

  const result = streamText({
    model: openai('gpt-4o'),
    messages,
    system: 'You are a helpful shopping assistant.',
    tools: {
      searchProducts: {
        description: 'Search for products',
        parameters: z.object({ query: z.string() }),
        execute: async ({ query }) => searchProductsDB(query),
      },
    },
  });

  return result.toDataStreamResponse();
}

// ─── Client: streaming chat UI ─────────────────────────────────────────────
'use client';
import { useChat } from 'ai/react';

function AIChatInterface() {
  const { messages, input, handleInputChange, handleSubmit, isLoading, stop } = useChat({
    api: '/api/chat',
    onError: (err) => showToast({ type: 'error', message: err.message }),
  });

  return (
    <div className="flex flex-col h-screen">
      <div className="flex-1 overflow-y-auto p-4 space-y-4">
        {messages.map(message => (
          <div key={message.id} className={`flex ${message.role === 'user' ? 'justify-end' : 'justify-start'}`}>
            <div className={`max-w-lg rounded-2xl px-4 py-2 ${
              message.role === 'user'
                ? 'bg-blue-600 text-white'
                : 'bg-gray-100 text-gray-900'
            }`}>
              {/* Text content — streams in word by word */}
              {message.content}

              {/* Tool call results */}
              {message.toolInvocations?.map(tool => (
                tool.state === 'result' && (
                  <ProductSearchResults key={tool.toolCallId} results={tool.result} />
                )
              ))}
            </div>
          </div>
        ))}

        {isLoading && (
          <div className="flex justify-start">
            <div className="bg-gray-100 rounded-2xl px-4 py-2">
              <TypingIndicator />
            </div>
          </div>
        )}
      </div>

      <form onSubmit={handleSubmit} className="border-t p-4 flex gap-2">
        <input
          value={input}
          onChange={handleInputChange}
          placeholder="Ask about products..."
          className="flex-1 border rounded-lg px-3 py-2"
        />
        {isLoading
          ? <button type="button" onClick={stop}>Stop</button>
          : <button type="submit">Send</button>
        }
      </form>
    </div>
  );
}

// ─── Streaming structured output ───────────────────────────────────────────
import { useObject } from 'ai/react';
import { z } from 'zod';

const ProductSchema = z.object({
  recommendations: z.array(z.object({
    id: z.string(),
    name: z.string(),
    reason: z.string(),
    score: z.number(),
  })),
});

function ProductRecommender({ userId }: { userId: string }) {
  const { object, submit, isLoading } = useObject({
    api: '/api/recommend',
    schema: ProductSchema,
  });

  return (
    <div>
      <button onClick={() => submit({ userId })}>Get Recommendations</button>

      {/* Streams in as JSON is partially parsed — items appear one by one */}
      {object?.recommendations?.map((rec, i) => rec && (
        <ProductRecommendationCard key={i} recommendation={rec} />
      ))}
    </div>
  );
}
React 19's streaming architecture makes it the natural home for AI UIs. Suspense + streaming + Server Components + use() all compose together: the AI response streams from the edge, React progressively hydrates the content, and the user sees text appearing in real-time without any visible loading state for the parts that have already arrived.

Chapter 11 — Final Takeaway

React 19 blurs the server/client boundary in a fundamental way. Server Components run with zero client JavaScript, Server Actions replace API routes for mutations, and streaming turns React into a real-time content delivery system. useActionState and use() complete the picture on the client side. These features compose beautifully with the Agentic Apps pattern — AI responses stream in progressively, tool calls render inline, and the whole system feels instant because content arrives incrementally rather than in one batch. This is where React is heading — less JavaScript on the client, more work done close to the data.
