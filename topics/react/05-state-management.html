State Management

When Context API is enough, when to reach for Redux Toolkit, and the modern alternatives — Zustand, Jotai, and TanStack Store. Plus the global state design principles that keep large apps maintainable.

Concept 1

Context API — built-in React state sharing

Context solves prop drilling — passing props through many layers to reach a deeply nested child. It is best for low-frequency updates like theme, locale, and auth state. It is NOT a high-performance global store.

// ─── Theme context — typical Context API use case ─────────────────────────
import { createContext, useContext, useState, useCallback } from 'react';

const ThemeContext = createContext(null);

export function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light');

  const toggle = useCallback(() =>
    setTheme(t => t === 'light' ? 'dark' : 'light'), []);

  return (
    <ThemeContext.Provider value={{ theme, toggle }}>
      {children}
    </ThemeContext.Provider>
  );
}

export const useTheme = () => {
  const ctx = useContext(ThemeContext);
  if (!ctx) throw new Error('useTheme must be inside ThemeProvider');
  return ctx;
};

// ─── The performance problem with Context ─────────────────────────────────
// Every consumer re-renders when ANY value in the context changes
// Solution: split into multiple contexts by update frequency

const UserContext     = createContext(null);  // changes: login/logout only
const NotifContext    = createContext(null);  // changes: every new notification
const SettingsContext = createContext(null);  // changes: user preferences

// Now a component consuming UserContext won't re-render when a notification arrives
Context is not a state management solution for frequent updates. A cart with 100 items that changes on every keystroke will re-render every consumer on every change. Use Zustand or Redux Toolkit for high-frequency mutable global state.
Concept 2

Redux Toolkit — the modern Redux

Redux Toolkit (RTK) is the official, opinionated way to write Redux. It eliminates the boilerplate of vanilla Redux — no more action type strings, action creators, or immutable update patterns. Use it when you need a centralized, debuggable, serialisable state store.

// npm install @reduxjs/toolkit react-redux
import { createSlice, createAsyncThunk, configureStore } from '@reduxjs/toolkit';
import { Provider, useSelector, useDispatch } from 'react-redux';

// ─── Slice — combines reducer + actions in one ────────────────────────────
const cartSlice = createSlice({
  name: 'cart',
  initialState: { items: [], status: 'idle', error: null },
  reducers: {
    addItem(state, action) {
      // RTK uses Immer — you CAN mutate directly here (it produces a new state)
      const existing = state.items.find(i => i.id === action.payload.id);
      if (existing) { existing.qty += 1; }
      else { state.items.push({ ...action.payload, qty: 1 }); }
    },
    removeItem(state, action) {
      state.items = state.items.filter(i => i.id !== action.payload);
    },
    clearCart(state) {
      state.items = [];
    },
  },
  extraReducers: (builder) => {
    builder
      .addCase(checkoutCart.pending,   (state) => { state.status = 'loading'; })
      .addCase(checkoutCart.fulfilled, (state) => { state.items = []; state.status = 'idle'; })
      .addCase(checkoutCart.rejected,  (state, action) => { state.error = action.error.message; state.status = 'idle'; });
  },
});

// ─── Async thunk for API calls ─────────────────────────────────────────
const checkoutCart = createAsyncThunk('cart/checkout', async (cartItems) => {
  const res = await fetch('/api/checkout', {
    method: 'POST',
    body: JSON.stringify({ items: cartItems }),
    headers: { 'Content-Type': 'application/json' },
  });
  if (!res.ok) throw new Error('Checkout failed');
  return res.json();
});

export const { addItem, removeItem, clearCart } = cartSlice.actions;

// ─── Store ────────────────────────────────────────────────────────────────
const store = configureStore({
  reducer: { cart: cartSlice.reducer },
});

// ─── In components ────────────────────────────────────────────────────────
function CartSummary() {
  const items    = useSelector(state => state.cart.items);
  const dispatch = useDispatch();

  const total = items.reduce((s, i) => s + i.priceCents * i.qty, 0);

  return (
    <div>
      {items.map(item => (
        <div key={item.id}>
          {item.name} × {item.qty}
          <button onClick={() => dispatch(removeItem(item.id))}>×</button>
        </div>
      ))}
      <p>Total: ${(total / 100).toFixed(2)}</p>
      <button onClick={() => dispatch(checkoutCart(items))}>Checkout</button>
    </div>
  );
}

function AppRoot() {
  return <Provider store={store}><App /></Provider>;
}
Concept 3

Zustand — minimal, boilerplate-free global state

Zustand is a tiny state management library (8kb) that combines the best ideas from Redux (external store, selectors) with the simplicity of Context. No providers, no reducers, no action creators.

// npm install zustand
import { create } from 'zustand';
import { persist } from 'zustand/middleware';

// ─── Cart store ───────────────────────────────────────────────────────────
const useCartStore = create(
  persist(
    (set, get) => ({
      items: [],

      addItem: (product) => set(state => {
        const existing = state.items.find(i => i.id === product.id);
        if (existing) {
          return { items: state.items.map(i => i.id === product.id ? { ...i, qty: i.qty + 1 } : i) };
        }
        return { items: [...state.items, { ...product, qty: 1 }] };
      }),

      removeItem: (id) => set(state => ({
        items: state.items.filter(i => i.id !== id),
      })),

      clearCart: () => set({ items: [] }),

      get total() {
        return get().items.reduce((s, i) => s + i.priceCents * i.qty, 0);
      },
    }),
    { name: 'cart-storage' }  // persist to localStorage automatically
  )
);

// ─── Usage — subscribe only to what you need ─────────────────────────────
function CartIcon() {
  const count = useCartStore(state => state.items.length);  // only re-renders when count changes
  return <span>🛒 {count}</span>;
}

function CartPage() {
  const items     = useCartStore(state => state.items);
  const addItem   = useCartStore(state => state.addItem);
  const clearCart = useCartStore(state => state.clearCart);

  return (
    <div>
      {items.map(item => (
        <CartRow key={item.id} item={item} />
      ))}
      <button onClick={clearCart}>Clear</button>
    </div>
  );
}

// ─── Multiple stores is fine — Zustand encourages domain separation ───────
const useAuthStore = create((set) => ({
  user: null,
  setUser: (user) => set({ user }),
  logout: () => set({ user: null }),
}));

const useUIStore = create((set) => ({
  sidebarOpen: false,
  toggleSidebar: () => set(s => ({ sidebarOpen: !s.sidebarOpen })),
}));
Concept 4

Jotai — atomic state management

Jotai is inspired by Recoil. Instead of one big store, state is broken into small atoms. Components subscribe only to the atoms they need — no unnecessary re-renders. Great for fine-grained reactivity.

// npm install jotai
import { atom, useAtom, useAtomValue, useSetAtom } from 'jotai';
import { atomWithStorage } from 'jotai/utils';

// ─── Define atoms ─────────────────────────────────────────────────────────
const userAtom         = atom(null);
const themeAtom        = atomWithStorage('theme', 'light');  // auto-persisted
const cartItemsAtom    = atom([]);

// Derived atom — computed from other atoms, no setter needed
const cartCountAtom = atom(get => get(cartItemsAtom).length);
const cartTotalAtom = atom(get =>
  get(cartItemsAtom).reduce((s, i) => s + i.priceCents * i.qty, 0)
);

// Write atom — derived atom with a custom setter
const addToCartAtom = atom(
  get => get(cartItemsAtom),
  (get, set, product) => {
    const items = get(cartItemsAtom);
    const existing = items.find(i => i.id === product.id);
    if (existing) {
      set(cartItemsAtom, items.map(i => i.id === product.id ? { ...i, qty: i.qty + 1 } : i));
    } else {
      set(cartItemsAtom, [...items, { ...product, qty: 1 }]);
    }
  }
);

// ─── Usage ────────────────────────────────────────────────────────────────
function CartBadge() {
  const count = useAtomValue(cartCountAtom);  // read only, no setter overhead
  return <span>{count}</span>;
}

function AddToCartButton({ product }) {
  const addToCart = useSetAtom(addToCartAtom);  // setter only, no re-render on value change
  return <button onClick={() => addToCart(product)}>Add to cart</button>;
}

function ThemeToggle() {
  const [theme, setTheme] = useAtom(themeAtom);
  return <button onClick={() => setTheme(t => t === 'light' ? 'dark' : 'light')}>{theme}</button>;
}
Concept 5

TanStack Store — framework-agnostic reactive store

TanStack Store is the headless state management solution from the TanStack ecosystem. It is framework-agnostic (React, Vue, Solid, Angular) and very lightweight.

// npm install @tanstack/store @tanstack/react-store
import { Store } from '@tanstack/store';
import { useStore } from '@tanstack/react-store';

// ─── Define the store ─────────────────────────────────────────────────────
const cartStore = new Store({
  items: [],
});

// Actions are just functions that call store.setState
function addToCart(product) {
  cartStore.setState(state => {
    const existing = state.items.find(i => i.id === product.id);
    if (existing) {
      return { items: state.items.map(i => i.id === product.id ? { ...i, qty: i.qty + 1 } : i) };
    }
    return { items: [...state.items, { ...product, qty: 1 }] };
  });
}

function clearCart() {
  cartStore.setState(() => ({ items: [] }));
}

// ─── Usage ────────────────────────────────────────────────────────────────
function CartCount() {
  // Second arg is a selector — component only re-renders when this value changes
  const count = useStore(cartStore, state => state.items.length);
  return <span>{count} items</span>;
}

function CartPage() {
  const items = useStore(cartStore, state => state.items);
  return (
    <div>
      {items.map(i => <div key={i.id}>{i.name}</div>)}
      <button onClick={clearCart}>Clear</button>
    </div>
  );
}
Concept 6

Global State Design — principles for large apps

State type → where it belongs Server state (data from APIs, loading/error status) → TanStack Query or SWR — they are purpose-built for this → NOT in Redux/Zustand — you're reinventing what Query already does Global UI state (theme, sidebar open, modal open) → Zustand or Context (low frequency) → Small, rarely needs time-travel debugging Form state (input values, validation errors) → React Hook Form + local state → Keep it local — forms rarely need to be global URL state (filters, pagination, search query) → URL search params via React Router → Free: shareable, bookmarkable, back-button works Local component state (open/closed, hover, focus) → useState inside the component → Don't lift it if nothing else needs it Decision tree Does only one component need it? → useState Do multiple sibling components need it? → lift to parent Do deeply nested components need it? → Context or Zustand Is it server data? → TanStack Query Is it form state? → React Hook Form Does it need undo/redo or time travel? → Redux Toolkit 
// ─── Combining TanStack Query with Zustand ─────────────────────────────────
// TanStack Query handles server state, Zustand handles UI state
// They complement each other perfectly

// Server state — TanStack Query
function useUser(userId) {
  return useQuery({
    queryKey: ['user', userId],
    queryFn: () => fetch(`/api/users/${userId}`).then(r => r.json()),
    staleTime: 5 * 60 * 1000, // 5 min
  });
}

// UI state — Zustand
const useUIStore = create(set => ({
  editingUserId: null,
  openEditModal: (id) => set({ editingUserId: id }),
  closeEditModal: () => set({ editingUserId: null }),
}));

// Component uses both cleanly
function UserRow({ userId }) {
  const { data: user } = useUser(userId);
  const openEdit = useUIStore(s => s.openEditModal);

  return (
    <tr>
      <td>{user?.name}</td>
      <td><button onClick={() => openEdit(userId)}>Edit</button></td>
    </tr>
  );
}
The single biggest state management mistake in React apps is putting server data (users, products, orders) into Redux/Zustand and manually managing loading/error/stale state. TanStack Query does this in 10 lines of code. Separate server state from UI state and your state management becomes dramatically simpler.

Chapter 5 Takeaway

Choose the simplest state solution that works: local state first, Context for low-frequency sharing, Zustand for global UI state, TanStack Query for server data. Redux Toolkit is overkill for most apps today but is the right choice when you need strict action-history, time-travel debugging, or a very large team with coordination needs. Jotai and TanStack Store are excellent modern alternatives if you want fine-grained reactivity or framework portability.
