Concept 1

React Compiler — automatic memoization

React Compiler (formerly React Forget) is a Babel/SWC plugin that automatically applies memoization where React previously required manual memo, useCallback, and useMemo. It shipped stable in React 19.

// Before React Compiler — manual memoization everywhere
const Parent = memo(function Parent({ data }) {
  const processedData = useMemo(() => expensiveTransform(data), [data]);
  const handleClick = useCallback((id) => selectItem(id), []);
  return <Child data={processedData} onClick={handleClick} />;
});

// With React Compiler — write normal code, compiler does the rest
function Parent({ data }) {
  const processedData = expensiveTransform(data);  // compiler memoizes automatically
  const handleClick = (id) => selectItem(id);        // compiler wraps in useCallback
  return <Child data={processedData} onClick={handleClick} />;
}
// The compiled output is equivalent to the manual version above

// ─── Enabling React Compiler ──────────────────────────────────────────────
// vite.config.js
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [
    react({
      babel: {
        plugins: [['babel-plugin-react-compiler', {}]],
      },
    }),
  ],
});

// ─── What React Compiler can't optimize ──────────────────────────────────
// Code that violates React's rules of hooks (compiler skips these components)
// Direct DOM mutations, external mutable state
// Use React Compiler's eslint plugin to find violations:
// npm install eslint-plugin-react-compiler
React Compiler vs manual memoization: If you're on React 19+, enable the compiler and remove manual memo/useCallback/useMemo unless the profiler shows it's still needed. The compiler is more precise — it memoizes at the expression level, not the function level.
Concept 2

Memoization — manual React.memo, useMemo, useCallback

Before React Compiler (or when you're on React 18), manual memoization is how you prevent expensive re-renders. The key is measuring before adding memos — premature memoization adds complexity without benefit.

import { memo, useMemo, useCallback } from 'react';

// ─── React.memo — skip re-render when props haven't changed ──────────────
// Use when: component is expensive to render AND props are stable references
const ExpensiveChart = memo(function ExpensiveChart({ data, width, height }) {
  // 200ms D3 render — only runs when data/width/height change
  return <svg>{/* complex D3 visualization */}</svg>;
});

// memo with custom comparison
const UserRow = memo(
  function UserRow({ user, onEdit }) { return <tr>{/* ... */}</tr>; },
  (prevProps, nextProps) => prevProps.user.updatedAt === nextProps.user.updatedAt
);

// ─── useMemo — memoize expensive computed values ─────────────────────────
function OrderSummary({ orders, filter }) {
  // Recomputes ONLY when orders or filter changes
  const filteredOrders = useMemo(
    () => orders.filter(o => filter === 'all' || o.status === filter),
    [orders, filter]
  );

  const totals = useMemo(
    () => filteredOrders.reduce((acc, o) => ({
      count:   acc.count + 1,
      revenue: acc.revenue + o.totalCents,
    }), { count: 0, revenue: 0 }),
    [filteredOrders]
  );

  return <div>{totals.count} orders, ${(totals.revenue / 100).toFixed(2)}</div>;
}

// ─── useCallback — stable function reference ─────────────────────────────
// Use when: passing a callback to a memoized child component
function ProductList({ categoryId }) {
  const [selectedId, setSelectedId] = useState(null);

  // Without useCallback: new function on every render → memo(ProductCard) re-renders anyway
  const handleSelect = useCallback((id) => setSelectedId(id), []);

  return <ProductGrid onSelect={handleSelect} />;
}

// ─── When NOT to memoize ──────────────────────────────────────────────────
// These add overhead WITHOUT benefit:
const text    = useMemo(() => `Hello, ${name}`, [name]);     // trivial — just use template literal
const doubled = useMemo(() => count * 2, [count]);           // trivial computation
const onClick = useCallback(() => setCount(c => c + 1), []); // child not memoized
memo does not prevent re-renders if props are object/array literals. <Comp config={'{'}{'}'} /> passes a new object on every parent render — memo is useless. Either lift the object outside the component, use useMemo, or rely on React Compiler.
Concept 3

Lazy Loading — load components on demand

import { lazy, Suspense } from 'react';

// ─── Route-level code splitting ───────────────────────────────────────────
// Each route becomes a separate JS chunk — only downloaded when navigated to
const HomePage         = lazy(() => import('./pages/HomePage'));
const ProductsPage     = lazy(() => import('./pages/ProductsPage'));
const AdminDashboard   = lazy(() => import('./pages/admin/Dashboard'));
const HeavyDataViz     = lazy(() => import('./components/HeavyDataViz'));

// Suspense provides the fallback while the chunk loads
function App() {
  return (
    <BrowserRouter>
      <Suspense fallback={<PageLoadingSpinner />}>
        <Routes>
          <Route path="/"         element={<HomePage />} />
          <Route path="/products" element={<ProductsPage />} />
          <Route path="/admin"    element={<AdminDashboard />} />
        </Routes>
      </Suspense>
    </BrowserRouter>
  );
}

// ─── Component-level lazy loading ────────────────────────────────────────
function ProductPage({ product }) {
  const [showChart, setShowChart] = useState(false);

  return (
    <div>
      <h1>{product.name}</h1>
      <button onClick={() => setShowChart(true)}>Show Price History</button>

      {showChart && (
        <Suspense fallback={<div className="chart-skeleton">Loading chart...</div>}>
          <HeavyDataViz productId={product.id} />
        </Suspense>
      )}
    </div>
  );
}

// ─── Lazy loading images ──────────────────────────────────────────────────
function ProductImage({ src, alt }) {
  return (
    <img
      src={src}
      alt={alt}
      loading="lazy"          // native browser lazy load
      decoding="async"        // non-blocking decode
      width={400}
      height={300}
      style={{ aspectRatio: '4/3', objectFit: 'cover' }}
    />
  );
}
Concept 4

Code Splitting — reducing initial bundle size

// ─── Vite code splitting ───────────────────────────────────────────────────
// vite.config.js — manual chunks for vendor splitting
export default defineConfig({
  build: {
    rollupOptions: {
      output: {
        manualChunks: {
          'react-vendor':  ['react', 'react-dom'],
          'router':        ['react-router-dom'],
          'query':         ['@tanstack/react-query'],
          'charts':        ['recharts', 'd3'],  // split heavy libs
        },
      },
    },
    chunkSizeWarningLimit: 500,  // warn if chunk > 500kb
  },
});

// ─── Dynamic import with preload hint ─────────────────────────────────────
// Preload a chunk in the background before the user navigates to it
function NavBar() {
  const handleProductsHover = () => {
    // Preload the chunk when user hovers the link
    import('./pages/ProductsPage');
  };

  return (
    <nav>
      <Link to="/products" onMouseEnter={handleProductsHover}>Products</Link>
    </nav>
  );
}

// ─── What to split, what to keep together ────────────────────────────────
// SPLIT:   Admin pages (not all users visit)
//          Heavy visualization libraries (chart.js, d3, mapbox)
//          PDF/Excel export functionality
//          Rich text editors (TipTap, Quill)
//          Settings/preferences pages

// KEEP:    Core UI components (Button, Input, etc.) — loaded on every page
//          Authentication code — needed before anything else renders
//          Navigation — visible on all pages
Concept 5

Virtualization — render only what's visible

Rendering a 10,000-row table creates 10,000 DOM nodes. Virtualization renders only the rows visible in the viewport (~20–30) and swaps them as the user scrolls. Essential for any list with more than ~100 items.

// npm install @tanstack/react-virtual
import { useVirtualizer } from '@tanstack/react-virtual';
import { useRef } from 'react';

function VirtualTable({ rows }) {
  const parentRef = useRef(null);

  const rowVirtualizer = useVirtualizer({
    count: rows.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 48,   // estimated row height in pixels
    overscan: 5,               // render 5 extra rows outside viewport
  });

  return (
    <div
      ref={parentRef}
      style={{ height: '600px', overflow: 'auto' }}
    >
      {/* Total scrollable height — keeps scrollbar correct */}
      <div style={{ height: rowVirtualizer.getTotalSize(), position: 'relative' }}>
        {rowVirtualizer.getVirtualItems().map(virtualRow => (
          <div
            key={virtualRow.index}
            style={{
              position: 'absolute',
              top:    virtualRow.start,
              left:   0,
              width:  '100%',
              height: virtualRow.size,
            }}
          >
            <TableRow row={rows[virtualRow.index]} />
          </div>
        ))}
      </div>
    </div>
  );
}

// 10,000 rows → only ~20 DOM nodes at any time
Concept 6

Bundle Analysis — understanding what's in your build

// npm install rollup-plugin-visualizer
// vite.config.js
import { visualizer } from 'rollup-plugin-visualizer';

export default defineConfig({
  plugins: [
    react(),
    visualizer({
      filename: 'stats.html',  // open this file after build
      open: true,
      gzipSize: true,
      brotliSize: true,
    }),
  ],
});

// Run: npm run build → stats.html opens automatically
// Look for: large vendor chunks, duplicate packages, unexpected inclusions

// ─── Common bundle bloat patterns and fixes ────────────────────────────────
// Problem: importing all of lodash
import _ from 'lodash';                      // 70kb gzipped
import { debounce, throttle } from 'lodash'; // still 70kb — tree-shaking doesn't work for lodash

// Fix: lodash-es or individual packages
import debounce from 'lodash-es/debounce';  // ~3kb
import { debounce } from 'lodash-es';       // tree-shakeable

// Problem: moment.js with all locales
import moment from 'moment';  // 280kb gzipped

// Fix: use date-fns or dayjs
import { format, parseISO } from 'date-fns';  // import only what you use (~2kb per function)
import dayjs from 'dayjs';                     // 7kb total

// ─── Measuring runtime performance ────────────────────────────────────────
// React DevTools Profiler — record a session, look for:
// - Components that render too often (grey = skipped, colored = rendered)
// - Long render times (hover for ms)
// - Wasteful re-renders from unstable prop references

// Web Vitals in code
import { onCLS, onINP, onLCP } from 'web-vitals';

function sendToAnalytics({ name, value, id }) {
  analytics.track('web_vital', { name, value, id });
}

onCLS(sendToAnalytics);
onINP(sendToAnalytics);
onLCP(sendToAnalytics);
Performance checklist Initial load ☑ Route-level code splitting (React.lazy + Suspense) ☑ Images: loading="lazy", correct dimensions, WebP/AVIF ☑ Vendor chunk splitting (react, router, heavy libs separate) ☑ Gzip/Brotli on server (automatic on Vercel/Netlify/Cloudflare) Runtime re-renders ☑ React Compiler enabled (React 19+) ☑ Stable prop references for memoized components (useCallback/useMemo) ☑ Context split by update frequency ☑ Virtualize lists > 100 items Measurement tools ☑ React DevTools Profiler: record interactions, check render counts ☑ Chrome Performance tab: flame chart for JS execution ☑ Lighthouse: LCP, CLS, INP scores ☑ Bundle visualizer: spot bloat before it ships

Chapter 8 Takeaway

Performance optimization in React has a clear priority order: (1) enable React Compiler if on React 19 — it handles most memoization automatically, (2) split code at route boundaries with React.lazy, (3) virtualize long lists with TanStack Virtual, (4) analyze your bundle with rollup-plugin-visualizer and cut bloat. Only reach for manual memo/useMemo/useCallback when the profiler shows a specific problem. Premature optimization adds complexity without measurable benefit.
