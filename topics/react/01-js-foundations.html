JavaScript Foundations

The JavaScript you must own before React makes sense. Arrow functions, destructuring, modules, promises, and async/await aren't optional extras — React code is built from all of them. This chapter covers each concept with the exact patterns you will see in every React codebase.

Concept 1

ES6+ — The modern JavaScript baseline

ES6 (2015) introduced features that changed how every JavaScript developer writes code. React assumes you know all of them. The most important are let/const, template literals, spread/rest, optional chaining, and nullish coalescing.

// const for values that don't change, let for those that do — never var
const API_URL = 'https://api.example.com';
let retryCount = 0;

// Template literals — string interpolation without concatenation
const greeting = (name) => `Hello, ${name}! Welcome back.`;
const url = `${API_URL}/users/${userId}/orders`;

// Spread operator — clone and merge without mutation
const defaults = { theme: 'light', language: 'en', fontSize: 14 };
const userPrefs = { theme: 'dark', notifications: true };
const merged = { ...defaults, ...userPrefs };
// → { theme: 'dark', language: 'en', fontSize: 14, notifications: true }

// React: spreading props onto a component
function Button({ className, ...rest }) {
  return <button className={`btn ${className}`} {...rest} />;
}

// Optional chaining — short-circuits on null/undefined without a chain of &&
const city = user?.address?.city ?? 'Unknown';
const firstTag = post?.tags?.[0];
const result = response?.data?.items?.map(formatItem) ?? [];

// Nullish coalescing — defaults only on null/undefined (not 0, '', false)
const pageSize = options.pageSize ?? 20;  // 0 would NOT trigger this default
const label = props.label ?? 'Submit';    // '' would NOT trigger this default

// Short-circuit evaluation
const isAdmin = user && user.roles.includes('admin');
// React pattern: conditional rendering
const element = isLoading && <Spinner />;

// Array methods every React dev uses daily
const users = [{ id: 1, name: 'Alice', active: true }, { id: 2, name: 'Bob', active: false }];
const activeUsers  = users.filter(u => u.active);
const names        = users.map(u => u.name);
const userById     = users.find(u => u.id === 1);
const allActive    = users.every(u => u.active);
const anyActive    = users.some(u => u.active);
const summary      = users.reduce((acc, u) => { acc[u.id] = u.name; return acc; }, {});
const does not mean immutable. const arr = [] means you cannot reassign arr, but you can still call arr.push(). In React you use const everywhere by default, relying on setState (not direct mutation) to trigger re-renders.
Concept 2

Arrow Functions — concise syntax and lexical this

Arrow functions are everywhere in React — as event handlers, map/filter callbacks, and inline components. The key difference from regular functions: they do not have their own this binding.

// Regular function vs arrow function syntax
function double(n) { return n * 2; }       // regular
const double = (n) => n * 2;               // arrow (implicit return)
const double = n => n * 2;                 // arrow (single param — parens optional)
const add    = (a, b) => a + b;            // arrow with two params
const getUser = () => ({ id: 1 });         // returning an object: wrap in ()

// React event handlers — arrow functions are the standard pattern
function Counter() {
  const [count, setCount] = React.useState(0);

  // Arrow function as event handler
  const handleClick = () => setCount(c => c + 1);
  const handleReset = () => setCount(0);

  return (
    <div>
      <p>{count}</p>
      <button onClick={handleClick}>+</button>
      <button onClick={handleReset}>Reset</button>
    </div>
  );
}

// Inline arrow functions in JSX callbacks
const list = items.map((item, index) => (
  <li key={item.id} onClick={() => handleSelect(item.id)}>
    {index + 1}. {item.name}
  </li>
));

// Lexical this — arrow functions close over the outer this
// This matters when passing methods as callbacks
class Timer {
  constructor() { this.seconds = 0; }

  start() {
    // Arrow function: this refers to the Timer instance (correct)
    setInterval(() => { this.seconds++; }, 1000);

    // Regular function would be WRONG here:
    // setInterval(function() { this.seconds++; }, 1000); // this === undefined in strict mode
  }
}

// Function composition with arrows
const pipe = (...fns) => (x) => fns.reduce((v, fn) => fn(v), x);
const processUser = pipe(
  user => ({ ...user, name: user.name.trim() }),
  user => ({ ...user, email: user.email.toLowerCase() }),
  user => ({ ...user, createdAt: new Date() }),
);
Don't inline new arrow functions in render if avoidance is trivial. onClick={() => fn(id)} creates a new function every render. For most cases this is fine. When it causes measurable perf issues (long lists), extract with useCallback or move the handler out of JSX.
Concept 3

Destructuring — unpack values cleanly

Destructuring is used in every React function: component props, hook returns, API responses. It makes code more readable by naming what you actually care about.

// Object destructuring
const user = { id: 1, name: 'Alice', email: 'alice@example.com', role: 'admin' };

// Without destructuring
const name  = user.name;
const email = user.email;

// With destructuring
const { name, email } = user;

// Rename on destructure
const { name: userName, email: userEmail } = user;

// Default values (only when value is undefined — not null or 0)
const { theme = 'light', language = 'en' } = userSettings;

// Nested destructuring
const { address: { city, zip } } = user;

// ─── React props destructuring ──────────────────────────────────────────────
// Instead of using props.title, props.onClose, etc.
function Modal({ title, children, onClose, size = 'medium' }) {
  return (
    <div className={`modal modal--${size}`}>
      <header>
        <h2>{title}</h2>
        <button onClick={onClose}>×</button>
      </header>
      <div className="modal__body">{children}</div>
    </div>
  );
}

// ─── Array destructuring ─────────────────────────────────────────────────────
const [first, second, ...rest] = [10, 20, 30, 40, 50];
// first=10, second=20, rest=[30,40,50]

// Skip elements
const [,, third] = [10, 20, 30];

// useState returns an array — this is why we destructure it
const [count, setCount] = useState(0);
const [isOpen, setIsOpen] = useState(false);
const [user, setUser] = useState(null);

// ─── Destructuring in loops ───────────────────────────────────────────────────
const entries = [['a', 1], ['b', 2], ['c', 3]];
for (const [key, value] of entries) {
  console.log(`${key}: ${value}`);
}

// Object.entries with destructuring
const config = { host: 'localhost', port: 5432, db: 'myapp' };
Object.entries(config).forEach(([key, value]) => {
  console.log(`${key} = ${value}`);
});

// ─── Function parameter destructuring ────────────────────────────────────────
// Common in React component callbacks
const users = [{ id: 1, name: 'Alice' }, { id: 2, name: 'Bob' }];
users.map(({ id, name }) => <li key={id}>{name}</li>);
Concept 4

Modules — ES modules and the import/export system

Every file in a React project is a module. Understanding named vs default exports matters because they have different import syntax, and mixing them up is a common source of bugs.

// ─── Named exports ─────────────────────────────────────────────────────────
// utils/format.js
export function formatCurrency(cents, currency = 'USD') {
  return new Intl.NumberFormat('en-US', { style: 'currency', currency })
    .format(cents / 100);
}

export function formatDate(date, locale = 'en-US') {
  return new Intl.DateTimeFormat(locale, { dateStyle: 'medium' }).format(new Date(date));
}

export const DATE_FORMATS = { short: 'MM/DD', long: 'MMMM DD, YYYY' };

// Import named exports — must match the exported name
import { formatCurrency, formatDate } from './utils/format';
import { formatCurrency as fc, formatDate as fd } from './utils/format';  // alias

// ─── Default export ────────────────────────────────────────────────────────
// components/Button.jsx
function Button({ children, variant = 'primary', ...props }) {
  return <button className={`btn btn--${variant}`} {...props}>{children}</button>;
}
export default Button;

// Import default — name can be anything
import Button from './components/Button';
import MyButton from './components/Button';  // Also valid

// ─── Mixing both ──────────────────────────────────────────────────────────
// api/users.js
export async function fetchUser(id) { /* ... */ }
export async function updateUser(id, data) { /* ... */ }
const usersApi = { fetchUser, updateUser };
export default usersApi;

// Import both at once
import usersApi, { fetchUser } from './api/users';

// ─── Re-exports (barrel files) ───────────────────────────────────────────────
// components/index.js — re-export all components from one entry point
export { default as Button } from './Button';
export { default as Input } from './Input';
export { default as Modal } from './Modal';
export * from './icons';

// Now import from the barrel instead of individual files
import { Button, Input, Modal } from './components';

// ─── Dynamic imports — lazy loading ─────────────────────────────────────────
// The basis for React.lazy() and code splitting
const HeavyChart = React.lazy(() => import('./components/HeavyChart'));

// In a function (route-based code splitting)
async function loadAdminModule() {
  const { AdminDashboard } = await import('./admin/Dashboard');
  return AdminDashboard;
}

// ─── What React itself uses ──────────────────────────────────────────────────
import React, { useState, useEffect, useCallback, useMemo, useRef } from 'react';
import ReactDOM from 'react-dom/client';
import { BrowserRouter, Routes, Route } from 'react-router-dom';
Convention in React projects: React components are almost always default exports. Utility functions, constants, hooks, and types are named exports. Barrel files (index.js) clean up import paths but can hurt tree-shaking if not written carefully — import from specific files when bundle size matters.
Concept 5

Promises — async operations without callback hell

Every API call in React returns a Promise. Understanding how promises work — especially chaining, error handling, and Promise.all — is essential for data fetching patterns.

// A Promise represents an async operation that will resolve or reject
const promise = new Promise((resolve, reject) => {
  setTimeout(() => resolve('data'), 1000);
  // or: reject(new Error('something failed'));
});

// .then() / .catch() / .finally()
fetchUser(1)
  .then(user => fetchUserOrders(user.id))   // chain — next .then gets the return value
  .then(orders => setOrders(orders))
  .catch(err => setError(err.message))       // any error in the chain lands here
  .finally(() => setLoading(false));         // always runs

// ─── fetch() returns a Promise ───────────────────────────────────────────────
// Two-step pattern: fetch returns response, .json() returns another Promise
function getUser(id) {
  return fetch(`/api/users/${id}`)
    .then(res => {
      if (!res.ok) throw new Error(`HTTP ${res.status}`);
      return res.json();
    });
}

// ─── Promise.all — run multiple promises concurrently ───────────────────────
// Resolves when ALL resolve. Rejects if ANY reject.
async function loadDashboard(userId) {
  const [user, orders, notifications] = await Promise.all([
    fetchUser(userId),
    fetchOrders(userId),
    fetchNotifications(userId),
  ]);
  return { user, orders, notifications };
}

// ─── Promise.allSettled — run all, get all results even if some fail ─────────
const results = await Promise.allSettled([
  fetchUser(1),
  fetchUser(2),
  fetchUser(999), // 404 — will reject
]);

results.forEach(result => {
  if (result.status === 'fulfilled') console.log(result.value);
  else console.error('Failed:', result.reason);
});

// ─── Promise.race — first to resolve/reject wins ────────────────────────────
// Useful for timeouts
function withTimeout(promise, ms) {
  const timeout = new Promise((_, reject) =>
    setTimeout(() => reject(new Error('Request timed out')), ms)
  );
  return Promise.race([promise, timeout]);
}

const user = await withTimeout(fetchUser(1), 5000);

// ─── Creating resolved/rejected promises ────────────────────────────────────
const resolved = Promise.resolve({ id: 1, name: 'Alice' });
const rejected = Promise.reject(new Error('Not found'));

// Useful in tests and mock implementations
function mockFetchUser() { return Promise.resolve(mockUser); }
Concept 6

Async/Await — promises with synchronous-looking syntax

Async/await is syntactic sugar over promises. It makes async code read like synchronous code, which dramatically reduces cognitive load. All modern React data fetching uses async/await.

// async/await vs .then() chains — same behaviour, much more readable
// With .then():
fetchUser(1)
  .then(user => fetchOrders(user.id))
  .then(orders => console.log(orders))
  .catch(console.error);

// With async/await:
async function loadUserOrders(userId) {
  try {
    const user   = await fetchUser(userId);
    const orders = await fetchOrders(user.id);
    console.log(orders);
  } catch (err) {
    console.error(err);
  }
}

// ─── Error handling in async functions ───────────────────────────────────────
async function fetchUserData(userId) {
  try {
    const res = await fetch(`/api/users/${userId}`);

    if (!res.ok) {
      const errorBody = await res.json().catch(() => ({}));
      throw new Error(errorBody.message ?? `HTTP ${res.status}`);
    }

    return await res.json();
  } catch (err) {
    // Re-throw with context
    throw new Error(`Failed to fetch user ${userId}: ${err.message}`);
  }
}

// ─── Parallel vs sequential async operations ────────────────────────────────
// Sequential (each awaits the previous — total time = sum of all)
async function sequential(ids) {
  const results = [];
  for (const id of ids) {
    const user = await fetchUser(id); // one at a time
    results.push(user);
  }
  return results;
}

// Parallel (all fire concurrently — total time = slowest one)
async function parallel(ids) {
  const promises = ids.map(id => fetchUser(id)); // all fire at once
  return Promise.all(promises);
}

// ─── In React useEffect ──────────────────────────────────────────────────────
// useEffect callback cannot be async directly
// Wrong:
useEffect(async () => { /* this leaks the Promise */ }, []);

// Right: define async function inside and call it
useEffect(() => {
  let cancelled = false;

  async function loadData() {
    setLoading(true);
    try {
      const data = await fetchUserData(userId);
      if (!cancelled) setUser(data);
    } catch (err) {
      if (!cancelled) setError(err.message);
    } finally {
      if (!cancelled) setLoading(false);
    }
  }

  loadData();
  return () => { cancelled = true; }; // cleanup: prevent state update after unmount
}, [userId]);

// ─── Async IIFE pattern for one-off async operations ────────────────────────
(async () => {
  const data = await fetchConfig();
  renderApp(data);
})();
Never await inside useEffect's cleanup function. The cleanup function runs synchronously on unmount — it cannot be async. Use the cancelled flag pattern above to prevent state updates after unmount instead of trying to cancel the promise.

Async/await strengths

  • Reads like synchronous code — easier to reason about
  • try/catch works naturally for error handling
  • Works with all promise-based APIs natively
  • Easier to use with loops (for...of)

Common pitfalls

  • Easy to accidentally make parallel ops sequential
  • useEffect cannot be async (wrap in inner function)
  • Unhandled rejected promises can silently fail
  • Missing await causes subtle bugs (returns Promise not value)

Chapter 1 Takeaway

React code is modern JavaScript. You will encounter destructuring in every component signature, arrow functions in every event handler, modules in every import statement, and async/await in every API call. Get these patterns into muscle memory before starting React — the framework syntax will click much faster when the underlying JS is automatic.
