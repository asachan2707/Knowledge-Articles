Concept 1

JSX — HTML-like syntax that compiles to JavaScript

JSX is not HTML. It is a syntax extension that Babel/React compiler transforms into React.createElement() calls. Understanding this transformation prevents common mistakes and makes error messages legible.

// JSX
const element = <h1 className="title">Hello, {name}!</h1>;

// What Babel compiles it to:
const element = React.createElement('h1', { className: 'title' }, `Hello, ${name}!`);

// Key JSX rules:
// 1. className not class (class is a JS reserved word)
// 2. htmlFor not for (for is a JS reserved word)
// 3. Expressions in curly braces {}, not template literals
// 4. Self-closing tags must close: <img />, <input />, <br />
// 5. Return a single root element — use fragments to avoid extra divs

// Fragment syntax — no extra DOM node
function UserCard({ user }) {
  return (
    <>
      <h2>{user.name}</h2>
      <p>{user.email}</p>
    </>
  );
}

// Expressions in JSX — anything JS evaluates
const element = (
  <div>
    <p>Count: {count}</p>
    <p>Total: {(price * quantity).toFixed(2)}</p>
    <p>Status: {isActive ? 'Active' : 'Inactive'}</p>
    <ul>{items.map(item => <li key={item.id}>{item.name}</li>)}</ul>
  </div>
);

// Style in JSX — object with camelCased properties
const style = { backgroundColor: '#f0f0f0', fontSize: '14px', padding: '1rem' };
const el = <div style={style}>Styled</div>;
// Or inline (less common, avoid for complex styles):
const el2 = <div style={{ color: 'red', fontWeight: 'bold' }}>Red</div>;

// Event attributes are camelCase
<button onClick={handleClick} onMouseEnter={handleHover}>Click</button>
<input onChange={handleChange} onBlur={handleBlur} />
<form onSubmit={handleSubmit}>...</form>
Concept 2

Components — functions that return UI

A React component is a JavaScript function that accepts props and returns JSX. React re-calls this function every time the component's state or props change. Understanding when and why React re-renders is the foundation of performance thinking.

// A component is a function that returns JSX
function Greeting({ name }) {
  return <h1>Hello, {name}!</h1>;
}

// Component names MUST start with a capital letter
// <greeting /> is treated as an HTML tag — React won't find it
// <Greeting /> is treated as a component — React calls the function

// ─── Render cycle ─────────────────────────────────────────────────────────
// React calls your component function on every render.
// Think of it as React saying: "give me the latest UI for this state/props"

function Counter() {
  const [count, setCount] = React.useState(0);

  console.log('Counter rendered'); // runs on EVERY render

  return (
    <div>
      <p>{count}</p>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  );
}

// ─── Composing components ─────────────────────────────────────────────────
function Avatar({ src, alt, size = 40 }) {
  return <img src={src} alt={alt} width={size} height={size} style={{ borderRadius: '50%' }} />;
}

function UserProfile({ user }) {
  return (
    <div className="profile">
      <Avatar src={user.avatarUrl} alt={user.name} size={64} />
      <div>
        <h2>{user.name}</h2>
        <p>{user.bio}</p>
      </div>
    </div>
  );
}

// ─── Component tree ────────────────────────────────────────────────────────
// When UserProfile renders, Avatar also renders
// Data flows DOWN (parent → child via props)
// Events flow UP (child → parent via callback props)

// ─── Real-world example: product card ─────────────────────────────────────
function ProductCard({ product, onAddToCart }) {
  return (
    <div className="product-card">
      <img src={product.imageUrl} alt={product.name} />
      <h3>{product.name}</h3>
      <p className="price">${(product.priceCents / 100).toFixed(2)}</p>
      <button onClick={() => onAddToCart(product.id)}>Add to Cart</button>
    </div>
  );
}

function ProductGrid({ products, onAddToCart }) {
  return (
    <div className="product-grid">
      {products.map(product => (
        <ProductCard key={product.id} product={product} onAddToCart={onAddToCart} />
      ))}
    </div>
  );
}
Concept 3

Props — data flowing from parent to child

Props (properties) are how you pass data to components. They are read-only in the child — a component must never mutate its own props. To change a parent's data, call a callback function passed as a prop.

// Passing props — same syntax as HTML attributes
<Button label="Submit" disabled={isLoading} onClick={handleSubmit} />

// Receiving props — destructure in the function signature
function Button({ label, disabled = false, onClick, variant = 'primary' }) {
  return (
    <button
      className={`btn btn--${variant}`}
      disabled={disabled}
      onClick={onClick}
    >
      {label}
    </button>
  );
}

// ─── All props are expressions ─────────────────────────────────────────────
// Strings: value="hello" OR value={'hello'}
// Numbers: count={42}
// Booleans: disabled={true}  OR just disabled (shorthand)
// Objects:  user={{ name: 'Alice', email: 'alice@x.com' }}
// Arrays:   items={[1, 2, 3]}
// Functions: onChange={handleChange}
// null/undefined: hides the prop (same as not passing it)

// ─── Spreading props ──────────────────────────────────────────────────────
const buttonProps = { type: 'submit', className: 'btn', disabled: isLoading };
<button {...buttonProps}>Save</button>

// Spreading and overriding
function Input({ className, ...rest }) {
  return <input className={`input ${className ?? ''}`} {...rest} />;
}
// Usage: <Input type="email" placeholder="Enter email" onChange={handler} />

// ─── Callback props (child → parent communication) ────────────────────────
function SearchBar({ onSearch }) {
  const [query, setQuery] = React.useState('');

  const handleSubmit = (e) => {
    e.preventDefault();
    onSearch(query); // call parent's handler with the data
  };

  return (
    <form onSubmit={handleSubmit}>
      <input
        value={query}
        onChange={e => setQuery(e.target.value)}
        placeholder="Search..."
      />
      <button type="submit">Search</button>
    </form>
  );
}

// Parent receives the value through the callback
function App() {
  const [results, setResults] = React.useState([]);

  const handleSearch = async (query) => {
    const data = await searchProducts(query);
    setResults(data);
  };

  return (
    <div>
      <SearchBar onSearch={handleSearch} />
      <ResultsList results={results} />
    </div>
  );
}
Never mutate props. props.user.name = 'new name' modifies the parent's object directly — React won't re-render because the reference didn't change, and you've introduced subtle state bugs. Always create new objects: setUser({'{'} ...user, name: 'new name' {'}'}).
Concept 4

State — component memory that triggers re-renders

State is data that changes over time and that React should track. When you call setState, React schedules a re-render — it calls your component function again with the new state value. This is the core mechanism of React.

import { useState } from 'react';

// useState returns [currentValue, setterFunction]
// The argument to useState is the initial value (only used on first render)
const [count, setCount]     = useState(0);
const [name, setName]       = useState('');
const [user, setUser]       = useState(null);
const [items, setItems]     = useState([]);
const [config, setConfig]   = useState({ theme: 'light', lang: 'en' });

// ─── Updating state ───────────────────────────────────────────────────────
setCount(5);              // Replace with value
setCount(c => c + 1);     // Functional update (safe when next state depends on prev)

// Always use functional update for increments/decrements
// (avoids stale closure bugs when multiple updates happen quickly)
const handleTripleClick = () => {
  setCount(c => c + 1);  // 3 separate updates, all see the latest c
  setCount(c => c + 1);
  setCount(c => c + 1);
  // → count increases by 3
};

// ─── Never mutate state directly ─────────────────────────────────────────
// WRONG: React won't see this change
items.push(newItem);         // mutation — no re-render
user.name = 'new name';      // mutation — no re-render

// RIGHT: create new values
setItems([...items, newItem]);                          // add to array
setItems(items.filter(i => i.id !== deletedId));       // remove from array
setItems(items.map(i => i.id === id ? {...i, ...updates} : i)); // update item
setUser({ ...user, name: 'new name' });                // update object field

// ─── State for a shopping cart (real example) ─────────────────────────────
function ShoppingCart() {
  const [cart, setCart] = useState([]);

  const addItem = (product) => {
    setCart(prev => {
      const existing = prev.find(i => i.id === product.id);
      if (existing) {
        return prev.map(i => i.id === product.id ? { ...i, qty: i.qty + 1 } : i);
      }
      return [...prev, { ...product, qty: 1 }];
    });
  };

  const removeItem = (productId) => {
    setCart(prev => prev.filter(i => i.id !== productId));
  };

  const total = cart.reduce((sum, item) => sum + item.priceCents * item.qty, 0);

  return (
    <div>
      {cart.map(item => (
        <CartItem key={item.id} item={item} onRemove={removeItem} />
      ))}
      <p>Total: ${(total / 100).toFixed(2)}</p>
    </div>
  );
}

// ─── Lazy initial state ───────────────────────────────────────────────────
// If computing initial state is expensive, pass a function
const [data] = useState(() => JSON.parse(localStorage.getItem('data') ?? '[]'));
// The function runs only on the first render, not on every re-render
Concept 5

Events — React's synthetic event system

React wraps browser events in synthetic events that normalise cross-browser differences. The API is nearly identical to native DOM events — you use e.target.value, e.preventDefault(), e.stopPropagation() — but event names are camelCase.

// ─── Common event handlers ───────────────────────────────────────────────
function Form() {
  const [form, setForm] = useState({ email: '', password: '' });
  const [error, setError] = useState('');

  // Controlled input pattern — React owns the value
  const handleChange = (e) => {
    const { name, value } = e.target;
    setForm(prev => ({ ...prev, [name]: value }));  // computed property key
  };

  const handleSubmit = async (e) => {
    e.preventDefault();  // prevent browser form submission / page reload
    if (!form.email.includes('@')) {
      setError('Invalid email');
      return;
    }
    setError('');
    await login(form.email, form.password);
  };

  return (
    <form onSubmit={handleSubmit}>
      <input
        type="email"
        name="email"
        value={form.email}
        onChange={handleChange}
        placeholder="Email"
      />
      <input
        type="password"
        name="password"
        value={form.password}
        onChange={handleChange}
        placeholder="Password"
      />
      {error && <p className="error">{error}</p>}
      <button type="submit">Log In</button>
    </form>
  );
}

// ─── Keyboard events ─────────────────────────────────────────────────────
function SearchInput({ onSearch }) {
  const [query, setQuery] = useState('');

  const handleKeyDown = (e) => {
    if (e.key === 'Enter') onSearch(query);
    if (e.key === 'Escape') setQuery('');
  };

  return (
    <input
      value={query}
      onChange={e => setQuery(e.target.value)}
      onKeyDown={handleKeyDown}
    />
  );
}

// ─── Event delegation — passing data through event handlers ─────────────
// Pattern 1: inline arrow function (simple, fine for most cases)
{items.map(item => (
  <button onClick={() => handleDelete(item.id)}>Delete</button>
))}

// Pattern 2: data- attribute (avoids inline arrow functions in hot loops)
const handleDelete = (e) => {
  const id = e.currentTarget.dataset.id;
  deleteItem(id);
};
{items.map(item => (
  <button data-id={item.id} onClick={handleDelete}>Delete</button>
))}

// ─── stopPropagation — prevent bubbling ──────────────────────────────────
function Card({ onClick, children }) {
  return (
    <div onClick={onClick} className="card">
      {children}
      <button onClick={e => { e.stopPropagation(); handleAction(); }}>
        Action
      </button>
    </div>
  );
}
Concept 6

Conditional Rendering — show or hide UI based on state

React has several patterns for conditionally rendering UI. Choosing the right one depends on whether you want to show/hide, switch between alternatives, or render lists.

// ─── Pattern 1: && (short-circuit) — show when true ──────────────────────
// Best for: "show this if condition is true"
function Notification({ message, show }) {
  return (
    <div>
      {show && <p className="notification">{message}</p>}
    </div>
  );
}

// ⚠️ Falsy values: 0 renders as "0" — guard with !!
// Wrong: {items.length && <List />}  — renders "0" when empty
// Right: {items.length > 0 && <List />}

// ─── Pattern 2: ternary — show one of two alternatives ───────────────────
function AuthButton({ user }) {
  return (
    <div>
      {user
        ? <button onClick={logout}>Log Out, {user.name}</button>
        : <button onClick={openLogin}>Log In</button>
      }
    </div>
  );
}

// ─── Pattern 3: if/else outside JSX ─────────────────────────────────────
// Best for: complex logic with multiple branches
function DataDisplay({ status, data, error }) {
  if (status === 'loading') return <Spinner />;
  if (status === 'error')   return <ErrorMessage error={error} />;
  if (!data || data.length === 0) return <EmptyState />;

  return <DataTable rows={data} />;
}

// ─── Pattern 4: switch (multiple states) ─────────────────────────────────
function OrderStatus({ status }) {
  const label = {
    pending:    'Awaiting payment',
    processing: 'Being prepared',
    shipped:    'On the way',
    delivered:  'Delivered',
    cancelled:  'Cancelled',
  }[status] ?? 'Unknown';

  const color = {
    pending:    'orange',
    processing: 'blue',
    shipped:    'purple',
    delivered:  'green',
    cancelled:  'red',
  }[status] ?? 'gray';

  return <span style={{ color }}>{label}</span>;
}

// ─── Rendering lists ──────────────────────────────────────────────────────
// key prop is required — helps React identify which items changed
function ProductList({ products }) {
  return (
    <ul>
      {products.map(product => (
        <li key={product.id}>  {/* key must be unique and stable */}
          {product.name} — ${(product.priceCents / 100).toFixed(2)}
        </li>
      ))}
    </ul>
  );
}

// ─── Real-world loading/error/data pattern ───────────────────────────────
function UserDashboard({ userId }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    fetchUser(userId)
      .then(setUser)
      .catch(err => setError(err.message))
      .finally(() => setLoading(false));
  }, [userId]);

  if (loading) return <div className="skeleton">Loading...</div>;
  if (error)   return <div className="error-banner">{error}</div>;
  if (!user)   return <div>User not found.</div>;

  return (
    <div className="dashboard">
      <h1>Welcome, {user.name}</h1>
      {user.isAdmin && <AdminPanel />}
      {user.orders.length > 0
        ? <OrderList orders={user.orders} />
        : <EmptyOrders />
      }
    </div>
  );
}
The loading/error/data tri-state pattern is ubiquitous in React. You will write it hundreds of times, which is why libraries like TanStack Query exist — they manage this state for you. But you need to understand the manual version first.

Chapter 2 Takeaway

React's mental model: UI = f(state). Your component function is called every time state or props change, returning a description of what the UI should look like right now. Props flow down, events flow up, and setState triggers the cycle. Master these six fundamentals and you have the foundation for everything in chapters 3–11.
