Claude OS — A System That Turns Claude into an Execution Engine

Claude OS is not an Anthropic product — it is an architectural pattern. By composing CLAUDE.md, hooks, workflows, skills, and sub-agents into a deliberate system, you transform Claude Code from a responsive assistant into a structured execution engine that handles multi-step tasks autonomously, consistently, and verifiably.

ConceptCommunity pattern, Claude Code 2025–2026
RequiresClaude Code with hooks + skills + workflows enabled
LevelAdvanced

A standard Claude Code session is reactive — you type, Claude responds. Claude OS flips that: Claude is the orchestrator. You define what tasks should trigger which agents, what quality gates must pass, how state is persisted between sessions, and what the output contract is. Claude executes. You review.

Core idea: Claude OS = CLAUDE.md (persistent context) + hooks (automatic triggers) + skills (reusable operations) + workflows (orchestrated sub-agents) + MCP tools (external integrations). Together they make Claude behave like a programmable OS, not a chat window.

The origin of the pattern

Engineers who used Claude Code heavily in 2025 noticed a problem: powerful sessions were not reproducible. You got great results by carefully setting context, but the next session started from scratch. The solution was to externalise the "good session setup" into files Claude always reads — which is exactly what CLAUDE.md was designed for. Then hooks let you fire agents automatically on events. Then skills let you package multi-step operations as slash commands. Then workflows let you fan out to dozens of sub-agents in parallel. Each addition turned Claude from a chat tool into a runtime.

The four layers

Architecture of a Claude OS system

L1 Context Layer CLAUDE.md files — project conventions, architecture decisions, anti-patterns, tool preferences. Always in context.
L2 Event Layer Hooks — shell commands that fire on PreToolUse, PostToolUse, Stop, Notification events. Deterministic reactions.
L3 Operation Layer Skills — slash commands that load specialised instructions and tools for a domain (deploy, review, test, document).
L4 Orchestration Layer Workflows — JavaScript scripts that spawn, coordinate, and verify parallel sub-agents with structured output schemas.
Layer 1: Context

CLAUDE.md as an operating system kernel

The kernel sets the rules for everything else. A well-designed CLAUDE.md for an OS-style setup includes execution contracts: 

# ProjectOS

> Node.js REST API. PostgreSQL. Redis. Deployed on Fly.io.

## Execution contracts
- All DB queries go through `src/db/query.js` — never raw SQL strings in route files.
- Tests run with `npm test` before any commit. If tests fail, STOP and report.
- Migrations live in `db/migrations/`. Always generate with `npm run migrate:make`.

## Agent autonomy gates
- READ + analyse: autonomous, no confirmation needed.
- WRITE to src/: confirm file list before writing.
- EXEC shell commands: confirm if outside npm scripts.
- PUSH to git: always require explicit user approval.

## Context pointers
- Architecture decisions: see DECISIONS.md
- Deployment runbook: see ops/RUNBOOK.md
- API contract: see openapi.yaml
Layer 2: Hooks

Event-driven automation

Hooks run outside Claude's context — they are shell commands triggered by Claude Code events. This means they are always deterministic and cannot be overridden by prompt injection. 

// .claude/settings.json — hook configuration
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{
          "type": "command",
          "command": "npm run lint -- --quiet 2>&1 | head -20"
        }]
      }
    ],
    "Stop": [
      {
        "hooks": [{
          "type": "command",
          "command": "node .claude/hooks/session-summary.js"
        }]
      }
    ]
  }
}
// .claude/hooks/session-summary.js
// Runs after every session ends — writes a summary to memory
import fs from 'fs';
import path from 'path';

const transcriptDir = process.env.CLAUDE_TRANSCRIPT_DIR ?? '.claude/transcripts';
const memoryDir     = '.claude/memory';

const today = new Date().toISOString().slice(0, 10);
const summaryPath = path.join(memoryDir, `session-${today}.md`);

// Append session end marker
fs.appendFileSync(summaryPath, `\n## Session ended ${new Date().toISOString()}\n`);
console.log('Session summary written to', summaryPath);
Layer 3: Skills

Reusable operation packages

Skills are Markdown files in .claude/skills/ that Claude loads when you invoke /skill-name. They contain specialised instructions, prompt patterns, and tool chains. 

# .claude/skills/deploy.md
# Deploy to Fly.io

When this skill is invoked:

1. Run `npm test` — if any test fails, stop and report. Do not proceed.
2. Run `npm run build` and confirm exit 0.
3. Run `flyctl status` and confirm the current deploy is healthy.
4. Run `flyctl deploy --ha=false` and stream output.
5. After deploy, run `flyctl status` again and confirm all instances are running.
6. Report: deploy duration, instance count, health check status.

If any step fails, report the exact error and suggest the most likely fix.
Do NOT roll back automatically — require explicit user confirmation.
// Invoking a skill from a workflow script (orchestration layer)
const result = await agent('/deploy', {
  label: 'deploy:production',
  schema: {
    type: 'object',
    properties: {
      success:  { type: 'boolean' },
      duration: { type: 'number' },
      instances:{ type: 'number' },
      error:    { type: 'string' },
    },
    required: ['success'],
  },
});
Layer 4: Workflows

Orchestrating parallel sub-agents

The workflow layer is where Claude OS becomes genuinely powerful. A workflow script fans out work across many specialised agents, collects structured output, and synthesises a result. 

// .claude/workflows/pre-release-check.js
export const meta = {
  name: 'pre-release-check',
  description: 'Full pre-release quality gate: tests, security, perf, docs',
  phases: [
    { title: 'Scan',     detail: 'Parallel domain checks' },
    { title: 'Verify',   detail: 'Adversarial verification of findings' },
    { title: 'Report',   detail: 'Synthesise go/no-go decision' },
  ],
};

const CHECKS = [
  { key: 'security',   prompt: 'Audit src/ for OWASP Top 10: injection, XSS, insecure deps, hardcoded secrets. Return findings array.' },
  { key: 'tests',      prompt: 'Run `npm test` and report pass/fail counts, any flaky tests, coverage delta vs main.' },
  { key: 'perf',       prompt: 'Check for N+1 queries, missing indexes, synchronous blocking calls in request paths.' },
  { key: 'api-drift',  prompt: 'Compare openapi.yaml against route implementations. List endpoints that diverged.' },
  { key: 'docs',       prompt: 'Check README, DECISIONS.md, and inline JSDoc for stale references to removed APIs.' },
];

const FINDINGS_SCHEMA = {
  type: 'object',
  properties: {
    key:      { type: 'string' },
    ok:       { type: 'boolean' },
    findings: { type: 'array', items: { type: 'string' } },
    blocker:  { type: 'boolean' },
  },
  required: ['key', 'ok', 'findings', 'blocker'],
};

phase('Scan');
const results = await parallel(CHECKS.map(c => () =>
  agent(c.prompt, { label: `check:${c.key}`, phase: 'Scan', schema: FINDINGS_SCHEMA })
    .then(r => ({ ...r, key: c.key }))
));

const blockers = results.filter(Boolean).filter(r => r.blocker);

phase('Report');
const report = await agent(
  `Synthesise these pre-release check results into a go/no-go decision.
   Blockers found: ${blockers.map(b => b.key).join(', ') || 'none'}.
   Full results: ${JSON.stringify(results, null, 2)}`,
  {
    schema: {
      type: 'object',
      properties: {
        verdict:  { type: 'string', enum: ['GO', 'NO-GO'] },
        summary:  { type: 'string' },
        blockers: { type: 'array', items: { type: 'string' } },
      },
      required: ['verdict', 'summary'],
    },
  }
);

log(`Verdict: ${report.verdict} — ${report.summary}`);
return report;
MCP integration

Connecting external systems

MCP (Model Context Protocol) tools extend Claude OS into external systems — databases, Slack, GitHub, Linear, Figma. A Claude OS setup typically wires these into workflows so agents can query real data rather than guessing. 

// .mcp.json — project-level MCP configuration
{
  "mcpServers": {
    "github":   { "command": "npx", "args": ["@modelcontextprotocol/server-github"],
                  "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" } },
    "postgres": { "command": "npx", "args": ["@modelcontextprotocol/server-postgres"],
                  "env": { "POSTGRES_URL": "${DATABASE_URL}" } },
    "slack":    { "command": "npx", "args": ["@modelcontextprotocol/server-slack"],
                  "env": { "SLACK_BOT_TOKEN": "${SLACK_BOT_TOKEN}" } }
  }
}

Key Takeaway

Claude OS is the realisation that Claude Code is not just a coding assistant — it is a runtime with four programmable layers. Layer 1 (context) sets the rules. Layer 2 (hooks) reacts to events. Layer 3 (skills) packages reusable operations. Layer 4 (workflows) orchestrates teams of agents. Together they turn a chat session into a reproducible, auditable execution engine.
