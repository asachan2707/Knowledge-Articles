How to Cut Claude Code Costs by at Least 2–3x

The specific habits, configuration changes, and session hygiene practices that reduce Claude Code token consumption by 2–3x without reducing output quality — and the mental model that makes cost awareness intuitive rather than tedious.

Portal Internal · Curated
Focus Claude Code cost optimisation: context management, model selection, session hygiene
Published Jun 2026

Claude Code cost is almost entirely a context problem. Every token in your session window costs money — both the input tokens Claude reads and the output tokens it writes. Long sessions with bloated context burn 3–5× more than equivalent sessions with disciplined context management. The techniques below address the root causes, not the symptoms.

Core idea: Cost reduction and quality improvement point in the same direction. A focused context makes Claude more accurate and cheaper simultaneously. Most cost waste comes from Claude reading things it doesn't need, not from Claude writing too much.

Story · The $200 session

The common pattern: a developer opens Claude Code on Monday morning and keeps the same session running all week, adding context, fixing bugs, changing direction, re-reading files. By Friday the session has consumed 800k+ tokens and cost $200. A disciplined developer doing the same work in focused, bounded sessions costs $40–60. The difference is not capability — it is hygiene.

The root causes of cost waste

Where tokens go unnecessarily

  • Conversation history accumulation: Every prior message stays in context. A session that started three hours ago is carrying thousands of tokens of resolved discussion that are no longer relevant to the current task.
  • CLAUDE.md bloat: A 500-line CLAUDE.md is loaded on every request. If 400 of those lines are irrelevant to the current file you are working on, you are paying for 400 lines of context noise on every single turn.
  • Raw file reads: When Claude reads a 2000-line file to answer a question that only requires reading 50 lines, 1950 lines of tokens are wasted. Providing line-specific context is dramatically more efficient.
  • Using Opus for everything: Opus is 5× more expensive than Sonnet per token. Tasks that do not require Opus-level reasoning (simple refactors, formatting, boilerplate) are wasteful on Opus.
  • Abandoned explorations: Asking Claude to "try an approach," then switching direction without clearing context. The abandoned approach sits in context burning tokens for the rest of the session.
The high-impact fixes

Techniques that cut cost 2–3x

1. Use /clear aggressively

Running /clear wipes the conversation history, keeping only the CLAUDE.md and any files explicitly in context. Do this at every task boundary — when you finish a bug fix, when you start a new feature, when you change the file you are working on. It is the highest-leverage cost reduction available.

2. Keep CLAUDE.md under 200 lines

Every line of CLAUDE.md is injected into every request. The 200-line rule is not arbitrary — it is the point at which the overhead becomes significant relative to the task context. Audit your CLAUDE.md and delete anything that is descriptive rather than instructional. Claude does not need to know your project's history; it needs to know your conventions.

3. Use /compact before context runs high

/compact summarises the conversation history into a compressed form. Run it when the session is long but you are not ready to clear completely. It reduces context by 60–80% while preserving the key decisions and open threads. Think of it as a save point before continuing.

4. Match model to task

  • Haiku: Simple completions, formatting, renaming, boilerplate generation. 20× cheaper than Opus.
  • Sonnet: Standard engineering tasks — bug fixes, feature implementation, code review. The right default for 80% of work.
  • Opus: Complex architectural reasoning, novel problem-solving, tasks where you have tried Sonnet and it failed. Not the default.

5. Reference files by line range, not full file

Instead of letting Claude read an entire file, use @filename.py:45-90 to reference only the relevant lines. For a question about one function, providing the function and its direct dependencies is all Claude needs. Full-file reads are appropriate for structural questions, not for targeted changes.

6. Write targeted prompts with explicit scope

"Fix the bug in the process_payment function on line 87 of payments.py" costs a fraction of "something seems wrong with payments, can you take a look?" The second prompt causes Claude to read multiple files, form a broad hypothesis, and write a lengthy diagnosis before attempting any fix. Specificity is free.

7. Use /goal to anchor long sessions

The /goal command sets a task objective that Claude tracks throughout the session, reducing the need to re-explain context. When Claude knows the goal explicitly, it stops re-reading prior context to re-infer what you are trying to accomplish.

Session design

The bounded session pattern

The highest-efficiency Claude Code sessions follow this pattern:

  1. Clear context (/clear) — start fresh with only CLAUDE.md loaded.
  2. State the goal (/goal: implement X feature in Y file) — scoped, specific.
  3. Work the task — one focused thread without tangents.
  4. Compact before switching (/compact) — if you need to continue but task is done.
  5. Clear again for the next task.

This pattern keeps every session under 50k tokens for most typical engineering tasks. An equivalent unmanaged session often exceeds 200k.

Common questions

FAQ

  • Won't /clear make Claude forget important project context?

    That context should be in your CLAUDE.md, not in the session history. CLAUDE.md is persistent across clears. Architectural decisions, conventions, and project structure belong there. Conversation history is working memory — it should not be the primary store of project knowledge.

  • How do I know what is actually costing the most?

    Use /cost to see the running session token count and estimated cost. The two biggest line items are almost always long conversation history and large CLAUDE.md files. Check both before investigating anything else.

  • Does this apply to the Max subscription or only API billing?

    Both. Max subscription has usage limits rather than per-token billing, but the same context hygiene habits prevent hitting those limits mid-session. The discipline is universally useful regardless of billing model.

Key Takeaway

Claude Code cost is a context hygiene problem. Clear aggressively, keep CLAUDE.md under 200 lines, match model to task, and reference files by line range. These four changes alone will cut costs by 2–3× and, counterintuitively, improve output quality at the same time.
