The Enum Trick Every Python Developer Needs to Master

A Medium article from The Pythonworld arguing that the real upgrade with Enums is not “use Enums instead of constants.” It is treating Enums as small domain types that carry behavior, validate input at the boundary, and serialize cleanly across APIs.

Author Aashish Kumar
Published 2026
Focus Using Python Enums as expressive, type-safe domain objects

Most Python codebases drift the same way: a few harmless string literals or magic numbers, scattered comparisons, a few typos that almost get caught, and eventually a tangle of conditional logic that nobody wants to refactor. Enums fix this — but only if you stop treating them as a fancier way to write constants.

Core idea: an Enum is not a constant. It is a small type. Give it behavior, validate with it, and let it kill invalid states at the boundary.

Story · The bug that wouldn't die

A team kept hitting a bug where users marked as "actve" (note the typo) were silently failing some permission checks but passing others. The string literal "active" was scattered across 40 files. One file had the typo. None of the tests caught it because the typo passed through string comparisons cleanly — they just always returned False.

The fix was a 2-line refactor: introduce a UserStatus Enum, replace the string comparisons, and let Python crash loudly the next time someone typed "actve". This is the article's argument compressed into one anecdote: Enums don't make code more elegant — they make bugs impossible.

The problem with strings and magic numbers

Why scattered constants quietly damage codebases

Plain strings and magic numbers cause a predictable set of pains:

  • Any value is accepted — no validation at all.
  • Typos pass through silently and cause logic that “almost works.”
  • Refactors are dangerous because nothing is centralized.
  • IDE autocomplete cannot help you with arbitrary strings.
  • Business logic leaks into every if branch that compares them.

Enums solve all of these — but the deeper trick is what you do with them after that.

The trick

Step 1: give Enums behavior

Instead of using an Enum purely as a name for a value, encode the rules around that value into the Enum itself. The classic example is a status type: 

from enum import Enum

class UserStatus(Enum):
    PENDING = "pending"
    ACTIVE = "active"
    SUSPENDED = "suspended"
    DELETED = "deleted"

    def can_login(self) -> bool:
        return self is UserStatus.ACTIVE

    def is_terminal(self) -> bool:
        return self in {UserStatus.SUSPENDED, UserStatus.DELETED}

The payoff is in the call sites. Instead of: 

if user.status == "active" and not user.is_deleted:
    ...

You write: 

if user.status.can_login():
    ...

The business rule lives in one place. The call site becomes readable English.

The trick, continued

Step 2: kill invalid states at the boundary

The second move is using the Enum constructor as a validator wherever data crosses into your system — API requests, database rows, message payloads, CLI arguments. Instead of guarding against bad strings with hand-rolled checks, let the Enum do it for you: 

raw = request_data["status"]
status = UserStatus(raw)   # raises ValueError on invalid input

Once a value is inside your code as a UserStatus, it is impossible for it to be invalid. Every downstream function can rely on that. This is what people mean by “make illegal states unrepresentable” — and Enums are the cheapest way to get there in Python.

The serialization trick

Step 3: use str + Enum for clean APIs

Plain Enums do not serialize well to JSON because UserStatus.ACTIVE is not a string. The common workaround — calling .value everywhere — is fragile. The clean solution is to inherit from both str and Enum: 

class OrderStatus(str, Enum):
    CREATED = "created"
    PAID = "paid"
    SHIPPED = "shipped"
    CANCELLED = "cancelled"

import json
json.dumps({"status": OrderStatus.PAID})   # '{"status": "paid"}'

Now your Enum is JSON-serializable, comparable to plain strings (OrderStatus.PAID == "paid" is True), and still validated when constructed. You get type safety without sacrificing API ergonomics.

Bonus patterns

A few small Enum upgrades worth knowing

auto() for value-agnostic enums

When the value does not matter (only the identity does), use auto() to avoid bookkeeping: 

from enum import Enum, auto

class Direction(Enum):
    NORTH = auto()
    SOUTH = auto()
    EAST = auto()
    WEST = auto()

IntEnum for legacy integer protocols

IntEnum behaves like an integer and an Enum at once. It is useful for HTTP status codes, C-style flags, and protocols where integers are required, but should be avoided when you do not need implicit integer comparison.

Flag for combinable options

For permissions, feature toggles, and other bitwise-combinable sets, Flag and IntFlag let you write Permission.READ | Permission.WRITE safely.

Iteration and membership are first-class

list(UserStatus) gives you every value, and "active" in UserStatus._value2member_map_ (or just try/except ValueError on the constructor) gives you cheap validation.

Practical learning

Where to actually use this in real code

  • Any field that has a small fixed set of valid values: status, role, type, mode, environment.
  • Anywhere you currently compare against string literals more than twice.
  • API request and response models — validate at the edge, trust internally.
  • Database column values that map to a finite set of states.
  • Anywhere business rules currently live as scattered if/elif chains.
Real-world refactor

Before / after on a typical service

Before — strings everywhere:

def can_user_login(user):
    if user.status == "active" and not user.deleted:
        return True
    if user.status == "pending" and user.email_verified:
        return False
    return False

def update_user_status(user, new_status):
    if new_status not in ("active", "pending", "suspended", "deleted"):
        raise ValueError(f"Invalid status: {new_status}")
    user.status = new_status

After — Enum with behavior:

class UserStatus(str, Enum):
    PENDING = "pending"
    ACTIVE = "active"
    SUSPENDED = "suspended"
    DELETED = "deleted"

    def can_login(self) -> bool:
        return self is UserStatus.ACTIVE

def can_user_login(user):
    return user.status.can_login()

def update_user_status(user, new_status: str):
    user.status = UserStatus(new_status)   # raises if invalid

The after version has fewer lines, no scattered string literals, no manual validation, type- checker friendliness, and behavior co-located with the data. The serialization to JSON keeps working because we inherit from str.

Common questions

Pushbacks Python devs raise about Enums

  • "Aren't dataclasses enough?"

    Dataclasses solve a different problem — they hold structured data. Enums constrain the set of valid values for one field. They compose: a dataclass can have an Enum field, and that's a very common pattern.

  • "What about Literal types or TypedDict?"

    Literal["active", "pending"] gives you static type checking, but no runtime validation and no place to attach behavior. Enums give you both. Use Literal for purely static contracts; use Enum when you also want runtime guarantees.

  • "Do Enums slow my code down?"

    The overhead is negligible — Enum lookups are O(1) hash lookups. In any code where this matters, you have a much bigger performance problem than Enums.

  • "How do I migrate a legacy codebase that uses strings everywhere?"

    Add the str + Enum at the boundaries first (API layer, DB layer). Because the Enum instances compare equal to their string values, the rest of the code keeps working. Migrate the internal call sites incrementally as you touch them.

Key Takeaway

The Enum trick is not “use Enums.” It is to treat them as small domain types: encode behavior on them, construct them at the boundary so invalid input fails immediately, and inherit from str when you need clean serialization. Done that way, Enums quietly remove a whole class of bugs from your codebase.
