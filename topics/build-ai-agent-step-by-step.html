Build an AI Agent Step by Step

A hands-on path from a plain LLM call to a production-ready, tool-using agent — built incrementally, one layer at a time. Every step includes the code, the reasoning behind the design choice, and the failure it prevents. JavaScript throughout.

Portal Internal · Curated
Focus Practical build, step-by-step with JS code
Published Jun 10, 2026
Level Beginner → Production-ready

Building an AI agent is not about finding the right framework and calling it. It is about adding capability one layer at a time — each layer solving a concrete problem the previous one created. Start with a plain LLM call. Add a tool when you need external data. Add a loop when one tool call isn't enough. Add memory when context falls off. Add safety when actions have consequences. That progression is this guide.

The most important thing to understand first: An agent is a decision-making system that speaks natural language — not a chatbot with extra features. Every interaction bottoms out in a decision. Design the decision before you design the agent. If you cannot state in one sentence what decision the agent is making, you are not ready to build it yet.

The three foundation questions

  • What decision is the agent making? Route this request, answer this question, trigger this action, ask for more information. State it exactly.
  • What does it need to know to make that decision well? What data, history, and context must be present before it acts?
  • What happens when it is wrong? What is the blast radius? How does the user recover? Design the failure mode before you build the happy path.
Step 0

Define the job before writing any code

The most common agent failure is a vague brief. "Build a customer support agent" is not a brief — it is a category. A brief is: "Given an authenticated user's message about an order, look up the order status and respond with the current status and estimated delivery date. Escalate to human if the order is flagged disputed." That sentence defines the input, the tools needed, the output shape, and the escalation path.

QuestionBad answerGood answer
What is the agent's one job? "Help customers" "Answer order-status questions using live order data"
What is the success condition? "The user is happy" "Correct status returned in <2 s, zero hallucinated order IDs"
What is the acceptable error rate? "Low" "<1% wrong order data; 0% unauthorized refunds"
What happens when it fails? "Try again" "After 2 failed tool calls, escalate to live agent with full transcript"

Decisions made vaguely at Step 0 will haunt you at Steps 4, 5, and 6. Write the brief down. Every system-prompt instruction and tool description you write later is a translation of this brief into machine-readable form.

Step 1

Start with a plain LLM call — no agent yet

Before adding any agent machinery, get a working single-turn LLM call. This forces you to write a system prompt that is correct and test the model's baseline behaviour on your task. Many things people want to build as agents should stay here permanently. 

import Anthropic from '@anthropic-ai/sdk';

const client = new Anthropic(); // reads ANTHROPIC_API_KEY from env

async function singleTurnAgent(userMessage) {
  const response = await client.messages.create({
    model:      'claude-sonnet-4-6',
    max_tokens: 1024,
    system: `You are a customer support assistant for an e-commerce company.
You help customers with questions about their orders, returns, and products.
Be concise and professional. If you do not know the answer, say so — do not guess.`,
    messages: [
      { role: 'user', content: userMessage }
    ]
  });

  return response.content[0].text;
}

// Test it
const answer = await singleTurnAgent("What is your return policy?");
console.log(answer);

// AT THIS POINT:
// - The model answers from its system prompt context only
// - No live data, no external tools, no memory
// - This is correct for static policy questions
// - Add tools only when the answer requires LIVE or USER-SPECIFIC data

The question to ask before proceeding: "Can the model answer this correctly from the information already in the system prompt?" If yes — stay here. Adding agent machinery to a single-prompt task adds latency, cost, and failure points with no benefit.

Step 2

Add your first tool — and design it properly

You add a tool when you need something the model cannot produce from training data alone: the user's live order status, current pricing, a database record, a calculation. The tool definition — not the function body — is where most tool bugs live.

The model selects tools by reading their descriptions. Your description is the selection logic. There is no code path that chooses a tool — the model pattern-matches the user's message against every tool description in the context. Write it for the model, not for humans.

Three things every tool definition needs

  • Name: a verb phrase that states what the tool does. lookupOrderStatus, not orders or getOrder.
  • Description: when to use it, what it returns, and crucially what NOT to use it for. The negative scope prevents selection collisions between similar tools.
  • Parameters: strict types. Required fields only where truly mandatory. For any required field: include "do not guess — ask the user" in the description.
// GOOD tool definition — selection signal is clear, negative scope is set,
// required parameter has a fallback instruction
const lookupOrderStatusTool = {
  name: 'lookupOrderStatus',
  description: `Look up the real-time status and details of a customer order.
Use when a customer asks about where their order is, when it will arrive,
what items are in it, or its current status.
Returns: order status, item list, shipping carrier, and estimated delivery date.
Do NOT use for refunds or cancellations — use the processRefund tool instead.`,
  input_schema: {
    type: 'object',
    properties: {
      orderId: {
        type: 'string',
        description: `The order ID. Expected format: ORD-XXXXX.
If the customer has not provided their order ID, ask for it before calling this tool.
Do NOT invent or guess an order ID.`
      }
    },
    required: ['orderId']
  }
};

// The tool executor — your actual business logic lives here
async function lookupOrderStatus({ orderId }) {
  // In production: call your order management API
  const order = await orderAPI.getById(orderId);
  if (!order) return { error: 'Order not found', orderId };
  return {
    orderId:           order.id,
    status:            order.status,
    items:             order.lineItems.map(i => `${i.name} × ${i.qty}`),
    carrier:           order.shipping.carrier,
    trackingNumber:    order.shipping.trackingNumber,
    estimatedDelivery: order.shipping.estimatedDelivery
  };
}

The "Do NOT use for X" line is as important as "use for Y." Without negative scope, when you add a second tool later that sounds similar, the model will randomly pick between them. Negative scope is how you draw clean boundaries between tools.

Step 3

Wire the agent loop — think → act → observe → adjust

A single LLM call with a tool definition does not make an agent. The agent is the loop: the model declares a tool call → your code executes it → the result goes back into the conversation → the model reads the result and decides what to do next. That cycle repeats until the model generates a text response (task complete) or the iteration limit is reached.

The critical thing: the model never executes a tool. It outputs a structured JSON intent — which tool, which arguments — and your scaffolding makes the real call. Model produces intent; your code produces action. 

import Anthropic from '@anthropic-ai/sdk';

const client = new Anthropic();

// All tool definitions in one place
const tools = [lookupOrderStatusTool /*, ...more tools later */];

// All tool executors mapped by name
const toolExecutors = {
  lookupOrderStatus
  // add more as you add tools
};

async function agentLoop(userMessage) {
  const messages = [{ role: 'user', content: userMessage }];
  const MAX_STEPS = 8; // ALWAYS cap — a loop without a ceiling is a bug waiting to fire

  for (let step = 0; step < MAX_STEPS; step++) {
    const response = await client.messages.create({
      model:      'claude-sonnet-4-6',
      max_tokens: 1024,
      system: `You are a customer support assistant.
Help customers with order status, refunds, and product questions.
Think step by step. Use tools when you need live data.
If you need information the customer hasn't provided, ask for it.`,
      tools,
      messages
    });

    // Model finished — return the final text answer
    if (response.stop_reason === 'end_turn') {
      const textBlock = response.content.find(b => b.type === 'text');
      return textBlock?.text ?? 'No response generated.';
    }

    // Model wants to use a tool
    if (response.stop_reason === 'tool_use') {
      // Push the model's full response (may include thinking text + tool call) into history
      messages.push({ role: 'assistant', content: response.content });

      // Process every tool call in this response (model can call multiple at once)
      const toolResults = [];
      for (const block of response.content) {
        if (block.type !== 'tool_use') continue;

        let result;
        try {
          const executor = toolExecutors[block.name];
          if (!executor) throw new Error(`Unknown tool: ${block.name}`);
          result = await executor(block.input);
        } catch (err) {
          result = { error: err.message };
        }

        toolResults.push({
          type:        'tool_result',
          tool_use_id: block.id,
          content:     JSON.stringify(result)
        });
      }

      // Feed all results back as the next user turn — model reads and continues
      messages.push({ role: 'user', content: toolResults });
      // Loop continues — model sees the observation and decides next step
    }
  }

  return 'I was unable to complete this request within the allowed steps. Please contact support directly.';
}

// Test it
const reply = await agentLoop("Hi, where is my order ORD-4521?");
console.log(reply);
// Loop:
// 1. Model reads message, decides to call lookupOrderStatus({ orderId: "ORD-4521" })
// 2. Your code calls the real API, gets status data
// 3. Model reads the data, generates a natural-language answer
// 4. stop_reason = 'end_turn' → returns the answer

Step 4

Add memory — don't make the agent ask twice

Without memory, every turn starts cold. The agent re-confirms the user's identity, re-asks for the order ID, re-retrieves data it already has. Memory is the mechanism that makes a multi-turn conversation feel coherent rather than amnesiac.

Memory is actually four distinct layers. Using the wrong one is a common design mistake.

LayerLifetimeUse forImplementation
In-context This turn only Current conversation history, tool results this session The messages array — already handled by the loop
Session / state Duration of the session Verified identity, confirmed order ID, active record, user name Key-value object injected into system prompt each turn
Long-term / vector Permanent User preferences, past issue history, domain knowledge Embeddings + vector search (RAG) — retrieved per query
Episodic Permanent, grows over time Log of what the agent did and what happened in past sessions Structured log DB — queried to inform future similar tasks

Session memory implementation — the most immediately useful layer

// Session store — lives in memory for the duration of a conversation
function createSession() {
  const store = {};
  return {
    set: (key, value) => { store[key] = value; },
    get: (key)        => store[key] ?? null,
    has: (key)        => key in store,
    // Renders the store as a block of context to inject into the system prompt
    toContextBlock: () => {
      const entries = Object.entries(store);
      if (entries.length === 0) return '';
      return '\n\nKnown context this session:\n' +
        entries.map(([k, v]) => `- ${k}: ${v}`).join('\n');
    }
  };
}

// Updated agent loop — now carries session state across turns
async function agentWithMemory(userMessage, session) {
  const systemPrompt = `You are a customer support assistant.
Help customers with order status, refunds, and product questions.
${session.toContextBlock()}

Use the context above when it is relevant — do not ask for information
you already have.`;

  const messages = [{ role: 'user', content: userMessage }];

  for (let step = 0; step < 8; step++) {
    const response = await client.messages.create({
      model: 'claude-sonnet-4-6', max_tokens: 1024,
      system: systemPrompt, tools, messages
    });

    if (response.stop_reason === 'end_turn') {
      return response.content.find(b => b.type === 'text')?.text;
    }

    if (response.stop_reason === 'tool_use') {
      messages.push({ role: 'assistant', content: response.content });
      const toolResults = [];

      for (const block of response.content) {
        if (block.type !== 'tool_use') continue;
        const result = await toolExecutors[block.name](block.input);

        // Persist useful facts to session memory so they survive the turn
        if (block.name === 'lookupOrderStatus' && result.orderId) {
          session.set('lastLookedUpOrderId', result.orderId);
          session.set('lastOrderStatus', result.status);
        }

        toolResults.push({ type: 'tool_result', tool_use_id: block.id, content: JSON.stringify(result) });
      }

      messages.push({ role: 'user', content: toolResults });
    }
  }
}

// Usage across multiple turns in a conversation
const session = createSession();
session.set('userId',        'USR-4521');
session.set('authenticated', 'true');
session.set('userName',      'Abhinav');

// Turn 1: user asks about an order
await agentWithMemory("Where is my order ORD-8833?", session);
// Agent calls lookupOrderStatus, stores lastLookedUpOrderId = ORD-8833

// Turn 2: user asks a follow-up — agent already knows the order ID
await agentWithMemory("When will it be delivered?", session);
// System prompt now includes: lastLookedUpOrderId: ORD-8833, lastOrderStatus: In Transit
// Agent answers directly without re-calling lookupOrderStatus

Design rule: match storage layer to the fact's lifetime

Ask for every piece of information: does this need to survive the turn? The session? Forever? Session memory for the order ID confirmed this conversation. Long-term store for the user's preference to always receive tracking links. Do not default everything to in-context — when the context window fills, older facts fall off silently.

Step 5

Add a validation layer — stop hallucinated parameters before they execute

This is the step most agent tutorials skip, and it is where the most dangerous bugs live. When the model constructs tool call arguments, it pulls values from three sources: the user's direct message, the conversation and memory context, and inference from surrounding context. If a required field has no clear source in any of these, the model invents a plausible-looking value. It does not say "I don't know" — it fills the gap confidently. 

// The hallucination scenario — without validation
// User: "Can you process a refund?"
// Required: orderId, amount, reason
// orderId was never mentioned → model generates: "ORD-0001" (invented)
// amount was never specified → model generates: 99.99 (invented)
// Both wrong values reach the payment API

// THE FIX — two parts working together

// PART 1: Description-level instruction in the tool schema
const processRefundTool = {
  name: 'processRefund',
  description: `Process a refund for a customer order.
ONLY call this tool after you have EXPLICITLY confirmed the order ID AND
the refund amount with the customer in this conversation.
Do NOT call this tool if either value has not been provided by the customer.`,
  input_schema: {
    type: 'object',
    properties: {
      orderId: {
        type: 'string',
        description: `The order ID to refund (format ORD-XXXXX).
Must be confirmed by the customer. If not provided, ask for it — do not guess.`
      },
      amount: {
        type: 'number',
        description: `Refund amount in USD. Must be stated by the customer.
If not specified, ask: "How much would you like refunded?" before calling this tool.`
      },
      reason: {
        type: 'string',
        enum: ['damaged', 'not_received', 'wrong_item', 'changed_mind', 'other'],
        description: 'Reason for the refund. Infer from the conversation if clear, otherwise ask.'
      }
    },
    required: ['orderId', 'amount', 'reason']
  }
};

// PART 2: Code-level validation layer — intercept every tool call before it fires
function validateToolCall(toolName, args) {
  const tool = tools.find(t => t.name === toolName);
  if (!tool) return { valid: false, error: `Unknown tool: ${toolName}` };

  for (const field of (tool.input_schema.required ?? [])) {
    const value = args[field];

    // Missing field
    if (value === undefined || value === null || value === '') {
      return {
        valid: false,
        field,
        recoveryInstruction: `You are missing the required field "${field}" for ${toolName}.
Ask the user to provide it before calling this tool.`
      };
    }

    // Plausibility checks for known patterns
    if (field === 'orderId' && typeof value === 'string' && !value.match(/^ORD-\d+$/i)) {
      return {
        valid: false,
        field,
        recoveryInstruction: `The orderId value "${value}" doesn't match the expected format (ORD-XXXXX).
Confirm the correct order ID with the customer before proceeding.`
      };
    }

    if (field === 'amount' && (typeof value !== 'number' || value <= 0)) {
      return {
        valid: false,
        field,
        recoveryInstruction: `The refund amount must be a positive number.
Ask the customer to confirm the exact amount they want refunded.`
      };
    }
  }

  return { valid: true };
}

// Updated loop — validation sits between the model's intent and your execution
for (const block of response.content) {
  if (block.type !== 'tool_use') continue;

  const validation = validateToolCall(block.name, block.input);

  if (!validation.valid) {
    // Don't execute the bad call — inject recovery instruction back to the model
    toolResults.push({
      type:        'tool_result',
      tool_use_id: block.id,
      content:     validation.recoveryInstruction,
      is_error:    true
    });
    // The model reads this and asks the user for the missing information
    continue;
  }

  const result = await toolExecutors[block.name](block.input);
  toolResults.push({ type: 'tool_result', tool_use_id: block.id, content: JSON.stringify(result) });
}
Step 6

Add actions — and design risk gates before you ship write operations

Up to this point everything has been read-only: look up data, return information. The moment the agent can change state in the world — send an email, process a payment, update a record, delete data — the design requirements change fundamentally.

Tools (read)Actions (write)
Effect on world stateNone — safe to retry freelyChanges something — may be irreversible
Error recoveryCall againMay need rollback or human intervention
Design priorityAccuracy of resultAccuracy + blast radius + approval gate

Design the blast radius before writing any action code

For every action, ask: what is the worst case if this fires on wrong data? A wrongly-sent email is embarrassing. A wrongly-triggered refund is a financial incident. A wrongly-triggered account deletion is a crisis. The risk classification dictates the gate. 

// Action registry — risk metadata lives here, not in the executor
const actionRegistry = {
  updateShippingAddress: {
    risk:        'medium',
    reversible:  true,
    description: 'Updates the shipping address on a pending order',
    execute: async ({ orderId, newAddress }) =>
      orderAPI.updateShipping(orderId, newAddress)
  },

  processRefund: {
    risk:        'high',
    reversible:  false, // issued refunds can't be recalled
    description: 'Issues a monetary refund for an order',
    execute: async ({ orderId, amount, reason }) =>
      paymentAPI.issueRefund(orderId, amount, reason)
  },

  cancelOrder: {
    risk:        'high',
    reversible:  false,
    destructive: true,
    dryRun: async ({ orderId }) => {
      const order = await orderAPI.getById(orderId);
      return `Would cancel order ${orderId} (${order.items.join(', ')}). This cannot be undone.`;
    },
    execute: async ({ orderId }) => orderAPI.cancel(orderId)
  }
};

// Action executor with risk gates
async function executeAction(actionName, args, context) {
  const action = actionRegistry[actionName];
  if (!action) throw new Error(`Unknown action: ${actionName}`);

  // Gate 1: high-risk actions require explicit human approval
  if (action.risk === 'high') {
    const approved = await requestHumanApproval({
      action: actionName,
      args,
      context,
      description: action.description,
      message: `Agent requests permission to run "${actionName}" with: ${JSON.stringify(args, null, 2)}`
    });

    if (!approved) {
      return { status: 'cancelled', reason: 'Approval denied. No changes were made.' };
    }
  }

  // Gate 2: destructive actions show a preview before executing
  if (action.destructive && action.dryRun) {
    const preview = await action.dryRun(args);
    return {
      status:  'preview',
      preview,
      message: `Preview of what would happen:\n${preview}\n\nConfirm to proceed.`
    };
  }

  try {
    const result = await action.execute(args);
    return { status: 'success', result };
  } catch (err) {
    return {
      status:    'error',
      error:     err.message,
      retriable: action.reversible
    };
  }
}

Blast radius principle: Design the confirmation strategy and rollback path before writing the action code. The moment you ship an action without these, you are one bad tool call away from a production incident.

Step 7

Package compound work into skills — keep the decision space manageable

When the agent's tool list grows past roughly 10–12 entries, selection accuracy starts to degrade. The model faces too many choices with overlapping descriptions and starts making inconsistent picks. The solution is skills: compound, reusable capabilities that combine multiple tools and LLM calls into a single callable unit.

The value is cognitive load reduction — the agent chooses between 6 skills, not 30 tools. Each skill manages its own internal complexity.

Skill typeWhat it doesExample
Cognitive LLM reasoning packaged as a reusable function: summarise, classify, extract, compare summariseSkill(text, length)
Procedural Fixed ordered sequence of steps — deterministic, no model judgment in the middle onboardCustomerSkill(): create → send email → assign manager → log
Compositional Skills calling other skills or tools — handles complex multi-source tasks as one unit researchCompetitorSkill(name): search × 2 + extract + summarise 
// COGNITIVE SKILL — an LLM call with a focused, reusable prompt
async function summariseSkill(text, length = 'short') {
  const lengthMap = {
    short:  '2-3 concise sentences',
    medium: 'one focused paragraph',
    long:   '3-4 detailed paragraphs'
  };
  const response = await client.messages.create({
    model: 'claude-haiku-4-5-20251001', // cheaper model is fine for focused cognitive tasks
    max_tokens: 512,
    messages: [{
      role: 'user',
      content: `Summarise the following in ${lengthMap[length]}.
Stick strictly to what is in the text — do not add outside information.

TEXT:
${text}`
    }]
  });
  return response.content[0].text;
}

// COGNITIVE SKILL — extract structured entities from free text
async function extractEntitiesSkill(text, entityTypes = ['person', 'company', 'date', 'amount']) {
  const response = await client.messages.create({
    model: 'claude-haiku-4-5-20251001',
    max_tokens: 512,
    messages: [{
      role: 'user',
      content: `Extract the following entity types from the text below.
Return a JSON object with one key per entity type, each containing an array of found values.
Types: ${entityTypes.join(', ')}. Return empty arrays for types not found.
Respond with valid JSON only.

TEXT: ${text}`
    }]
  });
  return JSON.parse(response.content[0].text);
}

// COMPOSITIONAL SKILL — search + extract + summarise in one callable unit
// The agent calls this as a single step instead of managing 4 individual tool calls
async function researchCompetitorSkill(competitorName) {
  // Step 1: gather raw data using search tools
  const [productInfo, newsInfo] = await Promise.all([
    searchWeb(`${competitorName} product features pricing 2026`),
    searchWeb(`${competitorName} recent news announcements`)
  ]);

  const rawText = productInfo.text + '\n\n' + newsInfo.text;

  // Step 2: extract key entities from the raw data
  const entities = await extractEntitiesSkill(rawText, ['product', 'price', 'feature', 'date']);

  // Step 3: summarise into a readable output
  const summary = await summariseSkill(rawText, 'medium');

  return {
    competitor: competitorName,
    summary,
    entities,
    sources:    [...productInfo.urls, ...newsInfo.urls]
  };
}

// From the agent's perspective: one clean call, one structured result
const report = await researchCompetitorSkill('Acme Corp');
// Internally ran: 2 searches + entity extraction + summarisation
// Agent never needed to manage those 4 steps individually
Step 8

Add reflection — catch errors before they become answers

Without reflection, the agent treats every tool result as correct and every generated answer as complete. In the real world: APIs return empty results, data is stale, the model misses half of what was asked. Reflection is a second pass that evaluates the output against a concrete rubric before returning it to the user.

The key implementation detail: the critic needs specific criteria, not "is this good?" — vague critique produces vague improvement. Three concrete questions are enough: accurate? complete? grounded? 

async function agentWithReflection(userMessage, session, maxRetries = 2) {
  let currentMessage = userMessage;

  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    const result = await agentWithMemory(currentMessage, session);

    // On the final retry, return whatever we have
    if (attempt === maxRetries) return result;

    // Critic evaluates the result against three specific criteria
    const critique = await client.messages.create({
      model: 'claude-sonnet-4-6',
      max_tokens: 256,
      messages: [{
        role: 'user',
        content: `Evaluate this response to the original request: "${userMessage}"

Response to evaluate:
"${result}"

Answer YES or NO for each criterion:
1. ACCURATE   — Is every factual claim in the response correct based on the retrieved data?
2. COMPLETE   — Does the response fully address what was asked, with no important parts missing?
3. GROUNDED   — Does every claim trace back to retrieved tool data, not model assumptions?

If all three are YES: respond exactly: PASS
If any are NO: respond exactly: FAIL - [one sentence per failed criterion stating what is wrong]`
      }]
    });

    const evaluation = critique.content[0].text.trim();

    if (evaluation.startsWith('PASS')) return result; // Passed — return it

    // Failed — inject the critique and the original ask, retry with the issues in context
    currentMessage = `${userMessage}

Your previous response was reviewed and rejected:
"${result}"

Issues found:
${evaluation.replace('FAIL - ', '')}

Retry addressing these specific issues. Be precise and grounded in the retrieved data.`;
  }
}
Step 9

Scope the environment — control what the agent perceives and can change

The environment is everything the agent sees and everything it can affect. Get it wrong and you have either a blind agent (perception too narrow) or a dangerous one (affordances too broad). Both are design failures, not model failures. 

// Build the agent's environment explicitly — pass it at the start of every turn
function buildEnvironment(session, request) {
  const isAuthenticated = session.get('authenticated') === 'true';
  const orderConfirmed  = session.has('confirmedOrderId');

  return {
    // PERCEPTION — what the agent knows about the current context
    // Make this explicit in the system prompt: don't make the agent guess its own context
    contextBlock: `
CURRENT SESSION CONTEXT:
- User: ${session.get('userName') ?? 'unknown'} (${session.get('userPlan') ?? 'standard'} plan)
- Authenticated: ${isAuthenticated}
- Active order: ${session.get('confirmedOrderId') ?? 'none'}
- Session started: ${session.get('sessionStart') ?? 'unknown'}
- Recent errors: ${session.get('lastError') ?? 'none'}`,

    // AFFORDANCES — scoped by session state
    // The model cannot select tools it cannot see — use this as a security boundary
    availableTools: isAuthenticated
      ? (orderConfirmed
          ? [lookupOrderStatusTool, processRefundTool, updateShippingTool]  // Full set
          : [lookupOrderStatusTool, searchKnowledgeBaseTool])                // Pre-confirmation: read-only
      : [searchKnowledgeBaseTool],                                           // Pre-auth: knowledge only

    // FAILURE HANDLING — every tool call can fail, define the response before it happens
    onToolError: (toolName, error) => ({
      status:     'error',
      tool:       toolName,
      message:    error.message,
      suggestion: `The ${toolName} tool is unavailable. Ask the user to provide the information manually, or suggest they try again in a moment.`
    })
  };
}

// PRINCIPLE: Sandbox during development, expand affordances only after validation
// In dev/staging: only read tools (lookupOrderStatus, searchKnowledgeBase)
// In production: full scoped set based on session state
const affordanceSet = process.env.NODE_ENV === 'production'
  ? buildEnvironment(session, request).availableTools
  : [lookupOrderStatusTool, searchKnowledgeBaseTool]; // Read-only during dev

Four environment design rules

  • Minimise affordances. Only expose the tools the agent needs for its current state. Unauthenticated users get read-only tools. The model cannot call what it cannot see.
  • Make context explicit in the prompt. If the user is authenticated, say so. If an order is "pending," include that. Implicit state becomes hallucinated assumptions.
  • Design every tool's failure path. APIs go down. Databases return null. Define the fallback before the failure happens, not after.
  • Sandbox before production. During development, agents should only see read-only tools against test data. Gate write access to after the agent's behaviour is validated against a representative benchmark.

// ─── VALIDATION ──────────────────────────────────────────────────────────────
function validate(toolName, args) {
  const tool = allTools[toolName];
  if (!tool) return { ok: false, msg: `Unknown tool: ${toolName}` };
  for (const field of (tool.definition.input_schema.required ?? [])) {
    if (!args[field]) return {
      ok:  false,
      msg: `Missing required field "${field}" for ${toolName}. Ask the user to provide it.`
    };
  }
  return { ok: true };
}

// ─── FULL AGENT ───────────────────────────────────────────────────────────────
async function runAgent(userMessage, session) {
  // Scope tools based on session state
  const isAuth   = session.get('authenticated') === 'true';
  const hasOrder = session.has('confirmedOrderId');
  const toolKeys = isAuth
    ? (hasOrder ? ['lookupOrderStatus','processRefund','searchKnowledgeBase'] : ['lookupOrderStatus','searchKnowledgeBase'])
    : ['searchKnowledgeBase'];

  const tools = toolKeys.map(k => allTools[k].definition);

  const systemPrompt = `You are a customer support assistant.
Help with order status, refunds, and product questions.
Think step by step. Use tools when you need live data.
Do NOT guess order IDs or amounts — ask the customer.
${session.toContextBlock()}`;

  const messages = [{ role: 'user', content: userMessage }];

  for (let step = 0; step < 8; step++) {
    const response = await client.messages.create({
      model: 'claude-sonnet-4-6', max_tokens: 1024,
      system: systemPrompt, tools, messages
    });

    if (response.stop_reason === 'end_turn') {
      return response.content.find(b => b.type === 'text')?.text;
    }

    if (response.stop_reason === 'tool_use') {
      messages.push({ role: 'assistant', content: response.content });
      const results = [];

      for (const block of response.content) {
        if (block.type !== 'tool_use') continue;
        const { name, id, input } = block;

        // Validation gate
        const check = validate(name, input);
        if (!check.ok) {
          results.push({ type: 'tool_result', tool_use_id: id, content: check.msg, is_error: true });
          continue;
        }

        // Risk gate for high-risk actions
        if (allTools[name]?.risk === 'high') {
          const approved = await requestHumanApproval(name, input);
          if (!approved) {
            results.push({ type: 'tool_result', tool_use_id: id, content: 'Action cancelled — approval denied.' });
            continue;
          }
        }

        // Execute
        let result;
        try {
          result = await allTools[name].execute(input);
          // Persist useful facts to session
          if (name === 'lookupOrderStatus') session.set('confirmedOrderId', input.orderId);
        } catch (err) {
          result = { error: err.message };
        }

        results.push({ type: 'tool_result', tool_use_id: id, content: JSON.stringify(result) });
      }

      messages.push({ role: 'user', content: results });
    }
  }

  return 'Unable to complete this request. Please contact support directly.';
}

Common Failure Modes

Five failure modes and exactly how to fix each

FailureWhat you seeRoot causeFix
Wrong tool selected Agent searches the knowledge base for a live order status; returns stale generic info Two tools with overlapping descriptions — model can't distinguish them Add negative scope to both: "Do NOT use for X." Reduce tool count per context. Separate tools clearly by return type (live data vs. static docs).
Hallucinated parameters Agent calls processRefund with an order ID the user never mentioned Required field with no source in context — model invents a plausible value Add "do not guess — ask the user" to every required parameter description. Add a validation layer that checks required fields before any tool fires.
Correct tool, wrong timing Agent processes a refund before confirming the amount with the user No pre-condition enforced. Model follows the action description but skips the gathering step. Add explicit pre-condition in the tool description ("only call after confirming X"). Use session-state-based tool scoping to make the action unavailable until the pre-condition is met.
Context window overflow In long sessions: agent repeats questions, forgets earlier confirmations All history kept in-context. When the window fills, old content falls off silently. Summarise older conversation turns and move to session memory. Persist confirmed facts (order ID, user name) to the session store so they survive the turn-off.
Infinite loop / doom loop Agent calls the same failing tool over and over; never escalates No iteration cap. Error handling re-feeds into the same tool call without a different strategy. Always set MAX_STEPS. Track which tools have been tried. After N failures on the same tool, switch strategy or return a graceful fallback message with escalation to a human.
Decision Framework

When NOT to use an agent

Agents add latency, cost, and failure points. Many things people reach for an agent to solve are better served by something simpler. Use this test before committing to agent architecture.

If your task is…Use this insteadWhy
A single transformation with fixed inputs and outputs (summarise, classify, draft) One LLM call with a prompt template No loop needed. Straight line in → out. Adding a loop adds overhead with no benefit.
A fixed, ordered process where every step is known in advance Deterministic workflow (code, no LLM in the loop) If the path never branches dynamically, there is nothing for the agent to decide. Hard-code it.
A question answerable purely from training knowledge or static docs RAG + single LLM call Retrieve the relevant documents, generate one response. No reasoning loop needed.

The litmus test: Does completing this task require the system to make decisions based on information it doesn't have at the start? Does the path change depending on what it discovers at runtime? If yes — agent. If no — something simpler.

Key Takeaway

An agent is not "just prompting harder." It is a product: a loop with typed tools, a validation layer, memory tiers, risk-gated actions, and quality gates wrapped around a model. The model is the brain — it reads your tool descriptions, matches them to context, and declares intent. Your scaffolding is everything else: executing the calls, validating the arguments, persisting the state, scoping the affordances, and catching the failures before they reach the user. Build that scaffolding with the same rigour you would give any production system — because that is exactly what an agent is.
