Images

Working with images

docker build -t myapp:1.0 .

Build image from Dockerfile in current dir, tag as myapp:1.0

docker build --no-cache -t myapp .

Build without using the layer cache (forces fresh build)

docker images

List all local images

docker rmi myapp:1.0

Remove image by tag

docker image prune -a

Remove all unused images (reclaim disk space)

docker pull node:22-alpine

Pull image from Docker Hub

docker push myregistry.io/myapp:1.0

Push image to a registry

docker inspect myapp:1.0

Show detailed image metadata (layers, env vars, entrypoint)

docker history myapp:1.0

Show layer history — useful for debugging large images

docker save myapp:1.0 | gzip > myapp.tar.gz

Export image to a tar file for transfer

Containers

Working with containers

docker run -d -p 3000:3000 --name api myapp

Run container in background, map port 3000, name it "api"

docker run --rm -it node:22 sh

Run interactive shell, remove container on exit

docker run -e NODE_ENV=production myapp

Pass environment variable to container

docker run --env-file .env myapp

Load env vars from .env file

docker run -v $(pwd)/data:/app/data myapp

Mount current directory's data folder into container

docker ps

List running containers

docker ps -a

List all containers including stopped ones

docker stop api && docker rm api

Stop then remove a named container

docker logs api -f --tail 100

Follow last 100 lines of container logs

docker exec -it api sh

Open shell inside a running container

docker exec api cat /etc/os-release

Run one-off command inside container

docker stats

Live resource usage (CPU, memory, network) for all containers

docker cp api:/app/logs ./local-logs

Copy files out of a container to local filesystem

docker container prune

Remove all stopped containers

Volumes & Networks

Persistence and networking

docker volume create pgdata

Create a named volume for persistent storage

docker run -v pgdata:/var/lib/postgresql/data postgres

Use named volume for Postgres data directory

docker volume ls && docker volume prune

List volumes, then remove unused ones

docker network create mynet

Create a custom bridge network

docker run --network mynet --name db postgres

Connect container to custom network — reachable by name "db"

docker network inspect mynet

Show all containers on a network and their IPs

Dockerfile patterns

Writing good Dockerfiles

# Multi-stage Dockerfile for a Node.js app — production-ready pattern
# Stage 1: install dependencies
FROM node:22-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production       # install prod deps only

# Stage 2: build (TypeScript compilation, bundling)
FROM node:22-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci                          # install all deps (including dev)
COPY . .
RUN npm run build                   # compile TypeScript → dist/

# Stage 3: production image — minimal, no source code
FROM node:22-alpine AS production
WORKDIR /app
# Create non-root user for security
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
COPY --from=deps   /app/node_modules ./node_modules
COPY --from=build  /app/dist         ./dist
COPY package.json ./
USER appuser                        # never run as root in production
EXPOSE 3000
ENV NODE_ENV=production
CMD ["node", "dist/index.js"]

Key Dockerfile best practices

# .dockerignore — always include this (like .gitignore for Docker)
node_modules
dist
.env
.git
*.log
coverage
.nyc_output
README.md

# Why each matters:
# node_modules: rebuilt inside container from package.json — copying local version causes platform issues
# .env: secrets should never be baked into images
# .git: large and irrelevant to runtime

# Layer caching tip: order Dockerfile instructions from least to most frequently changed
# WRONG order (invalidates cache on every source change):
FROM node:22-alpine
COPY . .          # changes every time code changes — npm ci runs every build
RUN npm ci

# RIGHT order (only invalidates cache when package.json changes):
FROM node:22-alpine
COPY package*.json ./   # only changes when dependencies change
RUN npm ci              # cached until package.json changes
COPY . .                # source changes don't bust the npm ci cache
Docker Compose

Multi-container development environments

# docker-compose.yml — Node.js API + Postgres + Redis dev environment
version: '3.9'
services:
  api:
    build:
      context: .
      target: build          # use the "build" stage from multi-stage Dockerfile
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=development
      - DATABASE_URL=postgresql://postgres:password@db:5432/myapp
      - REDIS_URL=redis://redis:6379
    volumes:
      - .:/app               # mount source — enables hot reload
      - /app/node_modules    # anonymous volume keeps container's node_modules
    depends_on:
      db:    { condition: service_healthy }
      redis: { condition: service_started }
    command: npm run dev      # override CMD for development

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: password
      POSTGRES_DB: myapp
    ports:
      - "5432:5432"          # expose for local tools (TablePlus, psql)
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"

volumes:
  pgdata:

Docker Compose command reference

docker compose up -d

Start all services in background

docker compose up --build

Rebuild images before starting (picks up Dockerfile changes)

docker compose down -v

Stop services and remove volumes (clean slate)

docker compose logs api -f

Follow logs for a specific service

docker compose exec api sh

Open shell in running service container

docker compose run --rm api npm test

Run one-off command in a fresh container

docker compose ps

Show status of all services

docker compose pull

Pull latest versions of all base images

Production tips

What separates dev Docker from prod Docker

  • Always use specific image tags, never latest. node:22.5.1-alpine3.20 not node:latest. Latest changes unexpectedly and breaks builds.
  • Run as non-root user. Add USER appuser to your Dockerfile after creating the user. Most security scanners will flag root containers.
  • Use multi-stage builds for every production image. A single-stage Node.js image includes TypeScript, dev dependencies, and source maps — all unnecessary in production.
  • Set NODE_ENV=production in the image. This disables Express error stacks, enables production-mode optimisations in most frameworks.
  • Add a HEALTHCHECK to every service image. Orchestrators (Kubernetes, ECS) use this to know when a container is actually ready.
  • Set memory limits on containers. A Node.js process with a memory leak will not OOM the host — it gets killed and restarted cleanly.
  • Use docker scout for vulnerability scanning. Run before pushing to a registry: docker scout cves myapp:1.0.
# Healthcheck in Dockerfile
HEALTHCHECK --interval=30s --timeout=5s --start-period=30s --retries=3 \
  CMD node -e "require('http').get('http://localhost:3000/health', r => { process.exit(r.statusCode === 200 ? 0 : 1) })"

# Express health endpoint
app.get('/health', (req, res) => {
  // Check DB connectivity, etc.
  res.json({ status: 'ok', uptime: process.uptime() });
});

Key Takeaway

The three Docker habits that pay off most: always use specific image tags (reproducible builds), always write multi-stage Dockerfiles (small production images), and always include a .dockerignore file (fast builds, no secret leakage). For development, Docker Compose with a named volume for node_modules and a bind mount for source code gives you hot reload without the platform-mismatch issues that come from copying local node_modules.
