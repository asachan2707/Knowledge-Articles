A walkthrough of Karpathy's nanoGPT — a complete, trainable transformer language model in 243 lines of Python — and what the code teaches you about how every modern LLM actually works from the ground up.

Most engineers who use LLMs every day have no idea what happens inside them. Karpathy's nanoGPT cuts through all abstraction: 243 lines of Python that define a complete transformer, train it on Shakespeare, and generate text. Reading it once is worth more than a semester of AI theory because the mystery disappears — you can see exactly where attention is computed, where loss is calculated, and why the architecture works.

Core idea: The entire transformer architecture — tokenization, embedding, multi-head self-attention, feed-forward layers, residual connections, layer norm, and the training loop — fits in 243 readable lines. Understanding them makes every LLM paper, product decision, and debugging session sharper.

Story · Why Karpathy built it

Karpathy's pedagogical philosophy is that the best way to understand something is to build it from scratch with no shortcuts. nanoGPT is the third iteration of this philosophy — after micrograd (autograd in 100 lines) and makemore (character-level language models). The goal was a GPT-2-class model with zero external dependencies except PyTorch, small enough to read in an afternoon, fast enough to train on a laptop.

The companion lecture ("Let's build GPT: from scratch, in code, spelled out") on YouTube is three and a half hours and walks through every line. It has become one of the most referenced deep learning resources in existence.

The architecture, component by component

What the 243 lines actually contain

1. Tokenisation

nanoGPT uses character-level tokenisation for simplicity: every unique character in the training text gets an integer ID. The vocabulary size is tiny (65 characters for Shakespeare) but the concept is identical to GPT-4's tiktoken: text → integer sequence → back to text. 

chars = sorted(list(set(text)))
vocab_size = len(chars)
stoi = {ch: i for i, ch in enumerate(chars)}
encode = lambda s: [stoi[c] for c in s]
decode = lambda l: ''.join([itos[i] for i in l])

2. Embeddings — token + position

Every token ID maps to a learned vector (token embedding). Every position in the sequence maps to a separate learned vector (positional embedding). These two are summed — the model learns both what the token means and where it is in context. 

self.token_embedding_table = nn.Embedding(vocab_size, n_embd)
self.position_embedding_table = nn.Embedding(block_size, n_embd)
# At forward pass:
tok_emb = self.token_embedding_table(idx)
pos_emb = self.position_embedding_table(torch.arange(T, device=device))
x = tok_emb + pos_emb

3. Self-attention — the core mechanism

Each token produces three vectors: Query (what am I looking for?), Key (what do I contain?), Value (what do I communicate if attended to?). Attention scores are computed as Q × Kᵀ / √d_k, masked so tokens can only attend to past positions (causal masking), softmaxed, then used to weight the Value vectors. 

k = self.key(x)    # (B, T, head_size)
q = self.query(x)  # (B, T, head_size)
# Scaled dot-product attention
wei = q @ k.transpose(-2, -1) * k.shape[-1]**-0.5  # (B, T, T)
wei = wei.masked_fill(self.tril[:T, :T] == 0, float('-inf'))
wei = F.softmax(wei, dim=-1)
v = self.value(x)
out = wei @ v  # (B, T, head_size)

4. Multi-head attention

Run n_head independent attention heads in parallel, each attending to different aspects of the input, then concatenate and project the results. More heads = richer representation.

5. Feed-forward layer

After attention aggregates information across tokens, a two-layer MLP processes each token independently: Linear → ReLU → Linear. The hidden dimension is 4× the embedding dimension — a constant Karpathy calls the "4× rule" inherited from the original transformer paper.

6. Residual connections + layer normalisation

Each sub-layer (attention, feed-forward) is wrapped in a residual connection: x = x + sublayer(x). This lets gradients flow directly to earlier layers, making deep networks trainable. Layer norm stabilises the activations before each sub-layer.

7. The training loop

Standard PyTorch: sample a batch of sequences, forward pass, compute cross-entropy loss (predicted token distribution vs actual next token), backward pass, AdamW optimiser step. The loss is averaged over all positions in all sequences in the batch. 

for iter in range(max_iters):
    xb, yb = get_batch('train')
    logits, loss = model(xb, yb)
    optimizer.zero_grad(set_to_none=True)
    loss.backward()
    optimizer.step()
What this teaches you about real LLMs

From 243 lines to GPT-4

  • Scale is the only real difference. GPT-4 uses the same architecture — more layers, larger embedding dimension, more heads, a larger vocabulary, and orders of magnitude more data. The components are identical to nanoGPT.
  • Attention is the bottleneck. The O(T²) complexity of the attention matrix is why context windows are expensive. Every doubling of context length quadruples the attention compute.
  • Temperature lives in the softmax. Dividing logits by a temperature value before softmax is a one-line change. Now you understand exactly what every LLM's temperature parameter does.
  • The model only predicts the next token. Everything — code generation, reasoning, instruction following — emerges from one objective: what token comes next? RLHF fine-tuning shapes the distribution; it does not change the fundamental mechanism.
  • Tokenisation shapes everything downstream. Character-level tokenisation in nanoGPT is pedagogically clean but inefficient. BPE tokenisation (tiktoken) compresses common subwords into single tokens, making the effective context window much larger for the same compute.
How to run it

From clone to generated Shakespeare in 20 minutes

git clone https://github.com/karpathy/nanoGPT
cd nanoGPT
pip install torch numpy
# Prepare the dataset
python data/shakespeare_char/prepare.py
# Train (CPU takes ~10 min; GPU takes ~3 min)
python train.py config/train_shakespeare_char.py
# Generate text
python sample.py --out_dir=out-shakespeare-char

The training script is train.py. The model definition is model.py. The configuration for Shakespeare character-level training sets context length to 256, embedding size to 384, 6 attention heads, 6 transformer blocks — a tiny model that still generates recognisable Shakespeare-style text after a few minutes of training.

Common questions

FAQ

  • Do I need a GPU to run this?

    No. The Shakespeare character-level model trains on CPU in about 10 minutes. A GPU (even a consumer RTX 3080) reduces that to under 3 minutes. For learning purposes, CPU is fine.

  • How does this relate to instruction-following models like Claude?

    nanoGPT is a base language model — it learns to predict text, not to follow instructions. Claude adds RLHF and Constitutional AI on top of the same base architecture. The transformer in nanoGPT is architecturally identical to the one inside Claude.

  • What's the best way to learn from the code?

    Watch the companion lecture ("Let's build GPT" on YouTube) while reading the code. Karpathy explains every line. Then modify the model: change the number of heads, add dropout, change the context length, and observe what happens to the loss curve.

Key Takeaway

Every modern LLM — GPT-4, Claude, Gemini — is a scaled version of exactly what nanoGPT implements. Spend an afternoon reading the 243 lines. The mystery dissolves, and every subsequent interaction with an LLM is informed by actually understanding what it's doing.
