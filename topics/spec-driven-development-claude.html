Claude Code: Spec-Driven Development — Why Your AI Coding Sessions Fall Apart at Hour Three

AI coding sessions follow a predictable degradation curve: sharp focus at the start, gradual drift in the middle, incoherence by hour three. The cause is not the model — it is the absence of a persistent spec that survives beyond the context window. Spec-Driven Development (SDD) fixes this. Here is the pattern, the spec format, and how to structure Claude Code sessions so quality holds for hours, not minutes.

The hour-three problem is so consistent it is almost a law: start an AI coding session on a non-trivial feature, and by the third hour the model is contradicting its own earlier decisions, asking for context you already provided, or drifting away from the architecture it established an hour ago. Most developers blame the model. The actual cause is the session structure — specifically, the absence of a spec.

Core idea: A context window is not a project memory. It is a sliding window that forgets. Spec-Driven Development moves the project state out of the context window and into a persistent, structured spec file. The context window reads from the spec; it doesn't store it.

The degradation curve explained

What actually happens inside a 3-hour session

Hour 0: Session start — fresh, sharp, coherent

Full context available. The model holds the entire problem clearly. Decisions are deliberate. Architecture choices are explained. Code quality is high.

Hour 1: Middle phase — compression begins

Earlier turns start getting compressed. The specific error message from 40 turns ago is summarised away. The architectural decision from turn 12 is no longer verbatim. The model is reasoning from summaries, not from original constraints.

Hour 2: Drift — re-learning what was established

The model starts asking for context you gave in hour one. It makes decisions that contradict hour-one architecture. If you don't notice and correct immediately, the codebase accumulates contradictions.

Hour 3: Incoherence — what you end up with

The code works on the happy path but has inconsistent error handling, mixed patterns, and architectural decisions that conflict. Debugging it requires understanding two separate mental models of the same system.

The cure is not shorter sessions. Long sessions are where most of the value is. The cure is externalising state — writing down every decision, constraint, and progress marker so the model reads from the file rather than from its compressed memory.

The spec format

What a good SDD spec contains

A spec is not a PRD or a design doc. It is a session state file — a minimal structure that lets an agent orient itself, know what's done, and know what's next, without any prior context.

# Webhook Service — Spec

## What we're building
An idempotent Stripe webhook handler. Receives events, verifies signatures,
processes payment_intent.succeeded and charge.refunded events, records every
event in webhook_events table, sends to dead-letter queue on failure.

## Stack & constraints
- Node.js 22 / TypeScript 5 strict
- Express 5, Prisma ORM, BullMQ, PostgreSQL
- All DB access via src/db/ modules — no raw SQL outside db layer
- No console.log — always src/lib/logger.ts
- Every DB write in a transaction
- Tests must pass before any commit

## Architecture decisions (record every non-obvious decision here)
- Using express.raw() before JSON middleware — Stripe needs raw body for signature verification
- Processing in BullMQ worker, not inline — allows retries without re-calling Stripe
- Idempotency check BEFORE processing — prevents duplicate writes under concurrent webhook delivery
- Dead-letter queue threshold: 3 failures — Stripe also retries, so we don't need more

The SDD session protocol

How to structure every Claude Code session

Session start (always)

// What you say at the start of every session:
"Read SPEC.md. Orient yourself on where we are (Progress section).
Your current task is in the 'In progress' section.
Before writing any code, confirm your understanding of the current task
and the constraints in the Architecture Decisions section."

// Why this works:
// - Forces the model to read the spec before touching any file
// - The confirmation step catches misunderstandings before they become code
// - Architecture decisions section prevents hour-three re-invention

During the session

// Tell the model to update the spec as it works
"After completing the current task, update SPEC.md:
1. Move the completed task from 'In progress' to 'Done'
2. If you made any non-obvious architectural decisions, add them to Architecture Decisions
3. Move the next TODO item to 'In progress'
Do this BEFORE writing the implementation — not after."

// Why before: updating the spec first forces the model to think about the task
// as a whole before diving into implementation detail. It's a forcing function
// for planning, not just executing.

Mid-session recovery (when you notice drift)

// When the model starts drifting, use this:
"Stop. Read SPEC.md Architecture Decisions section.
Explain how your last response is consistent with decision [X].
If it's not, revise the response before continuing."

// The key insight: don't just say "you got that wrong" — point to the spec.
// The spec is the source of truth. Referring to it directly re-grounds the model.

Session end

// At the end of every session:
"Update SPEC.md before we close:
1. Mark completed tasks as done
2. Add any new architectural decisions made this session
3. Update 'Current task' to the next unblocked item
4. If you discovered any constraints we should add, add them to the Constraints section"

// This 2-minute close ensures the next session can start immediately
// with accurate state — no re-reading commits or reconstructing what happened.

# CLAUDE.md — project-level SDD configuration

## Spec-Driven Development

This project uses SPEC.md as the single source of truth for session state.

### At the start of every session (mandatory)
1. Read SPEC.md fully — especially Architecture Decisions and Current Task
2. Confirm your understanding of the current task before writing code
3. Do NOT start implementing without this step

### During a session
- Update SPEC.md's Progress section as you complete tasks
- Add architectural decisions to SPEC.md as you make them — don't wait until the end
- If a task requires a decision not covered by SPEC.md, add it before implementing

### When you encounter a conflict
If you are about to do something that contradicts SPEC.md Architecture Decisions:
1. STOP
2. Re-read the relevant decision
3. Either follow the decision or explicitly ask whether to update it
Never silently override a spec decision

### Recovery prompt (if context seems corrupted)
"Ignore everything in our conversation history. Read SPEC.md and resume from Current Task."

## Files never to modify without updating spec
- Any file listed in SPEC.md's Architecture Decisions section
- src/lib/logger.ts — this is shared infrastructure
- src/db/ — all DB changes require spec update first
JavaScript: spec validation tool

Validate spec consistency with a small script

// scripts/validate-spec.js — catch spec drift before it becomes code drift
import fs   from 'fs/promises';
import path from 'path';
import { glob } from 'glob';

async function validateSpec(specPath = 'SPEC.md') {
  const spec     = await fs.readFile(specPath, 'utf8');
  const issues   = [];

  // Check 1: every file mentioned in spec actually exists
  const fileRefs = spec.match(/src\/[^\s`\]]+\.[jt]sx?/g) ?? [];
  for (const ref of fileRefs) {
    try   { await fs.access(ref); }
    catch { issues.push(`Missing file referenced in spec: ${ref}`); }
  }

  // Check 2: no task is both "Done" and "In progress"
  const doneSection    = spec.match(/### Done\n([\s\S]*?)(?=###|$)/)?.[1] ?? '';
  const progressSection = spec.match(/### In progress\n([\s\S]*?)(?=###|$)/)?.[1] ?? '';
  const doneTasks      = [...doneSection.matchAll(/\[x\] (.+)/g)].map(m => m[1].trim());
  const progressTasks  = [...progressSection.matchAll(/\[ \] (.+)/g)].map(m => m[1].trim());

  for (const t of progressTasks) {
    if (doneTasks.includes(t)) issues.push(`Task appears in both Done and In Progress: "${t}"`);
  }

  // Check 3: "Current task" matches something in In progress
  const currentTask = spec.match(/## Current task\n([\s\S]*?)(?=##|$)/)?.[1].trim();
  if (currentTask && progressTasks.length > 0) {
    const matched = progressTasks.some(t => currentTask.toLowerCase().includes(t.toLowerCase().slice(0,30)));
    if (!matched) issues.push('Current task does not match any In Progress item');
  }

  if (issues.length === 0) {
    console.log('✓ SPEC.md is consistent');
    return true;
  }

  console.error(`SPEC.md has ${issues.length} issue(s):`);
  issues.forEach(i => console.error(`  ✗ ${i}`));
  return false;
}

const ok = await validateSpec();
process.exit(ok ? 0 : 1);

Key Takeaway

AI coding sessions fall apart at hour three because the context window is not a project memory. Spec-Driven Development fixes this by externalising state into a SPEC.md file that the agent reads at the start of every session. The spec's five sections (what/stack-constraints/decisions/ progress/current-task) give the agent everything it needs to resume without any prior context. The session protocol — orient from spec before coding, update spec as you go, close with spec update — compounds over time: every session starts with perfect state, regardless of how long ago the last one was.
