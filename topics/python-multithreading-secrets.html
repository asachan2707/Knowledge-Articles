Story · The "more threads = faster" trap

A backend engineer parallelizes a CPU-heavy data transformation across 16 threads, expecting a 16× speedup. The benchmark comes back: marginally slower than single-threaded. Confused, they file a bug. The bug is the GIL — and the architecture, not Python.

Two days later they switch to multiprocessing with the same code. Now it really is faster. That single experience is what the article is trying to deliver in 12 takes: threads do not magically parallelize CPU work in CPython. The wins are elsewhere.

Foundational explanation

Why Python threading feels confusing

A lot of threading confusion comes from partial truths. People hear that “the GIL means only one thread runs at a time,” and they stop there. But that sentence leaves out an important detail: the GIL serializes Python bytecode execution, not every single operation in a threaded program.

During many I/O operations, sleep calls, and some extension-library operations written in C, Python releases the GIL. That means threads can still overlap meaningfully. This is exactly why downloading many URLs at once or waiting on multiple file operations can benefit from threads even though CPU-heavy number crunching often does not.

The article’s real contribution is that it replaces the simplistic CPU-bound vs I/O-bound rule with a better way of thinking: threads help when the program spends meaningful time waiting.

Detailed breakdown

What the “12 secrets” are really teaching

1. The GIL blocks Python bytecode, not all useful concurrency

This means threads can still improve throughput when work frequently pauses on network or system-level I/O.

2. The slogan is too simple

“Use threads for I/O, processes for CPU” is a good starting point, but reality is more nuanced when C extensions, mixed workloads, or external waiting are involved.

3. Context switching is expensive

Even if threads are allowed to overlap, creating too many of them can waste time on scheduling overhead, lock contention, and operating system task switching.

4. The GIL has benefits too

The article reminds us that the GIL simplifies object safety and memory management. It is not purely a flaw; it is a tradeoff in Python’s runtime design.

5. Exceptions and daemon threads are operational traps

Threaded programs can fail quietly. If a thread crashes or is terminated at shutdown, the main program may not automatically explain what went wrong.

6. Shared state is the real danger zone

Many threading bugs are not “thread bugs,” but design bugs caused by allowing too much mutable shared state. The safest approach is often message passing instead of direct shared mutation.

Practical learning

How to think about threaded design in real code

  • Use threads for waiting-heavy workflows like API calls, downloads, and file reads.
  • Do not assume more threads always means more speed.
  • Reach for ThreadPoolExecutor before manually managing thread lifecycles.
  • Use queues, events, and producer-consumer designs before defaulting to locks.
  • Use structured logging to debug concurrent execution instead of relying on interleaved prints.
  • For CPU-heavy work, evaluate multiprocessing or native libraries that release the GIL.
Why it matters

This topic improves your mental model

Multithreading becomes much less mysterious once you stop expecting magic speedups and start thinking in terms of waiting, scheduling overhead, shared-state risk, and runtime behavior. That mental model helps with both performance and debugging.

It also helps you choose the right tool. Many developers misuse threads when they actually need async I/O, multiprocessing, batching, or a queue-based architecture. Understanding the tradeoffs makes concurrency a design decision rather than a guess.

Threads vs processes vs async

Picking the right concurrency tool in Python

Workload type Best tool Why
Many slow I/O calls (HTTP, DB, files) asyncio Cheapest concurrency, no thread overhead, easy to reason about
Mixed I/O + blocking libs that don't support async ThreadPoolExecutor Threads release GIL during I/O; works with sync code
CPU-heavy: math, image processing, data transforms multiprocessing or native libs Sidesteps the GIL by running separate Python processes
Heavy numeric work (NumPy, PyTorch, polars) Native library (which already releases GIL) The C/Rust code does the work; Python orchestrates
Background jobs at scale Queue + worker process (Celery, RQ, Dramatiq) Process-based, durable, observable
Common questions

FAQ that comes up after reading the article

  • "Is the GIL going away in Python 3.13+?"

    Free-threaded Python (PEP 703) is being introduced as an opt-in build, not a default. For most codebases the GIL still applies in 2026. Even when free-threaded becomes mainstream, expect rough edges with C extensions for several years.

  • "Should I use threading or asyncio?"

    Default to asyncio for new I/O-heavy code, especially network services. Use threading when you must integrate with synchronous libraries (older DB drivers, blocking SDKs) that don't have async support.

  • "What's the right number of threads in a pool?"

    For I/O-bound work, start with min(32, os.cpu_count() * 5) as a baseline, then measure. For CPU-bound work in multiprocessing, use os.cpu_count() as the upper bound.

  • "How do I debug a deadlock or hung thread?"

    Use faulthandler.dump_traceback_later() or send SIGQUIT to dump every thread's stack. In production, attach py-spy to the running process — it requires no code changes and shows exactly where each thread is stuck.

Key Takeaway

Python threads are neither useless nor universally fast. They are a specific tool that works best when your program waits often, shares little mutable state, and is designed with concurrency in mind.
