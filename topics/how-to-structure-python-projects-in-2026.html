How to Structure Python Projects in 2026

Python project structure has matured. There is now a default-good stack — pyproject.toml, uv, ruff, mypy — and a layout that scales without rewrites.

The article's sharpest observation is cultural: most Python developers still treat project structure like a personal art form, improvising every time. That improvisation is the reason codebases feel "fine" for two weeks and brittle by month four. The 2026 ecosystem has finally produced an opinionated default stack, and the right move is to lean into it.

Core idea: Treat project layout as architecture, not improvisation — pick today's defaults early and stop accumulating utils.py debt.

Story · The utils.py spiral

You start clean — main.py, a config.py, maybe a single utils.py. A week later, helpers.py appears, suspiciously similar to utils.py. Then services.py. By month three, nobody on the team can predict where new code "belongs."

The article calls this the structure debt spiral. Refactoring later costs more than choosing a real layout on day one.

Detailed breakdown

Key insights and why they matter

1) pyproject.toml is the new center of gravity

Dependencies, build config, ruff rules, mypy settings — all live in one file. Stop scattering setup.cfg, requirements.txt, and pyproject configs across the repo.

2) uv for fast, reproducible environments

uv replaces pip + virtualenv for most workflows. It's dramatically faster and produces deterministic lockfiles for CI.

3) ruff + mypy as a non-negotiable pair

ruff handles lint + format in one tool; mypy adds typed contracts. Together they catch a huge class of bugs before runtime.

4) src/ layout, not flat package layout

A src-based layout prevents accidental imports of unpacked code during tests. It is the cleanest separation between source, tests, and tooling.

5) Domain boundaries inside the package

Split by domain (orders/, billing/, ingest/) rather than by technical role (models/, services/, utils/). Domain modules age better.

Practical learning

How to apply this in real work

  • Bootstrap new projects with: pyproject.toml + src/ layout + ruff + mypy + uv from commit #1.
  • Forbid top-level utils.py and helpers.py — push code into named domain modules.
  • Lock CI to run uv sync, ruff check, mypy, pytest in that order.
  • Document run commands in a README that a new dev can copy-paste in 60 seconds.
Common questions

FAQ

  • Do small scripts need all this?

    No. One-off scripts can stay flat. The structure pays off once a project has tests, multiple contributors, or production runtime.

  • uv vs poetry vs pip-tools?

    uv is the current default for speed and simplicity. Poetry is still fine if your team is invested. pip-tools is now niche.

  • Is mypy worth the friction?

    Yes at API boundaries (functions called from elsewhere). Worry less about deeply internal helpers.
