The Complete Claude Code Guide: CLAUDE.md, Skills, Hooks & Plugins

Four topics merged into one definitive reference. CLAUDE.md is the behavioral kernel. Skills are reusable operation packages invoked with slash commands. Hooks are event-driven shell automations that run outside Claude's context. Plugins extend Claude Code with external tools and MCP servers. Together they form a programmable AI development environment.

Claude Code is not a chat window with a code editor attached. It is a four-layer system. Each layer handles a different kind of persistent configuration: CLAUDE.md (what Claude always knows), Skills (what Claude can do on demand), Hooks (what the system does automatically), and Plugins (what external tools Claude can reach).

Core idea: Configure these four layers and Claude behaves consistently, follows your conventions without being reminded, executes complex operations with a single command, and integrates with your full toolchain — across every session.

Part 1 — CLAUDE.md

The behavioral operating system

CLAUDE.md is a Markdown file Claude reads at the start of every session. Everything in it is in Claude's context before you type a word. It acts simultaneously as:

Behavioral OS

Sets defaults for how Claude works — tone, commit style, testing approach, code style.

Context router

Tells Claude where to look for architecture decisions, runbooks, schemas, and API contracts.

Quality layer

Defines "done" — what tests must pass, what review checklist to follow, what conventions must hold.

Architectural memory

Preserves decisions that should outlast any single session — why you chose X over Y.

The anatomy of a good CLAUDE.md

# ProjectName

> One-sentence description of what this project is and its tech stack.

## Architecture
- Monorepo: apps/web (Next.js 15), apps/api (Node.js/Express), packages/shared
- Database: PostgreSQL via Drizzle ORM — all queries in src/db/
- Cache: Redis for sessions and job queues (BullMQ)
- Deployed on: Fly.io (API), Vercel (Web)

## Code conventions
- TypeScript strict everywhere — no `any`
- 2-space indent, single quotes
- Named exports only — no default exports
- Tests co-located with source: `foo.ts` → `foo.test.ts`

## Execution contracts (Claude: read carefully)
- Run `npm test` before every commit; STOP if any test fails
- DB migrations via `npm run db:migrate:make` only — never raw ALTER TABLE
- Secrets ONLY via environment variables — never in code

## Context pointers
- API types: packages/shared/types.ts
- DB schema: apps/api/src/db/schema.ts
- Architecture decisions: DECISIONS.md
- Deployment runbook: ops/RUNBOOK.md

## What NOT to do
- Do not use console.log in production code — use the logger at src/lib/logger.ts
- Do not add dependencies without checking the bundle size impact first
- Do not write comments that say WHAT the code does — only write WHY

The 200-line rule

Keep CLAUDE.md under 200 lines. Longer files dilute the important instructions. A 30-line CLAUDE.md Claude follows completely beats a 300-line one Claude ignores because it cannot identify what matters.

Multi-level CLAUDE.md hierarchy

  • ~/.claude/CLAUDE.md — global preferences (your voice, formatting preferences, tools you always use)
  • ./CLAUDE.md — project-level conventions (architecture, team standards)
  • ./src/auth/CLAUDE.md — directory-level context (security rules for auth module)
  • ./CLAUDE.local.md — personal overrides, git-ignored (your personal shortcuts)

/doctor

Part 2 — Skills

The evolution from slash commands to packaged operations

Skills are Markdown files in .claude/skills/ that Claude loads when you invoke /skill-name. They evolved from simple slash command aliases into structured operation packages with specialised instructions, tool chains, and output contracts.

Skill anatomy

# .claude/skills/code-review.md
# Code Review

When invoked with `/code-review [--fix] [--comment]`:

1. Run `git diff main` to see all changed files.
2. For each changed file, check for:
   - Security: injection, hardcoded secrets, OWASP Top 10
   - Correctness: logic bugs, off-by-one errors, null/undefined paths
   - Performance: N+1 queries, synchronous blocking in async contexts
   - Style: deviations from CLAUDE.md conventions

3. Produce a structured report:
   - Category (security/correctness/perf/style)
   - File and line number
   - Severity (critical/high/medium/low)
   - Finding description
   - Suggested fix

4. If `--fix` flag: apply all low-risk (medium/low severity) fixes automatically.
   Ask for confirmation before applying critical/high severity fixes.

5. If `--comment` flag: post findings as inline GitHub PR comments using the `gh` CLI.

End with a one-line verdict: APPROVE / REQUEST_CHANGES / NEEDS_DISCUSSION.

Skill scope levels

  • Global skills (~/.claude/skills/) — available in every project
  • Project skills (.claude/skills/) — only available in this project
  • Anthropic built-in skills — /code-review, /run, /verify, /init, /deep-research are shipped with Claude Code
  • Plugin skills — skills from installed plugins (Figma, Slack, etc.)

Chaining skills in workflows

// Call a skill from a workflow script
const reviewResult = await agent('/code-review --fix', {
  label: 'review-and-fix',
  schema: {
    type: 'object',
    properties: {
      verdict:  { type: 'string', enum: ['APPROVE', 'REQUEST_CHANGES', 'NEEDS_DISCUSSION'] },
      findings: { type: 'array', items: { type: 'object' } },
      fixed:    { type: 'number' },
    },
    required: ['verdict', 'findings'],
  },
});

Best skills to write for your project

  • /deploy — test → build → deploy to staging/prod with health check verification
  • /db-migrate — generate, review, and apply database migrations safely
  • /onboard — guide a new contributor through setup and first task
  • /incident — triage production issue: check logs, recent deploys, error rates
  • /changelog — generate changelog entries from git log since last tag
Part 3 — Hooks

Event-driven automation outside Claude's context

Hooks are shell commands that Claude Code executes in response to lifecycle events. They run in a separate process, outside Claude's reasoning — which makes them deterministic, tamper-proof, and ideal for quality gates.

Available hook events

  • PreToolUse — before Claude calls a tool (file write, bash command, MCP tool)
  • PostToolUse — after a tool call completes
  • Stop — when Claude finishes a response and is awaiting user input
  • Notification — when Claude sends a desktop notification
  • SubagentStop — when a sub-agent completes

Hook configuration

// .claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write|NotebookEdit",
        "hooks": [
          {
            "type": "command",
            "command": "npm run lint -- --quiet 2>&1 | head -30",
            "timeout": 10000
          }
        ]
      },
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "node .claude/hooks/log-bash-command.js"
          }
        ]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "node .claude/hooks/block-destructive-commands.js"
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display notification \"Claude is done\" with title \"Claude Code\"'"
          }
        ]
      }
    ]
  }
}

Hook context — reading what triggered

// .claude/hooks/block-destructive-commands.js
// Runs before every Bash tool call — can block dangerous commands
const input = JSON.parse(process.env.CLAUDE_TOOL_INPUT ?? '{}');
const command = input.command ?? '';

const DANGEROUS = [
  /rm\s+-rf\s+\//,        // rm -rf /
  /DROP\s+TABLE/i,         // SQL DROP
  /git\s+push\s+--force/,  // force push
  />\s*\/dev\/null\s*2>&1\s*&\s*disown/, // background detach
];

for (const pattern of DANGEROUS) {
  if (pattern.test(command)) {
    // Exit code 2 = hard block (Claude Code will not execute the tool)
    console.error(`BLOCKED: command matches dangerous pattern: ${pattern}`);
    process.exit(2);
  }
}

// Exit code 0 = allow
process.exit(0);
// .claude/hooks/log-bash-command.js
// PostToolUse — logs every Bash command with timestamp
import fs from 'fs';
import path from 'path';

const input  = JSON.parse(process.env.CLAUDE_TOOL_INPUT  ?? '{}');
const output = JSON.parse(process.env.CLAUDE_TOOL_OUTPUT ?? '{}');

const entry = {
  ts:      new Date().toISOString(),
  command: input.command,
  exit:    output.returncode,
  stdout:  (output.stdout ?? '').slice(0, 500),
};

const logPath = path.join('.claude', 'bash-audit.jsonl');
fs.appendFileSync(logPath, JSON.stringify(entry) + '\n');

Hook exit codes

  • 0 — allow the tool call to proceed
  • 1 — report a warning but proceed
  • 2 — hard block: Claude Code will not execute the tool
Part 4 — Plugins & MCP

Extending Claude Code with external tools

Plugins connect Claude Code to external systems via MCP (Model Context Protocol). Each plugin exposes tools Claude can call, resources Claude can read, and sometimes skills Claude can invoke.

MCP configuration

// .mcp.json — project-level MCP servers
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" }
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": { "POSTGRES_URL": "${DATABASE_URL}" }
    },
    "figma": {
      "command": "npx",
      "args": ["-y", "@figma/mcp-server-figma"],
      "env": { "FIGMA_API_TOKEN": "${FIGMA_API_TOKEN}" }
    },
    "slack": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-slack"],
      "env": { "SLACK_BOT_TOKEN": "${SLACK_BOT_TOKEN}" }
    }
  }
}

Global vs project MCP configuration

  • ~/.claude/mcp.json — always available in every project (personal tools: calendar, notes, terminal)
  • ./.mcp.json — project-specific tools (GitHub repo, project database, design system)
  • ~/.claude/settings.json — globally allowed/denied tool names

Building a custom MCP server

// my-custom-mcp/index.js
import { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js';

const server = new McpServer({ name: 'my-tools', version: '1.0.0' });

// Define a tool Claude can call
server.tool('get_deployment_status', {
  description: 'Get the current deployment status from Fly.io',
  inputSchema: {
    type: 'object',
    properties: {
      app: { type: 'string', description: 'App name on Fly.io' },
    },
    required: ['app'],
  },
}, async ({ app }) => {
  const { execSync } = await import('child_process');
  const status = execSync(`flyctl status --app ${app} --json`).toString();
  return { content: [{ type: 'text', text: status }] };
});

// Define a resource Claude can read
server.resource('recent-errors', 'app://logs/errors', {
  description: 'Recent production error logs',
  mimeType: 'text/plain',
}, async () => {
  const logs = await fetchRecentErrors(); // your implementation
  return { contents: [{ uri: 'app://logs/errors', text: logs }] };
});

const transport = new StdioServerTransport();
await server.connect(transport);
// Register it in .mcp.json
{
  "mcpServers": {
    "my-tools": {
      "command": "node",
      "args": [".claude/mcp/my-custom-mcp/index.js"]
    }
  }
}

Integration example

All four layers working together

// Scenario: Claude is asked to fix a production bug

// Layer 1: CLAUDE.md tells Claude where the logger is, what the coding conventions are,
//          and that tests must pass before any commit.

// Layer 2: PostToolUse hook runs eslint after every file write, catching linting errors
//          before Claude moves to the next step.

// Layer 3: /code-review skill is invoked by the workflow after the fix is written,
//          checking security and correctness before committing.

// Layer 4: GitHub MCP lets Claude read the bug report from the open issue and
//          post the fix as a PR comment when done.

// The workflow that orchestrates it:
export const meta = {
  name: 'fix-bug',
  description: 'Find, fix, test, review, and PR a bug from a GitHub issue',
  phases: [{ title: 'Diagnose' }, { title: 'Fix' }, { title: 'Review' }, { title: 'PR' }],
};

phase('Diagnose');
const issue = await agent(`Use the GitHub MCP to read issue #${args.issueNumber}. Return the bug description and reproduction steps.`);

phase('Fix');
const fix = await agent(`Fix this bug: ${issue}. Write tests. Run npm test.`);

phase('Review');
const review = await agent('/code-review', { schema: REVIEW_SCHEMA });
if (review.verdict === 'REQUEST_CHANGES') {
  log('Review requested changes — revising fix');
  await agent(`Apply these review comments: ${JSON.stringify(review.findings)}`);
}

phase('PR');
await agent(`Create a GitHub PR for this fix. Title: "fix: ${issue.title}". Reference issue #${args.issueNumber}.`);

Key Takeaway

CLAUDE.md, Skills, Hooks, and Plugins are the four configuration layers of Claude Code. CLAUDE.md makes Claude consistent. Skills make complex operations single-command. Hooks make quality gates automatic and tamper-proof. Plugins connect Claude to your entire toolchain. Configure all four and Claude Code transforms from an AI assistant into a programmable development runtime.
